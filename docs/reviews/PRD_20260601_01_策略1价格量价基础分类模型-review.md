## 发现

### P0-1：组合选股阶段使用 `t+1` 实际可成交性，会引入未来信息和替补偏差

**依据**：PRD 在标签层定义 `label_entry_tradable = can_buy_open[t+1] ...`（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:190`），并要求 `label_entry_tradable = FALSE` 的样本回测强制剔除（`:196`）。组合层又写成“在 `label_entry_tradable=TRUE`（用 `T+1` 实际可成交性）的候选中，取 score 最高 5 只”（`:231`）。但撮合层写“买不到则资金留现金，不顺延找替补”（`:301`）。

**影响**：在 `T` 日收盘生成信号时，`T+1` 开盘是否停牌、是否一字涨停、是否开盘临停尚不可知。先用 `T+1` 实际可成交性过滤再选 top 5，等价于事后跳过买不到的高分票并自动补入低一档可买票，回测会低估现金拖累和买入失败次数，高估组合收益与可执行性。

**建议**：组合候选应先按 `T` 日已知 universe + score 选出目标名单；撮合层再用 `T+1 can_buy_open` 判断是否成交，失败则留现金。若需要候补池，候补规则必须只依赖 `T` 日已知信息，并在 ADS 中记录原始 rank、目标 rank、成交 rank 与失败原因。

**与既有决策冲突**：冲突。既有策略方案要求 `t+1` 停牌或一字涨停“不买，资金保留或替补候选”，该替补政策必须显式参数化，不能事后用 `t+1` 可成交性重排候选（`docs/A股中低频小资金机器学习策略方案.md:437`）。

### P0-2：主标签用 `close_hfq[t+1]` 作入场价，但回测执行价是 `open_hfq[t+1]`

**依据**：PRD 标签定义为 `fwd_ret_Hd = close_hfq[t+1+H] / close_hfq[t+1] - 1`，并写“全部用 `close_hfq`”（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:172`）。同一 PRD 的执行假设却是默认 `t+1` 开盘价 `open_hfq[t+1]`（`:296`）。DWS/ADS 方案也要求入场价使用 `open_hfq` 或未来 `vwap_hfq`，不能用 `t` 日收盘（`docs/数据仓库建模方案-DWS-ADS.md:287`）。

**影响**：模型优化的是 `t+1` 收盘到未来收盘的收益，而回测交易的是 `t+1` 开盘开始的收益；`label_top30_5d`、RankIC、分层收益与实际组合收益不在同一价格路径上。若 `H=5` 还要服务“周频调仓、下个 `T+1` 换仓”，退出日与退出价格也需要和撮合层统一，否则会出现标签收益、组合收益和 NAV 三套口径。

**建议**：把主标签改成与 v0 执行一致的可执行口径，例如 `fwd_ret_5d_open_to_open` 或 `fwd_ret_5d_open_to_close`，分母使用 `open_hfq[t+1]`，分子使用明确的退出交易日与价格。`close_hfq` 标签可保留为研究对照，但不能作为主训练/验收标签。

**与既有决策冲突**：冲突。项目建模约束要求 `t` 日特征对应从 `t+1` 起算的入场价，DWS/ADS 方案已把 P0 标签定义为 `t+1` 开盘买入。

### P1-1：PRD 新建了一套策略专属 DWS/ADS 表名与主键，绕开既有 DWS/ADS 合约

**依据**：PRD 交付物定义 `dws_stock_feat_pv_daily`、`dws_stock_sample_pv_daily`、`ads_pv_clf_prediction_daily`、`ads_pv_clf_portfolio_daily`、`ads_pv_clf_backtest_nav`、`ads_pv_clf_backtest_metrics`（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:78`，`:87`，`:354`）。既有 DWS/ADS 方案的 P0 表为 `dws_stock_feature_price_daily`、`dws_stock_feature_valuation_daily`、`dws_stock_feature_daily_v0`、`dws_stock_sample_daily`，ADS 为 `ads_ml_training_panel_daily`、`ads_model_registry`、`ads_model_prediction_daily`、`ads_stock_candidate_daily`、`ads_portfolio_target_daily`、`ads_order_plan_daily`、`ads_backtest_*`（`docs/数据仓库建模方案-DWS-ADS.md:101`，`:414`，`:458`，`:521`）。

**影响**：如果按 PRD 实施，会产生一套并行表体系，既不关闭当前 TODO 中的 P0 DWS/ADS SQL，也会让后续策略无法复用统一训练面板、模型注册、候选池、订单计划和回测表。PRD 表格里的特征表主键还写成 `(sec_code, trade_date)`（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:81`），没有把 `feature_version` 纳入粒度；样本表主键也缺 `feature_version/label_version/universe_version`（`:83`），会削弱同一日期多版本并存和可复现能力。

**建议**：按既有 DWS/ADS 表名落地，策略 1 只定义 feature set、sample filter、model config 和 ADS 中的 `strategy_id/model_id/run_id`。若确实要采用 PRD 的策略专属表名，应先形成 owner 决策，明确它取代或补充现有 DWS/ADS 方案，并同步 TODO/架构记忆。

**与既有决策冲突**：冲突。DECISION-20260531-18 已确定 DWS/ADS 采用分族特征层 + 策略消费层；PRD 现在把策略专属命名推进到 DWS 和 ADS 基础表，会破坏该分层边界。

### P1-2：ADS 物理分区规则写成全部按 `trade_date`，对预测/组合/汇总表不可执行或语义错误

**依据**：PRD 写“全部 DWS/ADS 表：`PARTITION BY DATE_TRUNC(trade_date, MONTH)` + `CLUSTER BY sec_code`”（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:346`）。但 PRD 自己定义的组合表主键使用 `rebalance_date`（`:90`），回测汇总表使用 `metric_window`（`:92`）。既有 DWS/ADS 方案区分了股票 DWS 的 `trade_date`、ADS 预测/组合的 `predict_date/rebalance_date`、回测 NAV 的 `trade_date` 或不分区、元数据小表不分区（`docs/数据仓库建模方案-DWS-ADS.md:71`）。

**影响**：按 PRD 写 SQL 时，部分 ADS 表没有 `trade_date` 分区列会直接建表失败；即便强行加 `trade_date`，预测日、调仓日、成交日、NAV 日的语义也会混在一起，导致查询过滤、回测复现和 `require_partition_filter` 约束不清晰。

**建议**：逐表指定分区键：预测用 `predict_date` 或 `signal_date`，组合/候选/订单计划用 `rebalance_date` 或 `expected_trade_date`，NAV 用 `trade_date`，metrics/model registry/run config 不分区或按 `created_at`/`backtest_id` 管理。

**与既有决策冲突**：冲突。既有物理设计已经按表类定义不同分区键；PRD 的统一写法覆盖了该设计。

### P1-3：PRD 明确不读 2018 lookback，会改变现有 DWS 验收口径

**依据**：PRD 写 DWS 在现有 DWD 上自行 warm-up，完整样本约从 2019-04 起，且“不依赖 2017-2018 行情”（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:98`）。已知约束要求行情 DWD/DWS 最终写 2019+，但构建时按最大滚动窗口读取 2018 lookback buffer，buffer 不落最终 DWD/DWS（`.agent/memory/KNOWN_CONSTRAINTS.md:11`，`:12`）。DWS/ADS 验收还要求 `trade_date >= 2019-01-01` 且 2019 年初滚动特征不因 warm-up 系统性为空（`docs/数据仓库建模方案-DWS-ADS.md:579`）。

**影响**：PRD 实际把 2019Q1 从可训练样本中剔除，并将 fold 起点改为 2019-04；这是对当前 DWS 验收和样本范围的变更，不应只在策略 PRD 中隐式发生。若未来为补齐 2019Q1 让 DWS 直接读 ODS 2018 行情，又会违反“下游查 DWD/DIM，不直接打 ODS”的工程约束。

**建议**：二选一明确：要么 DWS 特征构建使用非落表的 2018 lookback 输入并保留 2019-01 起样本；要么正式修改 DWS/ADS 验收与 fold 起点，承认策略 1 的有效样本从 2019-04 起。该选择需要 owner 决策后同步到 DWS/ADS 文档和 TODO。

**与既有决策冲突**：冲突。DECISION-20260531-11 和 KNOWN_CONSTRAINTS 已把 2018 lookback 定为支持 2019+ 滚动特征的标准口径。

### P2-1：多个策略默认参数被 PRD 标成“已定”，但项目级 OQ-010 仍开放

**依据**：PRD 将默认池写死为 `board∈{SSE_MAIN,SZSE_MAIN}`、只保留沪深主板（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:134`，`:151`），并在开放问题表里把“仅保留沪深主板”标为已决策（`:405`）。PRD 还把组合固定为 5 只、单票 20%（`:231`，`:232`）。但项目级 OQ-010 仍记录 P0 策略默认参数待确认，包括持股数/单票权重上限、北交所是否纳入、训练工具链等（`.agent/memory/OPEN_QUESTIONS.md:16`）。既有策略方案对创业板/科创板默认是开，北交所为研究开关（`docs/A股中低频小资金机器学习策略方案.md:80`），并建议 top N 研究 20/30/50/80 四档（`:188`）。

**影响**：这些不是纯实现细节，会实质改变样本截面、风格暴露、容量、换手和回撤。若 PRD 直接落地，会把尚未 owner 决策的参数固化为策略行为，也让 `ml_pv_clf_v0` 与既有 `ml_ranker_v0` 基线不可横向比较。

**建议**：把主板-only、5 只、20% 单票上限标为“PRD 建议默认值，待 owner 确认”，并保留参数化实验矩阵。只有 owner 明确拍板后，才把对应 OQ 关闭并写入决策日志。

**与既有决策冲突**：部分冲突。PRD 的“已定”状态与项目级 OPEN_QUESTIONS 不一致。

### P2-2：模型预处理固化在 DWS 样本表，和“DWS 保留原始特征、训练面板冻结预处理”的边界不一致

**依据**：PRD 要求 `dws_stock_sample_pv_daily` 输出原始特征和截面标准化 `*_z` 特征，并在每日截面内完成 winsorize、缺失填充和 robust z-score（`docs/prd/PRD_20260601_01_策略1价格量价基础分类模型.md:201`，`:241`）。PRD 又说标准化放 SQL 还是 Python 二选一（`:370`）。既有 DWS/ADS 方案明确“不在 DWS 宽表中做模型训练专属的 one-hot、缺失填补、winsorize、z-score 固化”，训练面板才是冻结后的研究输入（`docs/数据仓库建模方案-DWS-ADS.md:314`，`:430`）。

**影响**：预处理如果落在 DWS 策略样本表，会把模型实验版本、特征缺失策略和横截面标准化口径固化到共享数据层；如果同一逻辑又允许在 Python 做，会产生 SQL/Python 两套口径漂移。后续策略复用 DWS 时难以区分原始特征、预处理版本和训练 run。

**建议**：DWS 保留原始特征、掩码、缺失标记和样本资格；把 `*_z`、winsorize、填充、one-hot 等固化到 `ads_ml_training_panel_daily` 的某个 `run_id/preprocess_version` 中。若策略 1 必须保留专属样本表，应把它定位为训练面板或 ADS 表，而不是通用 DWS 表。

**与既有决策冲突**：冲突。该写法模糊了 DWS 与 ADS/训练面板边界。
