# 策略 1 实验并发调度器运行手册

本文档说明 `scripts/strategy1/run_oq010_experiments.py` 调度器的安装、用法和注意事项。

## 前置条件

- Python 3.9+，已安装 `requirements.txt` 依赖
- `gcloud auth application-default login` 已完成（ADC 用于 BigQuery 和 GCS 访问）
- `bq` CLI 可用（内部使用 `subprocess` 调用 `bq query` 执行 SQL）
- GCS bucket `gs://ashare-artifacts` 已创建，ADC 有读写权限
- BigQuery 状态表 `ashare_meta.strategy1_experiment_run_status` 已创建（`sql/meta/02_strategy1_experiment_run_status.sql`）
- 实验 manifest `configs/strategy1/oq010_experiments_v0.json` 已就位

## 基本用法

### Dry-run 展开计划

```bash
python scripts/strategy1/run_oq010_experiments.py --dry-run
```

只输出完整调度计划，不执行任何 BigQuery 或 Python 脚本。dry-run 会预检可执行实验的 SQL `DECLARE p_* DEFAULT` 参数注入，发现缺失参数或类型不匹配时退出非 0。

### 按阶段运行

```bash
# 运行 stage_a 的所有实验
python scripts/strategy1/run_oq010_experiments.py --stage-id stage_a

# 运行 stage_a 的 2 个并行实验
python scripts/strategy1/run_oq010_experiments.py --stage-id stage_a --max-parallel 2
```

### 按实验 ID 运行

```bash
# 运行单个实验
python scripts/strategy1/run_oq010_experiments.py \
    --experiment-id oq010_a1_n10_w10

# 运行多个实验
python scripts/strategy1/run_oq010_experiments.py \
    --experiment-id oq010_a1_n10_w10 \
    --experiment-id oq010_a2_n20_w05
```

### 完整参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--manifest` | `configs/strategy1/oq010_experiments_v0.json` | manifest 路径 |
| `--stage-id` | — | 运行阶段，如 `stage_a` |
| `--experiment-id` | — | 可重复传入，限定实验 |
| `--max-parallel` | `2` | 总并发实验数 |
| `--max-parallel-backtest` | `1` | 08 ledger 最大并发数 |
| `--dry-run` | `False` | 只展开计划不执行 |
| `--force-replace` | `False` | 启用清理重跑 |
| `--resume` | `False` | 从状态表恢复失败实验 |
| `--resume-from-step` | — | 从指定 step 恢复 |
| `--fail-fast` | `False` | 实验失败后停止新实验 |
| `--allow-cross-stage` | `False` | 允许跨阶段并发 |
| `--log-dir` | `logs/strategy1/oq010_experiments` | 日志目录 |
| `--scheduler-instance-id` | 自动生成 | 调度器实例 ID |
| `--lock-ttl-minutes` | `30` | 锁 lease TTL |
| `--verbose / -v` | `False` | 输出 debug 日志 |

## 锁机制

### GCS 原子锁

调度器使用 GCS object `ifGenerationMatch=0` 实现原子锁获取：

```
gs://ashare-artifacts/locks/strategy1/oq010/<lock_key>.lock
```

| 锁 key 模式 | 保护步骤 | 说明 |
|---|---|---|
| `train:<prediction_run_id>` | 01-03 | 同一 `prediction_run_id` 互斥 |
| `predict:<prediction_run_id>` | 04 | 预测互斥 |
| `portfolio:<run_id>` | 05-07 | 组合互斥 |
| `backtest:<backtest_id>` | 08 | 回测互斥 |
| `summary:<run_id>:<backtest_id>` | 09-10 | 报告互斥 |
| `diagnosis:<run_id>:<backtest_id>` | 11-12 | 诊断互斥 |

锁内容（JSON）：

```json
{
  "lock_key": "backtest:<backtest_id>",
  "experiment_id": "...",
  "lock_owner": "<instance_id>",
  "acquired_at": "<timestamp>",
  "lease_expires_at": "<timestamp>"
}
```

### Lease & Heartbeat

- 锁默认 TTL 30 分钟（`--lock-ttl-minutes` 可配）
- 调度器每 60 秒刷新 lease（heartbeat）
- 过期锁可通过 stale reclaim 恢复
- 调度器不得在无锁状态下执行 step

### 状态表

`ashare_meta.strategy1_experiment_run_status` 只用于审计追踪和 resume 输入。
**不承担锁管理职责**——锁原语是 GCS object create-if-not-exists。

## 实验执行路径

### Portfolio-only 实验

`requires_retrain=false` 的实验从 `05_build_candidates` 开始执行，
跳过 01-04 训练/预测步骤。使用 `prediction_run_id` 引用上游模型和预测输出。

Steps: `05 → 06 → 07 → 08 → 09 → 10(render) → 11(diagnose) → 12(QA)`

### Retrain 实验

`requires_retrain=true` 的实验从 `01_build_training_panel` 开始执行完整流程。

Steps: `01 → 02 → 03 → 04 → 05 → 06 → 07 → 08 → 09 → 10(render) → 11(diagnose) → 12(QA)`

## Resume

### 自动 resume

```bash
python scripts/strategy1/run_oq010_experiments.py \
    --experiment-id oq010_a1_n10_w10 \
    --resume
```

调度器读取状态表，跳过已 `succeeded` 的 step，从 `failed` 或缺失的 step 继续。

### 从指定 step resume

```bash
python scripts/strategy1/run_oq010_experiments.py \
    --experiment-id oq010_a1_n10_w10 \
    --resume-from-step 05_build_candidates
```

### Resume 规则

- Resume 前不做完整性检查（由 QA 保证）
- Resume 时跳过 `succeeded` step
- Resume 后不会重复 ADS 输出（ADS 写入按 `run_id` / `backtest_id` 隔离）
- `force_replace` 会在 resume 时清理指定范围内输出

## 日志

调度器本地日志目录结构：

```
logs/strategy1/oq010_experiments/
  stage=<stage_id>/
    run_at=<timestamp>/
      <experiment_id>/
        01_build_training_panel.log
        05_build_candidates.log
        08_run_backtest.log
        ...
```

每个 step 的 stdout/stderr 写入独立的 `.log` 文件。

## 08 Backtest 并发

`08_run_backtest` 是账户级有状态 ledger，对并发敏感：

- `max_parallel_backtest` 默认 1（顺序执行）
- owner 可在验收 `sql/qa/07_strategy1_experiment_concurrency_checks.sql` 通过后提高到 2
- 不同 `backtest_id` 的 08 并发是安全的（08 的 DELETE/INSERT 都限定 `backtest_id`）
- 同一 `backtest_id` 的 08 不允许并发（由锁 `backtest:<backtest_id>` 保证）

## 依赖规则

| 场景 | 行为 |
|------|------|
| 同 stage portfolio-only | 互不依赖，可并发 |
| retrain 实验 | 必须先完成 01-04 再进入 05-12 |
| portfolio-only | 可复用 `prediction_run_id` 从 05 开始 |
| 跨阶段实验 | 默认禁止并发 |
| 显式跨阶段 | 配置 `depends_on_experiment_id`，仅上游 succeeded 后进入队列 |

## 安全限制

1. 不并发执行同一 `run_id` 的 BQML 训练步骤。
2. 清理模型对象前必须确认无 RUNNING job。
3. 不直接本地多进程裸跑 SQL。
4. 不碰正在运行的实验（以 GCS 锁状态为准）。
5. 不删除或覆盖 `reports/strategy1` 下已有产物。
6. 状态表记录不得包含 BigQuery key / Tushare token 等凭据。

## 后续 Cloud Composer 映射

| 本地调度器语义 | Composer 映射 |
|---|---|
| `stage_id` | DAG run config / task group |
| `experiment_id` | task group 内的实验参数 |
| `step_id` | Airflow task / BigQueryInsertJobOperator |
| `depends_on_experiment_id` | Airflow task dependency |
| `max_parallel` | DAG / pool concurrency |
| `max_parallel_backtest` | Airflow pool `strategy1_backtest_pool` |
| GCS lock object | task 前置 acquire-lock task |
| 状态表 | 审计表和 resume / report 输入 |

Composer 接入后，保留相同的 lock key、状态表字段和参数接口。
Composer 只替换调度执行面，不改变实验身份、写隔离、registry 稳定性或 08 ledger 并发边界。
