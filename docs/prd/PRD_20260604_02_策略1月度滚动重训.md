# PRD: 策略 1 月度滚动重训

## 1. 背景

策略 1 当前 v0 最优参数已固化为正式 baseline：

- 参数: `pv_fin_quality + 30/5% + biweekly + 5d`
- `run_id`: `s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`
- `backtest_id`: `bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`
- 回测区间: `2024-01-02` 至 `2025-12-31`
- 主基准: `000852.SH`
- 结果: total_return=41.10%, excess_return=12.09%, Sharpe=1.043, max_drawdown=-14.48%

当前 baseline 使用固定训练/验证/回测切分。随着时间推进，A 股市场风格、行业结构和交易制度会变化，固定模型可能逐渐失效。因此需要定义模型生命周期：什么时候重新训练、训练窗口如何滚动、当月预测使用哪个模型、训练失败如何回退、如何避免未来函数。

本文只定义月度滚动重训。交易执行语义由 `PRD_20260604_01_策略1LedgerV1交易执行语义.md` 定义，两者不得混在同一个实现 PR 中。

月度滚动重训的正式效果归因依赖 Ledger v1 的三个阶段先完成：

1. P0: Ledger v1 交易执行语义。
2. P1: `2024-01-02` 至 `2026-04-30` fixed-model 连续扩展回测。
3. P2: ledger state resume。

## 2. 目标

1. 定义策略 1 模型月度重训 cadence。
2. 定义 rolling train / valid 窗口和 label 可用边界。
3. 定义模型生效日和月内固定模型规则。
4. 定义训练失败时的回退策略。
5. 产出可供 Ledger v1 消费的 PIT-safe prediction stream。
6. 支持与固定模型 baseline 做 A/B 比较。

## 3. 非目标

1. 不改变 Ledger v1 成交规则。
2. 不改变当前最优参数本身。
3. 不切换模型族；P0 仍使用 BigQuery ML `LOGISTIC_REG`。
4. 不引入日度重训、周度重训或在线学习。
5. 不修改 DWS 特征定义；财务特征版本仍使用 `fin_default_v0_20260602`。
6. 不实现 ledger state resume；resume 状态恢复由 Ledger v1 PRD P2 定义。
7. 不定义 `2024-01-02` 至 `2026-04-30` 连续 fixed-model 回测；该扩展 baseline 由 Ledger v1 PRD P1 产出。
8. 不与 Ledger v1 或 resume 同 PR 实现，避免模型生命周期变化和交易执行语义变化混在一起。

## 4. 推荐默认口径

| 项 | 推荐 |
|---|---|
| 重训周期 | 每月一次 |
| 重训触发 | 每月第一个开市日收盘后，确认上月末 DWS/ADS 数据和 QA 完成后启动 |
| 模型生效 | 新模型通过 QA 后，从下一个开市日或下一个调仓信号日起生效，取较晚者 |
| 月内模型 | 月内固定，不每日切换 |
| 训练窗口 | rolling 5 年训练窗口 |
| 验证窗口 | 训练截止前最近 12 个月 label-available 样本 |
| 标签边界 | 训练/验证只使用截至 `train_cutoff_date` 已完整可见的 label |
| 失败回退 | 新模型训练或 QA 失败时沿用上一 active 模型 |
| 模型族 | BigQuery ML `LOGISTIC_REG`，继续使用 score orientation 校准 |

不建议 P0 使用每日或每周重训：版本过碎、噪声大、排错困难，也不适合当前 biweekly 调仓节奏。

## 5. 时间线定义

对任意生效月份 `M`：

1. `month_end_date`: `M-1` 月最后一个开市日。
2. `train_cutoff_date`: `month_end_date`，表示本轮训练最多只能使用该日收盘后已可见的数据。
3. `label_available_end_date`: `train_cutoff_date - (label_horizon + 1 个交易日缓冲)`，确保 open-to-close forward label 已完整形成。
4. `valid_end_date`: `label_available_end_date`。
5. `valid_start_date`: `valid_end_date` 往前约 12 个月的首个开市日。
6. `train_end_date`: `valid_start_date` 前一开市日。
7. `train_start_date`: `train_end_date` 往前约 5 年，且不早于当前样本最早稳定日期 `2019-04-03`。
8. `model_effective_from`: 新模型训练、选型、预测和 QA 全部通过后的下一个开市日；若该日已经错过当期调仓信号，则从下一调仓信号日起生效。
9. `model_effective_to`: 下一 active 模型生效日前一开市日。

示例：若 2026-07 模型在 2026-07 第一个开市日收盘后训练，`train_cutoff_date` 为 2026-06 最后一个开市日。训练/验证 label 不能使用 cutoff 后未来收益；预测可覆盖 2026-07 后续交易日。

## 6. 数据与 PIT 规则

1. 训练样本仍来自 `ads_ml_training_panel_daily`。
2. valid/test/live 预测池仍遵循 live-available 口径：
   - `in_universe_default`
   - `has_full_history_60d`
   - `has_valuation_data`
3. label 只用于训练、验证和事后评价，不能影响 live 预测入池。
4. 每个预测日使用的模型必须满足：

```text
model_train_cutoff_date < predict_date
model_effective_from <= predict_date <= model_effective_to
```

5. 不允许用预测日之后才披露的财务数据、标签或模型指标选择该日模型。
6. 月度重训模式下，部署决策只使用 train + valid；`test` split 不作为固定 holdout 参与当月模型激活。为兼容既有 `01_build_training_panel.sql`，实现可以保留 test 行，但这些行只能在标签成熟后做事后评价，不得影响当月模型选择、score orientation、active/fallback 决策或 prediction stream 生成。

## 7. 输出接口

### 7.1 模型注册

继续写 `ashare_ads.ads_model_registry`，并在 `model_params_json` / `metrics_json` 中补充以下字段：

| 字段 | 说明 |
|---|---|
| `retrain_cadence` | `monthly` |
| `train_cutoff_date` | 本轮训练可见数据截止日 |
| `label_available_end_date` | 本轮训练/验证可用 label 截止日 |
| `rolling_train_years` | 默认 5 |
| `valid_window_months` | 默认 12 |
| `model_effective_from` | 模型生效起始日 |
| `model_effective_to` | 模型生效结束日，可为空 |
| `fallback_from_model_id` | 若本轮失败，沿用的上一模型 |
| `deployment_status` | `candidate` / `active` / `rejected` / `fallback` |

P0 可先不新增物理字段，使用 JSON 保持 schema 兼容；P1 可新增模型部署表。

### 7.2 Prediction Stream

月度重训产出的预测必须拼接成一条 Ledger 可消费的 prediction stream。

推荐 P0 口径：

1. 新增参数 `p_prediction_stream_id`。
2. 每个预测日写入 `ads_model_prediction_daily.run_id = p_prediction_stream_id`。
3. `model_id` 保留该预测日实际使用的月度模型。
4. 对 Ledger 来说，仍只读取一个 `run_id` / prediction stream。
5. 同一 stream 内必须满足 `(predict_date, sec_code)` 唯一。

这样 Ledger v1 不需要理解月度训练细节，只消费一条 PIT-safe 预测流。

### 7.3 可选部署表

P1 可新增 `ashare_ads.ads_model_deployment_period`：

| 字段 | 说明 |
|---|---|
| `deployment_id` | 部署批次 id |
| `strategy_id` | 策略 id |
| `model_id` | 生效模型 |
| `prediction_stream_id` | 预测流 id |
| `train_cutoff_date` | 训练截止日 |
| `effective_from` | 生效起始日 |
| `effective_to` | 生效结束日 |
| `status` | active/rejected/fallback |
| `metrics_json` | 部署指标 |
| `created_at` | 写入时间 |

P0 如不新增表，则在 manifest 和 registry JSON 中表达同等信息。

## 8. 训练与部署流程

每月执行：

1. 确认上月末 DWD/DWS/ADS 已构建完成。
2. 跑基础 QA：
   - P0 smoke
   - 策略 1 DWS/ADS QA
   - 财务特征 QA
   - 单位 QA
3. 计算本轮 `train_start_date`、`train_end_date`、`valid_start_date`、`valid_end_date`。
4. 执行 `01_build_training_panel.sql`，生成本月训练面板。
5. 执行 `02_train_bqml_logistic_candidates.sql`。
6. 执行 `03_select_model_and_register.sql`，包含 score orientation 校准。
7. 对本月生效区间执行 `04_predict_daily.sql`。
8. 对本月预测结果做 prediction QA。
9. 若模型和预测 QA 通过，将预测写入或合并到 `p_prediction_stream_id`。
10. 若 QA 失败，则不激活新模型，沿用上一 active 模型并写 fallback 记录。

## 9. 通过 / 拒绝标准

### 9.1 候选模型通过标准

候选模型必须满足：

1. 训练和验证样本行数大于阈值。
2. valid prediction coverage >= 0.50；0.30-0.50 为 warning，不自动上线。
3. score orientation 决策成功，且 `score_orientation` 非空。
4. oriented valid RankIC 均值不低于 0；若 orientation 后仍为负，候选模型标记 `rejected`。raw RankIC 可为负，但必须由 score orientation 机制反转并在 registry 中记录 `oriented_valid_rank_ic_mean`。
5. valid top bucket 5d 平均收益不显著劣于整体样本。
6. BQML 训练 job 成功，registry 与模型对象一致。

### 9.2 部署通过标准

月度模型部署必须满足：

1. `ads_model_prediction_daily` 中本月每个预测日有预测。
2. `(predict_date, sec_code)` 唯一。
3. 每个预测日只使用当时已经 active 的模型。
4. `score` 已 orientation 校准。
5. 预测覆盖率和候选池行数未异常断崖。
6. 若任一 P0 QA 失败，不激活新模型。

## 10. 回退规则

1. 新模型训练失败：沿用上一 active 模型。
2. 新模型 QA 失败：沿用上一 active 模型。
3. 新模型预测写入失败：不激活，沿用上一 active 模型。
4. 上一 active 模型不存在：本月不生成 active prediction stream，runner 失败并要求人工处理。
5. 回退必须写入 registry/manifest：
   - `deployment_status=fallback`
   - `fallback_from_model_id`
   - `fallback_reason`

## 11. 与 Ledger v1 的边界

Ledger v1 只消费预测流，不参与模型选择。

1. 固定模型模式：Ledger 读取单一 `p_prediction_run_id`。
2. 月度重训模式：Ledger 读取单一 `p_prediction_stream_id`，但该 stream 内可包含多个 `model_id`。
3. Ledger 不判断模型是否该生效；月度重训流程必须提前保证 prediction stream PIT 正确。
4. 为比较清楚，Ledger v1 实现 PR 不应同时启用月度重训。

推荐实施顺序：

1. 先实现 `PRD_20260604_03_策略1因子贡献度分析.md`，生成当前正式 baseline 的因子贡献度报告；该步骤不代表高于 Ledger 或月度重训的优先级，只是先拿到解释基准。
2. 再实现 Ledger v1 P0 交易执行语义，并与正式 baseline A/B。
3. 再实现 Ledger v1 P1 fixed-model 连续扩展回测，产出 `2024-01-02` 至 `2026-04-30` 的 fixed-model extended baseline。
4. 再实现 Ledger v1 P2 state resume，并验证全段回测与 `2024-2025 + resume 2026` 拼接路径一致。
5. 最后实现月度滚动重训，并与 Ledger v1 P1/P2 产出的 fixed-model extended baseline 对比。

## 12. 实验与验收

### 12.1 P0 验收实验

用当前最优参数：

- `pv_fin_quality`
- `target_holdings=30`
- `max_single_weight=0.05`
- `rebalance_frequency=biweekly`
- `label_horizon=5`
- 主基准 `000852.SH`

跑一条月度重训 prediction stream，并用同一 Ledger 规则回测。

前置条件：

1. Ledger v1 P0/P1/P2 已完成。
2. fixed-model extended baseline 已由 Ledger v1 P1 产出，区间为 `2024-01-02` 至 `2026-04-30`。
3. ledger state resume 已通过 P2 等价性验收。

必须同时报告：

1. Ledger v1 fixed-model extended baseline 表现。
2. Ledger v1 monthly-retrain 表现。
3. 两者在全段 `2024-01-02` 至 `2026-04-30` 的收益、回撤、Sharpe、换手、成本、RankIC、top bucket 收益差异。
4. 两者在 `2026-01-02` 至 `2026-04-30` forward OOS 子区间的表现差异。

### 12.2 QA

新增或扩展 QA：

1. 每个 prediction stream 的 `(predict_date, sec_code)` 唯一。
2. 每个 predict_date 的 model_id 在该日已生效。
3. 任一 model 的 `train_cutoff_date < predict_date`。
4. 任一训练/验证样本不使用 cutoff 后 label。
5. 月度模型未通过 QA 时不得进入 active stream。
6. fallback 月份必须有 fallback reason。
7. 月内模型固定：同一月份内非 fallback 情况下模型不频繁日切。

## 13. 风险

1. 月度重训可能提高收益，也可能引入更高噪声和换手。
2. rolling 5 年窗口可能丢失早期样本；早期月份训练窗口不足时需自动降级为 expanding-from-2019。
3. 如果 monthly retrain 和 ledger v1 同时实现，结果变化不可归因，因此必须分 PR。
4. BQML 多模型训练成本会上升，需要记录每月训练 job 成本。
5. registry JSON 兼容实现简单，但长期可查询性不如部署表；P1 应考虑物理部署表。

## 14. 后续扩展

1. 对比 monthly / quarterly / yearly cadence。
2. 对比 rolling 3 年 / rolling 5 年 / expanding 窗口。
3. 引入市场 regime 条件下的重训频率调整。
4. 引入模型族对照，如 boosted tree 或 linear model ensemble。
5. 引入模型冻结和人工审批机制。

## 15. 验收结论

月度滚动重训通过条件：

1. 每月训练、选型、预测、部署状态可追溯。
2. 预测流满足 PIT，无未来函数。
3. 失败时能可靠回退到上一 active 模型。
4. Ledger 可用单一 prediction stream 完成回测。
5. 与固定模型 baseline 的收益变化能被清晰归因到模型生命周期，而不是交易执行语义变化。
