## 2. 目标与非目标

### 2.1 目标

1. 从 `ashare_dws.dws_stock_sample_daily` 生成冻结训练面板。
2. 用 BigQuery ML 训练 `ml_pv_clf_v0` 候选模型。
3. 按验证集 RankIC / 分层收益选择模型并登记。
4. 写入预测、候选池、目标组合、订单计划和回测 ADS 表。
5. 生成结构化绩效指标和报告输入，并输出 GCS 报告 URI 与本地镜像路径。
6. 提供 QA 脚本，证明无未来泄露、结果可复现、组合约束满足、NAV 连续。

### 2.2 非目标

- 不做 BigQuery 之外的模型训练。
- 不实现真实券商下单或实盘交易。
- 不新增策略 1 之外特征域，如财务、资金流、行业中性化。
- 不改变既有 DWS/ADS 表契约，除非实现中发现契约无法承载 runner 必要字段并单独提 PR。
- 不解决 OQ-010 的业务参数决策；未确认参数以 runner 配置项暴露。

## 6. 功能需求

### FR-1 构建训练面板

脚本：`01_build_training_panel.sql`

要求：

- 读取 `dws_stock_sample_daily`。
- 过滤 `feature_version`、`label_version`、日期范围。
- 只把白名单模型特征写入训练面板的可训练列。
- `board` 保留为分组/暴露字段，但不进入模型训练列。
- 写入 `split_tag`：`train`、`valid`、`test`、`live`。
- `run_id, sec_code, trade_date` 唯一。
- 默认训练样本使用 `sample_trainable_default=TRUE`。

### FR-2 训练 BQML 候选模型

脚本：`02_train_bqml_logistic_candidates.sql`

要求：

- 模型类型：BQML `LOGISTIC_REG`。
- 标签：`label_top30_5d`。
- 正则化：使用 BQML 原生 `L1_REG` / `L2_REG` 候选网格。
- 每组候选参数生成独立 BigQuery ML model object。
- 候选模型登记到 `ads_model_registry`，状态为 `candidate`.
- 不使用随机 split；训练 SQL 只读 `split_tag='train'`。

候选网格首版至少包含：

| candidate_id | `L1_REG` | `L2_REG` |
|---|---:|---:|
| `l1_0_l2_0` | 0 | 0 |
| `l1_0_l2_1e_4` | 0 | 0.0001 |
| `l1_0_l2_1e_3` | 0 | 0.001 |
| `l1_1e_5_l2_1e_4` | 0.00001 | 0.0001 |
| `l1_1e_4_l2_1e_3` | 0.0001 | 0.001 |

### FR-3 验证、选择并登记模型

脚本：`03_select_model_and_register.sql`

要求：

- 对每个候选模型在 valid 窗口执行 `ML.PREDICT`。
- 计算 valid：
  - AUC / log_loss
  - RankIC 均值与按年统计
  - 分 5/10 层收益
  - TopN 未来收益
  - 样本数、预测数、缺失率
- 主选择准则：`valid_rank_ic_mean`。
- tie-breaker：分层单调性、TopN 收益稳定性、log_loss。
- 胜出模型状态更新为 `registered` 或 `selected`；失败候选标为 `candidate_rejected`。
- `model_params_json` 必须记录候选网格、胜出原因、特征列清单。

### FR-4 生成预测分

脚本：`04_predict_daily.sql`

要求：

- 对 selected model 执行 `ML.PREDICT`。
- 覆盖 valid/test，后续可覆盖 live。
- 从 BQML 输出中抽取正类概率为 `score`。
- 按 `predict_date` 做横截面排序：
  - `rank_raw`
  - `rank_pct`
- 写入 `ads_model_prediction_daily`。

### FR-5 生成候选池

脚本：`05_build_candidates.sql`

要求：

- 调仓日为每周最后一个开市日。
- 只使用 `rebalance_date=t` 当日已知字段。
- 禁止用 `t+1` 的 `can_buy_open`、`label_entry_tradable` 或未来字段预过滤。
- 输出完整候选池和最终入选标记。
- `filter_reason` 必须可解释。

### FR-6 生成目标组合

脚本：`06_build_portfolio_targets.sql`

要求：

- 默认按 `score DESC` 选 TopN。
- 等权或按配置生成权重。
- 单票权重不得超过 `max_single_weight`。
- 目标权重之和不得超过 1。
- 不因 `t+1` 不可买而事后替补低排名股票。

### FR-7 生成订单计划

脚本：`07_build_order_plan.sql`

要求：

- 对比上一期持仓和本期目标组合生成 BUY/SELL。
- 输出 `expected_price`、`expected_amount_cny`、`order_reason`。
- 订单计划只作为回测输入，不代表真实下单。

### FR-8 BigQuery SQL 回测

脚本：`08_run_backtest.sql`

要求：

- 信号日 `t`，在 `t+1` 开盘撮合。
- 买入要求 `can_buy_open[t+1]=TRUE`；不可买则买入失败并保留现金。
- 卖出顺延首版采用**预计算 next-sellable 日期方案**，避免逐日 `WHILE` 状态循环。
- 对每个 `(sec_code, trade_date)` 预计算 `next_sellable_trade_date`：从 `trade_date` 之后的开市日中取最早满足 `can_sell_open=TRUE` 的日期。
- 回测生成卖出订单时，如果目标执行日不可卖，则把实际卖出日映射到 `next_sellable_trade_date`，并在顺延期间继续持有和估值。
- `next_sellable_trade_date` 查找窗口首版设为 60 个交易日；超过窗口仍不可卖时，不强制清仓，持仓继续按最后可用收盘价估值，并标记 `SELL_BLOCKED_NO_NEXT_SELLABLE_60D`。
- 连续停牌、连续一字跌停等多日不可卖情况必须进入 `sell_delay_days`、`sell_blocked_count` 和异常原因统计。
- 持仓每日按收盘价估值。
- 写入 trade / position / nav / summary 四类 ADS 回测表。
- 成本参数化，不硬编码业务最终值。

### FR-9 指标与报告输入

脚本：`09_build_metrics_and_report_inputs.sql`

报告渲染脚本：`scripts/strategy1/render_report.py`

要求：

- 生成回测汇总指标：
  - 年化收益、年化波动、Sharpe、最大回撤
  - 基准收益、超额收益、IR
  - 年化换手
  - 买入失败率、卖出顺延率
  - 成本敏感性
- SQL 只负责写 ADS 汇总和生成报告所需结构化数据，不直接生成 Markdown/HTML/PNG。
- `scripts/strategy1/render_report.py` 是必需交付物，负责：
  - 从 ADS 汇总或导出的 `metrics.json` 读取结构化指标。
  - 渲染 `report.md`、`report.html` 和 PNG 图表。
  - 上传报告产物到 GCS。
  - 写入或回传本地 `reports/` 镜像路径。
- GCS 上传可由 `render_report.py` 内部调用存储 API，也可由 README 规定用 `gsutil cp`；两者都必须把最终 `report_uri` 写回 ADS。
- 报告脚本不得承担模型训练、预测或回测计算职责。

### FR-10 QA

脚本：`10_qa_runner_outputs.sql`

必须断言：

- 同一 `run_id, sec_code, trade_date` 训练面板唯一。
- train/valid/test 日期互斥且有序。
- 模型特征不含 `fwd_*`、`label_*`、`rank_pct_*`、`*_qfq`、`board`。
- 预测日 `rank_raw=1` 唯一，`rank_pct` 在 `[0,1]`。
- 每个调仓日目标权重和 `<=1`，单票权重 `<=max_single_weight`。
- 代码审查项：`05_build_candidates.sql` 禁止引用 `t+1` 可交易字段做预过滤；该项不作为运行时 SQL 断言。
- 数据侧 QA：统计 `t+1` 实际不可买但仍被 `is_selected_candidate=TRUE` 选中的样本数和占比；若长期为 0，报告中标记“可能存在事后过滤风险”，需要人工复核候选池 SQL。
- 卖出顺延 QA：统计 `sell_delay_days` 分布、超过 60 个交易日仍不可卖数量、异常原因。
- NAV 覆盖开市日且无非预期断点。
- `ads_model_registry` 中 selected model 有唯一 `model_uri`。
- 回测 summary 有 `report_uri`。

## 7. 非功能需求

### 7.1 幂等

- 默认发现同 `run_id` 或 `backtest_id` 已存在结果时失败。
- `force_replace=TRUE` 时，脚本必须按 `run_id/backtest_id` 删除旧 ADS 行后重写。
- BigQuery model object 不覆盖 active model；同 run 重跑只允许覆盖同名候选模型。

### 7.2 成本控制

- 所有大表读取必须带日期分区过滤。
- 禁止训练 SQL 使用 `SELECT *`。
- 首次实现需提供 dry-run 或 bytes processed 记录方式。

### 7.3 安全

- 不写任何凭据、token、service account key。
- GCS bucket 名称通过参数注入，不写密钥。
- 本地 `reports/` 只保存生成型报告镜像，不提交 git。

### 7.4 可追溯

每个 run 必须能追溯：

- `run_id`
- `model_id`
- `backtest_id`
- git commit
- 参数 JSON
- feature / label / universe version
- 训练、验证、测试窗口
- GCS report URI

## 12. 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
