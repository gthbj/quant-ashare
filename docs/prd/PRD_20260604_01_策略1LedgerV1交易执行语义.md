# PRD: 策略 1 Ledger v1 交易执行语义

## 1. 背景

策略 1 当前 BigQuery ML runner 已完成固定参数正式基线：

- `run_id`: `s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`
- `backtest_id`: `bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`
- 参数: `pv_fin_quality + 30/5% + biweekly + 5d`
- 回测区间: `2024-01-02` 至 `2025-12-31`
- 主基准: `000852.SH`
- 结果: total_return=41.10%, excess_return=12.09%, Sharpe=1.043, max_drawdown=-14.48%
- 报告和诊断: `gs://ashare-artifacts/reports/strategy1/ml_pv_clf_v0/run_id=s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01/backtest_id=bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`

当前 `sql/ml/strategy1/08_run_backtest.sql` 已是账户级有状态 ledger，但仍存在一组需要产品化固化的交易执行语义：信号日与执行日边界、卖不出是否每日继续卖、买不进是否候补、实际持仓与目标持仓如何 netting、订单失败如何记录、NAV 是否每日按市值重估等。

本文中的 **Ledger v1** 指策略 1 后续正式对比使用的 `ledger_exec_v1` 交易执行语义契约。它在当前账户级 ledger 基础上进一步固化交易规则。该 PRD 会改变回测含义，因此必须先于实现和结果比较落地。

## 2. 目标

1. 固化策略 1 从信号到成交、持仓、NAV 的账户级交易规则。
2. 让后续修改 ledger 后的结果可以与当前正式基线做 A/B 比较，隔离“交易执行语义变化”的影响。
3. 为后续月度滚动重训预留输入接口，但不在本文实现训练逻辑。
4. 明确 `05_build_candidates`、`06_build_portfolio_targets`、`07_build_order_plan` 与 `08_run_backtest` 的职责边界。

## 3. 非目标

1. 不改变当前最优参数。
2. 不改变模型训练、score orientation、预测池口径。
3. 不实现月度滚动重训；月度重训由 `PRD_20260604_02_策略1月度滚动重训.md` 定义。
4. 不引入分钟线、真实 VWAP、撮合队列、盘口成交概率。
5. P0 不引入 A 股 100 股买入手数约束，仍保持当前 FLOAT shares 口径；手数约束后续可单独作为交易现实性增强。
6. P0 不显式建模 A 股 T+1 卖出锁定；当前策略使用 t-1 信号、t 开盘执行且调仓频率为 weekly/biweekly/monthly，同股当日买入又当日卖出的边界应由 netting 和 QA 自然规避。若后续引入日内/高频交易或手数约束，再单独补 T+1 锁仓模型。

## 4. 术语

| 术语 | 定义 |
|---|---|
| `signal_date` | 信号日期，使用该日收盘后已可见的数据和预测分数生成目标 |
| `execution_date` | 执行日期，使用 `signal_date` 后下一个开市日开盘执行 |
| `target portfolio` | 目标组合，只表达策略希望持有的权重，不代表真实成交结果 |
| `actual position` | ledger 中真实持仓，由历史成交和未成交 carry 决定 |
| `pending sell` | 股票不在目标组合或目标仓位降低，但因不可卖未能退出，后续每日继续尝试卖出的状态 |
| `netting` | 按目标市值与当前实际持仓市值差额生成净买卖单 |
| `order intent` | 计划交易意图，可被跳过、缩放、取消或成交 |
| `fill` | ledger 实际成交记录，是 NAV 与持仓变化的唯一来源 |

## 5. 总体设计

### 5.1 日期口径

推荐并固化以下口径：

1. `signal_date = t-1`：目标组合只能使用执行日前一交易日收盘后已可见的预测和特征。
2. `execution_date = t`：在下一个开市日开盘执行。
3. P0 为兼容既有 ADS 表，`ads_stock_candidate_daily.rebalance_date`、`ads_portfolio_target_daily.rebalance_date`、`ads_order_plan_daily.rebalance_date` 暂继续表示 `signal_date`。
4. `08_run_backtest.sql` 必须从 `rebalance_date` 推导 `execution_date = next_open_trade_date(rebalance_date)`。
5. 报告和 QA 必须展示 `signal_date` 与 `execution_date`，避免把信号日误读为成交日。

该口径等价于用户视角的“调仓日 t 使用 t-1 日信号，并在 t 日开盘成交”。

### 5.2 执行价口径

1. 买入成交价: `open * (1 + p_slippage_buy_bps / 10000)`。
2. 卖出成交价: `open * (1 - p_slippage_sell_bps / 10000)`。
3. 交易价格使用现金口径 `dwd_stock_eod_price.open`，不使用 `_hfq` / `_qfq`。
4. 若 `open IS NULL` 或方向性可交易标记为 false，则该订单不成交并写失败状态。
5. P0 不使用 VWAP；VWAP 需要分钟级或可靠成交量假设，后续单独评估。

### 5.3 目标生成与候补

1. 目标组合由 `signal_date` 的 oriented score 降序选出 top N。
2. 买不进时 P0 不做候补替换。
3. 买不进造成的目标缺口以现金保留，不从候补池重新补股票。
4. 理由：候补会引入新的时点边界和执行偏差，应先把主交易语义固化。

### 5.4 Ledger 执行顺序

每个 `execution_date` 按以下顺序执行：

1. 用执行日前最近可用收盘价对实际持仓估值，得到执行前 NAV。
2. 读取该日对应的目标组合。
3. 取股票全集：`actual_position ∪ target_portfolio ∪ pending_sell`。
4. 对每只股票计算 `target_value - actual_market_value`。
5. 先处理净卖出和 pending sell。
6. 卖出成交后更新现金和实际持仓。
7. 再处理净买入。
8. 若买入总需求超过可用现金，按目标买入金额等比例缩放。
9. 写成交、跳过、取消、缩放等订单状态。
10. 每个开市日按收盘价 mark-to-market 生成 NAV。

### 5.5 Netting 规则

对每只股票使用同一公式：

```text
delta_value = target_value - current_market_value
```

规则：

1. `delta_value > 0`：生成净买入。
2. `delta_value < 0`：生成净卖出。
3. `delta_value = 0` 或绝对值低于最小阈值：不交易，写 `NOOP_ALREADY_TARGET` 或不写订单。
4. 旧仓未卖出期间股票重新入选，必须用上述公式自动 netting：
   - 若当前持仓大于新目标，继续卖到目标。
   - 若当前持仓小于新目标，只补差额。
   - 若当前持仓接近新目标，取消 pending sell。
5. 严禁对未平旧仓重复全额建仓。

### 5.6 卖不出处理

推荐并固化：

1. 卖不出时进入 `pending_sell`。
2. `pending_sell` 在之后每个开市日继续尝试卖出，不等下一个调仓日。
3. 若股票在后续目标组合中重新出现，则按 netting 规则调整或取消 pending sell。
4. pending sell 持仓仍属于真实持仓，必须计入 NAV、风险暴露、行业/风格暴露。

### 5.7 买不进处理

1. 买不进时不候补。
2. 写 `BUY_SKIPPED_UNTRADABLE`。
3. 未使用资金保留现金。
4. 后续只在下一次目标生成或净买入机会时重新评估，不每日追买。

### 5.8 现金不足处理

1. 卖出先于买入。
2. 买入预算以卖出后可用现金为上限。
3. 佣金、印花税、滑点按 OQ-010 成本 profile 计入买卖需求。
4. 若买入总需求超过可用现金，则对全部买单按金额等比例缩放：

```text
scale = available_cash / required_cash_with_cost
actual_buy_amount_i = target_buy_amount_i * min(1, scale)
```

5. 缩放后订单写 `FILLED_SCALED_CASH` 或在 `metrics_json` / 订单字段记录 `cash_scale`。
6. 缩放后小于最小名义金额的订单可跳过，状态为 `SKIPPED_MIN_NOTIONAL`。P0 默认最小名义金额为 0，暂不触发。

### 5.9 非目标持仓

1. 非目标持仓允许继续存在，因为真实情况下不可卖股票无法消失。
2. 非目标持仓标记为 `is_target_position=false`、`exit_intent=pending_sell`。
3. 只要仍持有，必须计入 NAV 和风险暴露。
4. 报告中必须能解释“目标组合”和“实际持仓”不一致的原因。

### 5.10 订单状态

`ads_backtest_trade_daily.fill_status` 至少支持以下状态：

| 状态 | 含义 |
|---|---|
| `FILLED` | 全部成交 |
| `FILLED_SCALED_CASH` | 因现金不足按比例缩放后成交 |
| `BUY_SKIPPED_UNTRADABLE` | 买入不可交易，未成交 |
| `SELL_SKIPPED_UNTRADABLE` | 卖出不可交易，未成交并进入/维持 pending sell |
| `PENDING_SELL_CARRY` | 非调仓日延续 pending sell 并继续尝试 |
| `CANCELLED_BY_NETTING` | 旧 pending sell 因股票重新入选或目标提高而被 netting 取消 |
| `SKIPPED_CASH_INSUFFICIENT` | 现金不足且缩放后无法成交 |
| `SKIPPED_MIN_NOTIONAL` | 低于最小成交金额 |
| `NOOP_ALREADY_TARGET` | 当前实际持仓已经接近目标，无需交易 |

P0 可继续使用当前表结构，但必须在字段说明、README、QA 中补全状态枚举。若需要记录 `signal_date`、`execution_date`、`cash_scale`、`target_value_cny`、`actual_value_cny`，可通过后续 schema migration 添加字段。

### 5.11 NAV 口径

1. 每个开市日必须生成一条 NAV。
2. NAV 使用每日收盘 mark-to-market：

```text
net_value_cny = cash_cny + SUM(shares * close_or_ffill_close)
nav = net_value_cny / initial_capital
```

3. 停牌但有收盘价时使用当日收盘价。
4. 无收盘价时使用最近可用收盘价前向填充，并在 monitor 中记录 `ffill_price_count`。
5. 当日成交在开盘发生，NAV 在同日收盘重估。

## 6. 05-07 与 08 的职责边界

### 6.1 05 候选池

1. 05 只负责从 `prediction_run_id` / prediction stream 读取 `signal_date` 的预测并生成候选池。
2. 05 不读取真实持仓，不处理 pending sell。
3. 05 不判断买卖是否会成功。

### 6.2 06 目标组合

1. 06 只负责生成目标权重。
2. 目标组合是 desired state，不是实际持仓。
3. 若目标权重总和小于 1，剩余部分天然为现金目标。

### 6.3 07 订单计划

1. 07 在 P0 中定位为“静态预览订单计划”，不作为最终成交来源。
2. 07 可以根据上一调仓目标生成粗略 BUY/SELL intent，但不得替代 08 的真实 netting。
3. 08 必须以实际持仓和目标组合重新计算净订单。
4. 后续可将 07 下沉到 ledger 内部，或新增 `ads_order_intent_daily` 与 `ads_backtest_trade_daily` 做计划/成交对账。

### 6.4 08 Ledger

1. 08 是实际成交、持仓、现金、NAV 的唯一事实来源。
2. 08 必须逐日期维护 ledger 状态。
3. 08 必须消费目标组合，但不能假设目标组合一定成交。
4. 08 输出的 `ads_backtest_trade_daily`、`ads_backtest_position_daily`、`ads_backtest_nav_daily` 是后续报告和指标的事实来源。

## 7. 月度重训接口预留

Ledger v1 不负责训练模型，但必须为后续月度重训预留输入接口。

1. P0 支持固定 `p_prediction_run_id`：整段回测使用一个预测 run。
2. 未来支持 `p_prediction_stream_id`：一条预测流中可以包含多个 `model_id`，每个 `predict_date` 使用当时有效的模型。
3. Ledger 不关心模型是一次训练、月度重训还是人工指定；只要求输入预测流满足 PIT：
   - `predict_date` 的预测只能来自该日前已训练完成并已生效的模型。
   - 每个 `(predict_date, sec_code)` 在预测流中唯一。
   - score 已完成 orientation 校准。
4. 月度重训 PRD 必须负责生成满足上述约束的 prediction stream。

## 8. QA 与验收

### 8.1 Ledger QA

新增或扩展 QA：

1. `cash_cny >= -1`。
2. `gross_exposure <= 1.005`。
3. `ads_backtest_position_daily` 在 `(backtest_id, trade_date, sec_code)` 唯一。
4. `ads_backtest_nav_daily` 覆盖完整开市日。
5. 任一实际持仓不能无成交凭空消失。
6. 非目标持仓若仍存在，必须能追溯到 pending sell / skip 状态。
7. 同一股票同一 `execution_date` 不得同时存在未 netting 的独立 BUY 和 SELL。
8. 买入现金缩放后，成交现金影响与费用之和不得超过可用现金。
9. `BUY_SKIPPED_UNTRADABLE` 不产生成交额和现金影响。
10. `SELL_SKIPPED_UNTRADABLE` 不减少持仓。
11. pending sell 股票在下一个可卖开市日必须继续尝试卖出或被 netting 取消。
12. 报告必须展示目标持仓数、实际持仓数、pending sell 数、买入跳过数、卖出跳过数。

### 8.2 A/B 验收

以正式基线作为对照：

- baseline run: `s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`
- baseline backtest: `bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01`

Ledger v1 实现后必须使用相同参数、相同预测来源、相同成本 profile 跑一条新 backtest，并报告：

1. 收益、回撤、Sharpe、换手变化。
2. 与当前 baseline 的差异来源：
   - 执行日变化。
   - pending sell 每日继续卖。
   - netting 修正。
   - 现金缩放。
   - 非目标持仓 carry。
3. 至少列出前 20 个导致 NAV 差异最大的交易日和对应股票。

## 9. 实施计划

1. PRD review 后先改 SQL/ADS 契约说明，不直接重跑实验矩阵。
2. 调整 `08_run_backtest.sql`：
   - 固化 `signal_date -> execution_date`。
   - 支持每日 pending sell。
   - 强化 netting。
   - 输出完整订单状态。
3. 视字段需要调整 `ads_backtest_trade_daily`、`ads_backtest_position_daily` 或通过 `metrics_json` 兼容。
4. 扩展 `10_qa_runner_outputs.sql`。
5. 扩展报告和诊断展示。
6. 用正式 baseline 参数跑一条 ledger v1 A/B backtest。
7. 结果稳定后再进入月度重训实现。

## 10. 风险

1. Ledger v1 会改变历史回测收益，必须以 baseline A/B 解释差异。
2. 每日 pending sell 会增加 SQL 循环复杂度和执行时间。
3. 若后续加入手数约束，会再次改变结果，应单独 PRD/子项处理。
4. 如果目标组合和实际持仓差异变大，报告必须强化解释，否则用户会误解“为什么目标股没买到/为什么还持有旧股”。

## 11. 验收结论

Ledger v1 通过条件：

1. 同参数、同预测源的 A/B backtest 全链路 01-12 通过。
2. 交易状态可解释。
3. NAV 每日连续且无负现金/超暴露。
4. baseline 差异有报告归因。
5. 不改变模型训练和预测来源。
