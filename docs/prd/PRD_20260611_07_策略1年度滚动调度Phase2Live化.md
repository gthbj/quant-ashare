# PRD：策略 1 年度滚动调度 Phase 2 Live 化

> 状态：草案，待 owner review。
> 范围声明：把 Phase 1 dry-run scheduler 升级为可真实提交 Cloud Run execution 的 live scheduler，并完成 candidate-only live smoke；不跑完整年度滚动（Phase 3 按需另批）。
> 关联：`PRD_20260611_01_策略1年度滚动并发调度.md` §13 Phase 2、PR #161（Phase 1 dry-run）、KNOWN_CONSTRAINTS「gcloud --wait 非零 ≠ 失败」「maxRetries=0」。

---

## 1. 背景

Phase 1（PR #161）已落地 `quant_ashare.strategy1.annual_pipeline_scheduler` dry-run：跨年度 DAG、
互斥锁计划、generation-conditioned state 模型、共享资源池模拟（`simulation_model=synchronous_waves`），
单测覆盖 lock ownership / state generation / 资源 admission。但：

- 2021-2026 全量年度滚动实际是**手工逐步编排** Cloud Run executions 完成的，scheduler 从未 live 提交过任何 execution；
- dry-run 的 active fanout 计数是 **candidate-year proxy**，PRD_01 已明确 Phase 2 前必须改为
  **Cloud Run execution 粒度**（覆盖 retry / `--no-tail-fill-single-task` 尾部 batch 把同一年拆成多个 execution 的场景）；
- GCS lease lock 与 state 的 generation 条件写目前只有内存模拟与单测语义，没有真实 GCS 实现。

未来常态化年度复跑（每年新增一年滚动）依赖本阶段闭环。

## 2. 目标

1. **真实 GCS lease lock**：generation-guarded（`if-generation-match`）获取/续约/释放；失去
   ownership 后停止提交（语义已有单测，落到真实 GCS 实现并补集成测试）。
2. **真实 GCS state**：generation 条件写 + mismatch 后重读重算；scheduler 中途被 kill 重启后从
   state 恢复，**不重复提交已成功/进行中的 execution**。
3. **Execution 粒度 fanout 跟踪**：以 Cloud Run execution 为最小跟踪单元（execution name +
   describe 状态 + GCS artifact 双确认），替换 candidate-year proxy；遵守既有运维口径——
   `gcloud --wait` 非零退出不等于失败，必须 execution describe + artifact 双确认；`maxRetries=0` 不变。
4. **共享资源池 live admission**：prepare/select/backtest 与 candidate 共享 `40 CPU / 160Gi`
   全局池，按 execution 实际占用计数，带 reconcile（state 与 executions API 对账，修正漂移）。
5. **Candidate-only live smoke**（PRD_01 §13 Phase 2 原定范围）：选 2 个年度、每年 2-3 个
   candidate unit，验证部分 batch、状态恢复、GCS artifact skip、超时二次确认；
   **不跑 select/register/predict**。

## 3. 非目标

- 不跑完整 2021-2026 live pipeline（Phase 3，owner 另批）。
- 不改 step 语义、active step catalog 步骤集合、research/ADS 生命周期。
- 不实现 Phase 4 优化（慢候选优先、动态并发、显式重试、BigQuery state table）。
- 不改 Cloud Run job spec / IAM / 镜像发布纪律（仍走 one-off Cloud Build 不可变 digest）。

## 4. 设计要点

### 4.1 Lease 与 state（真实 GCS）

- lease 对象与 state 对象分离；两者都用 generation precondition 写入，冲突即重读。
- lease 含 holder id + 到期时间，续约失败或读到他人 holder → 立即停止提交、退出非零；
  绝不出现双 scheduler 同时提交（防脑裂的唯一依据是 generation precondition，不依赖时钟）。
- state 记录每个 execution unit 的 {unit key, execution name, 提交时间, 终态, artifact 校验结果}；
  恢复路径只信 state + executions API + artifact 三方对账结果。

### 4.2 Execution 跟踪与失败判定

- 提交后轮询 execution describe（限频、指数退避），结合 GCS artifact 存在性双确认成败；
  `--wait` 退出码只作信号不作结论。
- 同一 unit 的 artifact 已存在且校验通过 → skip 提交（幂等重入）。
- 失败 unit 记入 state 标记 `failed`，不自动重试（Phase 4 范围）；smoke 中人工触发重跑验证幂等。

### 4.3 Smoke 设计

- 年度选 2021、2022；每年 2-3 个 candidate unit（含一个故意拆小的尾部 batch 场景）。
- 场景清单：① 正常提交至完成；② 中途 kill scheduler → 重启恢复不重复提交；③ 预置已完成
  artifact → skip；④ 模拟 lease 竞争（第二个 scheduler 实例）→ 后者拒绝提交；⑤ describe
  超时/瞬断 → 二次确认不误判。
- 全程 research 口径产物，资源占用受全局池约束。

## 5. 验收标准

| 项 | 要求 |
|---|---|
| lease | 双实例竞争测试：同一时刻最多一个 holder 能提交；失 owner 即停 |
| state 恢复 | kill-restart 测试：零重复提交、零漏提交 |
| fanout 口径 | 计数单元 = Cloud Run execution；尾部 batch 多 execution 场景计数正确 |
| 失败判定 | `--wait` 非零 + describe 成功 + artifact 在 → 判成功（既有运维口径回归用例） |
| smoke | §4.3 五个场景全过，记录 execution ids 与 artifact 路径 |
| 一致性 | live 计划与 dry-run resolved plan 同源同序（同一 plan 输入产生同一提交序列） |
| 测试 | 单测 + 可重复的集成 smoke 脚本入 repo |

## 6. 风险与控制

| 风险 | 控制 |
|---|---|
| executions API 限频/分页遗漏 | 退避轮询 + 按 execution name 精确 describe，不靠 list 全量 |
| lease 时钟漂移 | 失效判定只用 generation precondition + 服务端时间戳，不信本地时钟 |
| 资源池计数漂移 | reconcile loop 定期与 executions API 对账并修正 state |
| smoke 资源占用影响其他实验 | 全局池 admission + smoke 限定 2-3 unit/年 |

## 7. 实施顺序

1. lease/state 真实 GCS 实现 + 单测/集成测试。
2. execution 粒度跟踪 + 双确认判定 + 幂等 skip。
3. 资源池 live admission + reconcile。
4. candidate-only live smoke（§4.3 五场景）→ 记录证据 → 更新记忆。
5. Phase 3（完整 live 年度滚动）是否启动交 owner 决策。
