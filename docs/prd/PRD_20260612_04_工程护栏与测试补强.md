## 1. 背景

2026-06-12 两轮重构评估（六维度 + 新合并 7 PR 增量，逐条对抗核实）确认一批「属实且当下值得做」的防护缺口。共性：都是小成本、零语义变化、不需要云上回归的护栏，但缺了它们，已验证的语义会随研究迭代静默漂移。

## 2. 改动项

### 2.1 dividend 事件链路新鲜度护栏（本 PRD 唯一 medium 风险项）

**事实**：DECISION-20260612-03 要求后续所有实验显式 CA-on，但 `dwd_stock_dividend_event`（`sql/dwd/12`）是 2026-06-12 一次性快照：`sql/README.md` 执行顺序未登记 dwd/12 与 qa/14；全量重建编排（`orchestration/workflows/ashare_warehouse_full_rebuild.yaml`）止于 dwd/11；窗口刷新 `sql/incremental/01` 不含该表；ODS 日常采集 scope（`configs/ingestion/ods_current_scope_v0.yml` 14 endpoint）不含 dividend。价格 DWD 每日刷新而事件表冻结，CA-on 回测窗口末端一旦越过快照覆盖日，新分红事件**静默漏记且无任何 QA 报警**（当前所有已落地 run 窗口止于 2026-06-09，实害为零，但会随时间兑现）。

**改动**：

1. `sql/README.md`：执行顺序登记 `dwd/12_dwd_stock_dividend_event.sql` 与 `sql/qa/14_corporate_action_event_checks.sql`，并注明两点依赖关系——ledger 消费视图 `v_dwd_stock_dividend_event_ledger_consumable` 由 qa/14 创建（刻意设计，见 PRD_20260612_02）、qa/14 依赖 dwd/12 先建；说明当前为手工刷新（重跑 dwd/12 后必须重跑 qa/14）。
2. `sql/strategy1/qa/qa_corporate_action_ledger_outputs.sql` 增加 **staleness 断言**：当 `p_corporate_actions != 'none_v1'` 时，断言事件表数据可见上界 ≥ `p_predict_end`。可见上界从表内血缘字段推导（候选：`PARSE_DATE('%Y%m%d', MAX(source_partition_date_max))` 或 `DATE(MAX(ingested_at))`——实现时确认 dividend 端点 `partition_date` 语义后选定口径，并在 PR 说明；两者都不合适时再议加列）。注意该 QA 引用的表若启用 `require_partition_filter`，断言子查询同样要带分区过滤。
3. **完整恢复路径**（断言失败信息与 README 必须一致陈述）：dividend endpoint 当前**不在** ODS 日常采集 scope（`ods_current_scope_v0.yml` 14 endpoint 无 dividend），单纯重跑 dwd/12 + qa/14 不会推进可见上界。完整链路为：**① 补采 dividend ODS 分区**（owner 决策落地前走手工 ingestion run）→ **② 重跑 dwd/12** → **③ 重跑 qa/14**（重建 mismatch 表与 ledger 消费视图）→ ④ 重跑本 QA。过渡政策同时写入 README 与 KNOWN_CONSTRAINTS：owner 对采集 scope / 编排接入做出决策前，CA-on 实验的 `predict_end` 不得超过事件表可见上界；确需更晚窗口时先执行 ①-③。staleness 断言即此政策的机器守门。
4. `configs/strategy1/active_step_catalog.yml` **必改项**（非条件项）：该 step 的 `inputs` 补登记 `v_dwd_stock_dividend_event_ledger_consumable`——该 SQL **现在就在读**这个视图（`qa_corporate_action_ledger_outputs.sql:38`），catalog 当前只列了 4 张 backtest 表，血缘已漏报；catalog / 渲染 pytest 同步覆盖这个非 run-scoped DWD 输入。
5. **不实现、列为 owner 决策项**：dividend endpoint 进日常采集 scope；dwd/12+qa/14 接入全量重建 / 窗口刷新编排；ledger CA-on 路径 runtime preflight。本 PRD 只立 QA 报警线与恢复 runbook。

### 2.4 窗口 SQL 文本同构 guard

**事实**：DECISION-20260612-01 退役 shadow parity 工具后，`sql/incremental/01`（1,746 行，整段复制 `sql/dwd/01、02` + `sql/dws/01-07` 共 9 张表的 SELECT 主体）、`incremental/02`（与 `dwd/04` 同构）、`incremental/03`（与 `dws/08` 同构）的双实现**没有任何自动化一致性兜底**，仅剩不变量 QA 与人工 review。评估实测：incremental/01 L720-807 与 dws/02 L16-102 归一化后仅 3 处预期差异（窗口边界变量、`feature_version` 引用方式、收尾 WHERE）。

**改动**：

1. 新增 pytest（建议 `tests/warehouse/test_windowed_sql_isomorphism.py`）：对上述 11 对「incremental SELECT 主体 vs canonical SELECT 主体」做归一化文本比对。归一化规则：统一空白与大小写敏感保持、按显式映射表替换窗口边界变量名（如 `p_feature_read_start_date`↔`dws_start_date`）、登记**显式豁免行清单**（每条豁免注明原因，如 `p_feature_version AS feature_version`、收尾窗口 WHERE）。提取边界（每对的起止锚点）写死在测试数据中。
2. **当前代码必须直接通过**：若实现中发现真实语义漂移（豁免清单解释不了的差异），停下报 owner，不得用扩大豁免来让测试通过。
3. 负向自检：测试内置至少一个用例，对 canonical 文本做 seeded mutation 后断言 guard 能 fail（证明 guard 有效，防呆归一化把差异洗没）。
4. 维护成本声明写进测试 docstring：今后改 9 张表的任何特征逻辑，必须同步改两份 SQL 且更新本测试的锚点 / 豁免——这正是 guard 的目的。

### 2.5 `resolve_experiment` 四份拷贝合一

**事实**：同名函数四处定义且已漂移——`src/quant_ashare/strategy1/train_predict.py:94-126`（独有 `--manifest-resolved` 解析与 run_id 覆写）、`select_register_predict.py:133-147`、`prepare_matrix.py:121-134`（与前者仅错误消息一词之差）、`reporting.py:117-147`（独有 8 字段 replacements；`parse_args` 声明了 `--manifest-resolved` 但 resolve 路径**静默忽略**该参数）。
