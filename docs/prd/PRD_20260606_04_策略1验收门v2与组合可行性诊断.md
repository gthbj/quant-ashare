## 1. 背景

策略 1 已经完成多轮模型、交易和风控实验：

1. BQML historical reference `pv_fin_quality + 30/5% + biweekly + 5d` 在 `2024-01-02` 至 `2025-12-31` 表现较好，但该路径已被 owner 决定仅保留为 reference / audit。
2. Ledger v1 P1 extended fresh run 覆盖 `2024-01-02` 至 `2026-04-30` 后，full-period total_return 35.16%，但跑输中证1000 `000852.SH` 7.22pct。
3. 2026 final holdout `2026-01-05` 至 `2026-04-30` 中，策略跑输中证1000 12.81pct。
4. sklearn native、LightGBM binary、LightGBM regression 三波 Cloud Run Python Top5 全部 rejected。
5. 尾部风险 P1 只有轻度改善，P2 `skip_new_buys` 降收益且未改善最大回撤。
6. 风险特征入模 PRD 的 Phase B0 只读诊断输出 `primary_blocker=mixed_evidence`，不能自动证明“只是模型 / 特征差距”，也不能自动证明“只是接受门太严”。

因此，继续直接做第 4 波风险特征训练之前，需要先把验收门和组合可行性诊断冻结。否则下一轮实验会继续在以下问题上混淆：

1. 信号 RankIC 为正，但 TopN 组合收益 / 超额收益不合格。
2. 2025 test 与 2026 final holdout 已被多轮查看，继续复用会增加多重检验风险。
3. 当前回测仍未验证 10 万资金下整数手、实际持股数、现金占比和低价股偏移。
4. 当前 `30/5%` 组合能否实际稳定买满 30 只，尚未变成硬决策门。

本文定义下一版验收门 v2。

## 2. 目标

### 2.1 P0 目标

1. 把现有 reference run 明确判为：当前 top-30 long-only 等权 / 5% 上限组合实现不接受为生产 baseline。
2. 明确该拒绝只针对当前组合实现，不等同于判定整个信号家族失效。
3. 新增“组合实盘可行性门”，先检查 10 万资金下整数手、实际持股数、现金占比、买入失败和低价股偏移。
4. 将持股数候选固定为：

```text
target_holdings in [10, 20, 30, 40]
```

5. 明确不纳入 50、100、150 等更高持股数候选。
6. 新增 eligible universe benchmark，用实际可交易股票池构造参考基准，避免只与中证1000比较时误判股票池 / 市值暴露差异。
7. 冻结 accepted / needs_more_evidence / rejected 状态机，供后续风险特征、组合参数和月度滚动重训复用。

### 2.2 P1 目标

在 P0 诊断完成后，用同一模型 / 同一 prediction stream 做组合层实验：

1. `target_holdings=10`、`20`、`30`、`40`。
2. 单票权重上限默认仍为 5%。
3. 回测必须使用整数手 / 100 股 lot 约束。
4. 输出每个持股数的实际持仓、现金占比、跳单、成交和收益 / 风险差异。

P1 不改变模型训练、不改变特征、不改变标签，不与风险特征入模混跑。

### 2.3 P2 目标

只有 P1 发现某个组合实现具备可交易性且接近验收门，才继续：

1. 风险特征入模 Phase B/C。
2. 新模型族或目标函数。
3. 月度滚动重训。
4. 风险评分 / 组合优化。

## 6. 验收状态机 v2

### 6.1 状态枚举

| 状态 | 含义 |
|---|---|
| `accepted` | 可登记为当前 Cloud Run Python production baseline |
| `needs_more_evidence` | 没有命中硬拒绝，但证据不足、样本不足或实现可行性未验证 |
| `rejected` | 当前候选 / 组合实现明确不可接受 |
| `failed` | 执行失败或 artifact / QA 缺失，不能用于模型结论 |

状态机优先级：

1. 执行失败、artifact 缺失、QA 失败：`failed`。
2. 命中 hard reject：`rejected`。
3. 未命中 hard reject 但缺必要证据：`needs_more_evidence`。
4. 满足全部 accepted 条件：`accepted`。
5. 未覆盖状态：`rejected`，并写 `unmatched_acceptance_state`。

### 6.2 Accepted 条件

候选登记为 `accepted` 必须同时满足：

| 领域 | 字段 | 门槛 |
|---|---|---|
| full-period | `full_period_excess_return_vs_000852` | `> +3%` |
| full-period | `full_period_information_ratio` | `> 0.25` |
| full-period | `full_period_max_drawdown` | `>= -18%` |
| full-period | `full_period_relative_max_drawdown_vs_000852` | `>= -18%` |
| final_holdout | `final_holdout_excess_return_vs_000852` | `> -5%` |
| signal | `valid_rank_ic_mean` | `> 0` |
| signal | `test_rank_ic_mean` | `> 0` |
| signal | `valid_rank_ic_t_stat` | `> 1.0` |
| signal | `test_rank_ic_t_stat` | `> 1.0` |
| top bucket | `valid_top_minus_bottom_fwd_ret_mean` | `> 0` |
| top bucket | `test_top_minus_bottom_fwd_ret_mean` | `>= 0` |
| implementation | `implementation_gate_status` | `passed` |
| QA | runner / diagnosis / acceptance QA | 全部通过 |

说明：

1. RankIC 为正但 top bucket 为负时，不可 accepted。
2. full-period 超额必须为正且留出 3pct 缓冲，避免把噪声正超额登记成 baseline。
3. final_holdout 允许小幅跑输，但不允许明显坏结果。
4. implementation gate 未通过时，即使收益数据好，也只能是 `needs_more_evidence` 或 `rejected`。

### 6.3 Hard reject 条件

任一条件触发即 `rejected`：

| 领域 | 条件 |
|---|---|
| 数据 | 发现未来泄露、PIT 错误、universe 幸存者偏差 |
| 执行 | 组合无法按整数手和现金约束可行执行 |
| full-period | `full_period_excess_return_vs_000852 <= -3%` |
| full-period | `full_period_information_ratio < 0` |
| full-period | `full_period_max_drawdown < -25%` |
| full-period | `full_period_relative_max_drawdown_vs_000852 < -25%` |
| final_holdout | `final_holdout_excess_return_vs_000852 <= -10%` |
| top bucket | `test_top_minus_bottom_fwd_ret_mean < 0` 且 `final_holdout_top_minus_bottom_fwd_ret_mean < 0` |
| signal | `test_rank_ic_mean < 0` 且 `final_holdout_rank_ic_mean < 0` |
| implementation | `implementation_gate_status='hard_failed'` |
| diagnosis | `signal_inverted`、`unusable_signal`、`sample_filter_risk high` 或等价失败状态 |

当前 reference run 因 full-period excess、information ratio、relative max drawdown、final_holdout excess 和 test top bucket 证据，在 v2 下为 `rejected`。

### 6.4 Needs-more-evidence 条件

未命中 hard reject，但任一条件成立时为 `needs_more_evidence`：

1. full-period 超额在 `[-3%, +3%]`。
2. final_holdout 交易日或调仓次数不足。
3. RankIC 为正，但 top bucket 不稳定。
4. implementation gate 缺少整数手 / 现金 / 实际持仓证据。
5. test 被复用超过 3 波，但没有新增 final holdout passed 证据。
6. eligible universe benchmark 未构建或无法解释与 `000852.SH` 的差异。

## 7. 组合实盘可行性门

### 7.1 资金与手数口径

P0/P1 组合可行性诊断必须使用：

| 参数 | 值 |
|---|---|
| 初始资金 | 100,000 CNY |
| 买入单位 | 100 股整数手 |
| 最小买入 | 1 手 |
| 买入价格 | Ledger v1 t+1 开盘价 + 买入滑点 |
| 卖出价格 | Ledger v1 t+1 开盘价 - 卖出滑点 |
| 成本 | 现有成本 profile |
| 现金 | 不允许负现金；现金不足按订单规则缩放 / 跳过 |

### 7.2 持股数候选

持股数候选固定为：

| `target_holdings` | 等权目标权重 | 单票权重上限 |
|---:|---:|---:|
| 10 | 10.00% | 5% |
| 20 | 5.00% | 5% |
| 30 | 3.33% | 5% |
| 40 | 2.50% | 5% |

解释：

1. `10/5%` 理论最多只能部署约 50% 资金，除非另行允许单票权重上限高于 5%。因此它主要用于低集中度 / 高现金对照，不应直接与满仓方案比较收益。
2. `20/5%` 是 5% 上限下的理论满仓边界。
3. `30/5%` 是当前历史参考组合。
4. `40/5%` 用于验证更分散组合能否降低尾部风险，同时仍兼顾 10 万资金整数手约束。
5. 不纳入 `50`。40 只已经会明显暴露 10 万资金下的手数和现金碎片问题；50 只会把单票等权降到 2%，更容易被 1 手最低买入额扭曲。

### 7.3 实现门槛

每个候选组合必须输出以下指标：

| 指标 | 通过门槛 |
|---|---:|
| `actual_holdings_median / target_holdings` | `>= 0.80` |
| `avg_cash_weight` | `<= 10%` |
| `max_cash_weight` | `<= 20%` |
| `skipped_buy_rate` | `<= 20%` |
| `negative_cash_days` | `= 0` |
| `max_single_weight_realized` | `<= 5.5%` |
| `low_price_tilt_from_lot_constraint` | 不得显著偏向低价股 |

Hard fail：

1. 中位实际持股数低于目标持股数的 60%。
2. 平均现金占比超过 20%。
3. 买入跳单率超过 35%。
4. 出现系统性负现金。
5. 整数手约束导致组合明显偏向低价股，且该偏移解释了主要收益。

## 8. Eligible Universe Benchmark

### 8.1 目的

当前策略股票池只含沪深主板，并带 ST、上市天数、停牌、一字板、流动性等约束。只和中证1000比较时，可能把股票池差异、市值暴露或可交易性差异误判成模型 alpha / beta。

因此必须新增 eligible universe benchmark，作为诊断基准。

### 8.2 构建规则

每个调仓日 `t` 构建：

1. 股票池：与策略 live prediction pool 一致。
2. 交易所：仅沪深主板。
3. 排除 ST。
4. 上市满现有最低交易日要求。
5. `amount_ma20_cny >= 50,000,000`，阈值可作为配置，但首版固定。
6. `close_hfq >= 3`，避免极低价股主导整数手诊断。
7. `can_buy_open=TRUE`，与 t+1 执行可买性对齐。
8. 权重：首版等权。
9. 调仓频率：与策略候选相同。
10. 成本：首版不计成本和计成本两版都输出，报告默认展示计成本版。

输出：

```text
eligible_universe_benchmark_nav.csv
eligible_universe_benchmark_summary.json
eligible_universe_benchmark_constituents.csv
```

### 8.3 使用规则

accepted 候选必须：

1. 跑赢中证1000 `000852.SH`。
2. 跑赢 eligible universe benchmark，或给出明确解释为什么该 benchmark 不适用。
3. 若跑赢 `000852.SH` 但跑输 eligible universe benchmark，只能是 `needs_more_evidence`。

## 9. 风格与暴露诊断

P0 必须补齐以下只读诊断：

| 诊断 | 输出 |
|---|---|
| beta / alpha | `beta_to_000852`、`alpha_regression_return`、`residual_vol` |
| 市值暴露 | 持仓市值分桶 vs live pool / eligible benchmark |
| 行业暴露 | 行业权重、active industry weight、top3 industry concentration |
| 流动性暴露 | 成交额、换手、可买 / 可卖失败率 |
| 波动暴露 | `vol_20d` / `drawdown_20d` 分布 |
| 近期收益暴露 | `ret_5d` / `ret_20d` 分布 |
| top bucket 质量 | Q1-Q5 / Q1-Q10 分桶收益、单调性、top-minus-bottom |

诊断触发：

1. `beta_to_000852 < 0.65` 或 `> 1.35`：`needs_more_evidence`，除非收益 / 回撤明显通过。
2. 单一行业权重超过 30%：`needs_more_evidence`。
3. Top3 行业权重超过 60%：`needs_more_evidence`。
4. 单一市值分桶权重超过 eligible benchmark + 15pct：`needs_more_evidence`。

以上是诊断门，不是首版 hard reject；但报告必须展示。
