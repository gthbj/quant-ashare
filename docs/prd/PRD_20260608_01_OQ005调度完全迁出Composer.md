# PRD：OQ-005 调度完全迁出 Cloud Composer

## 1. 背景

### 1.1 问题不是 DAG 次数，而是 Composer 常驻底座费

`data-aquarium.gcp_billing` 在 `2026-06-05`、`2026-06-06`、`2026-06-07` 三天的账单显示，Cloud Composer 费用主体稳定集中在：

- `Cloud Composer 3 standard milli DCU-hours (asia-east2)`

这三天的 Composer 毛费用分别约为：

- `2026-06-05`: `161.67 USD`
- `2026-06-06`: `161.72 USD`
- `2026-06-07`: `161.71 USD`

其中主费用项 `standard milli DCU-hours` 约为 `160.97 USD/天`。这说明当前成本主要来自 Composer 3 环境常驻底座，而不是某几个 DAG 的执行次数。

### 1.2 当前 Composer 实际只承担编排，不承担核心计算

当前项目中，Composer 主要负责：

- 定时触发 ODS ingestion
- 非交易日 gate / readiness gate / QA gate
- 串接 window refresh / full rebuild
- 状态写回、告警检查和少量分支控制

真正的计算仍在：

- `Cloud Run Jobs`: Tushare 兼容 API -> GCS Parquet ingestion
- `BigQuery SQL / Dataform`: ODS -> DIM/DWD/DWS/ADS 转换与 QA

因此，当前体系的高成本点是“常驻 Airflow 环境”，不是“编排复杂度”。

## 2. 目标

### 2.1 主目标

将 OQ-005 的调度与编排层完全迁出 Cloud Composer，并在 cutover 验收后删除 Composer 环境。

### 2.2 业务目标

迁移后必须保持以下业务语义不变：

- `ashare-ingest-current-scope` 仍是生产 ingestion 入口
- `daily_current` / `backfill` / `qa_only` / `full_rebuild` 语义保持不变
- 非交易日 gate、ODS readiness、window refresh、window QA 语义保持不变
- `pipeline_run` / `pipeline_task_status` / `upstream_pipeline_run_id` 等 metadata 契约保持不变
- 现有 BigQuery SQL、Dataform 生成物、Cloud Run Jobs、runbook 和 QA 表口径保持不变

### 2.3 成本目标

编排层成本降到当前 Composer 方案的 `5%` 以下。

以 `2026-06-05` 至 `2026-06-07` 的 Composer 毛费用快照估算：

- 当前 Composer 基线约 `161.7 USD/天`
- 约等于 `4,850 USD/月` 量级
- 迁移后编排层目标成本应低于 `243 USD/月`

本 PRD 的推荐方案预期远低于该门槛，目标是单月低双位数美元以内，不含本来就要支付的 BigQuery 与 Cloud Run 执行费。

## 3. 非目标

- 不重写现有 BigQuery SQL 业务逻辑
- 不在本阶段推动 Dataform 全面替代 canonical `sql/`
- 不改策略训练 / 回测 / acceptance gate 逻辑
- 不在本阶段改变 `ashare_pipeline_alert_checker` 的告警规则语义
- 不在本阶段新增更多 ODS endpoint 接入

## 4. 推荐方案

### 4.1 结论

推荐使用：

- `Cloud Scheduler`: 负责定时触发
- `Cloud Workflows`: 负责多步业务编排、重试、状态机和手工补跑入口
- `Cloud Run Jobs`: 继续负责 ingestion 执行
- `BigQuery SQL / Dataform`: 继续负责数仓转换与 QA
- `Cloud Scheduler + Cloud Workflows + Cloud Run`: 负责 alert checker 的定时入口、轻量编排和执行

不推荐把新的长期编排层继续放在 Composer，也不推荐优先自建 `Cloud Run orchestrator`。

### 4.2 为什么选 Workflows，不选自建 orchestrator

`Cloud Workflows` 更接近当前 Airflow/Composer 的职责边界：

- 原生支持多步状态机、分支、重试、等待、超时
- 可直接调用 BigQuery 和 Cloud Run API
- 无常驻底座费，空闲时不收费
- 维护成本低于自建 Python orchestrator

自建 `Cloud Run orchestrator` 只有在以下情况才应优先：

- 编排逻辑高度动态，Workflows YAML 表达明显吃力
- 需要大量复用复杂 Python 业务逻辑
- 需要平台侧不提供的自定义调度/状态机能力

当前 OQ-005 不满足这些条件。

### 4.3 单纯给 Composer 降配不是替代方案

把 Composer 环境降到更小规格，不改变“存在常驻 Airflow 环境就持续付底座费”这个事实。

因此：

- 降配 Composer 不是本 PRD 的备选长期方案
- 真正的降本动作必须是迁出编排层并删除 Composer 环境

## 5. 目标架构

```text
Cloud Scheduler
  -> Workflows: ashare_ods_ingestion_daily
      -> Cloud Run Job: ashare-ingest-current-scope
      -> BigQuery QA: 09_ods_daily_partition_readiness.sql
      -> Workflows: ashare_warehouse_window_refresh
          -> BigQuery SQL window refresh
          -> BigQuery QA suite

Cloud Scheduler
  -> Workflows: ashare_warehouse_window_refresh
     (manual/backfill/qa_only trigger path also uses Workflows API)

Cloud Scheduler
  -> Workflows: ashare_pipeline_alert_checker
      -> Cloud Run service: ashare-pipeline-control /v1/tasks/alert-check

Manual only
  -> Workflows: ashare_warehouse_full_rebuild
```

### 5.1 组件职责

| 组件 | 迁移后职责 |
|---|---|
| `Cloud Scheduler` | 定时入口，不做业务判断 |
| `Cloud Workflows` | 多步编排、参数传递、重试、状态机、分支控制 |
| `Cloud Run Jobs` | ingestion 执行 |
| `BigQuery` | DIM/DWD/DWS/ADS 转换、QA、metadata 写入 |
| `Cloud Workflows` alert checker | 单步告警轮询编排、调用控制层写 Cloud Logging heartbeat / alerts |

### 5.2 对现有 DAG 的映射

| 当前 Composer DAG | 迁移后目标 |
|---|---|
| `ashare_ods_ingestion_daily` | `Cloud Scheduler -> Workflows` |
| `ashare_warehouse_window_refresh` | `Workflows` |
| `ashare_warehouse_full_rebuild` | `Workflows`，仅手工触发 |
| `ashare_pipeline_alert_checker` | `Cloud Scheduler -> Workflows -> ashare-pipeline-control` |
| `airflow_monitoring` | 无迁移对象；删除 Composer 后自然消失 |
| `ashare_daily_pipeline_v0` | 不迁移；作为历史 legacy 关闭 |

## 6. 迁移后的流程设计

### 6.1 `ashare_ods_ingestion_daily` workflow

输入参数保持与当前 DAG 兼容，至少包括：

- `pipeline_mode`
- `pipeline_dry_run`
- `skip_ingestion`
- `warehouse_mode`
- `business_date`
- `date_from`
- `date_to`
- `force_non_trading_day_gate`
- `enable_ads_contract_init`

主流程：

1. 初始化并写 `pipeline_start_status`
2. 解析 scheduled run 的业务日
3. 执行非交易日 gate
4. 调用 `ashare-ingest-current-scope` Cloud Run Job
5. 执行 `sql/qa/09_ods_daily_partition_readiness.sql`
6. 按 `warehouse_mode` 决定是否同步调用下游 `ashare_warehouse_window_refresh` child workflow，并阻塞等待其终态
7. 写 `pipeline_finalize_status`

硬要求：

- Workflow 内每个业务步骤都必须显式写 `pipeline_task_status`，不能只写 start/finalize
- 对 Cloud Run Job、BigQuery SQL、gate、skip、warning、child workflow 调用都要落独立 task 状态
- 失败路径也必须显式写失败状态，不能依赖 Workflows 的顶层 execution failure 自动推断

生产 `daily_current` 路径的 ingestion -> refresh 耦合方式固定为：

- 同步 child workflow 调用
- 父 workflow 只有在 child workflow 成功后才进入 finalize success

这意味着 cutover 后：

- 生产 scheduled 路径不再依赖 `refresh-missing watchdog`
- refresh 缺失/失败会直接体现为父 workflow 失败
- `refresh-missing watchdog` 只在迁移过渡期保留，用于旧 Composer 路径观察；新生产路径验收后退役

### 6.2 `ashare_warehouse_window_refresh` workflow

保留当前 window refresh 的语义与顺序：

1. setup / 参数归一
2. 指数 DWD window refresh
3. 股票 DWD/DWS window refresh
4. market-state window refresh
5. 只读 QA / 窗口 QA
6. finalize status

要求：

- 仍保持串行执行，避免同窗口重复写
- 继续记录 `upstream_pipeline_run_id`
- 继续支持 `daily_current` / `backfill` / `qa_only`
- 不依赖 Workflows 原生“单实例串行”能力，因为 Workflows 没有 `max_active_runs=1` 等价约束

串行化硬要求：

- 所有会进入窗口 DML 的 execution，在执行前必须先获取显式分布式锁
- 推荐机制为复用现有 GCS lease lock 语义：原子获取、heartbeat、过期回收、显式 release
- `daily_current` scheduled run 与手工 `backfill` / `qa_only` 不得并发进入同一窗口写路径
- BigQuery transaction 冲突只能作为最后兜底，不能代替上层编排锁

锁粒度要求：

- `ashare_warehouse_window_refresh` 全 workflow 级串行，默认只允许一个生产 execution 进入窗口写路径
- 如后续要细化到窗口级锁，必须保证重叠业务日窗口不会并发写入

per-task 状态写回硬要求：

- 该 workflow 内每个 gate、每个 BigQuery step、每个 skip 分支都要显式写 `pipeline_task_status`
- 必须保留现有事实口径：`task_type`、success/failed/skipped/warning、错误摘要、以及 BigQuery/Cloud Run/Workflow 对应的可追踪 ID 与 URL

### 6.3 `ashare_warehouse_full_rebuild` workflow

只保留为手工维护入口，不做 schedule。

要求：

- 只有 owner / 指定维护者可触发
- 明确与 `daily_current` / `backfill` 隔离
- 复用现有 BigQuery SQL / metadata / QA 契约

### 6.4 `ashare_pipeline_alert_checker`

迁移目标：

- `Cloud Scheduler` 定时触发
- 调用单步 `Workflows` checker
- 再由 Workflow 调 `ashare-pipeline-control /v1/tasks/alert-check`

理由：

- 当前 checker 虽然是单步逻辑，但 `Workflows -> ashare-pipeline-control` 已有真实 smoke 成功证据
- `Cloud Scheduler -> authenticated Cloud Run` 在当前项目配置下存在 live 鉴权风险；用 Workflows 作为稳定的 Google API 触发面更稳
- 可保留现有 BigQuery 观测视图、Cloud Logging 结构和 alert policy，并与主编排层保持同一认证模型

已知 live 背景：

- `2026-06-08` 对 `Cloud Scheduler -> authenticated Cloud Run` 做真实验证时，请求持续在 Cloud Run 鉴权层返回 `403`
- 当前还没有把根因收敛到单一缺失权限、单一 audience 配置或单一 org policy；已尝试 canonical URL、多个 OIDC service account 和多组 IAM 绑定，问题仍复现
- 因此该直连路径当前不视为 production-ready；本 PRD 不再把它作为 alert checker 的目标方案

新路径 IAM / 认证前提：

- 触发 alert-check workflow 的 Scheduler caller service account 必须具备目标 workflow 的 `roles/workflows.invoker`
- Scheduler 调用的是 Workflows Executions API；认证对象变为 Google API，而不是直接向 Cloud Run 自行发 OIDC 请求
- 执行 alert-check workflow 的 Workflows runtime service account 必须继续具备 `ashare-pipeline-control` 的 `roles/run.invoker`
- Workflow 内部继续复用 phase 1 已 smoke 通过的 `Workflows -> authenticated Cloud Run` 调用方式

过渡期硬约束：

- `main` 上现有 [deploy_scheduler_jobs.sh](/Users/fisher/Desktop/git/quant-ashare/orchestration/workflows/deploy_scheduler_jobs.sh) 仍实现的是被放弃的 `Cloud Scheduler -> authenticated Cloud Run` 直连路径
- 在对应实现 PR 把它改写为调用 Workflows Executions API 之前，这个脚本视为 `superseded / do-not-run`
- cutover 期间不得再用该旧脚本重建直连 Cloud Run 的 scheduler job

Phase 1 保持现有语义：

- 仍按当前 lookback 和告警口径执行
- 不在迁移 PRD 中顺手改频率或规则

### 6.5 状态写回映射

Airflow/Composer 当前依赖 callback 自动写 task 状态；Workflows 没有这层免费能力。

因此迁移后的实现必须显式提供一个统一状态写回层，至少覆盖：

- `step_started`
- `step_succeeded`
- `step_failed`
- `step_skipped`
- `step_warning`

字段要求：

- 保留现有 `pipeline_task_status` 语义
- `task_type`
- `error_summary`
- `bigquery_job_id` / `bigquery_job_url`
- `cloud_run_execution_id` / `cloud_run_execution_url`
- `workflow_execution_id`
- `workflow_step_name`
- 对应控制台 / 日志 URL

原则：

- 如果某个步骤没有显式状态写回，该步骤就视为设计不合格
- `v_pipeline_failed_tasks`、`v_pipeline_qa_failures`、alert checker 所依赖的 task 级事实，不得因为迁移到 Workflows 而退化为只有 pipeline 级状态

### 6.6 BigQuery / Cloud Run 调用模式与 Workflows 限额

Workflows 调 BigQuery 和 Cloud Run 的实现模型，不得假设为“像 Airflow operator 一样阻塞执行完毕”。

实现要求：

- BigQuery 调用按“提交 job -> 捕获 job id -> 轮询终态 -> 写状态”实现
- Cloud Run Job 调用按“触发 execution -> 捕获 execution id -> 轮询终态 -> 写状态”实现
- 长任务必须显式设置 connector timeout / polling 参数，不能依赖默认等待配置

限额复核要求：

- Phase 1 必须复核 Workflows 的 execution duration、step count、payload 大小与 connector polling 开销
- `full_rebuild` 不得默认假设为一个单体 workflow 就能安全承载
- 如果 `full_rebuild` 接近 Workflows 限额，应改为父 workflow + 子 workflow 分段，或改为显式的分批驱动模式

## 7. 状态、幂等与告警要求

### 7.1 状态写回

迁移后继续使用现有：

- `ashare_meta.pipeline_run`
- `ashare_meta.pipeline_task_status`
- `upstream_pipeline_run_id`
- `triggered_by_dag_id` 或等价来源字段

要求：

- Workflow / Cloud Run 替代 Composer 后，不得丢失现有状态表语义
- 失败、skip、warning、success 都必须保持现有事实口径
- 这条要求下沉为实现硬约束：每个业务步骤必须显式写 task 状态，而不是只保留 pipeline 开始/结束状态

### 7.2 幂等

必须保留当前窗口 SQL / QA 的幂等前提：

- `daily_current` 可重复执行
- `backfill` 可重跑同窗口
- `qa_only` 不写业务表
- `full_rebuild` 必须显式手工触发

### 7.3 告警

迁移后告警体系要求：

- pipeline failure 告警保留
- task failure 告警保留
- ingestion failed 告警保留
- alert checker heartbeat 保留
- 生产 scheduled ingestion -> refresh 路径改为同步 child workflow 后，现有 `warehouse_refresh_missing` watchdog 不再作为长期保留项；其职责由父 workflow 对 child refresh 成功/失败的同步判定替代

删除 Composer 后，不再保留任何依赖 Composer 环境存活的健康信号。

## 8. 成本模型

### 8.1 当前 Composer 基线

当前 Composer 费用具有以下特征：

- 固定常驻成本高
- 与 DAG 数量弱相关
- 与业务计算量不匹配

### 8.2 目标方案成本结构

迁移后编排层费用主要由以下部分组成：

- `Cloud Scheduler`: 按 job 收费，不按执行次数收费
- `Cloud Workflows`: 按 step 收费，无常驻底座费
- `Cloud Run` alert checker: 按实际执行收费

BigQuery 与 Cloud Run ingestion 的执行费用属于原业务执行成本，不是此次需要优化掉的 Composer 底座费。

### 8.3 预期

在当前 OQ-005 调度规模下，预期编排层会从“数千美元/月量级”降到“低双位数美元/月以内”，显著低于 `5%` 目标门槛。

## 9. 迁移计划

### Phase 0：语义冻结

- 冻结现有 Composer DAG 参数与状态语义
- 冻结 ODS readiness、window refresh、QA、alert checker 的现有口径
- 明确哪些属于“编排层迁移”，哪些不在本阶段修改

### Phase 1：基础设施实现

- 新建 Workflows 定义
- 新建 Scheduler jobs
- 为 Workflows service account 配置 BigQuery / Cloud Run 所需 IAM
- 为 alert checker 的 Scheduler caller service account 配置 `roles/workflows.invoker`
- 为 alert checker 的 Workflows runtime service account 配置并保留 `ashare-pipeline-control` 的 `roles/run.invoker`
- 实现统一 `pipeline_task_status` 显式写回层
- 实现 `ashare_warehouse_window_refresh` 的显式分布式锁
- 复核 BigQuery / Cloud Run 轮询模式与 Workflows 限额，确认 `full_rebuild` 是否需要拆分

### Phase 2：手工 smoke

至少完成以下手工 smoke：

- `qa_only`
- `daily_current`
- `backfill` 小窗口
- 非交易日 skip
- alert checker heartbeat / alerts

要求：

- 结果与当前 Composer 事实语义一致
- 不得出现重复写、漏写或 metadata 断链

### Phase 3：生产 cutover

1. 暂停 Composer 的 scheduled 业务 DAG
2. 启用 Scheduler + Workflows 的 scheduled 入口
3. 验收：
   - 至少 `2` 个自然开市日 scheduled run
   - 至少 `1` 个自然非交易日 scheduled skip
4. 保留 Composer 环境作为短期 rollback 选项，直到上述验收完成

cutover 期间要求：

- 不允许新 Workflows scheduled run 与旧 Composer scheduled DAG 同时对同一生产窗口写入
- `warehouse_refresh_missing` 旧告警只用于迁移期观察，不得在新同步 child-workflow 路径上继续作为长期主告警机制

### Phase 4：删除 Composer

满足以下条件后删除 Composer 环境：

- cutover 验收完成
- 无阻塞告警缺口
- 无 metadata / QA 语义倒退
- rollback 窗口结束

## 10. 回滚方案

在 Composer 删除前，允许回滚到当前 Composer 入口。

回滚条件：

- Workflows scheduled run 连续失败
- 非交易日 / backfill / qa_only 语义不一致
- 状态表写回或告警链路出现不可接受缺口

回滚动作：

1. 暂停 Scheduler 新入口
2. 恢复 Composer scheduled DAG
3. 记录 cutover 失败原因并修复后再重试

删除 Composer 环境后，不再有快速回滚路径，因此必须先完成 Phase 3 验收。

## 11. 验收标准

### 11.1 功能

- `daily_current`、`backfill`、`qa_only`、`full_rebuild` 全部可执行
- 非交易日 gate 与 readiness gate 与当前一致
- `000001.SH` 指数链路、stock window、market-state window 均能在新编排下执行

### 11.2 数据

- BigQuery 业务表无重复写 / 漏写
- 现有 QA 全部通过
- metadata 与 lineage 字段保持可追溯

### 11.3 运维

- alert checker 脱离 Composer 后仍有 heartbeat
- 失败告警与缺刷新告警仍可工作
- 手工补跑与恢复路径仍可用

### 11.4 成本

- 删除 Composer 后，不再产生 `Cloud Composer 3 standard milli DCU-hours`
- 编排层月成本低于当前 Composer 方案的 `5%`

## 12. 风险

| 风险 | 说明 | 控制方式 |
|---|---|---|
| 重复写 | 两套编排同时写生产窗口 | cutover 前先停 Composer scheduled 业务 DAG |
| 状态断链 | `pipeline_run` / `pipeline_task_status` 语义丢失 | Phase 0 先冻结契约，Phase 2 逐项 smoke |
| IAM 不足 | Workflow 不能调 BigQuery / Cloud Run | 单独梳理 SA 权限并先做手工执行验证 |
| 逻辑漂移 | 迁移时顺手改业务口径 | 本 PRD 明确非目标，不改 SQL/QA 业务语义 |
| 告警缺口 | 删除 Composer 后丢失 checker heartbeat | 先完成 Composer-free checker 和 heartbeat 验证 |
| 串行失效 | Workflows 没有 `max_active_runs=1`，daily 与 backfill 可能并发写窗口 | 显式分布式锁 + cutover 期间禁止双入口同时写生产 |
| 长任务超限 | `full_rebuild` 或长轮询任务逼近 Workflows 限额 | Phase 1 先做限额复核，必要时拆成父子 workflow 或分批驱动 |
