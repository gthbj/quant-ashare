# PRD: 策略 1 验收门 v3 切换实施

> 状态：规划中；本 PRD 只定义从当前 `v1` 主写回门切换到 `v3` 的实施路径，不直接改动 live search、accepted 写回或历史 artifact。
> 范围声明：本文覆盖 `model_acceptance_contract_v3`、Cloud Run live search acceptance、read-only replay、QA 与切换顺序；不重定义策略模型、特征集、股票池、成本、Ledger 语义或月度滚动重训方案。
> 关联：`docs/prd/PRD_20260605_04_策略1CloudRunPython模型基线搜索.md`、`docs/prd/PRD_20260606_03_策略1风险特征入模与候选增强.md`、`configs/strategy1/model_acceptance_contract_v1.yml`、`.agent/memory/OPEN_QUESTIONS.md` OQ-010。

## 1. 背景

当前策略 1 在 Cloud Run live search 上的事实状态是：

1. 主写回门仍是 `model_acceptance_contract_v1.yml`。
2. 默认评估主 benchmark 已切到上证指数 `000001.SH`，但验收逻辑仍是 `v1` 指标体系。
3. `v2` 在仓库中只作为历史诊断路线存在，不再作为未来切门目标。
4. owner 已明确：后续讨论和实现应忽略 `v2`，直接从当前 `v1` 切到 `v3`。
5. `v3` 的门定义已在多轮 owner 讨论中收敛出明确口径，但仓库中还没有可执行的 `model_acceptance_contract_v3.yml`，也没有与之匹配的 acceptance 代码、artifact 字段和 QA。

因此，下一步不是继续讨论 `v2`，也不是直接把 manifest 的 `acceptance_contract_path` 改名，而是先把 `v3` 固化成可执行契约，再完成 replay 和 QA 后切主写回门。

## 2. 目标

### 2.1 P0 目标

1. 定义 `v3` 的唯一事实来源文件。
2. 将 owner 已确认的 `v3` 指标、公式、阈值和字段名写成可实现契约。
3. 明确当前 live search / registry / backtest summary / comparison artifact 缺失哪些 `v3` 字段。
4. 定义 `v1 -> v3` 的安全切换顺序：先 replay，再 QA，再切主写回。
5. 保证历史 `v1` artifact 与结论不被追溯改写。

### 2.2 P1 目标

1. 在 `v3` 切换完成后，让后续 Cloud Run Python search 直接使用 `v3` 判定 `accepted / rejected / needs_more_evidence`。
2. 为月度滚动重训和未来新模型族搜索保留统一的 `v3` acceptance contract 接口。

## 3. 非目标

1. 不启用或扩展 `v2`。
2. 不重写历史 `v1` 的 accepted / rejected artifact。
3. 不改变策略 1 当前固定实验口径：股票池、交易成本、Ledger 语义、训练/valid/test/final_holdout 切分。
4. 不在本 PRD 中直接实现 `v3` 代码、跑 Cloud Run 真实搜索或重训模型。
5. 不把 benchmark 切换、bool-feature 修复、特征工程和 `v3` 门实现混在同一个变更里。

## 5. v3 当前确认口径

本节只写 owner 已明确确认、并将在实现中冻结为 `v3` contract 的内容。

### 5.1 回测与 benchmark 口径

| 项目 | 口径 |
|---|---|
| 回测全周期 | `2024-01-02` 至 `2026-04-30` |
| 主 benchmark | 上证指数 `000001.SH` |
| 指数对比集合 | 上证50、沪深300、中证1000、上证指数、深证成指 |

### 5.2 复利口径

本项目中 `v3` 涉及的年化/日化/月化统一使用复利。

默认约定：

```text
年化：target_period_count = 252
月化：交易日口径默认 target_period_count = 21
日化：target_period_count = 1
```

说明：

1. `252` 和 `21` 是标准换算因子，不表示 A 股每个自然年或自然月固定有这些真实交易日。
2. 样本期 `return_period_count` 必须使用样本区间内实际有效收益期数。

复合年化收益率定义：

```text
compound_annualized_return = (ending_value / starting_value) ^ (252 / return_period_count) - 1
```

### 5.3 信号质量门

`v3` 保留当前 `v1` 中仍然有效的信号质量前置门：

| 条件 | 要求 |
|---|---|
| CV 状态 | `passed` |
| Valid signal | `stable` |
| Valid RankIC | `> 0` |
| Valid top-bottom | `> 0` |
| Test RankIC | `> 0` |
| Test top-bottom | `> 0` |
| Score orientation | `identity` 或 `reverse_probability` |
| 诊断 | 不能是 `signal_inverted`、`unusable_signal` 等硬拒绝诊断 |

### 5.4 收益与风险门

`v3` 当前确认只保留两个全周期绝对表现门：

| 条件 | 要求 |
|---|---|
| Sharpe | `>= 0.70` |
| Calmar Ratio | `> 1` |

其中：

```text
Calmar Ratio = 策略复合年化收益 / abs(策略最大回撤)
```

### 5.5 Final holdout 门

`v3` 当前确认只保留：

| 条件 | 要求 |
|---|---|
| Final holdout 交易日数 | `>= 40` |

### 5.6 五指数相对门

五个指数中，必须存在至少一个指数满足以下同指数条件：

```text
策略超额复合年化收益 > 0
AND
(
  Excess Calmar Ratio > 1
  OR 策略最大回撤同期超额 > 0
)
```

这里的“同指数条件”表示：

1. `策略超额复合年化收益`
2. `Excess Calmar Ratio`
3. `策略最大回撤同期超额`

必须全部相对于同一个指数计算，不能跨指数拼接通过。

#### 5.6.1 指标定义

```text
策略超额复合年化收益 = 策略复合年化收益 - 指数复合年化收益
```

```text
策略最大回撤同期超额 = 策略最大回撤 - 指数在该策略最大回撤窗口内的同期收益
```

```text
Excess Calmar Ratio = 策略超额复合年化收益 / abs(策略最大回撤同期超额)
```

#### 5.6.2 命名约定

实现层统一使用名称：

```text
strategy_max_drawdown_same_window_excess
```

中文报告和 PRD 中对应名称为：

```text
策略最大回撤同期超额
```

## 6. 切换策略

`v3` 切换必须分四步，不允许直接把 live manifest 从 `v1` 指到 `v3`。

### 6.1 Phase A：冻结 contract

新增：

```text
configs/strategy1/model_acceptance_contract_v3.yml
```

要求：

1. `v3` 的阈值、benchmark、窗口、公式字段名统一从该 contract 读取。
2. 不允许再把 `v3` 阈值散落在 Python 常量、QA SQL 默认值或 PRD 文案里。

### 6.2 Phase B：实现只读 replay

新增或扩展只读 replay 路径，输入历史已完成 search / Top5 / backtest，输出：

1. `v3` 判定结果
2. 五指数逐指数相对门明细
3. `Sharpe` / `Calmar` / `Final holdout days`
4. 通过指数列表或拒绝原因

要求：

1. replay 只读，不重训模型。
2. replay 不回写 historical `accepted` 状态。

### 6.3 Phase C：补 QA

新增 `v3` QA，至少覆盖：

1. contract version / hash 写入
2. `000001.SH` 主 benchmark 一致
3. 五指数相对门字段完整
4. `策略超额复合年化收益 > 0` 不能被 direct-pass 绕过
5. `Calmar` / `Excess Calmar` 公式口径正确
6. `Final holdout 交易日数 >= 40`

### 6.4 Phase D：切 live write-back

只有在 replay 和 QA 通过后，才允许：

1. 将 Cloud Run search manifest 的 `acceptance_contract_path` 从 `v1` 切到 `v3`
2. 将默认 contract path 从 `v1` 切到 `v3`
3. 将 live search accepted / rejected 的事实状态改由 `v3` 决定

## 7. 代码改动范围

实现阶段预计最少涉及：

1. `configs/strategy1/model_acceptance_contract_v3.yml`
2. `scripts/strategy1_cloudrun/acceptance.py`
3. `scripts/strategy1_cloudrun/orchestrate_sklearn_native_search.py`
4. `sql/ml/strategy1/19_qa_cloudrun_python_baseline_search_outputs.sql` 或新的 `v3` QA SQL
5. `configs/strategy1/cloudrun_python_*.yml`

必要时可新增独立 `v3` replay / QA 文件，但不应顺手改动无关训练、回测、报告逻辑。

## 8. 历史兼容与回滚

### 8.1 历史兼容

1. `v1` 历史 artifact 保持不变。
2. 已有 `accepted / rejected` 结论不追溯改写。
3. `v2` 只保留为历史诊断实现，不再作为未来切门目标。

### 8.2 回滚

若 `v3` 上线后发现 QA 或 live search 判定异常，回滚方式必须是：

1. 把 manifest / 默认 contract path 切回 `v1`
2. 保留已生成的 `v3` replay / QA artifact 作为审计证据
3. 不直接删除 `v3` contract 或代码

## 9. 验收标准

`v3` 只有同时满足以下条件，才算“可以开始用”：

1. `model_acceptance_contract_v3.yml` 已落地并成为唯一事实来源。
2. 历史已完成的正式搜索轮次已跑过 `v3` 只读 replay。
3. `v3` QA 全绿。
4. live search registry / comparison artifact 能稳定产出 `v3` 所需字段。
5. Cloud Run search manifest 已切到 `v3`，且首轮真实运行没有 contract/QA/字段缺失失败。

在上述 5 条完成前，`v3` 只视为：

```text
doc + replay gate
```

不是 production write-back gate。

## 10. 实施顺序建议

建议按以下顺序执行：

1. 新增 `PRD`（本文）
2. 落地 `model_acceptance_contract_v3.yml`
3. 实现 `v3` replay
4. 用已完成的 5 次正式搜索 replay `v3`
5. 新增并跑通 `v3` QA
6. 再把 live search 主写回门从 `v1` 切到 `v3`

禁止顺序：

```text
先改 live search 默认 contract path，再补 replay / QA
```

## 11. 风险

1. 当前 live artifact 还没有直接暴露 `v3` 全量字段，若实现时边跑边补字段，容易出现 comparison artifact、QA 和 accepted 状态不一致。
2. 五指数相对门比 `v1` 更复杂，若不先 replay 历史候选，很容易上线后才发现通过率和预期偏差过大。
3. `v3` 与 `v1` 并存期间，必须明确区分“历史审计结论”和“新主门结论”，避免同一 search 被两套门混读。
