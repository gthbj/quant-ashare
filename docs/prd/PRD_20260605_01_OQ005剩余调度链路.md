## 1. 背景

OQ-005 已完成每日生产采集的第一阶段：

1. `ashare-ingest-current-scope` Cloud Run Job 是当前生产入口。
2. 该 Job 通过 Direct VPC egress + Cloud NAT 固定出口访问 Tushare 兼容 API。
3. 当前 14 张已消费 ODS endpoint 已完成 `2026-05-20` 至 `2026-06-04` 生产采集和 readiness 验收。
4. Cloud Composer 环境 `ashare-composer` 已部署 `ashare_daily_pipeline_v0`。
5. 当前每日主链为 `setup -> ingestion -> ods_daily_partition_readiness -> finish`。
6. Airflow Variables 当前为：

```text
ashare_pipeline_dry_run=false
ashare_enable_full_refresh=false
ashare_warehouse_mode=daily_current
ashare_transform_backend=bq_sql
ashare_enable_ads_contract_init=false
```

剩余工作是把每日采集后的 ODS 数据纳入完整生产转换和验收链路，使 DAG 可以从 ODS readiness 开始，按依赖完成 DIM/DWD/DWS/ADS、metadata、QA、策略产物和最终状态记录。

## 2. 目标

1. 将 ODS readiness 之后的生产链路补齐为可调度、可补跑、可追溯的 ODS→ADS DAG。
2. Phase 2.2 增量化完成后，每日默认链路只处理最新业务日及必要影响窗口，不默认扫描 2019+ 全历史。
3. 完整重建、历史补跑和 schema 全量检查通过显式参数或变量进入。
4. ODS→DIM/DWD/DWS/ADS 转换由 Dataform / BigQuery Studio pipeline 管理；在 Dataform definitions 完成前，允许使用 Composer + BigQuery SQL 兼容执行器作为过渡路径。
5. ADS 表契约初始化与策略 runner 写入分离，避免每日调度覆盖已有实验和回测产物。
6. 每个阶段写入 `ashare_meta.pipeline_run` / `ashare_meta.pipeline_task_status`，并保留 Cloud Run execution id、BigQuery job id、Dataform invocation id。
7. DAG 结束前必须执行 metadata 与 QA 门禁；失败、跳过、补跑、人工豁免均有状态记录和告警摘要。

## 3. 当前链路与缺口

| 模块 | 当前状态 | 剩余要求 |
|---|---|---|
| ODS 采集 | 已部署 `ashare-ingest-current-scope`，每日 20:00 执行当前 14 个 endpoint | 保持单 execution 顺序采集，继续写 `ingestion_run` / `ingestion_partition_status` |
| ODS readiness | `sql/qa/09_ods_daily_partition_readiness.sql` 已接入每日主链 | 扩展为 ODS gate，记录空返回、财务滚动、业务日与分区语义 |
| ODS schema | `06` / `08` 作为维护检查存在 | 加入显式 full-refresh / backfill 分支，不进入每日默认主链 |
| DIM/DWD/DWS | 当前 SQL 已存在，Composer full refresh 分支可调用 | 转换为 Dataform actions 或 BigQuery SQL adapter，并明确刷新窗口 |
| ADS | `sql/ads/01_ads_strategy1_tables.sql` 已使用 `CREATE TABLE IF NOT EXISTS` 创建契约表 | 每日 DAG 不执行 ADS 契约初始化；该脚本只作为一次性初始化或手工补齐入口 |
| metadata | `sql/metadata/01` / `02` 已存在 | Dataform 后执行；字段说明缺失作为 QA 阻断 |
| QA | `01-05` 及策略 runner QA 已存在 | 按调度模式分层执行，失败写状态并阻断 finish success |
| 告警 | 尚未闭环 | 生成失败摘要，记录 task/job/execution/invocation id |
| 补跑 | 当前可通过 DAG conf 手工传 `business_date` | 支持单日、区间、只转下游、只跑 QA、full rebuild、backfill |

## 4. 生产 DAG 目标形态

### 4.1 DAG 名称

完整链路采用当前 DAG 演进路径：

```text
ashare_daily_pipeline_v0
```

当 Dataform repository 和增量转换稳定后，可新增：

```text
ashare_daily_pipeline_v1
```

`v0` 保持生产采集入口和 BigQuery SQL 兼容路径；`v1` 承接 Dataform-first 的正式 ODS→ADS 转换路径。两个 DAG 不同时写同一业务日期的同一生产输出，Composer 通过 `max_active_runs=1` 和 pipeline lock 控制互斥。

### 4.2 每日默认执行顺序

```text
start
  -> pipeline_start_status
  -> resolve_business_calendar
  -> setup
  -> ingestion.current_scope
  -> ods_gate.daily_partition_readiness
  -> transform_gate.resolve_refresh_window
  -> transform.dim
  -> transform.dwd
  -> transform.dws
  -> ads_contract_gate
  -> metadata
  -> qa.p0
  -> qa.oq004_index
  -> qa.finance
  -> qa.oq006_unit
  -> qa.strategy1_dws_ads
  -> optional_strategy_branch
  -> pipeline_finalize_status
  -> notify
  -> finish
```

每日默认链路不执行 2019+ 全历史 schema 检查，不执行完整 ODS 外部表可读性矩阵，不重跑所有策略实验，不覆盖 ADS runner 历史产物。

Phase 2.0 / Phase 2.1 使用现有 CTAS SQL 或等价 Dataform CTAS 时，DWD/DWS 转换仍是全量重建兼容路径。该路径作为 smoke、维护和 full rebuild 使用；进入每日生产转换前，必须在 Phase 2.2 完成增量 / 分区覆盖，或在 Phase 2.0 直接实现最小可用的分区覆盖写入。

### 4.3 显式维护分支

维护 / 补跑分支由 DAG conf 或 Airflow Variable 启用：

```text
ods_schema_full_check
warehouse_full_rebuild
warehouse_backfill
ads_contract_init
strategy_runner
report_render
notification_test
```

这些分支必须写入 `pipeline_task_status`，并在 run label 中标明 `manual`、`backfill`、`maintenance` 或 `production_daily`。

## 6. ODS Gate

ODS gate 位于采集之后、转换之前，负责把“采集成功”转成“ODS 可供下游消费”。

### 6.1 每日检查

每日默认执行：

1. `sql/qa/09_ods_daily_partition_readiness.sql`
2. 查询 `ashare_meta.ingestion_partition_status`，确认当前业务日涉及的 partition endpoint 为 `success` 或合理 `empty_return`
3. 对财务表确认近期滚动窗口内有状态记录，空返回不等于缺数
4. 对交易日表确认 `partition_date = trade_date`
5. 对维度快照确认最新 snapshot 可读

### 6.2 维护检查

以下检查在 `warehouse_mode=full_rebuild`、`backfill` 或 `qa_scope=full` 时执行：

1. `sql/qa/08_ods_external_readability_checks.sql`
2. `sql/qa/06_ods_parquet_schema_checks.sql`
3. ODS source URI 与 canonical GCS 路径一致性检查
4. 同分区精确重复检查
5. API 返回上限命中复核

### 6.3 阻断规则

| 问题 | 处理 |
|---|---|
| 外部表不可读 | 阻断转换 |
| 业务日行情分区缺失且当天为交易日 | 阻断转换 |
| 财务接口 empty-return 且处于可解释窗口 | 记录并继续 |
| schema mismatch | 阻断转换，进入 OQ-012 修复流程 |
| source URI 口径偏离 canonical 路径 | 阻断转换 |
| API 返回上限命中未拆细复查 | 阻断转换 |

## 7. ODS→DIM/DWD/DWS 转换

### 7.1 转换后端

生产支持两个后端：

| 后端 | 使用阶段 | 入口 |
|---|---|---|
| `bq_sql` | Dataform definitions 完成前的兼容路径 | Composer `BigQueryInsertJobOperator` 执行现有 `sql/` |
| `dataform` | 长期正式路径 | Composer 创建 Dataform compilation result + workflow invocation |

`bq_sql` 与 `dataform` 必须产出相同表名、字段名、字段说明、分区和聚簇口径。迁移期同一日期先双跑到临时 dataset 或 isolated compilation target 做差异对比，再切生产目标。

字段说明在迁移期以 `sql/metadata/01_core_table_column_descriptions.sql` 和 `sql/metadata/02_finance_table_column_descriptions.sql` 为生产覆盖来源。Dataform SQLX 内的 descriptions 必须与这两个 metadata 脚本保持一致；Phase 2.1 需要明确切换点：切换前由 metadata ALTER 脚本覆盖，切换后由 Dataform SQLX 或同一份字段说明 manifest 生成生产说明。

### 7.2 Dataform action tags

### 7.4 ADS 契约处理

ADS 表分三类处理：

| 类型 | 表 | 调度规则 |
|---|---|---|
| 契约表 | `ads_ml_training_panel_daily` 等 `sql/ads/01` 创建的表 | 只在初始化或 owner 明确要求时执行 |
| runner 输出表 | model registry、prediction、candidate、portfolio、order、backtest、monitor | 由策略 runner 按 `run_id/backtest_id` 写入 |
| 每日监控 / 生产信号表 | 后续 `ads_signal_monitor_daily`、默认策略 daily prediction | 在 OQ-010 默认参数确认后纳入可选策略分支 |

ADS 契约初始化脚本在进入生产调度前必须保持：

1. 使用 `CREATE TABLE IF NOT EXISTS` 或 Dataform declaration / operation 形式。
2. 不覆盖已有 `run_id` / `backtest_id` / model registry。
3. 每次执行只补缺失表或字段说明。
4. 初始化动作有单独 `ads_contract_init` task id 和手工确认参数。

## 8. 刷新窗口

### 8.1 模式

| 模式 | 用途 | 写入范围 |
|---|---|---|
| `daily_current` | Phase 2.2 起每日生产 | 最新业务日 + 受影响历史窗口 |
| `backfill` | 区间补跑 | `date_from` 至 `date_to` + lookback / label impact |
| `full_rebuild` | 维护重建 | 2019+ 最终写入范围；财务/事件按 2017+ 支撑范围 |
| `qa_only` | 只验收 | 不改生产表 |

Phase 2.0 / Phase 2.1 使用现有 CTAS SQL 时，`daily_current` 只能作为参数规划和 ODS gate smoke 名称使用；实际 DWD/DWS 写入等同 `full_rebuild`，必须记录为 `full_rebuild_compat` 或 `maintenance` run label。Phase 2.2 增量化完成后，`daily_current` 才成为默认每日写入模式。

### 8.2 影响窗口计算

每日转换不得使用固定 `lookback_start_date`。调度器需要从配置计算：

```text
read_start_date = write_start_date - max_feature_lookback_days - calendar_buffer_days
write_start_date = business_date - max(label_horizon_days, downstream_repair_window_days)
write_end_date = business_date
```

当前策略 1 至少纳入：

| 参数 | 来源 |
|---|---|
| `max_feature_lookback_days` | DWS 价格特征最大滚动窗口 |
| `label_horizon_days` | 策略 1 当前支持 5 / 10 / 20 交易日标签 |
| `calendar_buffer_days` | 非交易日和停牌补偿 |
| `finance_asof_window_days` | 财务 as-of 近端重算窗口 |
| `downstream_repair_window_days` | late data / 修正版数据的重算窗口 |

Dataform / BigQuery SQL 执行时必须显式传入这些窗口参数，禁止依赖 SQL 文件内手改默认值。

### 8.3 表级刷新口径

| 表层 | `daily_current` 口径 |
|---|---|
| DIM 日历/股票/指数 | 可每日重建全量维表，行数小且依赖最新快照 |
| DWD 行情 | Phase 2.2 起写业务日及影响窗口；Phase 2.0/2.1 CTAS 兼容路径为全量重建 |
| DWD 财务 | 读取近期公告/修正窗口，更新受影响报告期与 `_latest` |
| DWS 特征 | 读取 lookback 窗口，写业务日及 label impact 窗口 |
| DWS 标签 | 新价格到达后回填对应 horizon 的历史标签有效性 |
| ADS runner 输出 | 只按显式 `run_id/backtest_id` 写，不参与默认 warehouse refresh |

### 9.2 QA 结果记录

每个 QA task 至少记录：

```text
pipeline_run_id
task_id
task_type = qa
business_date
bigquery_job_id
status
started_at
finished_at
error_summary
```

QA 通过后才允许 `pipeline_finalize_status` 写入 `success`。任何阻断性 QA 失败时，DAG run 失败并发送告警。

### 9.3 串号检查

进入策略 runner 分支时必须追加：

1. `run_id/backtest_id` 唯一性检查。
2. 当前 run 产物行数完整性检查。
3. ADS 表按 `run_id/backtest_id` 统计，确认无其他实验串号。
4. report / diagnosis / factor attribution artifact URI 与当前 run_id 对齐。

## 10. 状态与可观测性

### 10.1 `pipeline_run`

DAG start 写 `running`，finish 写 terminal status：

```text
running -> success
running -> failed
running -> partial
running -> cancelled
```

`partial` 用于手工补跑时部分 branch 被显式跳过，不能作为每日生产成功状态。

### 10.2 `pipeline_task_status`

每个 task 写入一行或多行状态。最小字段沿用 `sql/meta/01_create_meta_tables.sql`，后续可扩展：

```text
business_date
date_from
date_to
run_label
warehouse_mode
transform_backend
dataform_tags
gcs_uri
row_count_before
row_count_after
warning_count
```

### 10.3 日志与链接

最终摘要必须能定位：

1. Airflow DAG run URL
2. Cloud Run execution URL
3. BigQuery job URL
4. Dataform workflow invocation URL
5. GCS raw prefix
6. GCS report / diagnosis artifact prefix
7. 失败 task 的脱敏错误摘要

日志、状态表和报告均不得写入 Tushare token、service account key 或代理认证信息。

## 13. 策略 Runner / Report 分支

策略分支在以下条件均满足后进入每日或周期调度：

1. OQ-010 默认参数已由 owner 确认。
2. Ledger v1 P1/P2 和 Cloud Run runner 必要验收已完成。
3. `run_id/backtest_id` 生成规则已固定。
4. ADS 写入严格按 `run_id/backtest_id/strategy_id` 隔离。
5. runner QA 通过。

策略分支支持三种模式：

| 模式 | 触发频率 | 产物 |
|---|---|---|
| `predict_only` | 每个交易日 | 当日 prediction / candidate / target / signal monitor |
| `monthly_retrain` | 每月首个配置交易日 | model registry / prediction stream |
| `backtest_report` | 手工或维护调度 | backtest summary / report / diagnosis / attribution |

策略分支失败不应回滚当天已通过的 ODS→DWS warehouse 输出；DAG terminal status 可标为 `partial`，并在告警中标出 warehouse success + strategy failed。

## 14. 权限

| Service Account | 权限范围 |
|---|---|
| `sa-ashare-ingestion` | 读 Secret Manager token，执行 Cloud Run Job，写 raw GCS，写 ingestion meta |
| `sa-ashare-composer` | 触发 Cloud Run / Dataform / BigQuery，写 pipeline meta，发告警 |
| `sa-ashare-dataform` | 读 ODS / meta，写 DIM/DWD/DWS/ADS，执行 Dataform assertions |
| `sa-ashare-runner` | 执行策略 runner，写 ADS runner 输出和 GCS artifact |

权限按最小范围授予。Dataform 执行使用自定义 service account；Composer 触发 Dataform workflow invocation 时使用可审计的 impersonation chain。

## 15. 实施阶段

### Phase 2.0: BigQuery SQL 兼容路径闭环

交付：

1. 在现有 `ashare_daily_pipeline_v0` 中补 `pipeline_run` / `pipeline_task_status` 写入。
2. 把 full refresh branch 的 BigQuery SQL task 改为显式 `warehouse_mode` 控制，并将现有 CTAS DWD/DWS 标记为 `full_rebuild_compat`。
3. ADS 契约初始化从每日链路剥离，改为手工 task。
4. 整理 `sql/meta/` 脚本编号，确保生产执行序号唯一。
5. 补告警摘要和 task URL。
6. 用 `skip_ingestion=true` 完成一次 ODS→DWS/ADS QA smoke。

验收：

1. `warehouse_mode=full_rebuild` 或 `full_rebuild_compat` 能跑通 ODS gate、转换、metadata、`01-05` QA。
2. `warehouse_mode=qa_only` 不改生产表。
3. ADS 历史 runner 产物未被覆盖。
4. `pipeline_run` 和 `pipeline_task_status` 有完整 terminal 状态。
5. 生产日调度未把 CTAS 全量重建伪装成增量 `daily_current`。

### Phase 2.1: Dataform definitions

交付：

1. 新增 Dataform repository / workspace / release config。
2. 迁移 DIM/DWD/DWS SQL 到 SQLX。
3. 添加 sources、dependencies、tags、assertions、table descriptions。
4. Composer 使用 Dataform operators 或 API 创建 workflow invocation。
5. Dataform 输出与 `bq_sql` 输出做差异对比。
6. 明确字段说明生产来源，迁移期继续由 metadata ALTER 脚本覆盖，切换后由 Dataform SQLX 或统一字段说明 manifest 生成。

验收：

1. Dataform compiled graph 覆盖 P0 DIM/DWD/DWS。
2. tags 可按 `dim_core`、`dwd_market`、`dwd_finance`、`strategy1_dws` 单独运行。
3. 同一业务日期 Dataform 输出与现有 SQL 输出差异为 0，或差异有书面解释。
4. `01-05` QA 通过。

### Phase 2.2: 增量 / 影响窗口

交付：

1. 新增刷新窗口配置。
2. 日频 DWD/DWS 支持按影响窗口写入。
3. 标签 horizon 到达后自动回填受影响历史样本。
4. 财务 recent window 更新对应报告期和 as-of 特征窗口。

验收：

1. 每日运行不扫描 2019+ 全历史。
2. 与 full rebuild 对比，影响窗口内主键、行数、核心指标一致。
3. `lookback_start_date` 由调度配置计算，不再固定写死。

### Phase 2.3: 策略产物纳入调度

交付：

1. 策略 runner Cloud Run / BigQuery runner branch。
2. daily prediction / monthly retrain / report render 三类模式。
3. runner QA 和 artifact QA。
4. ADS signal monitor 和 report URI 回写。

验收：

1. `predict_only` 只写当前策略 run，不影响历史实验。
2. `monthly_retrain` 有 model registry 状态和 fallback 规则。
3. `backtest_report` 产物 URI 与 `run_id/backtest_id` 对齐。
4. `10/12/14/16/17` 中适用 QA 通过。

## 16. OQ-005 关闭标准

OQ-005 可关闭的条件：

1. `ashare_daily_pipeline_v0` 或 `v1` 能从 ODS gate 运行到 final success，并写完整 pipeline 状态。
2. 当前 14 张 ODS 的每日采集、ODS readiness、转换、metadata、QA 全部在 GCP 上闭环。
3. ODS→DIM/DWD/DWS/ADS 转换使用 Dataform 或明确的 BigQuery SQL 生产 adapter，依赖和参数可审计。
4. ADS 契约初始化不覆盖 runner 历史产物。
5. Phase 2.2 增量化完成后，每日默认链路不扫描 2019+ 全历史；full rebuild / backfill 需要显式参数。
6. `01-05` QA 在生产 DAG 中通过；策略分支启用时，相应 runner QA 通过。
7. 失败告警包含 Cloud Run execution id、BigQuery job id、Dataform invocation id 和脱敏错误摘要。
8. 单日补跑、只重建下游、区间补跑、全量重建四类路径均有运行记录和验收记录。
9. `OPEN_QUESTIONS.md` 中 OQ-005 状态更新为 closed，并把新增 endpoint 接入改为常规流程。
