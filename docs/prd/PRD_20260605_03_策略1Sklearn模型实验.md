## 2. 核心决策

本文把 Cloud Run sklearn 后续工作从“复刻 BQML parity”切换为“建立 sklearn native baseline”。

含义：

1. BQML baseline 仍作为历史参考和对照展示。
2. sklearn native 实验不再以逐项贴近 BQML RankIC 为成功条件。
3. sklearn native baseline 必须通过自己的 valid/test 信号、回测、成本、稳定性和 QA 门槛。
4. 若最终 sklearn native baseline 被接受，后续月度滚动重训优先复用 Cloud Run sklearn backend。
5. 既有 BQML runner 保留为 reference / fallback，不删除。

本文不修改 `PRD_20260604_04_策略1CloudRun训练回测.md` 中的原始 parity gate；那是“执行环境迁移是否可声明为 BQML 等价替代”的门槛。本文新增的是另一条路径：当 parity 失败时，重新建立 sklearn native baseline。

## 3. 目标

### 3.1 P0 目标

1. 固定交易参数和数据口径，只比较 scikit-learn 模型 / 预处理 / 参数。
2. 利用已实现的 Cloud Run task fan-out，一次 prepare matrix 后并发训练 30-50 个候选。
3. 先用 valid 期信号质量筛选 shortlist，再只对 Top 5 候选跑完整预测、组合、回测、报告和诊断。
4. 输出候选排名、valid/test 信号诊断、Top 5 回测对比和 sklearn native baseline 建议。
5. 定义 sklearn native baseline 的接受 / 拒绝标准。
6. 保持所有产物可追溯到 `experiment_id`、`search_id`、`candidate_id`、`run_id`、`backtest_id`、matrix hash、model artifact 和 container image。

### 3.2 P1 目标

1. 若 P0 线性模型不够，扩展到 `SGDClassifier` 和 `HistGradientBoostingClassifier`。
2. 若 native baseline 通过，补正式 baseline 登记、报告归档和后续月度重训入口。
3. 后续再考虑 LightGBM / XGBoost / Optuna / SHAP 等外部库或解释工具。

## 4. 非目标

1. 不改变 DWD/DWS 特征定义。
2. 不改变首轮已选业务参数：`pv_fin_quality + 30/5% + biweekly + 5d`。
3. 不改变股票池：仅沪深主板，不含北交所、创业板、科创板。
4. 不改变成本口径：佣金万一免五、卖出印花税 5 bps、买 / 卖滑点各 5 bps。
5. 不改变 Ledger v1 交易执行语义。
6. 不引入 GPU、本地训练、深度学习、Ray、Dask 或 Spark。
7. 不用 2025 test 反复调参。test 只用于样本外验收和风险复核；跨模型族波次复用 test 必须遵守 §10.4。
8. 不让 30-50 个候选全部跑完整回测；完整回测只给 shortlist，控制成本。
9. 不把 BQML 模型对象、BQML `ML.WEIGHTS` 或 BQML preprocessing 作为 scikit-learn native 的实现依赖。

## 7. 第一轮候选矩阵

### 7.1 默认预处理

第一轮默认使用：

```text
preprocess_version = sklearn_median_winsor_zscore_v1
```

理由：

1. A 股横截面特征存在极值和缺失，median / winsor / zscore 对线性模型更稳。
2. 该口径是 scikit-learn native 建模选择，不再以贴近 BQML 自动预处理为目标。
3. `prepare_matrix` 必须只用 train split 计算 median、winsor 分位点、mean 和 std；valid/test/predict 只应用 train 统计量。

可选对照：

```text
preprocess_version = sklearn_mean_zscore_v1
```

该对照只用于确认预处理是否明显影响结果。若第一轮默认预处理已通过 native baseline 门槛，可不跑 mean/zscore 全量网格。

### 7.2 LogisticRegression 第一轮网格

P0 第一轮先把线性 logistic 模型跑透。

推荐候选数：36 个。

| 组 | 参数 | 数量 |
|---|---|---:|
| 无正则 | `penalty=None`, solver=`lbfgs`, `class_weight in [None, balanced]` | 2 |
| L2 | `C in [0.03, 0.1, 0.3, 1, 3, 10, 30]`, `class_weight in [None, balanced]` | 14 |
| ElasticNet | `C in [0.1, 0.3, 1, 3, 10]`, `l1_ratio in [0.15, 0.5]`, `class_weight in [None, balanced]` | 20 |

solver 规则：

| penalty | solver |
|---|---|
| `None` | `lbfgs` |
| `l2` | `saga` 或 `lbfgs`，manifest 必须显式记录 |
| `elasticnet` | `saga` |

实现要求：

1. 不把 BQML `L1_REG` / `L2_REG` 数值直接翻译成 sklearn 参数。
2. `C` 是 sklearn 原生正则强度倒数，必须按 sklearn 语义记录。
3. `class_weight='balanced'` 是 native 候选，不再被视为“贴近 BQML”的默认项。
4. 每个候选必须写 `model_family`、`solver`、`penalty`、`C`、`l1_ratio`、`class_weight`、`max_iter`、`random_state`。

### 7.3 第二轮候选：SGDClassifier

若 LogisticRegression 第一轮没有候选通过 native baseline 门槛，P1 增加 `SGDClassifier(loss='log_loss')`。

SGD 是第二个模型族波次。进入该波次前，必须按 §10.4 显式记录 2025 test 复用风险和 owner 批准；不得把 P0 LogisticRegression 失败后的继续搜索视为无成本重试。

推荐候选：

| 参数 | 值 |
|---|---|
| `loss` | `log_loss` |
| `penalty` | `l2`, `elasticnet` |
| `alpha` | `1e-6`, `3e-6`, `1e-5`, `3e-5`, `1e-4` |
| `l1_ratio` | `0.15`, `0.5`（仅 elasticnet） |
| `class_weight` | `None`, `balanced` |
| `average` | `True` |

SGD 候选不得伪装成 LogisticRegression。`model_family` 必须写为 `sgd_log_loss`。

### 7.4 第三轮候选：HistGradientBoostingClassifier

若线性模型信号弱或 RankIC 不稳，P1 允许小规模非线性对照：

HistGradientBoosting 是新的模型族波次。若它复用同一 2025 test 作为接受门，必须按 §10.4 记录 `test_reuse_wave_no`、批准依据和后续 holdout 约束。

| 参数 | 值 |
|---|---|
| 模型 | `sklearn.ensemble.HistGradientBoostingClassifier` |
| `max_iter` | 100, 200 |
| `max_leaf_nodes` | 15, 31, 63 |
| `learning_rate` | 0.03, 0.05, 0.10 |
| `l2_regularization` | 0, 0.01 |

限制：

1. 先跑小网格，不超过 18 个候选。
2. 若训练时间或内存明显高于线性模型，应单独记录成本。
3. 非线性模型通过后，因子贡献度解释需要单独扩展；不能沿用线性系数解释。

## 8. 训练和筛选流程

### 8.1 一次 prepare matrix

同一个 `search_id + preprocess_version` 下，`prepare_matrix` 只运行一次。

产物：

```text
gs://ashare-artifacts/models/strategy1/ml_pv_clf_v0/
  search_id=<search_id>/
  preprocess_version=<preprocess_version>/
  matrix_id=<matrix_id>/
    matrix_manifest.json
    feature_schema.json
    preprocess_stats.json
    work_units.json
    train_features.parquet
    train_labels.parquet
    valid_features.parquet
    valid_labels.parquet
    test_features.parquet
    test_labels.parquet
    predict_features.parquet
    predict_index.parquet
```

candidate task 不得重新计算预处理统计量，不得直接扫描 `ads_ml_training_panel_daily`。

### 8.2 并发训练全部候选

owner 未设置 `--candidate-parallelism` 或设置为 0 时：

```text
resolved_candidate_parallelism = candidate_count
```

例如第一轮 36 个候选就启动 36 个 candidate tasks。若 GCP quota 不足，runner 记录失败，owner 决定提高 quota 或显式限流；runner 不得静默降到 2、4 或其他隐藏默认值。

### 8.3 候选信号指标

每个候选至少计算：

| 指标 | split | 说明 |
|---|---|---|
| `oriented_rank_ic_mean` | valid/test | 日截面 Spearman RankIC 均值 |
| `oriented_rank_ic_icir` | valid/test | RankIC 均值 / 标准差 |
| `top_bucket_fwd_ret_mean` | valid/test | 高分 bucket 平均未来收益 |
| `top_minus_bottom_fwd_ret_mean` | valid/test | 高低分差 |
| `bucket_monotonicity` | valid/test | 分层单调性 |
| `topn_avg_fwd_xs_ret_5d` | valid/test | Top N 未来横截面超额收益 |
| `prediction_coverage` | valid/test | live-available 预测覆盖 |
| `score_orientation` | valid | `identity` / `reverse_probability` |
| `roc_auc` | valid/test | 分类辅助指标 |

候选 ranking 主依据只使用 valid。test 指标只做样本外风险复核，不能参与第一轮 shortlist 排名。

### 8.4 Shortlist 规则

第一轮 shortlist 固定取 Top 5。

排序规则：

```text
valid_oriented_rank_ic_mean
  -> valid_topn_avg_fwd_xs_ret_5d
  -> valid_top_minus_bottom_fwd_ret_mean
  -> valid_roc_auc
  -> lower_model_complexity
```

硬过滤：

1. valid `oriented_rank_ic_mean <= 0` 的候选不得进入 Top 5，除非所有候选都 <= 0；此时实验标记 `failed_no_positive_valid_signal`。
2. `prediction_coverage` 明显低于候选中位数 5 个百分点以上的候选不得进入 Top 5。
3. score orientation 证据缺失的候选不得进入 Top 5。
4. 训练未收敛但没有写清收敛状态的候选不得进入 Top 5。

## 10. sklearn native baseline 接受标准

### 10.1 必须通过

候选要被接受为 sklearn native baseline，必须同时满足：

1. valid `oriented_rank_ic_mean > 0`。
2. test `oriented_rank_ic_mean > 0`。
3. valid 和 test 的 `top_minus_bottom_fwd_ret_mean` 不能同时为负。
4. 2025 test-year 策略收益为正。
5. 2025 test-year 相对 `000852.SH` 的超额收益为正。
6. 2024-2025 全区间 Sharpe >= 0.70。
7. 2024-2025 全区间 max drawdown 不低于 -25%。
8. 年化换手和总经济成本不显著高于 BQML baseline；若更高，必须由收益 / 回撤改善解释。
9. `10_qa_runner_outputs.sql`、`12_qa_model_diagnosis_outputs.sql`、Cloud Run `16/17` QA 全部通过。
10. Python ledger vs SQL ledger 等价验收已通过或该候选只声明“模型候选”，不声明“可替代执行路径 baseline”。
11. `valid_signal_status` 不能为 `weak` 或 `failed`。

`valid_signal_status` 的 P0 默认判定：

| 状态 | 判定 |
|---|---|
| `failed` | valid `oriented_rank_ic_mean <= 0` |
| `weak` | valid RankIC 仅勉强为正（默认 `<= 0.01`），或 valid ICIR `<= 0`，或 valid topN / 高低分差两项均 `<= 0` |
| `stable` | valid RankIC、ICIR 和至少一项 valid 经济信号均为正，且不触发 `weak` |

若候选满足 test-year 收益、超额、Sharpe 和回撤门槛，但 valid 信号被标为 `weak`，不得标记为 `accepted`，只能标记为 `needs_more_evidence`。

### 10.2 BQML 对照

BQML baseline 继续展示以下对照：

| 对照 | 用途 |
|---|---|
| BQML valid RankIC | 历史 reference，不作为 sklearn native hard gate |
| BQML total_return / excess_return | 经济表现 reference |
| BQML max_drawdown / Sharpe | 风险表现 reference |
| BQML turnover / cost | 成本 reference |

sklearn native 候选低于 BQML valid RankIC 不自动失败；但如果 test-year 收益、回撤、成本或 QA 明显劣化，应拒绝。

### 10.3 推荐最终选择规则

若多个候选都通过硬门槛：

1. 优先选 valid RankIC 和 valid topN 收益更稳的候选。
2. 若 valid 表现接近，选 test 不劣化、回撤更低、成本更低的候选。
3. 若 LogisticRegression 与更复杂模型表现接近，优先选 LogisticRegression。
4. 不得因为 2025 test 收益最高而忽略 valid 信号弱、换手高或回撤大的问题。
5. 多个候选同时过门时，`valid_signal_status=stable` 是第一决胜条件；`needs_more_evidence` 候选不得压过 stable accepted 候选。

若没有候选通过硬门槛：

1. 不建立 sklearn native baseline。
2. 保留 BQML baseline 为当前正式 baseline。
3. 记录失败原因：`no_positive_test_signal`、`negative_test_excess_return`、`drawdown_too_high`、`cost_too_high`、`qa_failed` 等。
4. 进入 P1 模型族扩展或回到特征 / 标签 PRD，而不是继续扩大无边界网格。

### 10.4 跨模型族 test 复用控制

2025 test 是当前 sklearn native 接受门的唯一 test split。它可以用于 P0 LogisticRegression 第一波验收；但每多跑一个模型族波次并继续用同一 2025 test 做接受门，都会削弱样本外保证。

规则：

1. `test_reuse_wave_no=1` 固定为 P0 LogisticRegression 第一波，不需要额外批准。
2. 第二波及以后（例如 SGD、HistGradientBoosting、外部库）若继续使用 2025 test 作为接受门，必须先由 owner 显式批准，并在 manifest、candidate comparison 报告和 registry / summary JSON 中写入 `test_reuse_wave_no`、`test_reuse_approval_ref` 和批准原因。
3. 默认最多允许 3 个波次使用同一 2025 test：LogisticRegression、SGD、HistGradientBoosting。超过 3 个波次后，不能仅凭 2025 test 建立新 baseline，必须新增未参与选择的最终 holdout 证据。
4. 2026 数据落地后，最终被推荐的 sklearn native baseline 必须在未用于候选选择的 2026 holdout 上做生产前确认；若 2026 holdout 不通过，只能标记 `needs_more_evidence` 或 `rejected`。
5. 报告必须展示已经复用 test 的模型族波次数，避免把多波次搜索后的 test 结果解释成单次独立样本外检验。

## 16. 风险与控制

| 风险 | 控制 |
|---|---|
| 单波次 test 泄露 / 过拟合 | shortlist 只用 valid；test 只做验收 |
| 跨模型族重复使用同一 test | 记录 `test_reuse_wave_no`；第二波及以后必须 owner 批准；超过 3 个波次必须新增最终 holdout |
| Top 5 共享 test 多重比较 | accepted 必须 `valid_signal_status=stable`；valid 弱但 test 过门只能标 `needs_more_evidence` |
| 网格无限扩张 | P0 固定 36 个 logistic 候选，P1 才扩模型族 |
| 成本失控 | prepare matrix 一次，候选 task 小规格，Top 5 才回测 |
| Cloud Run quota 不足 | 显式失败并记录；owner 决定提升 quota 或限流 |
| 复杂模型难解释 | 线性模型优先；非线性通过后另补解释方案 |
| 和 BQML 结论混淆 | 报告分清 BQML reference、parity gate、native baseline gate |
| 回测语义漂移 | 必须通过 Python ledger vs SQL ledger 等价验收后才能声明执行路径替代 |
