## 1. 背景

策略 1 live-available 预测池修正后，正式 livepool run 已完成执行和诊断：

```text
source_run_id = s1_bqml_livepool_20260602_01
source_backtest_id = bt_s1_bqml_livepool_20260602_01
selected_model_id = s1_bqml_livepool_20260602_01__l1_1e_4_l2_1e_3
```

该 run 的模型诊断显示 `primary_diagnosis=signal_inverted`，且 valid / test RankIC 均为负。随后执行了不重训模型的 reverse-score shadow run：

```text
shadow_run_id = s1_bqml_livepool_revscore_20260603_01
shadow_backtest_id = bt_s1_bqml_livepool_revscore_20260603_01
shadow_score = 1.0 - source_score
```

事实对照：

| 指标 | source run | reverse-score shadow run |
|---|---:|---:|
| `total_return` | -0.9712 | 0.2787 |
| `annual_return` | -1.6286 | 0.1490 |
| `sharpe` | -2.5316 | 0.7258 |
| `max_drawdown` | -0.9769 | -0.2217 |
| `excess_return` | -1.2614 | -0.0114 |
| `valid_rank_ic.mean` | 约 -0.099 | 0.098666 |
| `test_rank_ic.mean` | 约 -0.056 | 0.055965 |
| `primary_diagnosis` | `signal_inverted` | `usable_signal` |

这些结果说明：当前 runner 不能无条件把 BigQuery ML `ML.PREDICT` 输出的正类概率当作“越高越好”的排序分数。需要在模型选择和预测阶段加入 score orientation 校准，并持久化方向口径。

## 2. 问题定义

当前正式 runner 的 score 语义混在一起：

1. `raw_score`：BigQuery ML 对 `label_top30_5d=1` 的预测概率。
2. `score`：下游 `05_build_candidates.sql` 实际用于 `ORDER BY score DESC` 的排序分。

现有 ADS 契约只有 `score` 字段，且描述为“模型原始分数或正类概率”。这导致三个问题：

- 当 raw 正类概率在 valid 期呈现稳定反向信号时，下游仍按 `score DESC` 买入 raw 高分股票。
- 模型选择阶段的 TopN / bucket 指标也沿用 raw 方向，可能选中“反向最稳定”的模型，却没有记录需要反向使用。
- 报告和诊断无法区分“原始模型概率”和“最终排序分”，后续调参和审计容易误读。

## 3. 目标

1. 明确字段契约：
   - `raw_score` = BigQuery ML 正类概率，即 `P(label_top30_5d=1)`。
   - `score` = 最终用于选股排序的方向校准分数。
   - `score_orientation` = `identity` 或 `reverse_probability`。
2. 在 valid 期为每个候选模型评估 raw 与 reversed 两种方向，并把方向选择写入 `ads_model_registry`。
3. 模型选择指标使用 oriented score，而不是无条件使用 raw score。
4. 预测阶段按 selected model 的 `score_orientation` 生成最终 `score`，下游继续按 `score DESC`。
5. QA、报告、诊断都能展示和验证 score orientation，保证结果可追溯。
6. 保持 test 只用于样本外验收，不用于选择方向。

## 4. 非目标

- 不把策略永久硬编码为反向分数。
- 不用 test 期结果决定 `score_orientation`。
- 不修改 `label_top30_5d` 语义。
- 不修改训练样本、valid/test prediction pool、特征列、模型族、成本参数或组合参数。
- 不保证本次修复后策略收益达到上线标准；本文只解决 score 方向契约和 runner 行为。
- 不引入新的外部数据源或调度系统。

## 5. 术语与字段契约

| 名称 | 类型 | 定义 | 用途 |
|---|---|---|---|
| `raw_score` | `FLOAT64` | `ML.PREDICT` 输出中 `label='1'` 的概率 | 模型原始输出审计，不直接保证排序方向 |
| `score_orientation` | `STRING` | `identity` / `reverse_probability` | 说明 raw score 如何转换为最终排序分 |
| `score` | `FLOAT64` | 最终排序分。`identity` 时等于 `raw_score`；`reverse_probability` 时等于 `1.0 - raw_score` | 下游候选池、组合、报告和诊断使用 |
| `raw_valid_rank_ic_mean` | `FLOAT64` | valid 期 raw score 与未来收益排序的平均 RankIC | 方向决策证据 |
| `oriented_valid_rank_ic_mean` | `FLOAT64` | valid 期最终 score 与未来收益排序的平均 RankIC | 模型选择主指标 |

### 5.1 ADS 预测表契约

P0 推荐对 `ashare_ads.ads_model_prediction_daily` 增加字段：

```sql
raw_score FLOAT64
score_orientation STRING
```

字段含义：

- `raw_score` 保留 BQML 正类概率。
- `score` 改为最终用于 `rank_raw` / `rank_pct` 的 oriented score。
- `score_orientation` 记录该行预测使用的方向。正常情况下同一 `run_id/model_id` 下所有行一致。

如果实现阶段因兼容性选择不改 ADS schema，则必须满足最低要求：

- `ads_model_registry.model_params_json` 或 `metrics_json` 持久记录 `score_orientation`；
- `render_report.py` 和 `diagnose_model_quality.py` 能展示 selected model 的方向；
- 实现 PR 明确说明为什么暂不新增 `raw_score`，以及如何追溯 raw probability。

### 5.2 Registry 契约

selected model 的 `model_params_json` 或 `metrics_json` 必须可解析以下字段：

```json
{
  "score_source": "positive_class_probability",
  "score_orientation": "identity",
  "raw_valid_rank_ic_mean": 0.012345,
  "reverse_valid_rank_ic_mean": -0.012345,
  "oriented_valid_rank_ic_mean": 0.012345,
  "raw_valid_top_minus_bottom": 0.001234,
  "reverse_valid_top_minus_bottom": -0.001234,
  "orientation_decision_reason": "raw_valid_rank_ic_non_negative",
  "orientation_decision_split": "valid"
}
```

`score_orientation` 只允许：

- `identity`：最终 `score = raw_score`。
- `reverse_probability`：最终 `score = 1.0 - raw_score`。

## 6. 方向选择规则

### 6.1 候选模型评估

`sql/ml/strategy1/03_select_model_and_register.sql` 对每个候选模型在 valid 期生成 raw prediction 后，必须同时计算两套评价：

```text
raw_score = P(label_top30_5d=1)
reverse_score = 1.0 - raw_score
```

两套评价都必须只在 valid 期 label-available evaluation subset 上计算，延续 `PRD_20260602_05_策略1预测池口径修正.md` 的口径：

```text
prediction row exists
AND label_valid_5d = TRUE
AND fwd_xs_ret_5d IS NOT NULL
AND label_top30_5d IS NOT NULL
```

必须计算：

- raw / reverse 的 daily RankIC mean、ICIR；
- raw / reverse 的 top-minus-bottom；
- raw / reverse 的 TopN 平均未来收益；
- raw / reverse 的 bucket monotonicity。

### 6.2 P0 方向判定

P0 使用 valid 期 RankIC + bucket lift 的保守规则：

```text
IF raw_valid_rank_ic_mean <= -0.03
   AND reverse_valid_rank_ic_mean >= 0.03
   AND reverse_valid_top_minus_bottom > raw_valid_top_minus_bottom
THEN score_orientation = 'reverse_probability'
ELSE score_orientation = 'identity'
```

说明：

- 阈值 `0.03` 用于避免在弱信号或近似随机信号上频繁翻转方向。
- test 期不得参与方向判定，只能用于验收报告。
- 若 raw 和 reverse 都弱，即 `ABS(raw_valid_rank_ic_mean) < 0.03` 且 `ABS(reverse_valid_rank_ic_mean) < 0.03`，默认 `identity`，并在 metrics 中标记 `orientation_confidence='low'`。
- 若 reverse 满足 RankIC 阈值但 bucket lift 不支持，应默认 `identity` 或标记 `manual_review_required`，不得静默反转。

### 6.3 模型选择指标

候选模型最终排序必须使用 oriented 指标：

```text
primary = oriented_valid_rank_ic_mean
secondary = oriented_valid_topn_fwd_ret_mean
tie_breaker = valid_roc_auc
```

`raw_valid_rank_ic_mean` 仍需保留在 metrics 中，但不能再作为默认选型主指标。

## 7. Runner 行为

### FR-SCORE-1 03 选型登记方向

`sql/ml/strategy1/03_select_model_and_register.sql` 必须：

1. 对候选 valid prediction 保留 raw score。
2. 为每个候选计算 `identity` 与 `reverse_probability` 两种方向的 valid 指标。
3. 按 §6.2 为每个候选决定 `score_orientation`。
4. 按 oriented 指标选择 selected model。
5. 将 selected model 的 direction 和证据写入 registry。
6. registry 中 candidate 行也应保留各自方向证据，便于复核选型。

### FR-SCORE-2 04 预测应用方向

`sql/ml/strategy1/04_predict_daily.sql` 必须：

1. 从 selected model registry 读取唯一 `score_orientation`。
2. 若 selected model 找不到方向，脚本失败，不允许默认继续跑。
3. `ML.PREDICT` 仍抽取 `label='1'` 概率作为 `raw_score`。
4. 按方向生成最终 `score`：

```sql
CASE score_orientation
  WHEN 'identity' THEN raw_score
  WHEN 'reverse_probability' THEN 1.0 - raw_score
END AS score
```

5. `rank_raw` / `rank_pct` 必须按最终 `score DESC` 计算。

### FR-SCORE-3 下游继续消费最终 score

`sql/ml/strategy1/05_build_candidates.sql` 及之后脚本继续按 `ads_model_prediction_daily.score DESC` 排序，不直接读取 raw score。

下游字段语义更新为：

- candidate / report / diagnosis 中的 `score` 均表示 oriented score；
- 如需要展示模型原始概率，应展示 `raw_score`，不能把 raw probability 与 final score 混称。

### FR-SCORE-4 报告和诊断展示方向

`scripts/strategy1/render_report.py` 和 `scripts/strategy1/diagnose_model_quality.py` 必须展示：

- `score_source`
- `score_orientation`
- `raw_valid_rank_ic_mean`
- `oriented_valid_rank_ic_mean`
- 若有 ADS `raw_score` 字段，报告预测样本摘要可同时显示 raw score 与 final score 分布。

诊断分类逻辑必须基于最终 `score`；同时建议输出 raw-vs-oriented 对照 artifact，避免后续误判。

## 8. QA 要求

### QA-SCORE-1 registry 方向字段完整

selected model 必须满足：

```text
score_source = positive_class_probability
score_orientation IN ('identity', 'reverse_probability')
oriented_valid_rank_ic_mean IS NOT NULL
orientation_decision_split = valid
```

同一 `run_id` 必须只有一个 selected model。

### QA-SCORE-2 prediction 分数转换正确

若 `ads_model_prediction_daily` 新增 `raw_score` 和 `score_orientation`，QA 必须断言：

```text
identity: ABS(score - raw_score) < 1e-12
reverse_probability: ABS(score - (1.0 - raw_score)) < 1e-12
```

并断言 `raw_score` 与 `score` 均在 `[0,1]`。

### QA-SCORE-3 rank 基于最终 score

每个 `predict_date` 内：

- `rank_raw=1` 唯一；
- `rank_raw` 与 `ORDER BY score DESC, sec_code` 一致；
- `rank_pct` 在 `[0,1]`；
- 不能按 `raw_score` 排名。

### QA-SCORE-4 诊断/报告状态完整

`ads_backtest_performance_summary.metrics_json` 或相关 artifact 必须能读取：

- `score_orientation`
- `model_diagnosis_status`
- `report_upload_status`
- report / diagnosis artifact manifest

## 10. 风险与约束

| 风险 | 控制方式 |
|---|---|
| valid 期方向选择过拟合 | 使用保守阈值；test 不参与选择；报告 raw/oriented/test 对照。 |
| 弱信号被误反转 | `ABS(RankIC)<0.03` 默认不反转，并标记低置信度。 |
| `score` 字段语义变更影响下游读者 | PRD、ADS 字段描述、README、报告同步说明 `score` 是 final oriented score。 |
| 不新增 `raw_score` 导致审计困难 | P0 推荐新增；若不新增，PR 必须说明替代追溯方式。 |
| 反向后仍跑输 benchmark | 不影响本文验收；本文只修方向契约，模型质量继续归入 OQ-010。 |
