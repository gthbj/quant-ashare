## 2. 问题定义

OQ-010 剩余项混合了两类变量：

- **组合参数变量**：调仓频率、持股数、单票权重上限。这类变量不改变模型分数，只改变如何把分数转成组合。
- **模型质量变量**：标签 horizon、特征集合、训练 / 选型口径。这类变量会影响模型本身，需要重新训练、选型和诊断。

如果一次实验同时改标签、特征、持仓数和调仓频率，将无法判断收益变化来自哪里，也容易用 2025 test 结果反复挑参数。因此第一轮必须分层执行：先做组合参数敏感性，再做标签和特征实验，并且每个实验都保留完整 run/backtest/报告/诊断/QA 产物。

## 3. 目标

1. 固化第一轮实验矩阵，减少后续实现时的任意调参。
2. 区分不重训模型的组合参数实验与需要重训的标签 / 特征实验。
3. 定义每个实验必须产出的 `run_id` / `backtest_id` / artifact / ADS metrics。
4. 定义比较指标和晋级规则，避免只看 2025 test 收益做选择。
5. 给后续实现 PR 提供 runner 参数化、实验 manifest、对比报告和 QA 验收标准。

## 4. 非目标

- 不在本文实现 SQL 或 Python 修改。
- 不引入本地训练；仍使用 BigQuery ML + SQL runner。
- 不引入 XGBoost、神经网络、AutoML 或外部模型服务。
- 不加入北交所、创业板、科创板。
- 不新增 P1 资金流、行业、事件或外部文本数据。
- 不改变成本 profile；成本口径继续使用 `PRD_20260602_02_OQ010交易成本口径.md`。
- 不用 2025 test 作为调参主依据。test 只做样本外验收。
- 不把第一轮最好结果直接定义为可上线策略；上线前仍需独立稳定性判断。

## 5. 实验原则

### 5.1 固定基线

所有实验必须和以下基线比较：

```text
baseline_experiment_id = oq010_base_oriented_weekly_h5_n5_w20_pv
baseline_run_id = s1_bqml_livepool_oriented_20260603_01
baseline_backtest_id = bt_s1_bqml_livepool_oriented_20260603_01
```

基线含义：

- horizon = 5d；
- 周频调仓；
- 持股数 = 5；
- 单票权重上限 = 20%；
- 特征集 = 价格量价 + 估值；
- score orientation 已启用；
- valid/test 使用 live-available 预测池。

`baseline_experiment_id` 是全局 canonical baseline id，只代表上述已跑通 oriented run。阶段 A/B/C/D 内部的 `*0` 行是各阶段对照行，必须使用自己的 `experiment_id`，并通过 `parent_experiment_id = oq010_base_oriented_weekly_h5_n5_w20_pv` 或上游晋级实验 id 追溯来源，不得复用 canonical baseline id。

### 5.2 先 valid，后 test

实验判断顺序：

1. valid 指标用于实验晋级。
2. test 指标用于样本外验收和风险复核。
3. 若某方案 valid 明显变差但 test 变好，不得直接采纳，只能标记为 `possible_overfit`。

### 5.3 单轮只比较有限矩阵

第一轮只允许三组实验：

1. 组合集中度：持股数 / 单票权重。
2. 调仓频率。
3. 标签 horizon 与财务特征。

不在第一轮同时扩展模型族、股票池和外部数据源。

### 5.4 全链路验收

每个进入比较表的实验都必须至少完成：

```text
01 -> 02 -> 03 -> 04 -> 05 -> 06 -> 07 -> 08 -> 09 -> render_report -> 10 -> diagnose_model_quality -> 12
```

如果组合参数实验选择复用基线模型和预测，也必须为实验生成独立 `run_id` / `backtest_id`，并保证 ADS 表、报告和诊断可独立追溯。

## 6. 实验矩阵

### 6.1 阶段 A: 组合集中度实验

目的：判断当前 5 只股票、单票 20% 是否过于集中，是否导致回撤、不可交易和个股损失贡献过高。

保持不变：

- horizon = 5d；
- 周频调仓；
- 特征集 = 价格量价 + 估值；
- 成本 profile 不变；
- score orientation 不变。

| experiment_id | parent_experiment_id | 持股数 | 单票权重上限 | 目标权重口径 |
|---|---|---:|---:|---|
| `oq010_a0_n5_w20` | `oq010_base_oriented_weekly_h5_n5_w20_pv` | 5 | 20% | 阶段 A 对照行，复现当前基线组合 |
| `oq010_a1_n10_w10` | `oq010_base_oriented_weekly_h5_n5_w20_pv` | 10 | 10% | 等权，最高 10% |
| `oq010_a2_n20_w05` | `oq010_base_oriented_weekly_h5_n5_w20_pv` | 20 | 5% | 等权，最高 5% |
| `oq010_a3_n30_w05` | `oq010_base_oriented_weekly_h5_n5_w20_pv` | 30 | 5% | 等权 3.33%，上限不触发 |

晋级规则：

- valid 期 `topn_fwd_ret_mean` 不低于基线；
- test 期最大回撤下降或个股亏损集中度下降；
- 换手和成本不显著恶化；
- 持仓数不足、不可交易 skip 或现金约束不明显恶化。

### 6.2 阶段 B: 调仓频率实验

目的：判断 5d 标签与周频调仓是否匹配，是否需要降低换手或延长持有期。

阶段 B 使用阶段 A 中 valid 表现最稳的持股数 / 权重组合；若阶段 A 无明显改进，则使用 `n10_w10` 作为保守分散默认候选。

阶段 B 的 `oq010_b0_weekly` 是阶段 B 的周频对照行，不是全局 baseline id。其 `parent_experiment_id` 必须写为阶段 A 的实际晋级实验 id；如果阶段 A 无明显改进并采用保守默认候选，则写 `oq010_a1_n10_w10`。

| experiment_id | parent_experiment_id | 调仓频率 | 调仓日定义 |
|---|---|---|---|
| `oq010_b0_weekly` | `selected_stage_a_experiment_id` | 周频 | SSE 每个 ISO 周最后一个开市日 |
| `oq010_b1_biweekly` | `selected_stage_a_experiment_id` | 双周 | 以 valid/test 首个调仓周为锚点，每 2 个 ISO 周最后一个开市日 |
| `oq010_b2_monthly` | `selected_stage_a_experiment_id` | 月频 | SSE 每月最后一个开市日 |

实现 PR 必须把当前 `05_build_candidates.sql` 中固定周频的 `rebalance_dates` 逻辑参数化为：

```sql
DECLARE p_rebalance_frequency STRING DEFAULT 'weekly';
```

允许值：

```text

```

### 6.3 阶段 C: 标签 horizon 实验

目的：判断 5d open-to-close 标签是否过短，是否和周频 / 双周 / 月频持有目标不匹配。

保持不变：

- 使用阶段 A/B 晋级后的组合参数；
- 调仓频率固定使用阶段 B 晋级频率，不按 horizon 重新指派；
- 特征集先保持价格量价 + 估值；
- 模型族继续 `LOGISTIC_REG`；
- train/valid/test 时间切分不变。

阶段 C 的目的先是隔离 label horizon 变量，因此所有 C 阶段实验的执行调仓频率都取阶段 B 的晋级频率。manifest / ADS metrics 中的 `rebalance_frequency` 必须写入阶段 B 晋级后的真实值（`weekly` / `biweekly` / `monthly` 之一），下表用 `selected_stage_b_frequency` 表示该占位含义。`horizon_natural_frequency` 只作为解释字段写入 manifest / 对比报告，不作为本阶段执行频率，也不覆盖阶段 B 的结论。

| experiment_id | parent_experiment_id | 训练标签 | 目标收益 | 执行调仓频率 | horizon_natural_frequency |
|---|---|---|---|---|---|
| `oq010_c0_h5` | `selected_stage_b_experiment_id` | `label_top30_5d` | `fwd_xs_ret_5d` | `selected_stage_b_frequency` | weekly |
| `oq010_c1_h10` | `selected_stage_b_experiment_id` | `label_top30_10d` | `fwd_xs_ret_10d` | `selected_stage_b_frequency` | biweekly |
| `oq010_c2_h20` | `selected_stage_b_experiment_id` | `label_top30_20d` | `fwd_xs_ret_20d` | `selected_stage_b_frequency` | monthly |

实现 PR 必须让 `01_build_training_panel.sql`、`02_train_bqml_logistic_candidates.sql`、`03_select_model_and_register.sql`、`04_predict_daily.sql`、诊断脚本和 QA 按 `p_horizon` 选择目标标签和收益列，而不是硬编码 5d。

如果现有 DWS/ADS 缺少 `label_top30_10d` 或 `label_top30_20d`，实现 PR 必须先补 DWS/ADS/QA；不能用 5d label 伪装 10d/20d 实验。

### 8.3 晋级规则

第一轮不设“上线通过”标准，只设“进入下一轮候选”标准。

组合参数实验进入下一轮需满足：

- valid `topn_fwd_ret_mean` 不低于基线；
- test `max_drawdown` 或亏损集中度明显改善；
- 经济成本 / 换手没有显著恶化；
- QA 全部通过。

标签 / 特征实验进入下一轮需满足：

- valid `rank_ic_mean`、`topn_fwd_ret_mean` 或 bucket lift 至少两项优于基线；
- test 未出现相反方向的大幅恶化；
- prediction/eval coverage 不显著下降；
- score orientation 决策仍来自 valid，test 不参与方向选择。

若多个方案相近，优先选择：

1. 更简单的特征集；
2. 更低换手；
3. 更低回撤；
4. 更稳定的 valid/test RankIC。

## 9. 执行顺序

建议第一轮按以下顺序执行：

1. **确认基线**：复核 `s1_bqml_livepool_oriented_20260603_01` 的 10/12 QA、报告和诊断 artifact。
2. **阶段 A**：跑持股数 / 权重上限实验。
3. **阶段 B**：用阶段 A 晋级参数跑调仓频率实验。
4. **阶段 C**：用阶段 A/B 晋级组合参数跑 label horizon 实验；调仓频率固定使用阶段 B 晋级频率，不随 horizon 硬绑改写。
5. **阶段 D**：用阶段 C 晋级 horizon 跑财务特征实验。
6. **生成对比报告**：输出中文对比结论和结构化 metrics。
7. **owner 决策**：确认下一轮默认参数或继续开第二轮 PRD。

执行 BigQuery runner 时必须遵守已知约束：同一 run 不并发训练，清理模型对象前确认无 RUNNING job。

## 10. 验收标准

本文 PRD 合并后，后续实现 PR 的验收标准为：

1. runner 支持本文定义的实验参数或有等价 manifest 驱动方式。
2. 每个实验有独立 `run_id` / `backtest_id`。
3. 每个实验完成 01-12 全链路或在 PR 中解释 portfolio-only 复用模型/预测的等价路径。
4. 每个实验的 report、diagnosis、10 QA、12 QA 均可复核。
5. 输出 `experiment_comparison.md/json/csv`。
6. `TODO.md` 和 `.agent/memory/OPEN_QUESTIONS.md` 更新 OQ-010 状态。
7. 不把 test 最优方案直接写成最终默认参数，除非 owner 明确确认。

## 11. 风险和边界

| 风险 | 处理 |
|---|---|
| 实验数过多导致 BigQuery 成本和执行时间上升 | 第一轮分阶段执行，每阶段只让晋级结果进入下一阶段。 |
| 2025 test 被反复用来挑参数 | valid 作为晋级依据，test 只验收；报告标记 `possible_overfit`。 |
| 财务特征缺失导致预测池收缩 | 不用 `has_fin_*` 做 WHERE 过滤；用掩码和 NULL 表达缺失。 |
| 标签 horizon 与调仓频率混在一起 | 阶段 C 固定使用阶段 B 晋级频率，先隔离 label horizon；`horizon_natural_frequency` 仅写入报告作解释。 |
| 重训模型带来随机或实现差异 | 每个实验记录 full run metadata；如为 portfolio-only 实验，优先复用同一模型/预测或在对比报告中说明重训差异。 |
| 参数变好但解释性差 | 对比报告必须同时展示诊断指标，不只展示收益。 |

## 12. Owner 需确认项

本文建议 owner 确认以下第一轮默认矩阵：

1. 持股数 / 权重：`5/20%`、`10/10%`、`20/5%`、`30/5%`。
2. 调仓频率：weekly、biweekly、monthly。
3. 标签 horizon：5d、10d、20d；阶段 C 默认固定沿用阶段 B 晋级调仓频率，不按 horizon 硬绑重设频率。
4. 财务特征第一轮只加入比率 / 新鲜度 / 可用性字段，不加入原始金额字段。
5. 第一轮仍使用 BigQuery ML `LOGISTIC_REG`，不切换模型族。

owner 确认后，下一步实现 PR 应先落地实验参数化和 manifest，再按阶段执行并提交对比报告。
