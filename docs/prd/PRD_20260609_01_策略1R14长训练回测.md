## 1. 背景

当前 Strategy1 Cloud Run Python 路线已跑出一个 v3 accepted 的风险特征 LightGBM regression 候选：

```text
risk_lgbm_r14_l63_lr002_n300_leaf800_ff07_bf10_l1_1_l2_01
```

该候选在原始实验中使用：

| 项 | 值 |
|---|---|
| model_family | `lightgbm_regression` |
| feature_set_id | `strategy1_pv_fin_risk_v0_20260606` |
| label_horizon | 5d |
| target | `target_return = fwd_xs_ret_5d` |
| 原训练窗口 | `2019-04-03 ~ 2023-12-31` |
| 原回测展示窗口 | `2024-01-02 ~ 2026-04-30` |

后续 portfolio-only 敏感性表明，同一 R14 prediction stream 下，`20` 只持仓、`7.5%` 单票权重上限在 `2024-01-02 ~ 2026-04-30` 窗口中相对均衡：收益高于原 `30/5%`，Sharpe 最高，回撤低于 `10/15%`。

owner 现在希望先回答一个更直接的问题：

```text
同一套 R14 训练方法，如果只用 2015-2019 训练，
在 2020-2022 这三年样本外回测表现如何？
```

本文定义该实验。

## 2. 目标

### 2.1 P0 目标

1. 固定 R14 模型族、参数、特征集合、标签 horizon 和训练方法，不重新搜索参数。
2. 使用名义训练窗口：

```text
train_nominal_start = 2015-04-01
train_nominal_end   = 2019-12-31
```

3. 先只跑三年样本外回测：

```text
backtest_start = 2020-01-02
backtest_end   = 2022-12-30
```

4. P0 回测必须同时跑 `10` 只和 `20` 只两个组合变体。
5. 输出每个组合变体的候选级绩效、五指数相对表现、MaxDD 窗口、持仓 / 成交 / NAV / report artifact。
6. 明确本轮 `2020-2022` 是 diagnostic smoke，不直接登记 production accepted baseline。

### 2.2 P1 目标

若 `2020-2022` 表现值得继续，再由 owner 决定是否追加后续窗口：

```text
optional_segment_start = 2023-01-03
optional_segment_end   = 2026-06-09
```

后续段只作为追加诊断，不自动执行。实际截止日必须以 DWD/DWS、指数 benchmark、交易日历和回测所需价格的完整可用日期为准；如果 `2026-06-09` 当日 EOD 数据或下一执行日价格尚未完整入库，则截到最近完整可回测交易日。

如果 owner 需要一个连续的正式 `2020 -> 2026` 结果，仍必须从 `2020-01-02` fresh-run 到目标截止日，不能把 P0 的 `2020-2022` 和后续 `2023-2026` 两个 fresh segment 拼接成连续回测。

## 3. 非目标

1. 不重新搜索 R14 参数。
2. 不使用 `2020-2022` 结果反向选择模型参数。
3. 不把 `2020-2022` 单独回测结果写入 production accepted registry。
4. 不把历史 BQML / `sql/ml/strategy1` runner 恢复为默认执行路径。
5. 不改变 ODS / DIM / DWD / DWS 的既有事实口径；若需要补 2015-2018，只通过明确的 backfill / rebuild 任务实现。
6. 不做分段回测拼接；当前 Cloud Run Python ledger P0 仍按 fresh-start 运行。

## 4. 固定 R14 方法

### 4.1 模型参数

本实验固定使用当前 R14 参数，不做网格搜索：

| `feature_fraction` | 0.7 |
| `bagging_fraction` | 1.0 |

### 4.2 特征、标签和股票池

| 项 | 值 |
|---|---|

| `feature_set_id` | `strategy1_pv_fin_risk_v0_20260606` |

### 4.3 组合参数

P0 回测组合必须同时覆盖最近 portfolio-only sensitivity 中最需要比较的两个集中度：

| 变体 | `target_holdings` | `max_single_weight` | `rebalance_frequency` | 用途 |
|---|---:|---:|---|---|
| `n10_w15` | 10 | 15% | `biweekly` | 高集中度 / 高收益潜力 / 高回撤压力诊断 |
| `n20_w075` | 20 | 7.5% | `biweekly` | 当前更均衡的主观察组合 |

原 R14 accepted 的 `30/5%` 可作为 P0 之后的 reference 对照，但不得阻塞 P0 的 `10` 只和 `20` 只双组合回测。

## 5. 数据窗口与泄漏控制

### 5.1 名义窗口

owner 指定的名义训练窗口固定为：

```text
2015-04-01 ~ 2019-12-31
```

P0 回测窗口固定为：

```text
2020-01-02 ~ 2022-12-30
```

### 5.2 5d 标签 embargo

因为 R14 训练目标是 `fwd_xs_ret_5d`，如果直接把 `trade_date=2019-12-31` 的样本放进训练，其标签会使用 2020 年初的未来收益。这会把 2020 回测期信息泄漏进训练。

因此 P0 默认使用无泄漏口径：

1. 训练样本的信号日仍从 `2015-04-01` 起。
2. 训练样本只保留 5d 标签完整落在 `2019-12-31` 及以前的行。
3. 产物必须同时记录：
   - `train_nominal_end = 2019-12-31`
   - `train_effective_last_signal_date`
   - `label_embargo_trading_days = 5`
4. 若 owner 明确要求把 `2019-12-31` 信号日也纳入训练，则只能标记为 `leakage_diagnostic`，不得和无泄漏样本外回测混作 production evidence。

### 5.3 2015 年特征 warm-up

当前价格 / 风险特征包含 `ret_60d`、`vol_60d`、`drawdown_20d`、`amount_ma20_cny` 等滚动窗口。`2015-04-01` 已避开 2015 年初 60 个交易日窗口不完整问题。

如果实现阶段发现 `2015-04-01` 仍有大量 `has_full_history_60d=FALSE`，则处理顺序是：

1. 先审计缺失原因和缺失比例。
2. 若是 DWD/DWS 未补 2015 前 lookback，优先补 2014Q4 / 2015Q1 的 lookback 构建输入。
3. 不得静默把缺失样本填 0 或当作完整历史。

## 6. 数据前置条件

当前 raw ODS 股票行情层已经具备 2015 起数据，但策略实际输入 DWD/DWS 当前只从 2019 起。因此本实验前必须补齐或新建 extended 策略输入层。

### 6.1 必须补建 / 复核的表

| 层 | 表 | 要求 |
|---|---|---|
| DWD | `dwd_stock_eod_price` | 支持 2015-2022 价格、复权、停牌、可交易标记 |
| DWD | `dwd_stock_eod_valuation` | 支持 2015-2022 估值、市值、换手 |
| DWS | `dws_stock_universe_daily` | 支持 2015-2022 universe / board / lifecycle / ST / 可交易 mask |
| DWS | `dws_stock_feature_daily_v0` | 支持 2015-2022 价格量价和风险基础特征 |
| DWS | `dws_stock_feature_fin_daily` | 支持 2015-2022 PIT 财务特征；允许早期财务稀疏，但必须显式记录 `has_fin_*` |
| DWS | `dws_market_state_daily` | 支持 2015-2022 `market_state_v0_20260606` 或明确兼容版本 |
| DWS | `dws_stock_sample_daily` | 支持 2015-2022 特征 + 5d 标签 |
| ADS | `ads_ml_training_panel_daily` | 冻结本实验 run scoped 训练面板 |

### 6.2 不改全局生产默认范围

补 2015-2018 不能静默改变现有 daily production 默认刷新范围。推荐方式：

1. 使用显式 backfill / full rebuild execution。
2. 使用清晰的 run label，例如 `strategy1_r14_long_train_backfill_2015_2022`。
3. 若需要新增 extended artifact 或 shadow 表，命名必须表明是研究实验，不覆盖当前 2019+ 生产事实口径。

## 7. 执行计划

### 7.1 P0-A: 只读覆盖审计

先输出一份覆盖审计，不训练、不回测：

1. raw ODS `daily / daily_basic / adj_factor / stk_limit / suspend_d` 在 `2015-01-01 ~ 2022-12-30` 的开市日覆盖。
2. DWD/DWS 目标表在 `2015-04-01 ~ 2022-12-30` 的日期覆盖。
3. `strategy1_pv_fin_risk_v0_20260606` 所需字段在 2015-2019 train 期的缺失率。
4. 5d 标签在 `2015-04-01 ~ 2019-12-31` 的有效样本数，以及 embargo 后的有效最后信号日。

### 7.2 P0-B: 补建 2015-2018 策略输入层

若审计确认 DWD/DWS 缺 2015-2018，则补建：

```text
build_start = 2015-01-01
sample_start = 2015-04-01
build_end = 2022-12-30
```

如需支持 60d warm-up，可把读取层向前扩到 2014Q4，但最终样本仍从 `2015-04-01` 开始。

### 7.3 P0-C: 固定 R14 重训

生成新的训练 run：

```text
run_id = s1_risk_lgbm_r14_train2015_2019_v20260609_01
candidate_id = risk_lgbm_r14_train2015_2019_l63_lr002_n300_leaf800_ff07_bf10_l1_1_l2_01
```

训练只使用 embargo 后的 2015-2019 样本。实现不得把 2020-2022 样本用于 fit、预处理统计、score orientation 或参数选择。

### 7.4 P0-D: 2020-2022 三年双组合回测

生成三年 diagnostic backtest，必须同时覆盖 `10` 只和 `20` 只：

```text
predict_start = 2020-01-02
predict_end = 2022-12-30
rebalance_frequency = biweekly
```

| 变体 | `backtest_id` | `target_holdings` | `max_single_weight` |
|---|---|---:|---:|
| `n10_w15` | `bt_s1_risk_lgbm_r14_train2015_2019_n10_w15_2020_2022_v20260609_01` | 10 | 0.15 |
| `n20_w075` | `bt_s1_risk_lgbm_r14_train2015_2019_n20_w075_2020_2022_v20260609_01` | 20 | 0.075 |

两条回测都从初始现金 fresh-start，不读取任何 2019 年末持仓。

### 7.5 P1: 2023-2026-06-09 可选追加诊断

如果 P0-D 结果值得继续，后续是否追加 `2023-01` 至 `2026-06-09` 由 owner 决策。若追加，则必须继续同时跑 `10` 只和 `20` 只两个组合变体：

```text
optional_segment_start = 2023-01-03
optional_segment_end = 2026-06-09
```

该追加段如果单独运行，也会从初始现金 fresh-start，因此只能作为“后续市场阶段诊断”，不能和 P0-D 的 `2020-2022` NAV / 持仓 / 现金 / pending sell 拼接成连续结果。

若最终需要正式连续结果，仍必须从 `2020-01-02` fresh-run 到目标截止日。当前 Cloud Run Python ledger P0 是 `fresh-start only`。

## 8. 验收与输出

### 8.1 P0-D 输出

P0-D 至少对 `n10_w15` 和 `n20_w075` 分别输出：

1. 候选级 summary：总收益、复合年化、年化波动、Sharpe、MaxDD、Calmar、换手、成本。
2. 五指数相对门明细：上证50、沪深300、中证1000、上证指数、深证成指。
3. MaxDD peak/trough 日期和对应持仓。
4. 月度 / 季度收益表。
5. 主要持仓、行业暴露、风格暴露。
6. 交易状态分布：`FILLED`、`FILLED_SCALED_CASH`、`BUY_SKIPPED_*`、`SELL_SKIPPED_*`、`PENDING_SELL_CARRY`。
7. 训练面板 audit：样本数、sec_code 数、缺失率、label embargo 后有效截止日。

### 8.2 状态判定

P0-D 只给 diagnostic status：

| 状态 | 条件 |
|---|---|
| `promising_diagnostic` | 2020-2022 Sharpe / Calmar / 五指数相对表现均有继续完整回测价值 |
| `mixed_diagnostic` | 收益或信号可用，但回撤、换手、成本或相对门有明显问题 |
| `failed_diagnostic` | 信号失效、亏损严重或无法通过基本 QA |

P0-D 不写 production accepted registry。是否进入 P1 由 owner 根据诊断结果决定。

### 8.3 QA

P0 必须通过：

1. Cloud Run backtest/report 基础 QA。
2. lot-aware ledger QA。
3. NAV 连续性、现金非负、持仓唯一、交易状态枚举 QA。
4. 训练数据泄漏 QA：确认训练 fit 未使用 2020-2022 的 feature rows、labels、预处理统计或 score orientation。
5. 分区过滤 QA：所有 BigQuery 读取必须带日期 / 分区范围，不能只按 `run_id` 或 `backtest_id` 过滤。

## 9. 风险与边界

| 风险 | 处理 |
|---|---|
| 2015-2018 DWD/DWS 当前不存在 | 先做覆盖审计，再显式 backfill / rebuild |
| 2019 年末 5d 标签泄漏到 2020 | 强制 label embargo，记录 effective last signal date |
| 早期财务特征稀疏 | 使用 `has_fin_*` 和缺失率审计，不静默填 0 |
| 2020-2022 表现被当作调参依据 | 本实验固定 R14，不搜索参数；P0-D 只做 diagnostic |
| 分段结果被误拼为连续回测 | 明确 `2023-2026-06-09` 追加段只是 diagnostic；正式连续结果必须从 2020 fresh-run |
| v3 gate 默认窗口不适配 2020-2022 | P0-D 使用 v3 指标定义和五指数相对表，但状态只标 diagnostic，不写 production accepted |
