# PRD：策略 1 Ledger 分红送转记账修复（corporate-action accounting）

> 状态：草案，待 Codex review。
> 范围声明：为 Python ledger 家族新增**默认关闭**的公司行为记账能力（送转调股数 + 现金分红入账）；
> v1/v2 既有语义逐字节不变；research-only 重跑与对照；不 promotion。
> 关联：PR #194（复权漏损量化，预登记判据触发）、DECISION-20260612-01（true5y baseline，PR #192）、
> DECISION_LOG 既有"未复权口径、持有期除权简化"接受约定（本 PRD 即其正式重开提案）、
> PRD_20260611_10 §6/§8（本修复排在其 Phase 2 之前）。

---

## 1. 背景

PR #194 用浮点恒等级别的对账（无成交日残差 p99 ≈ 1e-16）量化了 v1 ledger"未复权价 +
恒定股数"约定的 NAV 漏损：true5y baseline 修正后 CAGR `13.85% → 15.72%`（+1.86pp）、
contract Sharpe `0.6076 → 0.6894`、Calmar `0.3725 → 0.4275`，MaxDD peak 日期移动；
漏损主体为**分红型**（112 起、6.26pp、约 1.2pp/年），送转型在实际持仓中仅 2 起。
预登记判据（CAGR ≥ +1pp 或 Calmar ≥ +0.05）触发 → 按判据约定立本 PRD，
并排在 PRD_10 Phase 2 之前（避免 Phase 2 对比表因记账修正重锚）。

关键事实：修正后 contract Sharpe 距 v3 门（0.70）仅 0.011。但 #194 的 hfq 修正是
**税前、分红再投资**口径的上界；真实账户的现金分红要缴红利税且以现金形式滞留至下次
调仓。**本 PRD 不预设"修复后过门"的结论**——过不过门由 Phase C 重跑决定。

## 2. 设计

### 2.1 能力参数化而非新版本号（核心设计选择）

公司行为记账与组合构造（equal-weight v1 / topdown v2）是**正交维度**。不新开
`ledger_exec_v3`，而是新增 ledger 参数：

| 参数 | 取值 | 含义 |
|---|---|---|
| `corporate_actions` | `none_v1`（**默认**） | 现状：忽略分红送转（v1/v2 既有行为，逐字节不变） |
| | `cash_div_and_split_v1` | 送转调股数 + 现金分红（税后）入账 |

- 参数进 experiment / summary `metrics_json` / run 口径记录；
- **hash 纪律（吸取 PR #189 发现 1 教训）**：`corporate_actions` 仅在非默认值时进入
  `ledger_params_hash` payload——默认值下 payload 与现状逐字节一致，存量 parent 的
  resume 不受影响；
- 两种构造（v1 equal-weight、v2 topdown）同时获得该能力，QA 按参数断言。

### 2.2 记账语义（`cash_div_and_split_v1`）

事件源为 Phase A 的 DWD 事件表（见 §3），在**除权除息日（ex_date）开盘前**结算：

1. **送股/转增**：`shares *= (1 + 送股比例 + 转增比例)`。A 股送转对整数股按比例派发，
   10送5 → 500 股变 750 股；结果取整（`floor`），零股残余忽略（量级 < 1 股，记审计字段）。
   调整后股数继续满足"100 股整手"约束不作要求（送转后零股客观存在，卖出时沿用
   `allow_full_exit_odd_lot` / 部分卖出向下取整既有语义）。
2. **现金分红**：`cash += 登记日持仓股数 × 每股派息(税前) × (1 − 红利税率)`。
   现金计入当日可用现金，按既有现金语义滞留至下次调仓部署（不自动再投资——与 hfq
   口径的"再投资"差异即 Phase C 要量化解释的偏差之一）。
3. **红利税**：A 股个人红利税法定规则为**转让时**按持有期分档补缴（≤1月 20%、
   1月~1年 10%、>1年 0；国家税务总局股息红利差别化政策）。精确建模需要 tax-lot 级
   持仓状态（逐笔买入日期/股数、送转调整、卖出扣减、resume 持久化）——当前 ledger
   仅维护聚合股数，建 lot 子系统的工程量与其精度收益（相对 flat 口径约 ±0.1pp/年，
   而分红漏损总量约 1.2pp/年）不成比例。**主口径定为 `dividend_tax_mode=flat_10pct`**：
   分红入账时统一按 10% 计提（对应策略典型持有期 1月~1年 档），mode 进 summary 与
   run 口径记录。Phase C 报告必须附 0% / 20% 两端的敏感性界（即税口径不确定性的
   上下界）。tax-lot 精确化列为**非目标**（未来若需要另立 PRD）。
4. **配股/增发等其他公司行为**：不在本 PRD 范围（发生频率与漏损贡献见 #194 事件分解，
   占比可忽略；遗留为已知局限写入报告与 KNOWN_CONSTRAINTS）。

### 2.3 与既有语义的边界

- `corporate_actions=none_v1` 下记账输出（trade / position / NAV / ledger state /
  `ledger_params_hash`）与现状**逐字节一致**，以回归测试锁定（黄金 hash + 全套既有
  单测在默认值下不动）；不变量的精确范围与 summary additive 例外见 §4.3。
- resume：父子 run 的 `corporate_actions` 与 `dividend_tax_mode` 必须一致
  （validate + resume QA 校验）。flat 税口径下无需 lot 状态：送转调整后的**聚合股数**
  写入 ledger state 即事实持仓，既有 state schema（holdings_hash 等）不变即可支撑
  resume；CA-on parent 与 CA-off child（或反之）必须 fail-fast。
- 除权日 NAV 连续性：送转日 raw close 跳水被股数调整精确抵消（残差仅零股取整），
  现金分红日 NAV 增量 = 税后分红额——**不再有假悬崖**。

## 3. Phase A：DWD 分红送转事件表

1. 新表 `dwd_stock_dividend_event`（建议，命名从仓库惯例）：**消费键
   `(sec_code, ex_date)`，由 canonical 聚合保证唯一**。源数据同股同 ex_date 存在多条
   分配（BQ 实查 2021+ `div_proc='实施'` 22,028 行 vs 22,009 个 distinct 键：如年报 +
   一季报同日实施、送股与现金同日），canonical 化规则：过滤 `TRIM(div_proc)='实施'`
   后按 `(sec_code, ex_date)` 聚合 `cash_div_per_share_pretax = SUM(cash_div_tax)`、
   `bonus_ratio = SUM(stk_bo_rate)`、`conversion_ratio = SUM(stk_co_rate)`，并保留
   `source_event_count` / `source_end_dates` / `source_ann_dates` 审计字段。
   其余字段：`record_date`（股权登记日）、`ann_date`（最早公告日，PIT 审计用）。
   源：`ods_tushare_dividend`（OQ-012 已修复 schema contract 的 P2 扩展表；ledger
   不得直读 ODS，必须经本 DWD 表——分层纪律）。**多事件聚合的重复样例**（如
   `601966.SH@20250710`、`600188.SH@20230717` 送股+现金同日）必须进单测与 QA fixture。
2. PIT 口径：事件按 `ex_date` 消费（除权日是交易所事实，无前视）；`ann_date` 保留
   供审计。窗口覆盖 `2021-01-04 ~ 2026-06-09` 起步，向 2010+ 对齐既有回填范围。
3. **QA（含交叉校验硬门）**：`sql/qa/` 新增检查——键唯一、比例/金额范围、ex_date
   为开市日；**事件隐含的价格调整因子 vs `dwd_stock_eod_price` 的 hfq 因子跳变**
   逐事件对照（容差化），不一致行明细落表——这是事件数据质量的决定性校验，
   也是 Phase B 记账正确性的前置。

## 4. Phase B：ledger 实现

1. `LedgerParams` 增 `corporate_actions` / `dividend_tax_mode` 参数；run_ledger 主循环
   在每个交易日开盘结算前应用当日事件（先送转调股数、再按登记日股数派息；除权日
   恰逢调仓日时**先 CA 后交易**）。
2. **参数传播清单（实现必须逐项覆盖，缺一即 review 退回）**：
   `scripts.strategy1_cloudrun.config.Experiment` 与 manifest schema → `backtest_report`
   CLI（`--corporate-actions` / `--dividend-tax-mode`）→ Cloud Run command plan →
   `build_sql_params()` → `LedgerParams` → `ledger_params_hash`（仅非默认值入 payload）→
   `build_metrics_and_report_inputs`（SQL 参数 + summary `metrics_json`）→ catalog
   `required_params` → `qa_runner_outputs` / `qa_lot_aware_ledger_outputs`（兼容断言）→
   新 `qa_corporate_action_ledger_outputs` → resume 两套 QA。
3. **默认不变量的精确范围**：`corporate_actions=none_v1` 下，**trade / position / NAV /
   ledger state / `ledger_params_hash` 逐字节不变**（黄金回归锁定）；summary
   `metrics_json` 允许 additive 记录显式的 `corporate_actions='none_v1'` 与
   `dividend_tax_mode`（可审计性优先），存量行无该 key——所有 QA 断言一律用
   `COALESCE(JSON_VALUE(...), 'none_v1')` 口径，兼容新旧。
4. 单测覆盖：送转整数取整与零股审计、flat 税计提、同股同 ex_date 多事件聚合消费、
   登记日 vs 除权日股数口径、默认值回归（§4.3 范围黄金对照）、resume 参数一致性
   fail-fast、CA 与 BUY/SELL 同日交互（先 CA 后交易）。
5. 新 QA `qa_corporate_action_ledger_outputs`（登记 catalog）：
   - summary 记录 `corporate_actions` / `dividend_tax_mode` 与预期一致；
   - 送转日持仓股数变化 = 事件比例（按持仓 join 事件表复算）；
   - 分红日现金增量 = Σ登记日股数 × 税后每股派息（容差内）；
   - 除权日 NAV 无异常跳变（|当日收益 − 市场收益| 分布断言或逐事件复核）；
   - `corporate_actions=none_v1` 的 run 零 CA 审计行。
6. 运行手册补 CA 模式说明与参数表。

## 5. Phase C：research-only 重跑与对照

1. true5y continuous 以 `corporate_actions=cash_div_and_split_v1` 重跑（新 run/backtest
   id，研究 dataset，`--force-replace` 不触碰现役 baseline）。
2. **三方对照表**：原 true5y baseline / 本次 CA 重跑 / PR #194 的 hfq 修正估计。
   CA 重跑应落在前两者之间；与 hfq 估计的偏差按**可审计分解表**呈现（参照 #194
   对账先例）：`tax_effect`（flat 10% 计提）、`cash_not_reinvested_effect`（现金滞留
   vs hfq 隐含再投资）、`split_fractional_rounding_effect`（送转取整零股）、
   `same_ex_date_event_aggregation_effect`（多事件聚合）、`event_vs_adj_factor_residual`
   （事件口径 vs hfq 因子的容差内差异）、`unexplained_residual`。**验收门 =
   `unexplained_residual` 在容差内**，而非要求总差只由前两项解释。
3. v3 gates 重评：报告 contract Sharpe / Calmar 新值与门距。**预登记表述**：若 Sharpe
   过 0.70 单门，结论写"单门通过、Calmar 门未过，accepted 仍不成立"——不得因单门
   通过而触发任何 accepted/promotion 动作。
4. 结果交 owner 三个决策：① 是否以 CA 口径取代现役 baseline 数字（DECISION 条目）；
   ② DECISION_LOG 中"未复权口径"旧约定正式 superseded；③ PRD_10 Phase 2 与后续
   实验是否一律 CA-on（建议是，写入切换纪律）。

## 6. 验收标准

| 项 | 要求 |
|---|---|
| 默认不变 | `none_v1` 下 trade/position/NAV/state/hash 逐字节一致（§4.3 范围，回归 + 黄金 hash）；summary 仅 additive 记录 |
| hash 纪律 | 新参数仅非默认值进 hash payload；存量 parent resume 回归测试通过 |
| 数据质量 | Phase A 事件表 QA 全过，hfq 因子交叉校验不一致明细 = 0（或逐条解释） |
| 记账正确 | Phase B 单测 + QA 全过；除权日无假悬崖 |
| 对照可解释 | Phase C 偏差分解表完整产出，`unexplained_residual` 在容差内 |
| 口径纪律 | 全部 research-only；单门通过不触发 accepted/promotion；baseline 取代由 owner 决策 |

## 7. 风险与控制

| 风险 | 控制 |
|---|---|
| 事件数据错漏（ODS dividend 表质量） | Phase A hfq 因子交叉校验硬门 + 不一致明细落表 |
| 红利税建模偏差 | flat_10pct 主口径 + Phase C 附 0%/20% 敏感性界；tax-lot 精确化显式列为非目标 |
| 默认行为被无意改变 | 逐字节回归测试 + hash payload 条件化（PR #189 先例） |
| 除权日与调仓日同日的次序错误 | 固定"先 CA 后交易"并单测锁定 |
| 结果被解读为过门即 accepted | §5.3 预登记表述；KNOWN_CONSTRAINTS acceptance 约束原样 |
| paper 工具族口径分裂（#190/#181 等 raw 口径脚本） | 报告声明：paper 工具维持与其对照对象同口径；CA 口径成为 baseline 后新 paper 工具随之对齐 |

## 8. 实施顺序

1. Phase A（DWD 事件表 + 交叉校验 QA）→ 2. Phase B（ledger + 单测 + QA）→
3. Phase C（true5y CA 重跑 + 三方对照）→ 4. owner 三项决策 → 5. 此后才进入
PRD_10 Phase 2（按其 §6 基线条款衔接 CA 口径）。
