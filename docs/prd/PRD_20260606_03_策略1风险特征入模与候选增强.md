# PRD: 策略 1 风险特征入模与候选增强

> 状态：草案，待 owner review。
> 范围声明：本文定义策略 1 在 Cloud Run Python 路线下，把个股尾部风险、市场状态和风险画像信息纳入训练特征、frozen matrix、候选排序和验收报告的下一轮 OQ-010 方案。本文不启用 P2 v0 `skip_new_buys` 作为默认交易规则，也不新增或扩展 BQML / `sql/ml/strategy1` SQL runner。
> 关联：`docs/prd/PRD_20260605_04_策略1CloudRunPython模型基线搜索.md`、`docs/prd/PRD_20260606_01_策略1尾部风险控制.md`、`docs/prd/PRD_20260605_02_策略1CloudRun轻量Task并发.md`、`docs/prd/PRD_20260604_04_策略1CloudRun训练回测.md`、`docs/prd/PRD_20260604_01_策略1LedgerV1交易执行语义.md`、`.agent/memory/OPEN_QUESTIONS.md` OQ-010。

## 1. 背景

策略 1 当前已经完成以下事实闭环：

1. BQML 历史最优组合是 `pv_fin_quality + 30/5% + biweekly + 5d`，仅保留为 historical reference / audit。
2. Cloud Run Python runner、task fan-out、GCS artifact、报告、模型诊断、尾部风险诊断和 QA 链路已能运行。
3. sklearn native LogisticRegression 首轮 Top5 全部 rejected。
4. PRD04 LightGBM binary wave 2 Top5 全部 rejected。
5. PRD04 LightGBM regression wave 3 Top5 也全部 rejected，主要风险是最大回撤超过共享验收契约阈值。
6. 尾部风险 P1 `individual_risk_guard_v0` 对回撤有轻度改善，但仍跑输中证1000。
7. 尾部风险 P2 `market_risk_off_v0` / `individual_and_market_risk_guard_v0` 已完成 A/B，`skip_new_buys` 降低仓位和交易成本，但未改善最大回撤且显著拖累收益。

因此，当前不应把风控问题继续压到交易执行层的硬拦截，也不应继续只扩大模型参数网格。下一步应把已经诊断出的风险状态转成模型可学习的信息：

1. 个股层面的近期大跌、20 日回撤、跌停扩散、流动性与规模风险。
2. 市场层面的中证1000趋势、宽度、跌停扩散、市场 regime。
3. 风险特征对候选排序和回测表现的影响证据。

本文定义这轮风险特征增强的产品和实现口径。

## 2. 目标

### 2.1 P0 目标：风险特征入模

1. 新增一个可追溯的特征集合：

```text
feature_set_id = strategy1_pv_fin_risk_v0_20260606
```

2. 将个股尾部风险字段、市场状态字段和必要的风险标记字段纳入训练面板 `feature_values_json` 与 Cloud Run frozen matrix。
3. 使用 Cloud Run Python task fan-out 跑新一轮候选搜索，默认复用现有 LightGBM binary / regression 能力，不新增新的模型依赖。
4. 继续固定交易参数：`30/5% + biweekly + 5d`、沪深主板股票池、现有成本 profile、Ledger v1。
5. 继续使用共享验收契约 `configs/strategy1/model_acceptance_contract_v1.yml` 决定 accepted / needs_more_evidence / rejected，并叠加本文风险专项验收约束。
6. 输出风险特征 search comparison、Top5 报告、模型诊断、尾部风险诊断和 QA。

### 2.2 P1 目标：候选风险评分增强

如果 P0 风险特征入模仍无法建立 accepted baseline，或 P0 有信号但回撤仍偏大，则新增可配置候选风险评分增强。

P1 不做硬过滤，不复用 P2 v0 的 `skip_new_buys` 默认动作，而是在候选排序层引入风险惩罚：

```text
risk_adjusted_score = score - lambda_risk * risk_score
```

P1 必须作为独立 profile / experiment A/B 运行，不能静默改变 P0 feature-only 结果。

### 2.3 P2 目标：后续模型族或目标函数扩展

如果 P0/P1 仍未 accepted，再进入后续模型族或目标函数方案，例如 XGBoost、HistGradientBoosting、CatBoost 或 group-based ranking objective。P2 需要单独 PRD 或本文修订版，不在本文 P0 直接实现。

### 4.4 test 复用与 final holdout 规则

本轮风险特征入模不是第一轮模型搜索。2025 test 已被以下波次用作接受门：

| 波次 | search | 结论 |
|---:|---|---|
| 1 | sklearn native LogisticRegression | Top5 rejected |
| 2 | LightGBM binary | Top5 rejected |
| 3 | LightGBM regression | Top5 rejected |

因此本文 P0 search 必须显式记录：

```text
model_search_wave_no = 4
test_reuse_wave_no = 4
test_reuse_approval_ref = docs/prd/PRD_20260606_03_策略1风险特征入模与候选增强.md
```

规则：

1. 2025 test 仍可作为样本外接受门，但不能单独支撑 `accepted`。
2. 因 `test_reuse_wave_no > 3`，任何候选若要登记 `accepted`，必须同时具备 2026 final_holdout 证据，且 `final_holdout_status='passed'`。
3. final_holdout 缺失、交易日不足或证据未通过时，只能是 `needs_more_evidence` 或 `rejected`，不得 `accepted`。
4. test / final_holdout 均不得参与参数、特征、阈值、候选数量、`lambda_risk` 或 max drawdown 目标选择。
5. `matrix_manifest.json`、candidate metrics、comparison artifact 和 `21` QA 必须记录 `test_reuse_wave_no`、`final_holdout_status` 与 `risk_feature_max_drawdown_target`。
6. 因本轮 `test_reuse_wave_no=4`，若候选通过共享契约但未达到 final_holdout passed 或风险专项 max drawdown 目标，不得登记 `accepted`。

### 4.5 零成本前置诊断

在执行新的 Cloud Run 训练前，必须先做一次只读诊断，判断当前瓶颈更像“模型 / 特征不足”，还是“接受门 / 年份区间过紧”。

该诊断只读取已存在 ADS 和 GCS artifact：

1. 读取既有 BQML historical reference 回测 `bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01` 的日级 NAV，重切 `2025-01-02` 至 `2025-12-31`，计算 2025-only total return、excess return vs `000852.SH` 和 2025 段 max drawdown。
2. 读取已完成的 sklearn native、LightGBM binary、LightGBM regression Top5 候选 summary / comparison artifact，汇总 2025 test-year excess、full-period max drawdown 和 acceptance reason 分布。
3. 输出 `acceptance_window_diagnosis.md` / `.json`，明确：
   - BQML historical reference 在 2025 单年是否能通过当前 test excess 和 max drawdown 门。
   - Python 已拒候选是否集中输在同一接受门。
   - 本轮继续做风险特征入模是否有足够依据。

判断规则：

1. 如果 BQML historical reference 重切 2025 也不能通过 `2025 excess > 0` 或本轮 max drawdown 目标，且已拒 Python 候选几乎都落在同一错误侧，则先标记 `primary_blocker='acceptance_window_risk'`，进入 owner 复核；此时不得把新一轮 Cloud Run 训练登记为“必做”。
2. 如果 BQML historical reference 能通过 2025 单年门，而 Python 候选不能通过，则标记 `primary_blocker='model_feature_gap'`，风险特征入模按本文继续。
3. 该诊断不得重跑 BQML，不得执行 `sql/ml/strategy1` SQL runner；只允许读取已产出的 ADS / artifact。

## 5. 风险特征集合

### 5.1 特征集合身份

新增逻辑特征集合：

```text
feature_set_id = strategy1_pv_fin_risk_v0_20260606
base_feature_set_id = strategy1_pv_fin_quality_v0_20260603
feature_version = strategy1_pv_v0_20260601
fin_feature_version = fin_default_v0_20260602
market_state_version = market_state_v0_20260606
```

`feature_set_id` 代表训练面板和 frozen matrix 的特征列集合，不代表 DWD/DWS 价格特征表本身必须重命名。实现必须在 `feature_schema.json`、`matrix_manifest.json`、candidate metrics、model registry、summary 和 comparison artifact 中写入该 `feature_set_id`。

### 5.2 个股风险特征

P0 必须覆盖以下个股风险字段。若字段已经在 base feature set 中存在，也必须在新 feature set 的 `feature_schema.json` 中明确归入风险组，避免报告和贡献度分析仍把它们当作普通价格量价字段。

重要边界：多数个股风险字段本身来自既有 `dws_stock_feature_daily_v0`，很可能已经在 `strategy1_pv_fin_quality_v0_20260603` 的输入列中出现。把这些字段重新归入 risk group 不会自动给模型新增信息；它的作用是让训练矩阵、报告和贡献度分析能明确识别风险暴露。本文真正的增量主要来自：

1. `dws_market_state_daily` 市场状态字段。
2. 明确的 `risk_flags` 布尔特征。
3. 少量个股风险 × 市场状态交互特征。

实现必须输出 `feature_delta_vs_base.json`，列出相对 `strategy1_pv_fin_quality_v0_20260603` 的新增列、复用列、重分组列和删除列。若新增列几乎只来自 market-state，则 comparison 必须把 market-state feature group 的贡献单独展示，避免把“重分组”误读为新增 alpha。

| group | 字段 | 来源 | 说明 |
|---|---|---|---|
| `risk_recent_return` | `ret_5d`, `ret_20d`, `ret_60d` | `dws_stock_feature_daily_v0` | 近期收益，信号日 t 可见 |
| `risk_drawdown` | `drawdown_20d` | `dws_stock_feature_daily_v0` | 相对近 20 日高点回撤 |
| `risk_volatility` | `vol_20d`, `vol_60d`, `hl_range_20d` | `dws_stock_feature_daily_v0` | 收益波动与日内振幅 |
| `risk_limit_down` | `limit_down_days_20d`, `one_word_limit_days_20d` | `dws_stock_feature_daily_v0` | 近 20 日跌停 / 一字跌停次数 |
| `risk_liquidity` | `amount_ma20_cny`, `turnover_rate_ma20`, `volume_ratio` | `dws_stock_feature_daily_v0` | 成交额、换手和量比 |
| `risk_size` | `total_mv_cny`, `circ_mv_cny`, `log_total_mv`, `log_circ_mv` | `dws_stock_feature_daily_v0` | 市值和规模风险 |
| `risk_flags` | `risk_ret20_lt_30pct`, `risk_drawdown20_lt_30pct`, `risk_limit_down_20d_ge2`, `risk_one_word_limit_20d_ge1`, `risk_microcap_total_mv`, `risk_microcap_circ_mv` | 训练面板派生 | P1/P2 诊断阈值转换为模型可学习的布尔 / 0-1 特征 |

`risk_flags` 的默认阈值沿用尾部风险 P1 证据，但 P0 只把它们作为特征，不作为硬过滤：

| flag | 默认阈值 |
|---|---|
| `risk_ret20_lt_30pct` | `ret_20d < -0.30` |
| `risk_drawdown20_lt_30pct` | `drawdown_20d < -0.30` |
| `risk_limit_down_20d_ge2` | `limit_down_days_20d >= 2` |
| `risk_one_word_limit_20d_ge1` | `one_word_limit_days_20d >= 1` |
| `risk_microcap_total_mv` | `total_mv_cny < 30e8` |
| `risk_microcap_circ_mv` | `circ_mv_cny < 20e8` |

必需字段缺失不得被默认填成 0。缺失必须由预处理保留为 NaN，或额外加入 `*_is_missing` 标记；具体方式写入 `preprocess_stats.json`。

### 5.3 市场状态特征

P0 必须从 `ashare_dws.dws_market_state_daily` 读取同一 `trade_date=t`、同一 `market_state_version` 的市场状态字段。该表 PIT 口径为 t 日收盘后形成，可用于 t 信号对应的 t+1 开盘执行。

`market_state_v0_20260606` 在 P2 hard-action A/B 中表现为净负：`skip_new_buys` 降低收益且未改善最大回撤。本文只把 market-state 作为模型输入，不假设它天然有 alpha。P0 报告必须单独展示 market-state feature group 的重要性、方向和 Top5 候选中的贡献；若条件允许，应额外输出 `market_state_features_enabled=true/false` 的小规模 A/B 或 TopK 复核。

| group | 字段 |
|---|---|
| `market_index_trend` | `csi300_ret_5d`, `csi300_ret_20d`, `csi300_drawdown_20d`, `csi1000_ret_5d`, `csi1000_ret_20d`, `csi1000_drawdown_20d`, `csi1000_close_to_ma20`, `csi1000_close_to_ma60`, `csi1000_ma20_to_ma60` |
| `market_volatility` | `csi300_vol_20d`, `csi1000_vol_20d`, `avg_vol_20d` |
| `market_breadth` | `adv_ratio_1d`, `above_ma20_ratio`, `new_low_20d_ratio`, `ret_20d_p25`, `ret_20d_median`, `drawdown_20d_median` |
| `market_limit_down` | `limit_down_count`, `one_word_limit_down_count`, `limit_down_mv_ratio` |
| `market_regime_flags` | `is_smallcap_trend_down`, `is_breadth_weak`, `is_limit_down_diffusion`, `risk_off_trigger_count`, `is_risk_off` |

`market_regime` 字符串不得直接作为未经定义的类别塞入模型。P0 只使用数值 / 布尔字段；若后续需要 one-hot，应在 `feature_schema.json` 明确列名，例如 `market_regime_risk_off`、`market_regime_risk_neutral`、`market_regime_risk_on`。

### 5.4 风险交互特征

P0 默认先不新增大量交互特征，避免在 2024 valid 上过拟合。允许新增少量可解释交互：

| 字段 | 说明 |
|---|---|
| `stock_drawdown_x_market_riskoff` | `drawdown_20d * is_risk_off` |
| `stock_vol_x_market_vol` | `vol_20d * csi1000_vol_20d` |
| `microcap_x_breadth_weak` | `risk_microcap_circ_mv * is_breadth_weak` |
| `limitdown_history_x_limitdown_diffusion` | `risk_limit_down_20d_ge2 * is_limit_down_diffusion` |

如果 P0 smoke 中交互特征导致缺失率、异常值或贡献度不可解释，可以先禁用交互特征；禁用需写入 manifest，不得隐式改变。

## 6. 数据构建与 PIT 规则

### 6.1 backend-neutral training panel

Cloud Run Python 的 `prepare_matrix` 当前从 `ads_ml_training_panel_daily.feature_values_json` 还原特征。因此 P0 必须先扩展训练面板或提供等价 backend-neutral panel builder，使新 `feature_set_id` 的 `feature_values_json` 已包含风险特征。

实现不得把新开发绑回 BQML / SQL runner。推荐顺序：

1. 将训练面板构建口径抽成 Cloud Run / ADS 中性的 builder 或 SQL 资产。
2. 保持旧 `sql/ml/strategy1` 文件只作 historical reference / audit。
3. Cloud Run `prepare_matrix` 只读取一个已冻结、可追溯的训练面板 run，不在每个 candidate task 临时 join DWS。

### 6.2 分区过滤

所有读取 BigQuery 分区表的 SQL 或 Python query 必须显式写分区过滤：

1. `dws_stock_feature_daily_v0.trade_date BETWEEN ...`
2. `dws_market_state_daily.trade_date BETWEEN ...`
3. `ads_ml_training_panel_daily.sample_date` / `trade_date` 的对应分区范围。
4. `ads_model_prediction_daily.predict_date`、`ads_stock_candidate_daily.rebalance_date`、`ads_backtest_nav_daily.trade_date` 等后续 QA / 报告读取范围。

不得只用 `run_id`、`feature_set_id`、`market_state_version` 或普通聚簇列过滤。

### 6.3 市场状态 join 规则

训练面板中每条股票样本 `(sec_code, trade_date=t)` 只能 join 一条 `dws_market_state_daily`：

```text
market_state.trade_date = sample.trade_date
market_state.market_state_version = market_state_v0_20260606
```

如果某个 t 日缺失市场状态：

1. train / valid / test / final_holdout 任一 split 缺失率超过 `0.1%`，P0 必须失败。
2. 单日全市场缺失必须失败，不能用前值静默填充。
3. 若是 dws 构建范围不足，应先重建 `dws_market_state_daily`，不得在训练中临时补数。

### 6.4 预处理

P0 沿用树模型预处理：

```text
preprocess_version = tree_winsor_missing_passthrough_v1
```

规则：

1. winsor 边界只用 train split 计算。
2. 缺失值保留为 NaN，交给 LightGBM 处理。
3. 不做 zscore。
4. 布尔特征写为 0/1 或 NaN；不得写 `TRUE` / `FALSE` 字符串。
5. `preprocess_stats.json` 必须记录每个新增风险字段的 train 缺失率、valid 缺失率和 winsor 边界。

### 7.2 P1 risk-adjusted candidate scoring

P1 只在 P0 没有 accepted baseline，或 P0 候选信号可用但回撤仍触发风险门时执行。

新增 profile：

```text
candidate_risk_adjustment_profile_id = risk_score_penalty_v0
```

P1 风险分数建议：

```text
risk_score =
  0.25 * rank_pct(-ret_20d)
+ 0.25 * rank_pct(-drawdown_20d)
+ 0.20 * rank_pct(vol_20d)
+ 0.15 * rank_pct(limit_down_days_20d)
+ 0.15 * rank_pct(-log_circ_mv)
```

其中 `rank_pct` 必须在同一 `trade_date` 的 live prediction pool 内计算，且只用信号日 t 可见字段。P1 只调整排序，不改变原始 `raw_score` / model `score`：

| 字段 | 说明 |
|---|---|
| `score` | 模型方向校准后的原始排序分 |
| `risk_score` | 0-1 风险分 |
| `risk_adjusted_score` | 候选层排序分 |
| `candidate_score_source` | `model_score` / `risk_adjusted_score` |

候选 `lambda_risk` 只能在 train/CV/valid 上选择，不得使用 2025 test 或 2026 final_holdout 选值。P1 默认候选值：

```text
lambda_risk in [0.05, 0.10, 0.20]
```

P1 如果 accepted，必须同时给出与 P0 feature-only 同模型同参数的 A/B 差异，说明收益变化来自候选排序风险惩罚，而不是模型参数或训练数据变化。

## 8. 验收标准

### AC-RISK-1 特征覆盖

通过条件：

### AC-RISK-2 PIT 与泄露

通过条件：

1. 市场状态只用 `trade_date=t` 的收盘后信息，不能使用 t+1 执行日信息。
2. 个股风险特征只用 t 日及以前价格 / 估值 / universe 信息。

### AC-RISK-3 模型接受

### AC-RISK-4 报告与贡献度

1. search comparison 显示每个 Top5 候选的 total_return、excess_return、Sharpe、max_drawdown、acceptance status。

## 11. 风险与控制

| 风险 | 控制 |
|---|---|
| 用 2024 崩盘窗口过拟合风险特征 | 保持 CV + 2024 valid confirmation；2025 test 和 2026 final_holdout 只做接受 / veto，不参与选参 |
| 市场状态字段泄露 t+1 信息 | `dws_market_state_daily` 明确 t close 后形成，只用于 t 信号对应 t+1 执行；QA 检查 join 日期 |
| P1 风险惩罚变相硬过滤 | 保留原始 `score`，新增 `risk_adjusted_score`，并记录 `candidate_score_source` |
| 风险特征只是降低仓位而不是提升选股 | P0 feature-only 不改持股数；P1 也只改排序，不自动留现金；若持仓数不足必须解释 |
| BigQuery 分区过滤遗漏导致查询失败 | 所有新增 SQL / Python query review 必须显式带日期分区过滤，`21` QA 覆盖关键来源表 |
| 树模型解释性不足 | 输出 LightGBM gain / split importance，并复用因子贡献度模块做 feature-group 视角 |
| 与尾部风险 P1/P2 profile 混淆 | P0 固定 `tail_risk_profile_id=diagnostic_only`；P1 候选风险惩罚使用独立 `candidate_risk_adjustment_profile_id` |
| 第 4 波复用 2025 test 造成多重检验偏差 | 显式记录 `test_reuse_wave_no=4`；`accepted` 必须有 final_holdout passed；test / final_holdout 不参与选参 |
| 把已有个股风险字段重分组误读为新增 alpha | 输出 `feature_delta_vs_base.json`，comparison 区分新增、复用和重分组字段 |
| market-state 在硬动作 A/B 中已显示净负 | P0 只把 market-state 当特征，必须单独报告 market-state group 贡献；必要时做 enabled/disabled 小规模 A/B |
