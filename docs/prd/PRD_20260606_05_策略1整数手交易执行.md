# PRD: 策略 1 整数手交易执行

## 1. 背景

策略 1 已完成 Ledger v1 P0/P1/P2、验收门 v2 和组合可行性诊断。验收门 v2 暴露了一个必须先收口的交易现实性问题：

1. `docs/prd/PRD_20260604_01_策略1LedgerV1交易执行语义.md` 明确 P0 不引入 A 股 100 股买入手数约束，仍保持 FLOAT shares。
2. `docs/prd/PRD_20260606_04_策略1验收门v2与组合可行性诊断.md` 已把 10 万 CNY、100 股整数手、实际持股数、现金占比和跳单率纳入 production acceptance 前置诊断。
3. PR #98 生成的 `acceptance_gate_v2` 产物显示，`20/30/40` 组合的静态整数手可行性均进入 `needs_more_evidence`。
4. 进一步只读复核发现，当前 extended reference backtest `bt_s1_bqml_baseline_pvfq_n30_bw_h5_extended_20260604_01` 的成交仍是 FLOAT shares：1291 笔 `FILLED` 成交全部为小数股，约 98.2% 四舍五入后也不是 100 股整数倍。

这说明当前 35.16% extended reference 收益、历史 BQML baseline 收益和 v2 整数手 feasibility 诊断不在同一个交易执行语义下，不能继续把 FLOAT-shares 结果作为 production acceptance 的收益依据。

本文定义 **Ledger lot-aware v1**：在既有 Ledger v1 的 t+1 开盘执行、卖出先于买入、netting、现金约束和 mark-to-market 语义上，新增 A 股小资金必须面对的 100 股买入整数手、卖出 odd-lot 退出、现金碎片和 QA/report 口径。

## 2. 目标

1. 将策略 1 production acceptance 使用的回测执行语义从 FLOAT shares 升级为 lot-aware shares。
2. 让 Cloud Run Python ledger 支持 `lot_size=100`、`min_buy_lot=1`、买入向下取整、现金约束和可追溯跳单状态。
3. 固化卖出 odd-lot 处理规则，避免因买入整数手、历史持仓或未来 corporate action 产生无法退出的碎股。
4. 生成一个复用当前 prediction stream 的 lot-aware fixed-prediction reference run，用于替代 FLOAT extended reference 进入验收门 v2。
5. 更新 v2 acceptance gate，使 production candidate 必须基于 lot-aware ledger 结果，不再用 FLOAT-shares backtest 判定 accepted。
6. 在进入下一轮风险特征训练前，先确认 `20/30/40` 至少一个候选在真实有状态 lot-aware ledger 下通过 implementation gate，或明确由 owner 决策调整资金规模 / 持股数 / 权重。

## 3. 非目标

1. 不改变模型训练、特征、标签、score orientation 或 prediction stream。
2. 不重新启用 BQML / `sql/ml/strategy1` SQL runner 作为未来策略执行层。
3. 不追溯改写历史 BQML / FLOAT-shares backtest 结果；这些结果只保留为 historical reference / audit。
4. 不引入分钟线、VWAP、盘口队列、成交概率或部分成交模型。
5. 不实现 A 股 T+1 卖出锁仓；该问题后续如需要单独 PRD。
6. 不在本文扩大模型族、风险特征、调仓频率或标签 horizon。
7. 不在本文引入融资融券、可转债、ETF 或非沪深主板股票池。

### 4.3 静态 feasibility 与真实有状态 ledger 的差异

`acceptance_gate_v2` 的 feasibility 是“每个 rebalance 日从 10 万现金重新构造 topN 组合”的静态诊断；真实 ledger 是有状态的，会继承旧仓、按当前实际持仓与目标持仓做 netting。

因此下一步不能只看静态 feasibility。必须实现 lot-aware 有状态 ledger，并用同一 prediction stream 重新跑固定预测回测，再让 v2 acceptance gate 读取真实 lot-aware backtest 结果。

## 5. 交易规则

### 5.1 总体规则

Ledger lot-aware v1 继承 Ledger v1 以下规则：

1. `signal_date = t-1`，`execution_date = t`。
2. 执行价为 t 日开盘价，买入加买入滑点，卖出减卖出滑点。
3. 卖出先于买入。
4. 实际持仓和目标持仓按市值差额 netting。
5. 卖不出进入 pending sell，并在后续开市日继续尝试。
6. 买不进不候补，保留现金。
7. 现金不足时缩放买入。
8. NAV 每个开市日收盘 mark-to-market。

新增 lot-aware 规则只改变股份数量、现金碎片和跳单状态，不改变信号、目标组合或成本 profile。

### 5.2 参数

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `ledger_version` | `ledger_exec_v1_lot100` | lot-aware ledger 版本 |
| `lot_size` | 100 | 买入单位 |
| `min_buy_lot` | 1 | 最小买入 1 手 |
| `min_buy_shares` | 100 | 最小买入股数 |
| `sell_odd_lot_policy` | `allow_full_exit_odd_lot` | 允许卖出全部 odd-lot 持仓 |
| `partial_sell_rounding` | `floor_to_lot_keep_residual` | 部分减仓卖出向下取整到 100 股，余股保留 |
| `buy_rounding` | `floor_to_lot` | 买入向下取整到 100 股 |
| `cash_redistribution` | `none_v1` | P0 不做余现金二次分配 |
| `min_notional_cny` | 0 | P0 不额外设置最小名义金额 |

### 5.3 买入规则

对每个净买入意图：

```text
desired_buy_value = max(target_value - current_market_value, 0)
buy_fill_price = open * (1 + slippage_buy_bps / 10000)
raw_buy_shares = desired_buy_value / open
lot_buy_shares = floor(raw_buy_shares / lot_size) * lot_size
```

规则：

1. 若 `open IS NULL` 或 `can_buy_open = false`，写 `BUY_SKIPPED_UNTRADABLE`。
2. 若 `lot_buy_shares < min_buy_shares`，写 `BUY_SKIPPED_BELOW_LOT`。
3. 若现金不足，对所有可买订单先按目标买入金额缩放，再按 100 股向下取整。
4. 缩放后如果低于 1 手，写 `BUY_SKIPPED_BELOW_LOT_AFTER_SCALE`。
5. P0 不做余现金二次分配，向下取整后的现金碎片保留为 cash。
6. 每笔买入的实际现金消耗必须包含成交额、佣金、买入印花税（当前为 0）和买入滑点。
7. 买入后 `cash_cny` 不得为负；任何负现金直接 QA fail。

### 5.4 卖出规则

卖出分为清仓卖出和部分减仓：

1. 若 `target_value <= 0.01`，视为清仓卖出，允许卖出当前全部持仓，包括小于 100 股或非 100 整倍的 odd-lot。
2. 若 `target_value > 0.01` 且需要部分减仓，则：
   - `raw_sell_shares = (current_market_value - target_value) / open`
   - `lot_sell_shares = floor(raw_sell_shares / lot_size) * lot_size`
   - 若 `lot_sell_shares < lot_size`，不成交，保留超配残差。
3. 若股票卖不出，写 `SELL_SKIPPED_UNTRADABLE`，对应持仓进入或维持 pending sell。
4. 清仓失败的 odd-lot 仍进入 pending sell，后续可卖日继续尝试全额退出。
5. 卖出成交现金增加额必须扣除佣金、卖出印花税和卖出滑点。

选择这个卖出口径的原因：

1. 买入必须按 100 股整数手，是小资金组合现金碎片的主要来源。
2. 卖出全部 odd-lot 是为了避免未来 corporate action、历史 FLOAT 持仓迁移或部分减仓残留导致无法退出。
3. 部分减仓向下取整保留残差，避免为了精确目标权重制造违反 100 股 lot 的卖出。

### 5.5 Netting 与 lot 交互

每只股票仍按：

```text
delta_value = target_value - current_market_value
```

然后应用：

1. `delta_value > 0`：买入路径，按 100 股向下取整。
2. `delta_value < 0`：卖出路径，清仓允许 odd-lot 全卖，部分减仓向下取整到 100 股。
3. `abs(delta_value) <= 0.01`：不交易。
4. 若旧仓未卖出且重新入选，必须先按实际持仓 netting，严禁重复全额建仓。
5. 若由于 lot rounding 当前持仓略低于目标，但补差额不足 1 手，则保留现金，不追买。

### 5.6 现金缩放

卖出完成后计算可用现金。若所有可买订单按目标金额和 lot 后仍超过可用现金：

1. 先按目标买入金额计算 `scale = available_cash / required_cash_with_cost`。
2. 对每个订单使用 `scaled_buy_value = desired_buy_value * scale`。
3. 将 `scaled_buy_value / open` 向下取整到 100 股。
4. 重新计算成交额、成本和现金影响。
5. 若仍因 min commission 等原因超过可用现金，必须按 score / rank 从低优先级买单向高优先级回退，直到现金非负。

P0 不做正向余现金再分配；如果 lot rounding 留下现金，该现金进入 NAV。

## 6. 成交状态

新增或固化以下状态：

| 状态 | 说明 |
|---|---|
| `FILLED` | 完全成交 |
| `FILLED_SCALED_CASH` | 现金不足缩放后成交 |
| `BUY_SKIPPED_UNTRADABLE` | 买入不可交易 |
| `BUY_SKIPPED_BELOW_LOT` | 原始目标买入不足 1 手 |
| `BUY_SKIPPED_BELOW_LOT_AFTER_SCALE` | 现金缩放后不足 1 手 |
| `BUY_SKIPPED_CASH_INSUFFICIENT_AFTER_ROUNDING` | lot 与成本回退后仍无法成交 |
| `SELL_SKIPPED_UNTRADABLE` | 卖出不可交易，进入/维持 pending sell |
| `SELL_SKIPPED_BELOW_LOT_PARTIAL` | 部分减仓不足 1 手，保留持仓 |
| `NOOP_ALREADY_TARGET` | 已接近目标，不交易 |

报告和 QA 必须展示各状态数量、金额、涉及股票数和对现金占比的影响。

P0 完成标准：

1. Cloud Run Python ledger 默认使用 `ledger_exec_v1_lot100`。
2. 复用 fixed prediction stream 跑完 `2024-01-02` 至 `2026-04-30` lot-aware reference。

## 11. 风险与控制

| 风险 | 控制 |
|---|---|
| lot-aware 后收益明显下降 | 这是交易现实性修正，不应回避；先记录差异，再决定是否改持股数/资金规模 |
| 40 只组合现金长期偏高 | 用 `20/30/40` 的真实有状态 ledger 对比；若全部不达标，先调组合规则 |
| 为降低现金而过度二次分配 | P0 不做 cash redistribution，避免新增组合优化复杂度 |
| odd-lot 卖出规则不精确 | P0 只允许 full exit odd-lot；部分减仓仍按 100 股 lot |
| 旧 FLOAT 结果被误用 | QA 和报告必须标记 legacy/audit，v2 accepted 禁止使用 |
| SQL runner 与 Cloud Run 结果不一致 | SQL runner 已是 historical audit，不作为未来执行层；只要求 Cloud Run 路径正确 |
