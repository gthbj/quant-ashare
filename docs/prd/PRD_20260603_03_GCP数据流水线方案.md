# PRD: GCP 数据流水线与每日 ODS 采集方案

> 文档维护：GPT-5（最近更新 2026-06-03）
> 状态：草案，待 review / 合并。
> 范围声明：本文定义 `quant-ashare` 长期在 GCP 上运行每日数据采集与 ODS→ADS ETL 的目标架构、首批采集范围、编排边界和验收标准；不在本文实现 Cloud Run、Dataform、Cloud Composer 或 SQL 迁移。
> 关联：`.agent/memory/OPEN_QUESTIONS.md` OQ-005、`docs/数据仓库建模方案-DWD-DIM.md`、`docs/数据仓库建模方案-DWS-ADS.md`、`data_audit/ODS_GCS_DATA_AUDIT_PROMPT.md`。

---

## 1. 背景

当前项目已经在 BigQuery `data-aquarium` 上完成 P0 DIM/DWD、策略 1 DWS/ADS、BQML runner 与报告链路。现有执行方式仍以手工 `bq query` / 本地脚本为主，适合开发和实跑验证，但不适合长期每日自动化。

owner 已明确长期要求：

1. 每日从 Tushare/Tinyshare 拉取数据并写入 GCS Parquet。
2. BigQuery ODS 外部表读取 GCS raw 数据。
3. ODS→DIM/DWD/DWS/ADS 的 ETL 在 GCP 上完成。
4. 每日定时拉取任务只覆盖当前项目实际消费的 ODS endpoint；未被 DWD/DWS/ADS 使用的 ODS 暂不安排生产定时任务。

本文给出长期推荐方案：

```text
Cloud Run Jobs
  管 Tushare/Tinyshare -> GCS Parquet

BigQuery Studio pipeline / Dataform
  管 ODS -> DIM/DWD/DWS/ADS SQL 转换、依赖、assertions、文档

Cloud Composer
  串起来，做全流程调度、失败重试、告警、补跑入口
```

其中 Dataform 是 BigQuery 内部 SQL 转换层，Cloud Composer 是 GCP 托管 Airflow 编排层，Cloud Run Jobs 是每日 Python 采集执行层。三者职责不重叠。

## 2. 目标

1. 固化 OQ-005 的长期工程选型：GCP 原生流水线优先，采用 `Cloud Run Jobs + Dataform / BigQuery Studio pipeline + Cloud Composer`。
2. 明确首批每日采集只覆盖当前 SQL 实际消费的 ODS endpoint。
3. 定义 Tushare/Tinyshare→GCS Parquet→BigQuery ODS 外部表的采集契约。
4. 定义 ODS→DIM/DWD/DWS/ADS 迁移到 Dataform 的边界和阶段。
5. 定义 Cloud Composer DAG 的每日执行顺序、重试、补跑和告警要求。
6. 定义 QA 门禁：schema、分区、行数、单位、PIT、策略 1 产物都必须可阻断。
7. 避免把 notebook / 手工 `bq query` 作为长期生产机制。

## 3. 非目标

- 不在本文一次性迁移现有 SQL 到 Dataform。
- 不在本文实现 Cloud Run 镜像、Composer DAG 或 Dataform repository。
- 不安排 57 张 ODS 的全量每日生产采集。
- 不新增 P1 资金流、行业、事件、融资融券等未消费 DWD。
- 不改现有数据分层命名：继续使用 `ashare_ods`、`ashare_dim`、`ashare_dwd`、`ashare_dws`、`ashare_ads`。
- 不把 BigQuery Studio notebook 作为长期正式采集主方案；notebook 仅用于探索、抽样验证和临时审查。
- 不在仓库中保存 Tushare/Tinyshare token、service account key 或任何凭据。

## 4. 推荐架构

### 4.1 总体链路

```text
Tushare / Tinyshare API
  -> Cloud Run Jobs: ingestion endpoint groups
  -> GCS raw Parquet
  -> BigQuery ashare_ods external tables
  -> ODS readability / schema checks
  -> Dataform: DIM / DWD / DWS / ADS SQL workflow
  -> BigQuery QA: metadata / P0 / OQ / strategy QA
  -> BQML runner / backtest / report, as needed
  -> GCS artifacts + Composer notification
```

### 4.2 服务职责

| 服务 | 职责 | 本项目使用方式 |
|---|---|---|
| Cloud Run Jobs | 运行有明确开始/结束的 Python 采集任务 | endpoint group 级采集，写 GCS Parquet，不长期驻留 |
| Cloud Scheduler | 轻量定时触发 | 早期可直接触发 Cloud Run Job；长期由 Composer 接管主编排 |
| Secret Manager | 管理 Tushare/Tinyshare token 和敏感配置 | Cloud Run / Composer runtime 注入，不进代码和日志 |
| GCS | 保存 raw Parquet、审查报告、策略报告 artifact | raw ODS 数据与报告产物分 bucket / prefix 管理 |
| BigQuery ODS | Hive 分区外部表读取 GCS raw | 保持现有 `endpoint + partition_date` 强制过滤口径 |
| Dataform / BigQuery Studio pipeline | BigQuery SQL 转换、依赖、assertions、文档 | 管 ODS→DIM/DWD/DWS/ADS 的 SQL workflow |
| Cloud Composer | 全流程调度、重试、补跑、告警 | 串采集、ODS 检查、Dataform、复杂 QA、runner、报告 |
| BigQuery | SQL 执行、原生表存储、BQML | 所有 DIM/DWD/DWS/ADS 与 runner 结果继续在 BigQuery |

### 4.3 为什么不用纯 BigQuery Scheduled Queries

BigQuery Scheduled Queries 只适合少量定时 SQL。当前项目至少需要：

- 外部 API 采集；
- GCS Parquet schema 稳定性；
- ODS 外部表可读性检查；
- SQL 分层依赖；
- 多组 QA 门禁；
- 策略 runner 与报告；
- 失败重试、补跑、告警和运行记录。

这些已经超过单纯 Scheduled Queries 的职责。

### 4.4 为什么不把每日采集写在 BigQuery Studio notebook 里

BigQuery Studio / Colab Enterprise notebook 可以调度，也可以调用外部 API，适合 POC、抽样验证、数据审查和人工分析。但长期生产采集需要：

- endpoint group 级代码组织；
- 可复用 Python package；
- 稳定依赖镜像；
- 幂等写 GCS；
- 清晰日志与重试；
- service account 权限边界；
- CI / CD 与回滚。

这些更适合 Cloud Run Jobs。

### 5.2 暂不安排生产定时任务的 ODS

未被当前 DIM/DWD/DWS/ADS 消费的 ODS 不进入首批每日生产采集。例如：

- `moneyflow` / `moneyflow_hsgt`
- `margin` / `margin_detail`
- `hk_hold` / `ccass_hold`
- 龙虎榜、涨跌停扩展、股东增减持、质押回购、业绩预告 / 快报、分红、研报等 P1+ endpoint
- 行业 / 指数权重扩展 endpoint

这些 endpoint 只有在对应 DWD/DWS/ADS PRD 或实现 PR 进入开发时，才补充到采集 manifest、单位契约、QA 与 Dataform 依赖。

### 5.3 采集 manifest

后续实现应新增机器可读 manifest，例如：

```text
configs/ingestion/ods_current_scope_v0.yml
```

manifest 至少包含：

| 字段 | 含义 |
|---|---|
| `source_system` | `tushare` / `tinyshare` |
| `endpoint` | API endpoint |
| `ods_table` | BigQuery ODS 外部表 |
| `enabled` | 是否进入生产定时采集 |
| `cadence` | `trade_day_daily` / `calendar_daily_snapshot` / `finance_recent_window` |
| `business_date_field` | `trade_date` / `end_date` / snapshot date |
| `partition_date_semantics` | `business_date` / `report_period` / `snapshot_date` |
| `gcs_prefix` | raw Parquet 路径前缀 |
| `schema_contract` | 目标 Parquet schema 版本 |
| `unit_contract_required` | 是否需要 OQ-006 单位契约 |
| `current_consumer` | 当前 DIM/DWD/DWS/ADS 消费表 |
| `qa_required` | 必跑 QA 列表 |

未在 manifest 中 `enabled=true` 的 endpoint，不得由每日生产 DAG 自动采集。

## 6. 采集设计

### 6.1 Cloud Run Job 组织

首批不建议为每个 endpoint 单独维护一套完全独立代码。推荐一个 ingestion 镜像 + 多个 endpoint group job：

```text
ingest-market-eod
  daily, adj_factor, stk_limit, suspend_d, daily_basic

ingest-index-eod
  index_daily, index_dailybasic

ingest-dim-snapshot
  stock_basic(listed/delisted), trade_cal, namechange

ingest-finance-recent
  fina_indicator, income, balancesheet, cashflow
```

每个 job 使用相同 Python package 和共享工具层：

```text
scripts/ingestion/
  common/
    api_client.py
    gcs_writer.py
    parquet_schema.py
    manifest.py
    logging.py
  endpoints/
    daily.py
    daily_basic.py
    finance.py
    index.py
    dim_snapshot.py
```

### 6.2 生产采集请求口径

每日生产采集不按数据审查提示词的多 token 并发模式设计。默认口径：

- 使用生产配置中的一个 Tushare/Tinyshare token 或一个授权代理。
- 不依赖多 token 并发提速。
- 不实现 token 级并发限速器作为 P0 必需项。
- 仍保留基础请求节流、重试、超时和 API 返回上限检查。

API 返回行数若打满官方单次返回上限或脚本 page size / limit，必须标记为高风险并拆细请求。该规则来自 `data_audit/ODS_GCS_DATA_AUDIT_PROMPT.md`，生产采集也必须遵守。

### 6.3 GCS raw 写入契约

GCS 路径需保持 Hive 分区口径：

```text
gs://<raw_bucket>/tushare/<endpoint>/partition_date=<YYYYMMDD>/part-*.parquet
```

写入要求：

1. Parquet schema 必须显式 cast，不能依赖 pandas / pyarrow 自动推断。
2. 金额、比例、股本、市值等高风险字段必须按 endpoint schema 固定为目标类型。
3. 空 API 返回不写伪空 Parquet；记录 run log 和 empty-return event。
4. 同一 `endpoint + partition_date + ingestion_run_id` 可重试覆盖临时路径，但正式发布路径必须幂等。
5. 写正式路径前先写 staging prefix，完成 schema 校验后再 publish。
6. 每次 run 写入 ingestion manifest：

```text
endpoint
partition_date
business_date_start / business_date_end
request_params_hash
row_count
schema_version
source_system
ingestion_run_id
started_at / finished_at
status
error_summary
```

### 6.4 分区语义

| 类别 | partition_date 语义 | 生产采集要求 |
|---|---|---|
| 行情增量 | `partition_date = trade_date` | 每个交易日采当日数据；停牌无 daily 行不是缺数 |
| 指数行情 | `partition_date = trade_date` | 采可用 benchmark / dim_index 端点范围 |
| 财务报告期 | `partition_date = end_date` | 每日检查近期公告 / 修正，写回对应报告期分区 |
| 维度快照 | `partition_date = snapshot_date` | 每日或低频刷新最新快照，Dataform/DIM 取最新 |

财务表继续保留 raw 多版本，不在 raw 层做 canonical，不覆盖历史版本。DWD 版本事实表继续按 PIT 规则消费。

## 7. Dataform / BigQuery Studio Pipeline 设计

### 7.1 Dataform 接管范围

第一阶段只迁移 ODS→DIM/DWD/DWS/ADS 的 SQL 转换，不迁移 BQML runner。

| 范围 | 是否进入 Dataform P0 | 说明 |
|---|---|---|
| `dim/*` | 是 | 依赖清晰，适合 source + ref |
| `dwd/*` P0 + 财务三表 | 是 | 单位、PIT、分区、字段说明应固化 |
| `dws/*` 策略 1 + 财务特征 | 是 | 特征/标签/样本依赖图适合 Dataform |
| `ads/01_ads_strategy1_tables.sql` | 可选先保留 | 契约 DDL 可先保持 SQL，后续迁移 |
| `sql/ml/strategy1/*` | 暂不迁移 | BQML、ledger、报告链路过程性强 |
| `sql/qa/*.sql` 复杂 QA | 保留 | PIT、单位契约、runner 验收继续显式 SQL |

建议目标结构：

### 7.3 tags / release configs

Dataform action tags 建议：

| tag | 作用 |
|---|---|
| `p0_dim` | P0 维表 |
| `p0_dwd` | P0 明细 |
| `finance_dwd` | 财务三表 |
| `strategy1_dws` | 策略 1 DWS |
| `metadata` | 表/字段说明维护 |
| `assertion_p0` | P0 断言 |
| `assertion_strategy1` | 策略 1 断言 |
| `assertion_unit` | OQ-006 单位契约断言 |

每日 pipeline 可以按 tag 分阶段运行：

```text
run p0_dim + p0_dwd
run finance_dwd
run strategy1_dws
run assertions
```

### 7.4 字段说明

Dataform 应承担表/字段说明的主要维护职责。实现时有两种过渡方式：

1. Dataform definitions 内写 description。
2. 保留现有 `sql/metadata/*.sql`，由 Composer 在 Dataform run 后继续执行。

迁移完成后，应减少手工 `ALTER COLUMN SET OPTIONS` 脚本，只保留无法由 Dataform 直接表达或需要集中覆盖的 description。

## 8. Cloud Composer 编排设计

### 8.1 每日 DAG

建议 DAG 名：

```text
ashare_daily_pipeline_v0
```

每日执行顺序：

```text
start
  -> resolve_trade_date
  -> ingest_market_eod
  -> ingest_index_eod
  -> ingest_dim_snapshot
  -> ingest_finance_recent
  -> ods_external_readability_check
  -> ods_schema_contract_check
  -> dataform_run_p0_dim_dwd
  -> dataform_run_finance_dwd
  -> dataform_run_strategy1_dws
  -> run_metadata_descriptions
  -> run_p0_smoke_qa
  -> run_oq004_index_qa
  -> run_oq006_unit_qa
  -> run_finance_qa
  -> run_strategy1_dws_ads_qa
  -> optional_strategy_runner
  -> optional_report_render
  -> notify_success_or_failure
```

### 8.2 调度时间

建议按 Asia/Shanghai 时区调度。行情类任务应在 Tushare/Tinyshare 当日数据稳定后执行，避免市场收盘后立即拉取导致数据不全。

P0 建议：

```text
交易日 18:30 Asia/Shanghai: market/index eod
每日 20:00 Asia/Shanghai: dim snapshot + finance recent + Dataform + QA
```

如果上游数据可用性晚于该时间，按实测调整。

### 8.3 补跑

Composer 必须支持以下补跑参数：

| 参数 | 含义 |
|---|---|
| `business_date` | 单日补跑 |
| `date_from/date_to` | 行情区间补跑 |
| `endpoint_group` | 只补某个采集组 |
| `dataform_tags` | 只跑某组 Dataform actions |
| `skip_ingestion` | 仅重建下游 |
| `skip_dataform` | 仅采集 + ODS 检查 |
| `force_publish` | 手工确认后覆盖正式 GCS prefix |

补跑不得绕过 QA 门禁。若 owner 明确只做审查或临时验证，必须用独立 run label 标记，不得混入生产成功状态。

### 8.4 告警

失败告警至少包含：

- DAG id / run id；
- endpoint group；
- business date / partition date；
- 失败 task；
- BigQuery job id / Cloud Run execution id / Dataform workflow invocation id；
- 错误摘要；
- 是否已写 GCS staging / 是否 publish；
- 下一步建议。

## 9. QA 门禁

### 9.1 ODS / GCS 检查

每日采集后、Dataform 前必须检查：

1. GCS 新写入对象存在且非空。
2. Parquet schema 与 endpoint schema contract 一致。
3. BigQuery 外部表可读。
4. 新分区 row count 合理。
5. 空返回 endpoint 有明确 empty-return event，不写伪空 Parquet。
6. API 返回上限命中请求已拆细复查。
7. 同分区精确重复被记录并阻断或豁免。

### 9.2 Dataform assertions

适合迁入 Dataform 的断言：

- 主键唯一；
- 必填字段非空；
- 日期范围；
- 分区裁剪；
- 行数下限；
- 简单命名规则；
- source/ref 依赖完整性。

### 9.3 保留 SQL QA

以下复杂 QA 继续保留在 `sql/qa/`，由 Composer 显式执行：

- `01_p0_smoke_checks.sql`
- `02_strategy1_dws_ads_checks.sql`
- `03_oq004_index_checks.sql`
- `04_finance_caliber_checks.sql`
- `05_oq006_unit_checks.sql`
- runner `10_qa_runner_outputs.sql`
- diagnosis `12_qa_model_diagnosis_outputs.sql`

原因：这些 QA 包含 PIT、单位契约、benchmark 覆盖、runner artifact、复杂窗口和业务断言，保留显式 SQL 更便于审计。

## 10. 权限与安全

### 10.1 Service accounts

建议至少拆分：

| service account | 用途 |
|---|---|
| `sa-ashare-ingestion` | Cloud Run Jobs 采集，读 Secret Manager，写 raw GCS |
| `sa-ashare-dataform` | Dataform 执行 ODS→ADS SQL |
| `sa-ashare-composer` | Composer 编排，触发 Cloud Run / Dataform / BigQuery jobs |
| `sa-ashare-report` | 报告渲染与 artifact 上传，可与 runner 共享或独立 |

### 10.2 权限原则

- 采集 SA 不应拥有修改 DIM/DWD/DWS/ADS 的权限。
- Dataform SA 不应读取 Tushare/Tinyshare token。
- Composer SA 可以触发任务，但不直接持有业务 token。
- 所有 token 只存在 Secret Manager，不进入 repo、日志、memory、PRD 或 report。

## 11. 运行记录与可观测性

建议新增 BigQuery meta 表：

```text
ashare_meta.ingestion_run
ashare_meta.ingestion_partition_status
ashare_meta.pipeline_run
ashare_meta.pipeline_task_status
```

最小字段：

| 字段 | 含义 |
|---|---|
| `run_id` | Composer / ingestion run id |

## 13. 验收标准

本 PRD 对应实现完成后，OQ-005 可关闭。关闭标准：

1. 当前消费的 14 张 ODS 均有生产采集 manifest、schema contract 和 GCS raw 写入规则。
2. 每日生产 DAG 只调度 manifest 中 `enabled=true` 的 endpoint。
3. Cloud Run Jobs 可完成 API→GCS Parquet 采集，并通过 ODS readability/schema QA。
4. Dataform 或 BigQuery Studio pipeline 可按依赖构建 ODS→DIM/DWD/DWS/ADS。
5. Composer 可串起采集、Dataform、metadata、QA、可选 runner / report，并提供失败重试、补跑和告警。
6. P0 / OQ-004 / OQ-006 / OQ-003 / 策略 1 DWS/ADS QA 均可在 GCP 上执行。
7. 未消费 ODS 不进入生产每日采集；新增 endpoint 必须先更新 manifest、单位契约和 QA。
8. 全流程不在代码、日志、memory 或报告中泄露 token / key。

## 14. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Tushare/Tinyshare 数据延迟 | 当日分区不完整 | 调度延后 + row count QA + 可补跑 |
| API 返回上限截断 | 缺行 | row limit hit 检查 + 拆细请求 |
| Parquet schema 漂移 | ODS 外部表不可读 | schema contract + staging 校验 |
| 财务修正版迟到 | PIT 特征不完整 | finance recent rolling window + raw 多版本保留 |
| Dataform 迁移 SQL 产生差异 | 下游策略变化 | 迁移期双跑对比 + 既有 QA |
| Composer 成本 / 运维复杂度 | 早期负担 | Phase 1 可先用 Cloud Scheduler，Phase 3 再上 Composer |
| 未消费 ODS 过早采集 | 成本和维护膨胀 | manifest `enabled=true` 门禁 |
