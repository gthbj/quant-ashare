# PRD: GCP 数据流水线与每日 ODS 采集方案

> 文档维护：GPT-5（最近更新 2026-06-03）
> 状态：草案，待 review / 合并。
> 范围声明：本文定义 `quant-ashare` 长期在 GCP 上运行每日数据采集与 ODS→ADS ETL 的目标架构、首批采集范围、编排边界和验收标准；Cloud Run、Dataform、Cloud Composer 和 SQL 迁移按后续上线阶段交付。
> 关联：`.agent/memory/OPEN_QUESTIONS.md` OQ-005、`docs/数据仓库建模方案-DWD-DIM.md`、`docs/数据仓库建模方案-DWS-ADS.md`、`data_audit/ODS_GCS_DATA_AUDIT_PROMPT.md`。

## 1. 背景

当前项目已经在 BigQuery `data-aquarium` 上完成 P0 DIM/DWD、策略 1 DWS/ADS、BQML runner 与报告链路。下一阶段将每日数据采集、ODS→ADS ETL、QA 门禁、策略 runner 和报告链路纳入 GCP 托管生产流水线。

owner 已明确长期要求：

1. 每日从 Tushare/Tinyshare 拉取数据并写入 GCS Parquet。
2. BigQuery ODS 外部表读取 GCS raw 数据。
3. ODS→DIM/DWD/DWS/ADS 的 ETL 在 GCP 上完成。
4. 每日定时拉取任务只覆盖当前项目实际消费的 ODS endpoint；当前未消费 endpoint 进入后续接入池。

本文定义目标方案：

```text
Cloud Run Jobs
  管 Tushare/Tinyshare -> GCS Parquet

BigQuery Studio pipeline / Dataform
  管 ODS -> DIM/DWD/DWS/ADS SQL 转换、依赖、assertions、文档

Cloud Composer
  串起来，做全流程调度、失败重试、告警、补跑入口
```

其中 Dataform 是 BigQuery 内部 SQL 转换层，Cloud Composer 是 GCP 托管 Airflow 编排层，Cloud Run Jobs 是每日 Python 采集执行层。三者职责清晰分层。

## 2. 目标

1. 固化 OQ-005 的长期工程方案：采用 `Cloud Run Jobs + Dataform / BigQuery Studio pipeline + Cloud Composer`。
2. 明确首批每日采集只覆盖当前 SQL 实际消费的 ODS endpoint。
3. 定义 Tushare/Tinyshare→GCS Parquet→BigQuery ODS 外部表的采集契约。
4. 定义 ODS→DIM/DWD/DWS/ADS 迁移到 Dataform 的边界和阶段。
5. 定义 Cloud Composer DAG 的每日执行顺序、重试、补跑和告警要求。
6. 定义 QA 门禁：schema、分区、行数、单位、PIT、策略 1 产物都必须可阻断。
7. 定义 notebook、手工 `bq query`、Dataform、Composer 和 Cloud Run Jobs 的生产职责边界。

## 3. 实施边界

- 本文交付生产流水线 PRD；Cloud Run 镜像、Composer DAG、Dataform repository 和采集代码按上线阶段分别实现。
- 首批每日生产采集范围为当前实际消费的 14 个 ODS endpoint。
- P1 资金流、行业、事件、融资融券等 endpoint 按后续 DWD/DWS/ADS 需求进入采集扩展流程。
- 现有分层命名保持为 `ashare_ods`、`ashare_dim`、`ashare_dwd`、`ashare_dws`、`ashare_ads`。
- BigQuery Studio notebook 用于探索、抽样验证和临时审查；生产流水线状态由 Composer、Dataform 和 meta 表记录。
- Tushare/Tinyshare token、service account key 和凭据通过 Secret Manager / runtime 环境注入。

## 4. 目标架构

### 4.1 总体链路

```text
Tushare / Tinyshare API
  -> Cloud Run Jobs: ingestion endpoint groups
  -> GCS raw Parquet
  -> BigQuery ashare_ods external tables
  -> ODS readability / schema checks
  -> Dataform: DIM / DWD / DWS / ADS SQL workflow
  -> BigQuery QA: metadata / P0 / OQ / strategy QA
  -> BQML runner / backtest / report, as needed
  -> GCS artifacts + Composer notification
```

### 4.2 服务职责

| 服务 | 职责 | 本项目使用方式 |
|---|---|---|
| Cloud Run Jobs | 运行有明确开始/结束的 Python 采集任务 | endpoint group 级采集，写 GCS Parquet，任务完成后退出 |
| Cloud Scheduler | 轻量定时触发 | 早期可直接触发 Cloud Run Job；长期由 Composer 接管主编排 |
| Secret Manager | 管理 Tushare/Tinyshare token 和敏感配置 | Cloud Run / Composer runtime 注入，代码和日志只记录脱敏状态 |
| GCS | 保存 raw Parquet、审查报告、策略报告 artifact | raw ODS 数据与报告产物分 bucket / prefix 管理 |
| BigQuery ODS | Hive 分区外部表读取 GCS raw | 保持现有 `endpoint + partition_date` 强制过滤口径 |
| Dataform / BigQuery Studio pipeline | BigQuery SQL 转换、依赖、assertions、文档 | 管 ODS→DIM/DWD/DWS/ADS 的 SQL workflow |
| Cloud Composer | 全流程调度、重试、补跑、告警 | 串采集、ODS 检查、Dataform、复杂 QA、runner、报告 |
| BigQuery | SQL 执行、原生表存储、BQML | 所有 DIM/DWD/DWS/ADS 与 runner 结果继续在 BigQuery |

### 4.3 生产职责边界

生产流水线按以下边界实现：

| 层级 | 生产职责 | 状态事实来源 |
|---|---|---|
| Cloud Run Jobs | 执行 Tushare/Tinyshare API 采集、schema cast、GCS staging / publish | `ashare_meta.ingestion_run`、Cloud Run execution |
| Dataform / BigQuery Studio pipeline | 执行 ODS→DIM/DWD/DWS/ADS SQL workflow、基础 assertions、表字段文档 | Dataform workflow invocation、BigQuery job |
| Cloud Composer | 编排 Cloud Run Jobs、ODS 检查、Dataform、复杂 QA、runner、报告和告警 | `ashare_meta.pipeline_run`、Airflow DAG run |
| BigQuery SQL QA | 执行 PIT、单位、benchmark、runner artifact 等复杂断言 | BigQuery job、QA result log |
| Notebook / 手工 SQL | 执行探索、抽样验证、临时审查 | 人工审查记录、临时 run label |

## 6. 采集设计

### 6.1 Cloud Run Job 组织

首批采用一个 ingestion 镜像 + 多个 endpoint group job：

```text
ingest-market-eod
  daily, adj_factor, stk_limit, suspend_d, daily_basic

ingest-index-eod
  index_daily, index_dailybasic

ingest-dim-snapshot
  stock_basic(listed/delisted), trade_cal, namechange

ingest-finance-recent
  fina_indicator, income, balancesheet, cashflow
```

每个 job 使用相同 Python package 和共享工具层：

```text
scripts/ingestion/
  common/
    api_client.py
    gcs_writer.py
    parquet_schema.py
    manifest.py
    logging.py
  endpoints/
    daily.py
    daily_basic.py
    finance.py
    index.py
    dim_snapshot.py
```

### 6.2 生产采集请求口径

数据审查模式和生产采集模式分开管理。生产每日任务采用以下请求口径：

- 使用生产配置中的一个 Tushare/Tinyshare token 或一个授权代理。
- endpoint group 内按 endpoint 和日期窗口执行有限并发。
- 请求层内置基础节流、重试、超时和 API 返回上限检查。
- job 级限速参数按供应商或代理 SLA 配置，并写入 run log。

API 返回行数若打满官方单次返回上限或脚本 page size / limit，必须标记为高风险并拆细请求。该规则来自 `data_audit/ODS_GCS_DATA_AUDIT_PROMPT.md`，生产采集也必须遵守。

### 6.3 GCS raw 写入契约

GCS 路径需保持 Hive 分区口径：

```text
gs://<raw_bucket>/tushare/<endpoint>/partition_date=<YYYYMMDD>/part-*.parquet
```

写入要求：

1. Parquet schema 按 endpoint schema contract 显式 cast。
2. 金额、比例、股本、市值等高风险字段必须按 endpoint schema 固定为目标类型。
3. 空 API 返回记录 run log 和 empty-return event，正式 prefix 保持无新增数据对象。
4. 同一 `endpoint + partition_date + ingestion_run_id` 可重试覆盖临时路径，但正式发布路径必须幂等。
5. 写正式路径前先写 staging prefix，完成 schema 校验后再 publish。
6. 每次 run 写入 ingestion manifest：

```text
endpoint
partition_date
business_date_start / business_date_end
request_params_hash
row_count
schema_version
source_system
ingestion_run_id
started_at / finished_at
status
error_summary
```

### 6.4 分区语义

| 类别 | partition_date 语义 | 生产采集要求 |
|---|---|---|
| 行情增量 | `partition_date = trade_date` | 每个交易日采当日数据；停牌无 daily 行按交易状态口径记录 |
| 指数行情 | `partition_date = trade_date` | 采可用 benchmark / dim_index 端点范围 |
| 财务报告期 | `partition_date = end_date` | 每日检查近期公告 / 修正，写回对应报告期分区 |
| 维度快照 | `partition_date = snapshot_date` | 每日或低频刷新最新快照，Dataform/DIM 取最新 |

财务表 raw 层保留多版本和历史版本。DWD 版本事实表按 PIT 规则消费并生成 canonical 口径。

## 7. Dataform / BigQuery Studio Pipeline 设计

### 7.1 Dataform 接管范围

第一阶段迁移 ODS→DIM/DWD/DWS/ADS 的 SQL 转换。BQML runner 继续由 Composer 显式执行现有 SQL / Python 报告链路。

| 范围 | P0 处理方式 | 说明 |
|---|---|---|
| `dim/*` | Dataform SQLX | 维表 source + ref 依赖 |
| `dwd/*` P0 + 财务三表 | Dataform SQLX | 单位、PIT、分区、字段说明固化到 action |
| `dws/*` 策略 1 + 财务特征 | Dataform SQLX | 特征、标签、样本依赖图固化到 workflow |
| `ads/01_ads_strategy1_tables.sql` | Composer 执行契约 DDL，后续转 SQLX | ADS 契约表保持独立初始化入口 |
| `sql/ml/strategy1/*` | Composer 执行现有 runner SQL | BQML、ledger、报告链路按 run_id 参数化执行 |
| `sql/qa/*.sql` 复杂 QA | Composer 执行现有 QA SQL | PIT、单位契约、runner 验收按显式 SQL 阻断 |

### 7.4 字段说明

Dataform 承担表/字段说明的主要维护职责。实现采用两种过渡方式：

1. Dataform definitions 内写 description。
2. 保留现有 `sql/metadata/*.sql`，由 Composer 在 Dataform run 后继续执行。

迁移完成后，表/字段说明主要由 Dataform 维护；集中覆盖脚本负责少数字段说明和兼容场景。

## 8. Cloud Composer 编排设计

### 8.1 每日 DAG

DAG 名：

```text
ashare_daily_pipeline_v0
```

每日执行顺序：

```text
start
  -> resolve_trade_date
  -> ingest_market_eod
  -> ingest_index_eod
  -> ingest_dim_snapshot
  -> ingest_finance_recent
  -> ods_external_readability_check
  -> ods_schema_contract_check
  -> dataform_run_p0_dim_dwd
  -> dataform_run_finance_dwd
  -> dataform_run_strategy1_dws
  -> run_metadata_descriptions
  -> run_p0_smoke_qa
  -> run_oq004_index_qa
  -> run_oq006_unit_qa
  -> run_finance_qa
  -> run_strategy1_dws_ads_qa
  -> optional_strategy_runner
  -> optional_report_render
  -> notify_success_or_failure
```

### 8.2 调度时间

按 Asia/Shanghai 时区调度。行情类任务在 Tushare/Tinyshare 当日数据稳定后执行，调度时间设置为收盘后充足延迟。

P0 调度：

```text
交易日 18:30 Asia/Shanghai: market/index eod
每日 20:00 Asia/Shanghai: dim snapshot + finance recent + Dataform + QA
```

上游数据可用性晚于该时间时，按实测调整。

### 8.3 补跑

Composer 必须支持以下补跑参数：

| 参数 | 含义 |
|---|---|
| `business_date` | 单日补跑 |
| `date_from/date_to` | 行情区间补跑 |
| `endpoint_group` | 只补某个采集组 |
| `dataform_tags` | 只跑某组 Dataform actions |
| `skip_ingestion` | 仅重建下游 |
| `skip_dataform` | 仅采集 + ODS 检查 |
| `force_publish` | 手工确认后覆盖正式 GCS prefix |

补跑执行同一 QA 门禁。owner 明确要求的审查或临时验证使用独立 run label，并记录为非生产成功状态。

### 8.4 告警

失败告警至少包含：

- DAG id / run id；
- endpoint group；
- business date / partition date；
- 失败 task；
- BigQuery job id / Cloud Run execution id / Dataform workflow invocation id；
- 错误摘要；
- 是否已写 GCS staging / 是否 publish；
- 处置动作。

## 9. QA 门禁

### 9.1 ODS / GCS 检查

每日采集后、Dataform 前必须检查：

1. GCS 新写入对象存在且非空。
2. Parquet schema 与 endpoint schema contract 一致。
3. BigQuery 外部表读取成功。
4. 新分区 row count 合理。
5. 空返回 endpoint 有明确 empty-return event，正式 prefix 保持无新增数据对象。
6. API 返回上限命中请求已拆细复查。
7. 同分区精确重复被记录并阻断或豁免。

### 9.2 Dataform assertions

迁入 Dataform 的断言：

- 主键唯一；
- 必填字段非空；
- 日期范围；
- 分区裁剪；
- 行数下限；
- 简单命名规则；
- source/ref 依赖完整性。

### 9.3 Composer 显式执行的 SQL QA

以下复杂 QA 由 Composer 显式执行：

- `01_p0_smoke_checks.sql`
- `02_strategy1_dws_ads_checks.sql`
- `03_oq004_index_checks.sql`
- `04_finance_caliber_checks.sql`
- `05_oq006_unit_checks.sql`
- runner `10_qa_runner_outputs.sql`
- diagnosis `12_qa_model_diagnosis_outputs.sql`

这些 QA 覆盖 PIT、单位契约、benchmark 覆盖、runner artifact、复杂窗口和业务断言，作为生产 DAG 的阻断任务。

## 10. 权限与安全

### 10.1 Service accounts

Service accounts 拆分：

| service account | 用途 |
|---|---|
| `sa-ashare-ingestion` | Cloud Run Jobs 采集，读 Secret Manager，写 raw GCS |
| `sa-ashare-dataform` | Dataform 执行 ODS→ADS SQL |
| `sa-ashare-composer` | Composer 编排，触发 Cloud Run / Dataform / BigQuery jobs |
| `sa-ashare-report` | 报告渲染与 artifact 上传，可与 runner 共享或独立 |

### 10.2 权限原则

- 采集 SA 只读取 Secret Manager 中的采集密钥并写 raw GCS。
- Dataform SA 只读取 ODS / meta 并写 DIM/DWD/DWS/ADS。
- Composer SA 触发 Cloud Run、Dataform 和 BigQuery jobs。
- 所有 token 只存在 Secret Manager，并通过 runtime 注入任务。
- repo、日志、memory、PRD 和 report 均记录脱敏配置与运行状态。

## 12. 上线阶段

### Phase 0: PRD / manifest

### Phase 1: Cloud Run Jobs 采集

交付：

- 空返回记录 empty-return event，正式 prefix 保持无新增数据对象。

## 13. 验收标准

本 PRD 对应实现完成后，OQ-005 关闭标准如下：

1. 当前消费的 14 张 ODS 均有生产采集 manifest、schema contract 和 GCS raw 写入规则。
2. 每日生产 DAG 只调度 manifest 中 `enabled=true` 的 endpoint。
3. Cloud Run Jobs 可完成 API→GCS Parquet 采集，并通过 ODS readability/schema QA。
4. Dataform 或 BigQuery Studio pipeline 可按依赖构建 ODS→DIM/DWD/DWS/ADS。
5. Composer 串起采集、Dataform、metadata、QA、按配置 runner / report，并提供失败重试、补跑和告警。
6. P0 / OQ-004 / OQ-006 / OQ-003 / 策略 1 DWS/ADS QA 均可在 GCP 上执行。
7. 每日生产采集只调度当前实际消费 ODS；新增 endpoint 先更新 manifest、单位契约和 QA。
8. 全流程通过 Secret Manager / runtime 环境管理 token / key，代码、日志、memory 和报告只记录脱敏信息。

## 14. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Tushare/Tinyshare 数据延迟 | 当日分区缺行 | 调度延后 + row count QA + 补跑 |
| API 返回上限截断 | 缺行 | row limit hit 检查 + 拆细请求 |
| Parquet schema 漂移 | ODS 外部表读取失败 | schema contract + staging 校验 |
| 财务修正版迟到 | PIT 特征缺行 | finance recent rolling window + raw 多版本保留 |
| Dataform 迁移 SQL 产生差异 | 下游策略变化 | 迁移期双跑对比 + 既有 QA |
| Composer 成本 / 运维复杂度 | 早期负担 | Phase 1 可先用 Cloud Scheduler，Phase 3 再上 Composer |
| 未消费 ODS 过早采集 | 成本和维护膨胀 | manifest `enabled=true` 门禁 |
