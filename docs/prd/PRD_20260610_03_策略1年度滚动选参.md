# PRD: 策略 1 年度滚动选参回测实验

> 状态：草案，待 owner review。
> 范围声明：本文只定义 Strategy1 年度 walk-forward 参数选择、final refit 和连续回测方案；本文不改代码、不执行 BigQuery / Cloud Run、不生成或覆盖历史回测产物。
> 关联：`docs/prd/PRD_20260609_01_策略1R14长训练回测.md`、`docs/prd/PRD_20260605_04_策略1CloudRunPython模型基线搜索.md`、`docs/prd/PRD_20260604_04_策略1CloudRun训练回测.md`、`docs/prd/PRD_20260604_01_策略1LedgerV1交易执行语义.md`、`.agent/memory/OPEN_QUESTIONS.md` OQ-010。

---

## 1. 背景

最近已完成一轮固定 R14 的年度滚动训练回测：

1. 固定候选参数为 R14。
2. 每年重新训练同一组 LightGBM regression 参数。
3. valid 使用回测年前一年的后半段。
4. 年度预测合并后跑一条连续 ledger。

该实验回答了：

```text
固定 R14 参数，逐年重新训练，在 2020-2026 连续资金状态下表现如何？
```

但它没有回答：

```text
如果真实生产中每年用上一整年 valid 重新选择参数，再用选中参数 final refit，下一年表现是否更好？
```

owner 进一步确认希望评估一种更接近生产模型生命周期的年度滚动选参流程：

1. 用历史窗口训练一组候选参数。
2. 用下一整年作为 valid 选择参数、方向和是否可用。
3. 用选中参数在最近 5 年窗口上 final refit。
4. 回测下一年。
5. 年度预测可以分年生成，但最终必须跑一条连续 ledger，保持现金、持仓、pending sell 和 NAV 连续。

本文定义该实验。

## 2. 目标

### 2.1 P0 目标

1. 建立一套年度 walk-forward 参数选择协议，而不是固定 R14。
2. 可选候选参数池在实验开始前固定，后续不得按年度结果临时增删。
3. 每个回测年只使用该年初已经发生的数据选择参数。
4. valid 使用回测年前一整年。
5. final refit 使用选中参数在最近 5 年窗口上重训。
6. 回测覆盖 `2021-01-04 ~ 2026-06-09`，其中 2026 截止到当前 owner 指定可用日期。
7. 组合先固定为 `20` 只、单票上限 `7.5%`，避免同时把组合参数纳入年度过拟合。
8. 输出年度选中参数、valid 指标、年度回测结果、全周期连续回测结果，并和固定 R14 年度滚动结果对比。

### 2.2 P1 目标

1. 若 P0 结果有效，再评估是否把持仓数 `10/20/40` 或其他 portfolio 参数纳入下一轮年度选择。
2. 若 P0 仍表现不稳，再评估是否加入市场状态过滤、风险降权或 feature set 变体。
3. 若 P0 通过 owner review，再把方案实现为可复用 Cloud Run annual walk-forward workflow，而不是继续依赖一次性脚本。

## 3. 非目标

1. 不把 valid 年份本身作为最终样本外成绩。
2. 不在实验过程中动态扩展候选参数池。
3. 不同时搜索 feature set、label horizon、持仓数、调仓频率和权重上限。
4. 不恢复旧 BQML-only 或 SQL ledger runner。
5. 不把每年 fresh-run 回测结果拼接成连续结果。
6. 不把本实验自动登记为 production accepted baseline。
7. 不追溯覆盖已有 R14、v3 replay 或其他历史搜索结论。

## 4. 年度窗口定义

P0 固定年度窗口如下：

| 回测年 | 参数选择训练 | valid | final refit | 回测 |
|---:|---|---|---|---|
| 2021 | 2015-2019 | 2020 | 2016-2020 | 2021 |
| 2022 | 2016-2020 | 2021 | 2017-2021 | 2022 |
| 2023 | 2017-2021 | 2022 | 2018-2022 | 2023 |
| 2024 | 2018-2022 | 2023 | 2019-2023 | 2024 |
| 2025 | 2019-2023 | 2024 | 2020-2024 | 2025 |
| 2026 | 2020-2024 | 2025 | 2021-2025 | 2026-01-05 ~ 2026-06-09 |

### 4.1 交易日边界

上述年份是自然年描述。实现阶段必须转成交易日：

1. 每个 `train_start` / `valid_start` / `backtest_start` 取不早于自然边界的首个 SSE 开市日。
2. 每个 `train_end` / `valid_end` / `backtest_end` 取不晚于自然边界的最后一个 SSE 开市日。
3. 2026 `backtest_end` 固定为 `2026-06-09`，若该日数据不完整，则截到最近完整可回测交易日，并在 artifact 中记录原因。

### 4.2 5d label embargo

因为训练目标仍为 5 日未来收益，任何训练或 valid 评估都必须做 label horizon 防泄漏：

1. 参数选择训练的最后信号日，必须保证 5d 标签完整落在 valid 开始前。
2. valid 的最后信号日，必须保证 5d 标签完整落在下一年回测开始前。
3. final refit 使用上一年数据时，其最后信号日必须保证 5d 标签完整落在回测年前。
4. artifact 必须记录 `label_embargo_trading_days=5` 和每个窗口的 `effective_last_signal_date`。

### 5.2 Diagnostic reference

B26 binary 只作为 diagnostic-only reference：

| ID | 用途 | 规则 |
|---|---|---|
| `b26_baseline_binary_ref` | 历史 baseline binary 对照 | 只输出同窗口 valid / backtest 指标作为参考，不参与 valid hard filter、排序或 `selected_candidate_id` |

约束：

1. B26 的 binary probability score 可以按同一 RankIC / top-bottom 公式计算诊断指标，但这些指标只用于对照。
2. B26 不和 regression 候选横向排序。
3. B26 不触发 fallback，也不得成为年度 final refit 的 selected 参数。
4. 若后续要把 binary 候选纳入年度选择，必须另开 experiment version，并定义 binary 与 regression score 的统一排序和 acceptance 口径。

### 5.3 候选池冻结规则

1. 可选候选池和 diagnostic reference 必须写入 manifest，并生成 hash。
2. 同一实验 run 的每个年度必须使用相同可选候选池和相同 diagnostic reference 集合。
3. 如果候选实现失败，必须标记 `candidate_failed`，不得用临时新增候选补位。
4. 如果要新增候选，只能开启新的 experiment version，不能改写本实验结果。

## 6. valid 年度选参门

valid 只用于选择下一年参数，不是最终成绩。

### 6.1 valid 硬过滤

每个候选在 valid 年必须满足：

```text
valid_rank_ic > 0
valid_top_minus_bottom > 0
valid_excess_return_vs_any_5_indices > 0
valid_max_drawdown >= -33.33%
valid_sharpe >= 0.3
valid_calmar >= 0.3
valid_excess_calmar_ratio_vs_any_5_indices > 0.3
```

五个比较指数固定为：

| 指数 | `sec_code` |
|---|---|
| 上证指数 | `000001.SH` |
| 深证成指 | `399001.SZ` |
| 上证50 | `000016.SH` |
| 沪深300 | `000300.SH` |
| 中证1000 | `000852.SH` |

### 6.2 删除的 valid 条件

P0 不显式要求：

```text
valid_total_return > 0
```

原因是该条件在当前门中是重复硬门：`valid_sharpe >= 0.3` 和 `valid_calmar >= 0.3` 已经隐式要求策略在 valid 年具备正的绝对风险收益。删除显式 `valid_total_return > 0` 只是避免重复表达，不代表本版本允许负收益候选通过。若未来要允许“valid 绝对收益为负但显著跑赢指数”的候选进入年度选择，必须另开 experiment version，并把 Sharpe / Calmar 硬线改为 excess Sharpe / excess Calmar 或其他相对风险收益门。

### 6.3 valid excess return

```text
valid_excess_return_vs_index = strategy_valid_return - index_valid_return
valid_excess_return_vs_any_5_indices > 0
```

只要五个指数中任意一个 valid 超额收益大于 0，该条件通过。

### 6.4 valid Calmar

```text
valid_calmar = valid_strategy_return / abs(valid_strategy_max_drawdown)
```

若 `valid_strategy_max_drawdown = 0`，该指标必须特殊标记，不得静默除以 0。若 valid 年收益为负，Calmar 为负，无法通过 `>= 0.3`。

### 6.5 valid Excess Calmar Ratio

沿用 owner 已确认的“策略最大回撤同期超额”口径：

```text
strategy_drawdown_excess = strategy_max_drawdown - index_drawdown_during_strategy_max_drawdown_window
```

```text
excess_calmar_ratio = strategy_excess_return / abs(strategy_drawdown_excess)
```

其中：

1. `strategy_max_drawdown` 是策略在 valid 年内的最大回撤。
2. `index_drawdown_during_strategy_max_drawdown_window` 是同一个策略 peak-to-trough 区间内指数自身回撤，不是指数全年的最大回撤。
3. `strategy_excess_return = strategy_valid_return - index_valid_return`。
4. 如果 `strategy_drawdown_excess > 0`，代表策略最大回撤窗口内比指数更抗跌；该 ratio 不计算，标记为 `drawdown_excess_positive`。在硬过滤中可视为相对回撤风险通过，但排序时不能和普通 ratio 直接比较。
5. 如果 `strategy_drawdown_excess = 0`，ratio 不计算，标记为 `drawdown_excess_zero`。
6. valid 门要求五个指数中至少一个可计算 ratio `> 0.3`，或至少一个指数触发 `drawdown_excess_positive` 且该指数的 `strategy_excess_return > 0`。

### 6.6 选参排序

候选通过硬过滤后按以下顺序排序：

1. 优先 `valid_excess_calmar_ratio` 高。
2. 若存在 `drawdown_excess_positive` 候选，优先级高于普通 ratio，但同类之间按 `valid_excess_return` 和 `valid_calmar` 排序。
3. 其次 `valid_calmar` 高。
4. 其次 `valid_max_drawdown` 更浅。
5. 其次 `valid_excess_return` 高。
6. 最后 `valid_rank_ic` 高。

若所有候选均未通过 hard filter：

1. 当年不得强行选择“相对最好”的候选作为 accepted 参数。
2. 可以选择预先指定 fallback 参数 `r14_anchor` 跑 diagnostic-only 下一年回测。
3. fallback 结果必须标记为 `fallback_no_valid_candidate`，不能进入 accepted / promoted 结论。

## 7. final refit

每年 valid 选出参数后，必须使用该参数在最近 5 年窗口上 final refit。

以 2021 为例：

1. 用 2015-2019 训练 12 个候选。
2. 用 2020 valid 选择参数。
3. 用选中参数在 2016-2020 final refit。
4. 用 final refit 模型预测 2021。

### 7.1 final refit 约束

1. final refit 只能使用选中参数，不能重新调参。
2. final refit 可以使用 valid 年数据，因为 valid 已经结束，且只用于下一年预测。
3. final refit 输出必须记录 `selected_candidate_id`、`selected_by_valid_year`、`final_refit_train_start`、`final_refit_train_end`、`final_refit_effective_last_signal_date`。
4. final refit 后不得再用下一年回测表现反向修改参数。

### 7.2 训练缓存复用

训练任务可以按以下 key 复用：

```text
train_window + candidate_param_id + feature_set_id + label_horizon + preprocess_version
```

例如：

```text
2016-2020 + r14_anchor
2016-2020 + r20_attack_anchor
```

若 2021 final refit 已训练了 `2016-2020 + r14_anchor`，而 2022 参数选择训练也需要同一 key，则该模型可以复用。

但只能复用完全相同 key 的模型。不能因为上一年只 final refit 了一个选中参数，就跳过下一年对其他 11 个候选的训练。

## 8. 连续回测方式

### 8.1 年度预测生成

每个年度可单独生成 prediction stream：

```text
prediction_run_id = s1_annual_param_select_<year>_<selected_candidate>_vYYYYMMDD_NN
```

每个 prediction stream 只覆盖对应回测年。

### 8.2 合并预测

年度预测生成后，必须合并为一个 synthetic continuous prediction run：

```text
combined_prediction_run_id = s1_annual_param_select_pred_2021_20260609_vYYYYMMDD_NN
```

合并时必须：

1. 对 `ads_model_prediction_daily.predict_date` 使用静态分区过滤。
2. 为 synthetic run 写入 selected registry 或等价 model identity，避免 backtest path 找不到 selected model。
3. 若不同年份来自不同 selected model，synthetic prediction run 必须统一可被回测 SQL / Python path 识别，不得因单一 `model_id` 过滤丢年份。
4. artifact 记录每年来源 `source_run_id`、`source_model_id`、`selected_candidate_id` 和日期范围。

### 8.3 连续 ledger

最终评价只能来自一条连续 ledger：

```text
backtest_start = 2021-01-04
backtest_end = 2026-06-09
```

要求：

1. 从 2021 首日初始现金 fresh-start。
2. 后续年份不能重置现金、持仓、pending sell 或 active target。
3. 不能把每年 fresh-run 的收益率拼接成连续结果。
4. 若使用 resume segment 代替 full fresh，必须通过 Cloud Run Python ledger resume consistency QA。

## 10. 验收标准

P0 PRD 实现后，至少应满足：

1. 年度窗口、valid 年和 final refit 窗口与本文表格一致。
2. 可选候选池和 diagnostic reference hash 在所有年度一致。
3. valid 年没有被计入同一年最终样本外评价。
4. 每年 selected candidate 可追溯到 valid 指标和排序原因。
5. 所有 BigQuery 分区表查询显式带分区过滤。
6. 合并预测没有丢失年份，`2021-2026` 每个开市日都有可用 prediction / target / NAV 覆盖。
7. 连续 ledger 没有年度 reset。
8. 输出和固定 R14 结果可以一一对比。

## 11. 风险与注意事项

| 风险 | 说明 | 缓解 |
|---|---|---|
| 年度 valid 过拟合 | 每年用上一年选参数，仍可能追随短期 regime | 候选池固定、小规模、排序不只看收益 |
| valid 年份被误当成绩 | 2020 valid 不能算 2020 样本外成绩 | 输出中明确 valid / backtest 年份角色 |
| 候选池过大 | 大网格会放大 selection bias | P0 固定 11 个可选候选，B26 只作 diagnostic reference |
| 组合参数混入选参 | 同时选持仓数会增加过拟合 | P0 固定 20 只、7.5% |
| synthetic registry/model_id 错误 | 回测可能只读取某一年 model_id | 合并预测时显式处理 synthetic model identity |
| 分段拼接误用 | 每年 fresh-run 拼接会破坏资金连续性 | 只接受单条连续 ledger 或已验收 resume |
| 2026 数据不完整 | 最新日期可能缺 EOD / 标签 / benchmark | 截到最近完整可回测交易日并记录 |

## 12. 推荐执行顺序

1. 实现 manifest：固定 11 个可选候选、B26 diagnostic reference、年度窗口、valid gate 和输出 schema。
2. 先跑 `2021` 单年度 smoke：2015-2019 训练候选、2020 valid 选参、2016-2020 final refit、2021 回测。
3. 若 smoke 通过，再跑 2022-2026 年度链路。
4. 合并年度预测，跑单条连续 `2021-2026` ledger。
5. 生成年度选参表、年度回测表、全周期对比表。
6. 和固定 R14 annual walk-forward 结果并排 review。
7. owner review 后再决定是否进入实现复用化、扩大候选池或引入组合参数选择。
