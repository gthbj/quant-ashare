# PRD：策略 1 年度滚动执行工程化

## 1. 背景

`docs/prd/PRD_20260610_03_策略1年度滚动选参.md` 已定义年度 walk-forward 参数选择方案：

- 用上一整年 valid 选择参数。
- 用最近 5 年 final refit 训练最终模型。
- 用下一年作为样本外回测。
- 最终评价必须来自一条连续 ledger，不能拼接每年 fresh-run。

2026-06-10 的 2021 单年度 Cloud Run smoke 已跑通，但暴露出三类工程化问题：

| 问题 | 现象 | 影响 |
|---|---|---|
| ad-hoc experiment 未 catalog 化 | `select_register_predict` 只传 `experiment_id` 时找不到实验，需要手工传 `experiment-json` | 后续 `2021-2026` 不能依赖人工拼 base64 payload |
| ADS ledger state 表未部署 | `backtest_report` 第一次失败，缺 `ashare_ads.ads_backtest_ledger_state_daily` | resume / lot-aware ledger 路径依赖的生产 schema 不完整 |
| summary 复合年化字段未部署 | `backtest_report` 第二次失败，`ads_backtest_performance_summary` 缺 `compound_annual_return` 等字段 | PR #134 代码契约与生产 BigQuery schema 漂移 |

这些问题不是模型效果问题，而是年度滚动实验从“手工 smoke”走向“可重复正式执行”必须解决的执行工程问题。

## 2. 目标

P0 目标：

1. 年度滚动实验不再手工拼 `experiment-json`。
2. Cloud Run backtest/report 执行前能检查 ADS schema readiness。
3. ADS additive schema migration 可重复执行、可审计、不会重建或覆盖已有 ADS 表。
4. `2021` 单年度 smoke 能用正式 orchestrator / schema readiness 路径复现成功。
5. 完整 `2021-2026` 年度滚动执行具备清晰 run_id、artifact、prediction 和 continuous ledger 规则。

## 3. 非目标

本 PRD 不做以下事情：

- 不增加新模型。
- 不增加新 LightGBM 参数。
- 不调整 v3 gate。
- 不改变 valid 选参门。
- 不把默认输出切到 `ashare_research`。
- 不重建或回填历史 ADS 表。
- 不用年度 fresh-run 拼接出正式连续结果。

## 4. 术语

| 术语 | 含义 |
|---|---|
| selection train | 用于候选参数选择阶段训练候选模型的训练窗口 |
| valid year | 用于选择参数的上一整年样本外年份 |
| final refit | 选中参数后，用最近 5 年样本重新训练最终模型 |
| backtest year | final refit 后的下一年样本外回测年份 |
| resolved experiment | 已经填好 run_id、backtest_id、窗口、持仓参数、feature set 等字段的 `Experiment` payload |
| continuous ledger | 覆盖多年度的单一账户状态序列；现金、持仓、pending sell、NAV 连续传递 |

## 5. 年度窗口规则

P0 固定执行以下窗口：

| 回测年 | 参数选择训练 | valid | final refit | 回测 |
|---|---|---|---|---|
| 2021 | 2015-04-01 ~ 2019-12-31 | 2020 | 2016-01-01 ~ 2020-12-31 | 2021 |
| 2022 | 2016-01-01 ~ 2020-12-31 | 2021 | 2017-01-01 ~ 2021-12-31 | 2022 |
| 2023 | 2017-01-01 ~ 2021-12-31 | 2022 | 2018-01-01 ~ 2022-12-31 | 2023 |
| 2024 | 2018-01-01 ~ 2022-12-31 | 2023 | 2019-01-01 ~ 2023-12-31 | 2024 |
| 2025 | 2019-01-01 ~ 2023-12-31 | 2024 | 2020-01-01 ~ 2024-12-31 | 2025 |
| 2026 | 2020-01-01 ~ 2024-12-31 | 2025 | 2021-01-01 ~ 2025-12-31 | 2026-06-09 |

实现时必须按交易日和标签 horizon 做实际边界收缩：

- `label_horizon=5` 的训练样本必须做 label embargo。
- selection train / valid / final refit 的实际 `*_end` 可以早于自然年最后一天，以保证未来 5 日标签完整落在目标窗口内。
- predict / backtest 的实际起止日必须落在 SSE 开市日。
- 报告中必须同时记录名义窗口和实际样本窗口。

## 6. Experiment payload 生成

### 6.1 当前问题

2021 smoke 中，annual-selection experiment 没有登记在 `configs/strategy1/oq010_experiments_v0.json`，导致：

```text
ValueError: experiment_id ... not found in configs/strategy1/oq010_experiments_v0.json
```

临时处理是手工生成 base64 `experiment-json`。这不能作为正式执行路径。

### 6.2 P0 要求

新增年度滚动 orchestrator，由代码生成 resolved experiment payload。

建议入口：

```text
scripts/strategy1_cloudrun/orchestrate_annual_rolling_selection.py
```

P0 输入：

| 参数 | 说明 |
|---|---|
| `--start-year` | 默认 `2021` |
| `--end-year` | 默认 `2026` |
| `--as-of-date` | 2026 年截止日，默认 `2026-06-09` |
| `--target-holdings` | 默认 `20` |
| `--max-single-weight` | 默认 `0.075` |
| `--rebalance-frequency` | 默认 `weekly` 或按年度滚动 PRD 固定口径 |
| `--feature-set-id` | 默认 `strategy1_pv_fin_risk_v0_20260606` |
| `--candidate-set-id` | 默认 11 个冻结 LightGBM regression 候选 |
| `--force-replace` | 允许覆盖同 run_id 产物 |
| `--dry-run` | 只打印 resolved plan，不启动 Cloud Run |

P0 输出：

- 每个回测年的 resolved experiment JSON。
- 每个回测年的 matrix URI。
- 每个回测年的 selected candidate。
- 全周期 continuous ledger backtest id。
- 每一步 Cloud Run execution id。

### 6.3 run_id 命名

年度训练 / 预测 run_id 必须可追溯到年份：

```text
s1_annual_roll_y{backtest_year}_train{train_start_year}_{train_end_year}_valid{valid_year}_n20_w075_vYYYYMMDD_NN
```

连续 ledger backtest id 必须区别于单年 smoke：

```text
bt_s1_annual_roll_continuous_2021_2026_n20_w075_vYYYYMMDD_NN
```

P0 不要求一次性确定最终命名模板，但实现必须避免复用 2021 smoke run_id 作为正式 `2021-2026` 结果。

## 7. ADS schema migration

### 7.1 当前问题

Cloud Run runner 已依赖最新 ADS 契约，但生产 BigQuery 里缺少：

- `ashare_ads.ads_backtest_ledger_state_daily`
- `ashare_ads.ads_backtest_performance_summary.compound_annual_return`
- `ashare_ads.ads_backtest_performance_summary.return_period_count`
- `ashare_ads.ads_backtest_performance_summary.annualization_target_period_count`
- `ashare_ads.ads_backtest_performance_summary.annualization_method`

2021 smoke 已手工用 additive DDL 补齐。正式路径不能依赖手工补表。

### 7.2 P0 要求

新增 additive migration 脚本，必须满足：

- 只用 `CREATE TABLE IF NOT EXISTS`。
- 只用 `ALTER TABLE ADD COLUMN IF NOT EXISTS`。
- 不使用 `CREATE OR REPLACE TABLE` 作用于已有 ADS 表。
- 不删除历史数据。
- 不回填历史值。
- 可重复执行。

建议文件：

```text
sql/ads/02_strategy1_additive_migrations.sql
```

P0 至少覆盖：

```text
ashare_ads.ads_backtest_ledger_state_daily
ashare_ads.ads_backtest_performance_summary.compound_annual_return
ashare_ads.ads_backtest_performance_summary.return_period_count
ashare_ads.ads_backtest_performance_summary.annualization_target_period_count
ashare_ads.ads_backtest_performance_summary.annualization_method
```

## 8. Schema readiness QA

### 8.1 目标

在 Cloud Run backtest/report 运行前或 runner 部署后，明确检查生产 BigQuery schema 是否满足当前代码契约。

### 8.2 P0 要求

新增 schema readiness QA，建议文件：

```text
sql/strategy1/qa/qa_cloudrun_schema_readiness.sql
```

或按现有编号追加为：

```text
sql/strategy1/qa/26_qa_cloudrun_schema_readiness.sql
```

QA 必须检查：

- 必需 ADS 表存在。
- 必需字段存在。
- 字段类型符合契约。
- 分区字段符合契约。
- 聚簇字段至少不缺关键字段。

失败信息必须包含修复建议，例如：

```text
Run sql/ads/02_strategy1_additive_migrations.sql before backtest_report.
```

### 8.3 Cloud Run 集成

P0 可选两种方式：

1. 部署后人工或 CI 运行 readiness QA。
2. `backtest_report.py` 启动时先运行 readiness check，失败则 fail-fast。

推荐 P0 先做第 1 种，避免把 BigQuery schema 检查逻辑塞进 Python runner；后续如果仍频繁漂移，再加 runner preflight。

## 9. 连续 ledger 执行规则

### 9.1 禁止事项

正式 `2021-2026` 结果不得这样做：

```text
2021 fresh-run + 2022 fresh-run + ... + 2026 fresh-run
```

然后把每年 NAV 拼起来。这会破坏现金、持仓、pending sell、调仓锚点和收益序列连续性。

### 9.2 可接受路径

P0 接受两条路径。

路径 A：单一 continuous backtest

- 每年分别训练 / 预测。
- 把年度预测写入一个连续 prediction stream。
- 从 `2021-01-04` fresh-start 跑到 `2026-06-09`。
- 只生成一个 continuous backtest id。

路径 B：resume segment backtest

- 2021 fresh-start。
- 2022 从 2021 年末真实 ledger state resume。
- 2023 从 2022 年末真实 ledger state resume。
- 一直滚到 2026。
- 每段必须通过 resume consistency / continuity QA。
- 最终报告必须明确这是 resume-continuous 结果，不是 fresh segment 拼接。

推荐路径 A，除非实现预测流拼接成本高于 resume 路径。

## 10. Artifact 与报告要求

每个年度必须写出：

- matrix manifest
- work units
- candidate metrics
- selected candidate
- model artifact
- prediction row count
- yearly execution summary

完整连续回测必须写出：

- continuous backtest summary
- NAV / trade / position / ledger state 行数
- report URI
- diagnosis URI
- tail-risk artifact URI
- QA job ids
- selected candidate by year

报告必须区分：

- 单年度 smoke 结果
- 年度 rolling selection 的每年参数选择结果
- 完整 continuous ledger 结果

## 11. 验收标准

P0 实现完成后，必须满足：

| 验收项 | 要求 |
|---|---|
| schema migration | 可重复执行，不重建已有 ADS 表 |
| schema readiness QA | 能检查缺表 / 缺字段 / 类型漂移 |
| 2021 smoke | 不再手工生成 base64 payload |
| CV fold | 11 个候选均 `cv_fold_count=3` |
| select/register/predict | 成功生成预测并记录 selected candidate |
| backtest/report | 成功生成 ledger、summary、report、diagnosis、QA |
| full rolling plan | 能 dry-run 输出 `2021-2026` resolved experiments |
| continuous ledger | 正式结果必须是一条连续 ledger，或通过 resume segment continuity QA |

## 12. 风险与控制

| 风险 | 控制 |
|---|---|
| 手工 run_id 复用覆盖旧 smoke artifact | 正式 run 使用新 run_id，smoke run_id 不作为正式结果 |
| ADS schema 漂移再次出现 | additive migration + readiness QA |
| 年度 payload 与 matrix manifest 不一致 | resolved experiment 写入 manifest，并在 select/backtest 阶段校验 |
| 每年 fresh-run 被误当连续结果 | 报告和 QA 强制标记 `initial_state_mode` / continuity status |
| Cloud Run 成本上升 | P0 固定 11 候选，不扩大参数搜索 |
| 2021 smoke 结果较差被误读为模型最终结论 | 2021 smoke 只验链路，不作为 accepted baseline 判断 |

## 13. 实施顺序

建议按以下顺序实现：

1. 新增 ADS additive migration。
2. 新增 schema readiness QA。
3. 新增 annual rolling orchestrator 的 dry-run / resolved payload 生成。
4. 用 orchestrator 重跑 2021 smoke。
5. 扩展到 `2021-2026` 年度训练 / 预测。
6. 跑 continuous ledger 或 resume-continuous ledger。
7. 生成年度选择表与完整回测报告。
