# PRD: 策略 1 验收门 v3

> 文档维护：GPT-5 Codex（最近更新 2026-06-07）

## 1. 背景

策略 1 已完成 BigQuery SQL runner、Cloud Run Python runner、LightGBM binary / regression、风险特征入模和整数手 ledger 等多轮实验。现有验收门分两层：

- v1：当前 Python search 的主写回门，偏向基础信号质量、2025 test、full-period 收益 / 回撤 / Sharpe 和 final holdout veto。
- v2：下一代生产门诊断，新增 eligible signal / executable、lot-aware 交易可行性和 final holdout 约束，但尚不是当前 Python search 的主写回门。

owner 要求定义 v3：保留 v1/v2 中必要的信号质量和执行可行性门，同时引入完整季度维度的策略季化收益、Calmar Ratio、策略最大回撤同期超额和 Excess Calmar Ratio，用于更直接衡量策略在不同市场季度内的相对风险收益质量。

## 2. 目标

1. 固化策略验收门 v3 的完整条件和公式，避免只在临时 CSV 中维护口径。
2. 将项目内年化、季化、月化、日化等周期化收益默认口径统一为复利 / 几何换算。
3. 明确 10 / 20 / 30 / 40 持仓数组合均可参与 accepted，并分别定义单票最大权重。
4. 补回 v1/v2 中必须保留的 CV、valid、test、score orientation、诊断和交易可行性约束。
5. 将 v3 的核心表现门收敛为 Sharpe、按完整季度计算的 Calmar Ratio、按完整季度计算的 Excess Calmar Ratio，以及按完整季度计算的策略超额季化收益。
6. 为后续实现 `model_acceptance_contract_v3.yml`、诊断脚本和 QA SQL 提供可执行需求。

## 3. 非目标

- 不在本 PRD 中直接修改 Cloud Run runner、acceptance.py、ADS 表或 BigQuery SQL。
- 不重新回测历史候选，不自动把此前 rejected 候选改为 accepted。
- 不把下载目录中的个人 CSV 文件作为项目事实来源；CSV 只能作为临时导出或人工审阅副本。
- 不改变策略训练数据、特征集合、模型族或 ledger 执行语义。

## 4. 核心定义

### 4.1 复合年化收益

默认年化收益使用复利口径：

```text
compound_annual_return = (ending_value / beginning_value) ^ (252 / return_period_count) - 1
```

其中：

- `return_period_count` 是有效日收益期数，通常等于 NAV 点数减 1。
- `252` 是标准年化交易日因子，不表示 A 股每个自然年固定有 252 个真实开市日。
- 样本期分母必须使用实际可用的有效日收益期数，不能写死为 252。
- 若使用指数价格，`ending_value / beginning_value` 对应期末收盘价除以期初收盘价。

### 4.2 季化 / 月化 / 日化收益

项目内后续凡提到季化、月化、日化等周期化收益，默认同样使用复利 / 几何换算：

```text
compound_period_return = (ending_value / beginning_value) ^ (target_period_count / return_period_count) - 1
```

默认约定：

- 年化：默认使用 `252` 作为标准年化交易日因子；这只是换算约定，不表示 A 股每个自然年固定有 252 个真实开市日。
- 季化：默认使用 `63` 作为标准季度交易日因子；这只是换算约定，不表示每个自然季度固定有 63 个真实开市日。
- 月化：若按交易日口径，默认使用 `21` 作为标准月化交易日因子；这只是换算约定，不表示每个自然月固定有 21 个真实开市日。
- 日化：`target_period_count = 1`。
- 样本期：`return_period_count` 必须使用该样本区间内实际可用的有效日收益期数。
- 自然月 / 自然年口径：若报告明确使用自然日历口径，必须单独标注，并说明有效期数和换算方法。

禁止在未标注的生产验收门中混用 `平均日收益率 * 252` 作为年化收益。

### 4.3 完整季度

v3 的季度门只评估完整自然季度。

完整季度定义：

- 该自然季度的第一个交易日和最后一个交易日都落在回测 / 诊断覆盖区间内。
- 策略 NAV、主 benchmark 和五个对比指数在该季度内具有可用价格 / NAV 序列。
- 若回测区间只覆盖某个自然季度的一部分，该季度不参与季度门计算。

示例：回测结束日为 `2026-04-30` 时，`2026Q2` 只覆盖 4 月，不满一个完整自然季度，因此不参与季度 Calmar、季度 Excess Calmar 或策略超额季化收益对比。

### 4.4 策略最大回撤

```text
strategy_max_drawdown = min(strategy_nav / running_peak_nav - 1)
```

策略最大回撤窗口定义为：

```text
strategy_peak_date -> strategy_trough_date
```

其中 `strategy_peak_date` 是最大回撤发生前最近一次策略 NAV 创新高日期，`strategy_trough_date` 是最大回撤谷底日期。

季度门中，策略最大回撤和对应窗口必须在同一个完整自然季度内独立计算。

### 4.5 指数同期回撤

指数同期回撤使用策略自身最大回撤窗口内的指数涨跌幅：

```text
index_same_window_return = index_close(strategy_trough_date) / index_close(strategy_peak_date) - 1
```

说明：该值可能为正，也可能为负。字段名沿用“回撤”，但计算上是指数在该窗口内的 close-to-close return。

季度门中，指数同期回撤必须使用该季度内策略季度最大回撤窗口，而不是全周期策略最大回撤窗口。

### 4.6 策略最大回撤同期超额

v3 统一使用字段名：`策略最大回撤同期超额`。

```text
strategy_max_drawdown_same_window_excess
  = strategy_max_drawdown - index_same_window_return
```

解释：

- 若该值为负，表示策略在自身最大回撤窗口内比指数更差。
- 若该值为正，表示策略在自身最大回撤窗口内比指数更抗跌，或指数涨跌幅更差。
- 旧名称 `策略同期回撤超额` 停止使用。

季度门中，该指标必须逐完整季度、逐指数单独计算。

### 4.7 策略超额季化收益

对每个完整季度 `q` 和指数 `i`：

```text
strategy_quarterized_return_q
  = (strategy_nav_q_end / strategy_nav_q_start) ^ (63 / strategy_return_period_count_q) - 1

index_quarterized_return_q_i
  = (index_close_q_end_i / index_close_q_start_i) ^ (63 / index_return_period_count_q_i) - 1

strategy_quarterized_excess_return_q_i
  = strategy_quarterized_return_q - index_quarterized_return_q_i
```

v3 要求：每个完整季度内，策略相较五个指数中至少一个指数的 `strategy_quarterized_excess_return_q_i > 0`。

### 4.8 Excess Calmar Ratio

对每个完整季度 `q` 和指数 `i`：

```text
quarter_excess_calmar_ratio_q_i
  = strategy_quarterized_excess_return_q_i
    / abs(strategy_max_drawdown_same_window_excess_q_i)
```

v3 要求：每个完整季度内，策略相较五个指数中至少一个指数的 `quarter_excess_calmar_ratio_q_i > 0.9`。

为避免跨指数拼接，同一个完整季度内，建议由同一个指数同时满足：

```text
strategy_quarterized_excess_return_q_i > 0
AND quarter_excess_calmar_ratio_q_i > 0.9
```

实现层必须显式处理分母为 0 的情况，不得静默除零或用无限大直接通过。

### 4.9 Calmar Ratio

对每个完整季度 `q`：

```text
quarter_calmar_ratio_q
  = strategy_quarterized_return_q / abs(strategy_max_drawdown_q)
```

v3 要求：

```text
每个完整季度的 quarter_calmar_ratio_q > 0.9
```

实现层必须显式处理季度最大回撤为 0 的情况，不得静默除零。

## 5. v3 验收门

### 5.1 回测范围与主基准

| 条件 | 要求 |
|---|---|
| 回测全周期 | `2024-01-02 ~ 2026-04-30` |
| 主 benchmark | 上证指数，即 `000001.SH` |
| 指数对比集合 | 上证50、沪深300、中证1000、上证指数、深证成指 |

说明：主 benchmark 上证指数 `000001.SH` 当前不在项目 BigQuery `dwd_index_eod` 可用指数集合中。v3 实现前必须先将 `000001.SH` 纳入 ODS / DIM / DWD，或在诊断 artifact 中明确记录外部行情源和覆盖性证据。

### 5.2 候选资格门

| 条件 | 要求 |
|---|---|
| 候选池 | 必须来自 `eligible signal pool` |
| 可执行性 | 必须是 `eligible executable` |
| Score orientation | `identity` 或 `reverse_probability` |
| 诊断状态 | 不能是 `signal_inverted`、`unusable_signal`、`execution_infeasible` 等硬拒绝状态 |

### 5.3 信号质量门

| 条件 | 要求 |
|---|---|
| CV 状态 | `passed` |
| Valid signal | `stable` |
| Valid RankIC | `> 0` |
| Valid top-bottom | `> 0` |
| Test RankIC | `> 0` |
| Test top-bottom | `> 0` |

说明：v3 不允许只看回测收益接受候选；valid/test 的方向性和横截面排序质量必须保留。

### 5.4 收益与风险门

| 条件 | 要求 |
|---|---|
| Sharpe | `>= 0.90` |
| Calmar Ratio | 每个完整季度的 `quarter_calmar_ratio > 0.9` |

### 5.5 Final holdout 门

| 条件 | 要求 |
|---|---|
| Final holdout 交易日数 | `>= 40` |

Final holdout 在 v3 中只作为最小样本长度门，确保最新留出段有足够交易日可供审计；不再设置 final holdout 收益、超额收益或硬 veto 门。

### 5.6 持仓数与单票最大权重

| 目标持仓数 | 单票最大权重 | v3 状态 |
|---:|---:|---|
| 10 | 15.0% | 可参与 accepted |
| 20 | 7.5% | 可参与 accepted |
| 30 | 5.0% | 可参与 accepted |
| 40 | 5.0% | 可参与 accepted |

说明：

- 10 只组合不再沿用 5% 单票上限，因为 10 * 5% 最多只能部署 50% 仓位，天然不公平。
- 20 只组合使用 7.5% 单票上限，允许在候选不足或现金约束下更接近满仓。
- 30 / 40 只组合保持 5% 单票上限。
- 所有组合仍必须通过 100 股整数手、现金、持仓唯一性、skip reason 和实际成交约束。

### 5.7 五指数季度 Excess Calmar 与超额季化收益门

对以下五个指数分别计算：

- 上证50
- 沪深300
- 中证1000
- 上证指数
- 深证成指

每个完整季度必须满足：

```text
存在至少一个指数 i，使得：
strategy_quarterized_excess_return_q_i > 0
AND quarter_excess_calmar_ratio_q_i > 0.9
```

不满完整自然季度的区间不参与该门计算。例如回测截止到 `2026-04-30` 时，`2026Q2` 不参与季度对比。

注意：

- `strategy_quarterized_excess_return_q_i` 必须使用策略和对应指数在同一完整季度内的复合季化收益相减。
- `quarter_excess_calmar_ratio_q_i` 的分母必须使用同一完整季度内的 `策略最大回撤同期超额`。
- 不允许用一个指数满足超额季化收益、另一个指数满足 Excess Calmar 后拼接通过；推荐实现为同一指数同时满足两项相对门。
- 若五个指数在某个完整季度全部不满足，则候选不得 accepted。

### 5.8 Lot / 交易可行性门

| 条件 | 要求 |
|---|---|
| 100 股整数手 | 买入必须按 100 股整数手执行 |
| odd-lot 卖出 | 清仓 odd-lot 可全额卖出；部分卖出按实现约束处理 |
| 现金约束 | 不得产生不可解释的负现金 |
| 持仓唯一性 | 每日每股持仓事实唯一 |
| skip reason | 跳过买卖必须有可审计 reason |
| 执行状态 | 不得出现未知订单状态或不可解释成交 |

v3 必须基于 lot-aware ledger 输出，不应继续使用旧 FLOAT shares reference 作为 production acceptance 依据。

## 6. 推荐实现方案

### 6.1 Contract 配置

新增版本化配置：

```text
configs/strategy1/model_acceptance_contract_v3.yml
```

建议结构：

```yaml
version: model_acceptance_contract_v3
period:
  start_date: 2024-01-02
  end_date: 2026-04-30
benchmark:
  primary: 000001.SH
  excess_calmar_indices:
    - 000016.SH
    - 000300.SH
    - 000852.SH
    - 000001.SH
    - 399001.SZ
periodization:
  method: compound
  trading_days_per_year: 252
  trading_days_per_quarter: 63
  trading_days_per_month: 21
  complete_quarters_only: true
holdings:
  - target_holdings: 10
    max_single_weight: 0.15
    acceptance_status: accepted_eligible
  - target_holdings: 20
    max_single_weight: 0.075
    acceptance_status: accepted_eligible
  - target_holdings: 30
    max_single_weight: 0.05
    acceptance_status: accepted_eligible
  - target_holdings: 40
    max_single_weight: 0.05
    acceptance_status: accepted_eligible
thresholds:
  min_sharpe: 0.90
  min_quarter_calmar_ratio: 0.90
  min_quarter_excess_calmar_ratio: 0.90
  min_quarterized_excess_return: 0.0
  min_final_holdout_days: 40
```

### 6.2 Python acceptance 逻辑

`acceptance.py` 或后续 v3 诊断脚本必须：

1. 从 contract 读取所有阈值，不在代码中硬编码。
2. 用复合年化 / 季化收益替代 `AVG(daily_return) * 252` 等简单周期化口径。
3. 识别完整自然季度；不满完整自然季度的片段不得参与季度门。
4. 输出 Sharpe、每个完整季度的 Calmar Ratio、每个完整季度 × 指数的策略超额季化收益和 Excess Calmar Ratio。
5. 对季度最大回撤、策略最大回撤同期超额为 0 的情况做显式状态标记。
6. 输出每个完整季度、每个指数的：指数同期回撤、指数季度收益、指数季化收益、策略季度收益、策略季化收益、策略最大回撤、策略最大回撤同期超额、策略超额季化收益、Excess Calmar Ratio。
7. 输出每个完整季度的通过指数；若同一季度没有任何指数同时满足超额季化收益和 Excess Calmar Ratio，则该候选 v3 rejected。

### 6.3 QA SQL

建议新增：

```text
sql/ml/strategy1/24_qa_acceptance_gate_v3_outputs.sql
```

QA 至少覆盖：

- contract hash / version 已写入 artifact 或 registry。
- 周期化收益字段标记为 compound，且没有使用 simple annualization 标记。
- `trading_days_per_quarter=63`、`complete_quarters_only=true` 来自 contract。
- 10/20/30/40 持仓数和权重来自 contract。
- 五指数 × 完整季度 Excess Calmar 明细完整。
- 不完整季度被排除，并输出排除原因。
- `策略最大回撤同期超额` 字段存在，旧字段名 `策略同期回撤超额` 不再作为主输出。
- 每个完整季度都有同一指数同时满足 `strategy_quarterized_excess_return > 0` 和 `quarter_excess_calmar_ratio > 0.9`。
- 每个完整季度的 `quarter_calmar_ratio > 0.9`。
- final holdout 交易日数输出且满足最小样本长度要求。
- lot-aware ledger 证据存在，旧 FLOAT shares reference 不得作为 production acceptance 依据。

## 8. 迁移步骤

1. 新增 v3 contract YAML。
2. 更新 acceptance shared library，增加 compound periodization helper 和完整季度识别 helper。
3. 新增只读 v3 诊断脚本，先对历史 20 个 Python Top5 候选做 replay，不写 ADS accepted。
4. 新增 v3 QA SQL，验证 artifact、季度覆盖、公式字段和阈值。
5. owner 复核 replay 结果后，再决定是否把 v3 接入 Python search 主写回门。
6. 接入主写回门前，保留 v1/v2 诊断输出，避免口径切换后失去横向可比性。

## 9. 验收标准

v3 实现完成后，至少满足：

- `model_acceptance_contract_v3.yml` 可被 runner / 诊断脚本加载并记录 hash。
- 一个历史候选 replay 能输出完整 v3 gate summary。
- 完整季度识别正确；不满完整季度的片段不参与季度门。
- 五指数 × 完整季度 Excess Calmar 明细完整且公式可复核。
- 年化 / 季化收益全部使用复合周期化收益。
- 10/20/30/40 持仓配置来自 contract，而不是代码硬编码。
- v1/v2 的信号质量门没有被回测表现门替代。
- QA SQL 失败时能明确指出缺失字段、错误口径、季度覆盖问题或未注入 contract。

## 10. 风险与注意事项

- 五指数任一满足可能放宽风格约束，因此必须按“每个完整季度、同一指数同时满足超额季化收益和 Excess Calmar Ratio”实现，避免跨指数拼接。
- 季度门会比全周期门更严格，可能拒绝全周期表现较好但某个完整季度明显失效的候选。
- 复合年化 / 季化收益切换会改变此前使用 `AVG(daily_return) * 252` 的数值，历史表格需标注旧口径或重算。
- 主 benchmark 上证指数 `000001.SH` 当前不在项目 BigQuery `dwd_index_eod` 可用指数集合中；实现前需要将其纳入 ODS / DIM / DWD，或在诊断脚本中明确使用外部行情源并记录 source。
