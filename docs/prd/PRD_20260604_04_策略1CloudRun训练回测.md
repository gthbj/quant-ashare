# PRD: 策略 1 Cloud Run 训练回测执行器

> 状态：草案，待 review / 合并。
> 范围声明：本文只定义策略 1 训练、预测、回测和实验调度迁移到 Cloud Run Jobs 的目标方案；不实现代码、不创建 GCP 资源、不执行 BigQuery、不生成或覆盖任何报告产物。
> 关联：`docs/prd/PRD_20260601_02_策略1BQML回测闭环.md`、`docs/prd/PRD_20260603_05_策略1实验并发调度与隔离.md`、`docs/prd/PRD_20260604_01_策略1LedgerV1交易执行语义.md`、`docs/prd/PRD_20260604_02_策略1月度滚动重训.md`、`docs/策略1-ml_pv_clf_v0-runner设计.md`。

---

## 1. 背景

策略 1 当前执行链路是 BigQuery ML + BigQuery SQL runner：

1. `01_build_training_panel.sql` 冻结训练 / 预测面板。
2. `02_train_bqml_logistic_candidates.sql` 用 BQML 训练候选模型。
3. `03_select_model_and_register.sql` 做模型选型和 score orientation。
4. `04_predict_daily.sql` 写预测分。
5. `05-07` 生成候选、目标组合和订单计划。
6. `08_run_backtest.sql` 在 BigQuery scripting 中逐日执行 `ledger_exec_v1`。
7. `09-15` 生成 summary、报告、诊断、因子贡献和 resume consistency QA。

这套链路已经跑通并验证了策略逻辑，但存在两个工程问题：

1. BQML 训练成本偏高，且候选模型对象管理、并发重训和清理风险较高。
2. BigQuery scripting 做日级有状态 ledger 可行但慢，长区间、多实验和 resume 对比时调度成本高。

owner 已明确希望把训练和回测都迁移为 Cloud Run Job，并要求多实验对照时：

1. 不在项目代码中设置默认并发上限。
2. 默认并发数等于本次需要运行的实验数量。
3. owner 可以显式选择并发几个实验。

本文定义 Cloud Run 版本的策略 1 执行器。它不是 OQ-005 数据采集流水线的一部分；OQ-005 负责 Tushare/Tinyshare→GCS→ODS→DWS/ADS 的生产数据链路，本文负责策略训练、预测、组合、回测和报告执行链路。

## 2. PRD 拆分决策

本文只新增一篇 PRD，不拆成“训练 PRD”和“回测 PRD”。

原因：

1. 训练、预测、回测共享 `experiment_id`、`run_id`、`prediction_run_id`、`backtest_id`、artifact 路径、状态表和并发语义。
2. score orientation、prediction stream 和 ledger 输入必须一体定义，否则容易出现训练产物和回测消费口径漂移。
3. owner 的关键要求是“训练 + 回测都做成 Cloud Run Job，并支持多实验默认全并发”，本质是统一执行器问题。

月度滚动重训仍由 `PRD_20260604_02_策略1月度滚动重训.md` 定义。本文只提供 Cloud Run 训练 / 预测执行底座，月度重训后续在该底座上实现。

## 3. 目标

### 3.1 P0 目标

1. 新增策略 1 Cloud Run Job 执行器，支持一个实验从训练到回测报告的端到端运行。
2. 用 scikit-learn 在 Cloud Run 中训练 logistic regression，替代 BQML `LOGISTIC_REG` 训练步骤。
3. 在 Cloud Run 中完成预测分生成、score orientation 应用和 `ads_model_prediction_daily` 写入。
4. 在 Cloud Run 中实现 `ledger_exec_v1` 回测，语义必须与 `PRD_20260604_01_策略1LedgerV1交易执行语义.md` 对齐。
5. 保持既有 ADS 表契约、报告脚本、诊断脚本、因子贡献度脚本和 QA 可复用。
6. 支持多实验并发调度：默认并发数为本次 manifest 中可执行实验数量；owner 可通过参数显式限制。
7. 训练模型 artifact、预处理 artifact、运行日志、报告和诊断 artifact 写入 GCS，并保留本地镜像能力。
8. 每个实验按 `experiment_id` / `run_id` / `prediction_run_id` / `backtest_id` 完整隔离，失败可重试，不串号。

### 3.2 P1 目标

1. 把 Cloud Run 执行器接入月度滚动重训 prediction stream。
2. 支持 fixed-model extended baseline、resume segment 和 monthly-retrain 三类执行模式。
3. 支持更高性能的数据读取路径，例如 BigQuery `EXPORT DATA` 到 GCS Parquet + Polars / PyArrow streaming。
4. 支持 Cloud Composer 统一编排，但不改变 Cloud Run Job 的参数、状态和 artifact 契约。

## 4. 非目标

1. 不改变 DWD/DWS 特征定义。
2. 不改变当前最优业务参数：`pv_fin_quality + 30/5% + biweekly + 5d`。
3. 不改变默认股票池：仅沪深主板，不含北交所、创业板、科创板。
4. 不改变交易成本 profile：佣金万一免五、卖出印花税 5 bps、买 / 卖滑点各 5 bps。
5. 不在 P0 切换到 XGBoost、LightGBM、深度学习或 GPU 训练。
6. 不要求 sklearn 模型与 BQML 模型逐行预测完全一致；迁移验收看契约、PIT、QA、回测语义和关键指标，不做模型对象等价。
7. 不把 ODS/DIM/DWD/DWS 生产链路搬进策略 Cloud Run Job。
8. 不把 GCP quota、预算或组织级并发限制写死进代码；这些由 GCP 项目和 owner 配置决定。

## 5. 总体架构

### 5.1 组件

P0 使用一个容器镜像，多个命令入口：

```text
strategy1-cloudrun-runner image
  ├── train_predict
  ├── backtest_report
  └── experiment_orchestrator
```

对应 Cloud Run Jobs：

| Job | 职责 |
|---|---|
| `strategy1-train-predict-job` | 读取训练面板 / 预测池，训练 sklearn logistic model，做 valid 选型和 score orientation，写 model registry / prediction ADS，上传 model artifact |
| `strategy1-backtest-report-job` | 运行候选 / 组合 / 订单步骤，执行 Python ledger 回测，写 ADS backtest 表，生成 summary、报告、诊断、因子贡献度和 QA |
| `strategy1-experiment-orchestrator-job` | 读取 experiment manifest，按依赖启动每个实验的 train/predict 和 backtest/report job，维护状态表、GCS lock、artifact 路径和并发参数 |

P0 允许 `backtest_report` 内部调用现有 `05-07` BigQuery SQL 生成候选、组合和订单计划；但 `08` 有状态 ledger 必须在 Cloud Run Python 中执行。后续如 `05-07` 成为瓶颈，可再迁移为 Python / Polars 实现。

### 5.2 数据流

```text
BigQuery DWS / ADS
  -> Cloud Run train_predict
      -> sklearn model artifact on GCS
      -> ads_model_registry
      -> ads_model_prediction_daily
  -> Cloud Run backtest_report
      -> 05-07 candidates / targets / order plan
      -> Python ledger_exec_v1
      -> ads_backtest_trade / position / nav / summary
      -> report / diagnosis / factor attribution artifacts
      -> 10 / 12 / 14 / 15 QA
```

BigQuery 仍是结构化事实来源。GCS 是模型、预处理、报告、诊断和中间缓存的 artifact 仓库。

## 6. 并发契约

### 6.1 默认并发

Cloud Run 版本执行器的默认并发规则固定如下：

```text
max_parallel_experiments 未设置或为 0
  => max_parallel_experiments = 本次 manifest 中可执行实验数量
```

例：

| 本次可执行实验数 | owner 未设置并发 | 实际计划并发 |
|---:|---:|---:|
| 1 | 未设置 | 1 |
| 3 | 未设置 | 3 |
| 19 | 未设置 | 19 |
| 24 | 未设置 | 24 |

项目代码不得再为 Cloud Run 路径设置类似 `default max_parallel=2`、`default max_parallel_backtest=1` 的隐藏限流。当前本地 BigQuery SQL 调度器的保守默认值不适用于本文定义的 Cloud Run runner。

### 6.2 owner 显式限流

owner 可显式设置：

```text
--max-parallel-experiments N
```

语义：

1. `N > 0`：同一时刻最多运行 N 个实验链。
2. `N = 0` 或不传：运行全部可执行实验。
3. 不设置单独的默认 backtest 子上限；训练、预测、回测都按同一个实验并发预算调度。
4. 若后续需要资源分级限流，只能作为 owner 显式参数，例如 `--max-parallel-train` 或 `--max-parallel-backtest`，默认仍不生效。

### 6.3 GCP quota 边界

“不限制并发数量”指项目 runner 不设置人为默认上限，不代表绕过 GCP quota、Cloud Run Job 配额、BigQuery API 配额、GCS 请求配额或预算约束。

若默认全并发触发 GCP quota / 资源不足：

1. 单个失败实验必须记录 `failed`、错误摘要和 Cloud Run execution id。
2. 其他已启动实验继续按配置执行。
3. owner 可提高 GCP quota 或用 `--max-parallel-experiments N` 重跑。
4. runner 不得静默把并发数降到内部默认值。

## 8. 训练与预测

### 8.0 scikit-learn 替代边界

scikit-learn 在 P0 中替代的是当前 BQML `LOGISTIC_REG` 的**模型训练、候选模型评价和批量预测**能力，不替代 BigQuery 数据仓库、ADS 结果表、GCS artifact、报告、诊断和 QA。

换句话说：

| 能力 | P0 替代方式 |
|---|---|
| BQML `CREATE MODEL LOGISTIC_REG` | scikit-learn `LogisticRegression` |
| BQML `ML.PREDICT` | scikit-learn batch predict + 写 `ads_model_prediction_daily` |
| BQML model object | GCS `model.joblib` + `ads_model_registry.model_uri` |
| BQML `ML.WEIGHTS` | sklearn coefficient + preprocess artifact 还原标准化系数 |
| BigQuery DWS/ADS/QA | 继续使用 BigQuery |
| BigQuery ledger `08` | Cloud Run Python `ledger_exec_v1` |

因此实现依赖不止 `scikit-learn`。P0 推荐最小 Python 依赖：

| 类别 | 推荐库 | 用途 |
|---|---|---|
| 模型 | `scikit-learn`, `numpy`, `scipy` | logistic 训练、预测、RankIC / Spearman 等统计 |
| 数据读取 / 写回 | `google-cloud-bigquery`, `google-cloud-bigquery-storage`, `google-cloud-storage`, `db-dtypes` | 读 BigQuery、写 ADS、上传 GCS |
| 列式数据 | `pyarrow`, `polars`, `pandas` | Parquet、训练矩阵、ledger 和报告数据处理 |
| 持久化 / 配置 | `joblib`, `pyyaml` 或 `ruamel.yaml`, `orjson` 或标准 `json` | 模型、预处理、manifest、metrics |
| 报告沿用 | `matplotlib`, `requests` | 复用现有中文报告 / AI 诊断链路 |

P0 不引入 XGBoost、LightGBM、Optuna、SHAP、Dask 或 Ray。它们可作为 P1+，但第一步必须先把当前 logistic baseline 迁移到 Cloud Run，并保持模型族和业务参数不变，避免把执行环境迁移和模型升级混在一起。

### 8.1 输入数据

训练输入默认来自：

```text
ashare_ads.ads_ml_training_panel_daily
```

P0 可继续由 `01_build_training_panel.sql` 生成训练面板。Cloud Run train job 不直接从 ODS 或 DWD 拼特征，避免重复实现 PIT 逻辑。

推荐 P0 数据读取路径：

1. `01_build_training_panel.sql` 在 BigQuery 生成本次 `run_id` 的训练 / valid / test / live 面板。
2. train job 用 BigQuery Storage API 或 `EXPORT DATA` 导出到 GCS Parquet。
3. Cloud Run 从 GCS Parquet 读取训练矩阵和预测池。
4. 特征矩阵使用 `float32`，保留 `sec_code`、`trade_date`、`split_tag` 和 label 作为索引 / 评价字段。

### 8.2 模型族

P0 使用 scikit-learn logistic regression：

| 项 | 推荐默认 |
|---|---|
| 模型族 | `sklearn.linear_model.LogisticRegression` |
| solver | `saga` |
| penalty | `l2` 或 `elasticnet` 候选 |
| class_weight | `None` |
| max_iter | owner 可配置，默认先用 200 |
| random_state | 固定写入 manifest |
| n_jobs | 由 Cloud Run CPU 配置决定；不在代码中写死过高 |

P0 默认 `class_weight=None`。若 reference BQML registry 记录了不同的类别权重口径，Cloud Run 训练必须在 `implementation_audit.json` 中记录该差异；`class_weight='balanced'` 会改变损失函数权重和决策边界，只能作为后续独立建模实验 A/B，不得混在执行环境迁移中默认启用。

sklearn 的正则参数网格必须重新定义，不得把 BQML `L1_REG` / `L2_REG` 数值直接翻译成 sklearn 的 `C` / `l1_ratio`。原因是 BQML 使用加性正则强度，sklearn `C` 是正则强度倒数，且 solver / penalty 约束不同。P0 推荐先定义小网格：

| candidate_id | penalty | C | l1_ratio | 用途 |
|---|---|---:|---:|---|
| `l2_c_0_1` | `l2` | 0.1 | NULL | 较强 L2 |
| `l2_c_1` | `l2` | 1.0 | NULL | 中性 baseline |
| `l2_c_10` | `l2` | 10.0 | NULL | 较弱 L2 |
| `elastic_c_1_l1_0_15` | `elasticnet` | 1.0 | 0.15 | 轻 L1 + L2 |
| `elastic_c_1_l1_0_5` | `elasticnet` | 1.0 | 0.50 | 中 L1 + L2 |

若小网格 smoke 未通过 §8.5 parity gate，下一轮必须按 sklearn 原生参数扩展网格，不写成 BQML L1/L2 的等价替代。第一轮扩展网格为：

| penalty | C | l1_ratio |
|---|---|---|
| `l2` | `0.01, 0.03, 0.1, 0.3, 1, 3, 10` | NULL |
| `l1` | `0.01, 0.03, 0.1, 0.3, 1, 3, 10` | NULL |
| `elasticnet` | `0.01, 0.03, 0.1, 0.3, 1, 3, 10` | `0.1, 0.5, 0.9` |

共 35 个候选。实现必须记录 `candidate_grid_version='sklearn_logistic_parity_v1'`、`candidate_grid_size` 和每个候选的 valid 指标，便于判断 parity 失败是参数搜索不足还是模型族差异。

若 `LogisticRegression` 在全量样本上内存或收敛不可接受，允许后续 P1 增加：

```text
sklearn.linear_model.SGDClassifier(loss='log_loss')
```

但必须使用新的 `estimator_id`，不得把 SGD 结果伪装成原 logistic baseline。

### 8.3 预处理

sklearn 不像 BQML 自动处理所有缺失和类型问题，因此 P0 必须显式持久化预处理 artifact：

```text
preprocess_version = sklearn_median_winsor_zscore_v1
```

规则：

1. 特征列白名单来自 manifest / registry，不从 DataFrame 自动猜。
2. 缺失值填充只用 train split 统计量。
3. winsor 分位点只用 train split 计算。
4. 标准化均值 / 标准差只用 train split 计算。
5. valid / test / live 只应用 train 统计量，不重新估计。
6. `feature_schema.json`、`preprocess_artifact.json` 和 model artifact 必须一起上传。

### 8.4 模型选择与方向校准

Cloud Run 训练必须保留现有模型选择语义：

1. valid 期为模型选择主依据。
2. 排序指标优先级仍为：

```text
oriented_valid_rank_ic_mean
  -> valid_topn_fwd_ret_mean
  -> roc_auc
```

3. score orientation 仍按 `PRD_20260603_01_策略1分数方向校准.md`：
   - `raw_score` 是模型正类概率。
   - `score` 是最终排序分。
   - `score_orientation` 为 `identity` 或 `reverse_probability`。
4. `ads_model_registry.metrics_json` 必须记录 raw / oriented valid RankIC、bucket lift、orientation reason 和 selected candidate。
5. `ads_model_prediction_daily` 必须同时写 `raw_score`、`score`、`score_orientation`。

### 8.5 sklearn vs BQML 模型质量对等门槛

本文不要求 sklearn 与 BQML 逐行预测一致，但必须防止“执行环境迁移”静默推翻当前 OQ-010 baseline 参数结论。

P0 必须把 Cloud Run sklearn selected model 与当前 BQML 正式 baseline 做模型质量对等检查。参考 baseline：

```text
run_id = s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01
backtest_id = bt_s1_bqml_baseline_pvfq_n30_bw_h5_v20260604_01
参数 = pv_fin_quality + 30/5% + biweekly + 5d
```

参考指标以 `ads_model_registry.metrics_json` 中的实际字段为准；当前 oriented valid RankIC 参考值约为 0.099。

P0 对等门槛：

1. `sklearn_oriented_valid_rank_ic_mean >= bqml_oriented_valid_rank_ic_mean - 0.02`。
2. `sklearn_valid_topn_fwd_ret_mean` 不得低于 BQML baseline 超过以下两者较大值：

```text
max(0.002, 0.20 * ABS(bqml_valid_topn_fwd_ret_mean))
```

3. sklearn prediction coverage 不得低于 BQML baseline 超过 5 个百分点。
4. 若任一门槛未通过，Cloud Run 迁移仍可产出 artifact 和 QA 证据，但状态必须标记为 `model_quality_not_equivalent`，不得直接声明当前 BQML baseline 参数在 sklearn backend 下仍是默认参数。
5. 对等门槛失败后，下一步应在 sklearn backend 下重新跑参数 / 模型选择实验，而不是直接沿用 BQML 结论。

`training_metrics.json` 和 `ads_model_registry.metrics_json` 必须记录：

| 字段 | 说明 |
|---|---|
| `bqml_reference_run_id` | BQML reference run |
| `bqml_reference_model_id` | BQML selected model |
| `bqml_oriented_valid_rank_ic_mean` | BQML reference valid RankIC |
| `sklearn_oriented_valid_rank_ic_mean` | sklearn selected valid RankIC |
| `rank_ic_parity_delta` | sklearn - BQML |
| `bqml_valid_topn_fwd_ret_mean` | BQML reference topN 5d 收益 |
| `sklearn_valid_topn_fwd_ret_mean` | sklearn topN 5d 收益 |
| `topn_ret_parity_delta` | sklearn - BQML |
| `prediction_coverage_parity_delta` | sklearn - BQML |
| `model_quality_parity_status` | `passed` / `warning` / `failed` |

### 8.6 Model artifact

GCS 路径：

```text
gs://ashare-artifacts/models/strategy1/ml_pv_clf_v0/run_id=<run_id>/model_id=<model_id>/
```

最小文件：

| 文件 | 说明 |
|---|---|

## 9. 回测

### 9.1 Ledger 语义

Cloud Run backtest job 必须实现 `ledger_exec_v1`，并与当前 BigQuery `08_run_backtest.sql` 语义一致：

1. `rebalance_date` 是信号日。
2. `execution_date` 是信号日后下一开市日。
3. 按开盘价 + 分项滑点成交。
4. 卖出先于买入。
5. 买入受可用现金约束，现金不足按比例缩放。
6. 对实际持仓和目标持仓做 netting。
7. 卖不出进入 pending sell，并在后续每个开市日继续卖。
8. 买不进不候补。
9. 非目标持仓继续计入 NAV。
10. 每个开市日按收盘价 mark-to-market。
11. resume 模式必须按 `p_parent_backtest_id + p_state_as_of_date` 恢复现金、持仓、active target 和 pending sell。

### 9.2 输入输出

输入：

| 数据 | 来源 |
|---|---|
| prediction | `ads_model_prediction_daily` |
| candidate / target / order plan | `ads_stock_candidate_daily` / `ads_portfolio_target_daily` / `ads_order_plan_daily` 或 Cloud Run 内生成 |
| prices / tradability | `dwd_stock_eod_price` |
| benchmark | `dwd_index_eod` + `dim_index` |
| parent state | `ads_backtest_nav_daily` / `ads_backtest_position_daily` / `ads_backtest_trade_daily` |

输出：

| 表 | 要求 |
|---|---|
| `ads_backtest_trade_daily` | 成交、跳过、pending、netting、缩放状态 |
| `ads_backtest_position_daily` | 每日实际持仓，按 `backtest_id + trade_date + sec_code` 唯一 |
| `ads_backtest_nav_daily` | 每个开市日一行 NAV |
| `ads_backtest_performance_summary` | 汇总指标、artifact、backend、resume 信息 |
| `ads_signal_monitor_daily` | 预测 / 候选 / 成交 / 跳过诊断 |

Python ledger 可以用 pandas / Polars 实现，但输出必须和 ADS schema 对齐。

### 9.3 Ledger 等价性验收

P0 必须用同一 prediction / order input 对比：

1. BigQuery `08_run_backtest.sql` fresh-start。
2. Cloud Run Python ledger fresh-start。

验收阈值：

| 项 | 阈值 |
|---|---|
| NAV 覆盖日期 | 完全一致 |
| `cash_cny` | 绝对差 <= 0.01 |
| `net_value_cny` | 绝对差 <= 0.01 |
| `nav` | 绝对差 <= 1e-8 |
| 持仓股数 | 逐日逐股差异 <= 1e-8 |
| 成交状态集合 | 无新增未登记枚举 |
| `daily_return` | 绝对差 <= 1e-8 |

若 Cloud Run ledger 有意修复 BigQuery ledger bug，必须单独记录差异原因并更新 Ledger PRD / QA；不得在迁移 PR 中静默改变交易语义。

本节只验证同一预测输入下的 ledger 执行语义等价，不验证 sklearn 模型质量是否等价于 BQML。模型质量迁移门槛见 §8.5，二者必须同时通过，才能声明 Cloud Run runner 在 P0 上可替代当前 BQML + SQL baseline 路径。

## 14. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 默认全并发触发 GCP quota | 部分实验失败 | 失败可追踪；owner 提高 quota 或显式设置 `--max-parallel-experiments` |
| sklearn 与 BQML 结果不同 | 指标变化，可能推翻 BQML 下的最优参数结论 | 不要求逐行等价；必须通过 §8.5 模型质量对等门槛，否则标记 `model_quality_not_equivalent` 并重跑 sklearn backend 参数实验 |
| Cloud Run 内存不足 | 训练失败 | GCS Parquet 分片、float32、资源配置、必要时 P1 使用 SGDClassifier |
| Python ledger 与 BigQuery ledger 差异 | 回测语义漂移 | Phase 3 必跑等价性 QA，差异必须显式记录 |
| 多实验写 ADS 串号 | 结果污染 | run/backtest 唯一性、状态表、GCS lock、`16` QA |
| artifact 路径覆盖 | 报告或模型丢失 | 路径强制包含 `run_id` / `backtest_id` / `model_id`，GCS manifest 写 hash |
| 隐式并发限流回归 | 不符合 owner 要求 | `16` QA 断言未设置并发时 resolved 值等于可执行实验数 |

## 15. 通过标准

P0 完成时必须满足：

1. Cloud Run 单实验能完成 train/predict/backtest/report/diagnosis/QA。
2. selected sklearn model、preprocess artifact、prediction、backtest、report 均可用 `run_id` / `backtest_id` 追溯。
3. sklearn selected model 通过 §8.5 的 BQML baseline 模型质量对等门槛；若不通过，只能作为迁移失败 / 需重做 sklearn 参数实验的证据。
4. `ledger_exec_v1` Python 与当前 BigQuery ledger 在固定输入下通过一致性验收。
5. `10`、`12`、必要时 `14` 和 `15` QA 通过。
6. 新增 `16_qa_cloudrun_runner_outputs.sql` 通过。
7. 多实验 manifest 未设置并发时，resolved 并发数等于可执行实验数。
8. owner 设置 `--max-parallel-experiments N` 时，调度器按 N 控制实验链。
9. 文档和 README 明确 BQML runner 保留为 reference / fallback，Cloud Run sklearn runner 是后续降成本和提速的目标执行路径。
