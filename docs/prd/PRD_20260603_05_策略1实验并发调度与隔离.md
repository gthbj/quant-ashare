## 1. 背景

策略 1 OQ-010 首轮实验已经进入阶段化执行。当前基础路径为：

```text
Stage A: 4 个持股数 / 权重实验
Stage B: 3 个调仓频率实验
Stage C: 3 个 label horizon 实验
Stage D: 2 个财务特征实验
```

Stage A 的 A0 已完成，A1 / A2 已完成，A3 正在由另一个 agent 串行执行中。当前执行方式偏串行，主要原因是多个实验共享同一组 ADS 表和报告 artifact 根路径。虽然现有 runner 已使用 `run_id` / `backtest_id` 做基本隔离，但以下行为在并发环境中仍可能造成互相污染或排查困难：

1. `p_force_replace` 清理范围不够严格时，可能误删其他实验输出。
2. 幂等检查若只按默认参数或宽条件判断，可能把其他实验的残留当作当前实验输出。
3. `ads_model_registry` 中 selected model 状态和 `prediction_run_id` 复用关系需要稳定隔离。
4. report / diagnosis 回写 ADS summary 和 GCS / local artifact 时，路径和状态必须按实验唯一定位。
5. 失败重试若没有 step 级状态，可能复制输出、覆盖输出或跳过必要 QA。
6. `08_run_backtest.sql` 是当前最慢步骤之一，也是账户级 ledger 状态最敏感的步骤。

同一阶段内的多个实验天然互不依赖。例如 Stage A 的 A1 / A2 / A3 是组合层参数对照实验，理论上可以并发执行 05-12；但在写隔离、锁、状态表和 QA 未补齐前，直接本地多进程裸跑 SQL 会放大污染风险。

本文定义一个安全并发方案，使同阶段实验能在不牺牲可复现、幂等和 QA 可追溯的前提下节省执行时间。

## 2. 问题定义

### 2.1 当前问题

当前 OQ-010 实验串行执行导致总体耗时过长。同阶段实验之间通常没有业务依赖，但它们共享：

- `ashare_ads` 下训练、预测、候选、组合、订单、回测、summary、monitor 等表。
- `ads_model_registry` 中模型选择与 selected 状态。
- GCS report / diagnosis artifact 根路径。
- 本地 `reports/strategy1/` 镜像路径。
- 同一组 runner SQL 和 Python report / diagnosis 脚本。

### 2.2 不变要求

并发提速不能破坏以下要求：

1. 每个实验结果可按 `experiment_id` / `run_id` / `prediction_run_id` / `backtest_id` 复现。
2. 每个实验的 ADS 输出、GCS artifact、本地 artifact 和 QA 结果可单独审计。
3. 失败实验可从失败 step 恢复，不复制输出，不覆盖其他实验。
4. 并发实验的 QA 必须按各自参数执行，不能只跑默认 SQL 参数。
5. 当前正在执行的 A3 实验不受本文影响；本文不要求中断、接管或重跑正在运行的实验。

## 5. 并发原则

### 5.1 唯一身份

每个并发实验必须具备唯一身份：

| 字段 | 要求 |
|---|---|
| `experiment_id` | manifest 内唯一 |
| `run_id` | ADS 输出 run 唯一 |
| `prediction_run_id` | 模型 / 预测来源 run，可等于 `run_id` 或指向上游 run |
| `backtest_id` | 回测输出唯一 |
| `stage_id` | 当前实验阶段，如 `stage_a` / `stage_b` |
| `strategy_id` | 策略身份，策略 1 默认 `ml_pv_clf_v0` |

并发实验不得共享同一个 `run_id` 或 `backtest_id`。同一 `prediction_run_id` 可以被多个 portfolio-only 实验复用，但这些实验不得改写该 `prediction_run_id` 的训练、预测和 selected model 状态。

### 5.2 ADS 写隔离

所有 ADS 写入必须严格以当前实验身份隔离：

1. 训练面板、预测、候选、组合、订单、回测、summary、monitor 写入必须带 `run_id` / `backtest_id` / `strategy_id` / `experiment_id` 中的必要字段。
2. `DELETE` / `INSERT` / `MERGE` / `ASSERT` 必须限定当前 `experiment_id`、`run_id`、`backtest_id` 或 `prediction_run_id`。
3. `p_force_replace=TRUE` 只能清理当前实验输出。
4. 幂等检查必须识别当前实验完整输出，不能被其他实验输出满足。
5. 同 stage 内并发后，各 ADS 表按 `run_id` / `backtest_id` 统计行数不得串号。

### 5.3 Registry 稳定性

`ads_model_registry` 处理规则：

1. portfolio-only 实验复用 `prediction_run_id` 时，不执行 01-04。
2. portfolio-only 实验不得重跑或改写 `prediction_run_id` 对应的 selected model。
3. portfolio-only 实验在自身 ADS summary 中记录 `prediction_run_id`、`prediction_model_id`、`score_orientation` 和来源 run。
4. retrain 实验进入 01-04 时，必须获得当前 `prediction_run_id` 的训练 / 预测锁。
5. 同一 `prediction_run_id` 的 03 / 04 训练预测相关步骤同一时间只能有一个 active step。

### 5.4 Artifact 隔离

GCS 与本地 artifact 路径必须包含 `run_id` 和 `backtest_id`：

```text
gs://<artifact_bucket>/reports/strategy1/ml_pv_clf_v0/run_id=<run_id>/backtest_id=<backtest_id>/
reports/strategy1/ml_pv_clf_v0/run_id=<run_id>/backtest_id=<backtest_id>/
```

diagnosis artifact 使用同一隔离根路径或子目录：

```text
.../model_diagnosis/
```

任何 report / diagnosis 回写 ADS summary 时，必须只更新当前 `backtest_id` 或当前 `run_id + backtest_id` 对应行。

### 5.5 QA 参数隔离

QA 必须按单实验参数执行：

1. `10_qa_runner_outputs.sql` 必须显式接收当前 `run_id` / `prediction_run_id` / `backtest_id`。
2. `12_qa_model_diagnosis_outputs.sql` 必须显式接收当前 `run_id` / `prediction_run_id` / `backtest_id` / `p_label_horizon`。
3. QA 结果必须写入状态表或日志目录，并与当前 `experiment_id` 绑定。
4. 并发执行完成后，必须额外检查同 stage 内 `run_id` / `backtest_id` 唯一性和 ADS 表串号。

## 6. 状态表与锁机制

### 6.1 状态表

P0 新增实验运行状态表。推荐放在 meta 层：

```text
ashare_meta.strategy1_experiment_run_status
```

若为了贴近现有策略 ADS 契约，也可放在：

```text
ashare_ads.ads_strategy1_experiment_run_status
```

PRD 推荐 `ashare_meta.strategy1_experiment_run_status` 作为调度事实表，理由是该表记录调度和任务状态，不是策略产出本身。

最小字段：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| `experiment_id` | STRING | 实验 ID |
| `run_id` | STRING | 当前实验输出 run |
| `prediction_run_id` | STRING | 模型 / 预测来源 run |
| `backtest_id` | STRING | 当前实验回测 ID |
| `stage_id` | STRING | 阶段 ID |
| `step_id` | STRING | runner step，如 `01_build_training_panel`、`08_run_backtest` |
| `status` | STRING | `planned` / `running` / `succeeded` / `failed` / `cancelled` |
| `started_at` | TIMESTAMP | step 开始时间 |
| `finished_at` | TIMESTAMP | step 结束时间 |
| `job_id` | STRING | BigQuery job id / local process id / Cloud task id |
| `error_message` | STRING | 脱敏错误摘要 |
| `force_replace` | BOOL | 本次是否启用 force replace |
| `artifact_uri` | STRING | 当前 step 或实验 artifact 根路径 |
| `created_at` | TIMESTAMP | 记录创建时间 |
| `updated_at` | TIMESTAMP | 记录更新时间 |

扩展字段：

| 字段 | 含义 |
|---|---|
| `runner_version` | runner 脚本版本或 commit |
| `manifest_path` | manifest 文件路径 |
| `manifest_hash` | manifest 内容 hash |
| `params_json` | 当前实验参数 |
| `qa_status` | 当前 step / 实验 QA 状态 |
| `report_uri` | report artifact URI |
| `diagnosis_uri` | diagnosis artifact URI |
| `diagnosis_status` | diagnosis 状态 |
| `attempt` | 重试次数 |
| `lock_key` | 当前 step 使用的锁 key |
| `lock_owner` | 当前锁持有者，建议为 scheduler instance id |
| `lock_acquired_at` | 锁获取时间 |
| `lock_expires_at` | 锁 lease 过期时间 |
| `last_heartbeat_at` | 调度器最近一次心跳时间 |

### 6.2 状态流转

状态流转：

```text
planned
  -> running
  -> succeeded

planned
  -> running
  -> failed
  -> running
  -> succeeded

planned
  -> cancelled

running
  -> cancelled
```

状态约束：

1. 同一 `experiment_id + step_id` 同一时间只能有一个 `running`。
2. 同一 `run_id + step_id` 同一时间只能有一个 `running`.
3. 同一 `backtest_id + step_id` 同一时间只能有一个 `running`.
4. 同一 `run_id` / `backtest_id` 同一时间只能有一个 active step 修改同一 ADS 输出族。
5. failed step resume 时，调度器必须先验证该 step 的输出是否完整；完整则可标记为 `succeeded` 并继续，部分输出必须按当前实验范围清理或覆盖。

### 6.3 锁 key

P0 锁 key：

| 锁 key | 适用步骤 | 规则 |
|---|---|---|
| `train:<prediction_run_id>` | 01-03 | 同一 `prediction_run_id` 互斥 |
| `predict:<prediction_run_id>` | 04 | 同一 `prediction_run_id` 互斥 |
| `portfolio:<run_id>` | 05-07 | 同一 `run_id` 互斥 |
| `backtest:<backtest_id>` | 08 | 同一 `backtest_id` 互斥 |
| `summary:<run_id>:<backtest_id>` | 09-10 | 同一 `run_id + backtest_id` 互斥 |
| `diagnosis:<run_id>:<backtest_id>` | 11-12 | 同一 `run_id + backtest_id` 互斥 |
| `registry:<prediction_run_id>` | 03-04 retrain | selected model 状态互斥 |

### 6.4 原子锁获取

P0 锁必须有真实原子获取机制，不能只依赖 BigQuery 状态表里的查询后写入。BigQuery 状态表用于审计、可观测和 resume 判断，不单独承担低延迟锁管理职责。

P0 推荐锁原语：

```text
GCS object create-if-not-exists:
gs://<artifact_bucket>/locks/strategy1/oq010/<lock_key>.lock
```

调度器获取锁时，使用 GCS 对象创建条件 `ifGenerationMatch=0`。该条件表示只有对象不存在时才创建成功；若对象已存在，请求失败，调度器不得进入该 step。锁对象内容写入脱敏 JSON：

```json
{
  "lock_key": "backtest:<backtest_id>",
  "experiment_id": "<experiment_id>",
  "run_id": "<run_id>",
  "prediction_run_id": "<prediction_run_id>",
  "backtest_id": "<backtest_id>",
  "stage_id": "<stage_id>",
  "step_id": "<step_id>",
  "lock_owner": "<scheduler_instance_id>",
  "acquired_at": "<timestamp>",
  "lease_expires_at": "<timestamp>"
}
```

锁获取顺序：

1. 调度器先通过 GCS `ifGenerationMatch=0` 原子创建 lock object。
2. GCS 创建成功后，写入或更新状态表，将 step 标为 `running`，并记录 `lock_key` / `lock_owner` / `lock_acquired_at` / `lock_expires_at`。
3. 状态表写入失败时，调度器必须释放刚获取的 GCS lock，并把该 step 标记为未启动或 failed。
4. GCS 创建失败时，调度器必须认为锁已被其他 active step 持有，不得执行该 step。
5. step 完成或失败后，调度器先更新状态表，再删除 lock object。

lease 与心跳：

1. 每个 lock 必须有 lease TTL，P0 默认 30 分钟，08 可配置更长 TTL。
2. 调度器运行 step 时，至少每 5 分钟更新状态表 `last_heartbeat_at`，并刷新 lock object 中的 `lease_expires_at`。
3. lock object 过期但状态表仍为 `running` 时，调度器不得自动抢占执行中的 BigQuery job；必须先检查 `job_id` 是否仍在运行。
4. 若 `job_id` 不存在或已结束，且 `last_heartbeat_at` 超过 TTL，调度器可把旧 step 标为 `failed` 或 `cancelled`，记录 `error_message='stale lock reclaimed'`，再重新获取锁 resume。
5. 手工 cancel / resume 仍保留为 P0 运维入口，但 lease / heartbeat 是默认恢复机制。

多调度器约束：

1. 若只允许单写调度器运行，仍必须有全局 scheduler lock，例如 `scheduler:strategy1:oq010`，同样用 GCS `ifGenerationMatch=0` 获取。
2. 若允许多个 agent 或多个调度器同时运行，所有 step lock 必须走上述 GCS 原子锁；不得使用“查状态表无 running 后再写 running”的非原子流程。
3. 后续若迁移到 Cloud Tasks、Firestore 或 BigQuery 多语句事务 CAS，必须保留同等语义：原子 acquire、明确 owner、lease、heartbeat、release、stale reclaim。

### 6.5 08 ledger 并发策略

`08_run_backtest.sql` 是最慢步骤之一，也是状态隔离要求最高的 ledger 步骤。P0 策略：

1. 允许不同 `backtest_id` 的 08 并发。
2. 并发前必须通过写隔离检查：08 的所有 `DELETE` / `INSERT` / `ASSERT` 都限定当前 `backtest_id`，且不会清理其他实验回测表行。
3. 并发前必须通过幂等检查：当前 `backtest_id` 的成交、持仓、NAV、summary 前置状态可独立识别。
4. 调度器提供 `max_parallel_backtest` 参数。
5. 默认 `max_parallel_backtest=1`，owner 可配置为 `2`。
6. 当 P0 QA 证明两个不同 `backtest_id` 的 08 并发无串号后，可把默认值提高到 `2`。
7. `max_parallel_backtest` 不得无限放大；BigQuery slot、job 配额和 ledger 排查复杂度是硬约束。

验收条件：

1. 两个不同 `backtest_id` 的 08 并发后，`ads_backtest_trade_daily`、`ads_backtest_position_daily`、`ads_backtest_nav_daily`、`ads_backtest_summary` 按 `backtest_id` 统计无串号。
2. 任一 08 失败时，另一个 08 的状态和 ADS 输出保持完整。
3. 失败的 08 可以按同一 `backtest_id` resume，不复制成交、持仓或 NAV 行。

## 7. SQL Runner 改造要求

本文不实现 SQL runner，但定义未来实现要求。

### 7.1 参数接口

所有 step 必须支持统一参数接口，不依赖手改脚本顶部 `DECLARE` 默认值。调度器调用方式优先级：

1. 统一参数接口。
2. 调度器生成带显式参数 block 的临时 SQL。
3. 手工复制 SQL 后改默认值仅允许用于临时 debug，不允许作为并发生产路径。

必传参数：

| 参数 | 适用范围 |
|---|---|
| `p_experiment_id` | 01-12 |
| `p_experiment_group` | 01-12 |
| `p_stage_id` | 01-12 |
| `p_run_id` | 01-12 |
| `p_prediction_run_id` | 05-12，04 之后所有引用预测来源的步骤 |
| `p_backtest_id` | 07-12 |
| `p_force_replace` | 01-12 |
| `p_resume_from_step` | 调度器级参数 |
| `p_label_horizon` | 01 / 03 / 04 / 10 / 11 / 12 |
| `p_rebalance_frequency` | 06-09 |
| `p_target_holdings` | 06-09 |
| `p_max_single_weight` | 06-09 |
| `p_feature_set_id` | 01 / 03 / 11 / 12 |

### 7.2 清理与写入范围

所有 SQL step 改造要求：

1. 所有 `DELETE` 必须限定当前实验身份。
2. 所有 `INSERT` 写入必须带当前实验身份。
3. 所有 `ASSERT` 必须限定当前实验身份。
4. `p_force_replace=TRUE` 只能清理当前 `run_id` / `backtest_id` / `experiment_id` 的输出。
5. 训练和预测步骤清理模型 registry 时，只能影响当前 `prediction_run_id`。
6. portfolio-only 实验不得清理 `prediction_run_id` 的训练面板、模型对象、registry 或预测输出。

### 7.3 幂等与 resume

幂等检查必须能回答：

| 问题 | 要求 |
|---|---|
| 当前 step 是否已完整完成 | 只检查当前 `experiment_id` / `run_id` / `backtest_id` |
| 当前 step 是否有部分输出 | 输出不完整时标记 failed 或 partial，并允许按当前实验清理 |
| 当前 step 是否可跳过 | 只有完整输出 + QA 通过时才可跳过 |
| resume 从哪里开始 | 使用状态表 `step_id` 和 `status` 决定 |

失败重试要求：

1. 支持从指定 step 继续。
2. resume 时不复制输出。
3. resume 时不覆盖其他实验输出。
4. retry attempt 写入状态表。
5. retry 后 QA 仍按当前实验执行。

## 8. 调度器设计

### 8.1 脚本入口

新增调度器脚本：

```text
scripts/strategy1/run_oq010_experiments.py
```

### 8.4 依赖规则

依赖规则：

1. 同 stage 内 portfolio-only 实验默认互不依赖。
2. retrain 实验必须先完成 01-04，再进入 05-12。

### 8.7 失败策略

失败策略：

| 策略 | 行为 |
|---|---|
| `fail_fast=true` | 任一实验失败后，不再提交新实验；已运行实验继续到当前 step 结束 |
| `fail_fast=false` | 单实验失败不阻断其他实验；失败实验状态保留 |
| `resume=true` | 从状态表读取 failed / planned / partial step 继续 |
| `force_replace=true` | 只清理当前实验输出后重跑 |

失败实验保留状态供 resume。调度器不得自动删除 GCS / local artifact；清理必须通过当前实验路径和 `force_replace` 规则执行。

## 11. 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| BigQuery slot / job 配额消耗上升 | 并发实验可能拖慢其他任务 | `max_parallel` 和 `max_parallel_backtest` 默认 conservative |
| 08 ledger 并发排查复杂 | 回测失败更难定位 | P0 默认 `max_parallel_backtest=1`，验收后由 owner 提高到 2 |
| `p_force_replace` 范围错误 | 误删其他实验输出 | 清理语句必须限定当前实验身份，并纳入 QA |
| registry selected 状态被覆盖 | portfolio-only 实验复用预测源失真 | 复用 `prediction_run_id` 时禁止改写 registry |
| artifact 路径覆盖 | report / diagnosis 难以追溯 | 路径强制包含 `run_id` / `backtest_id` |
| 失败 step 部分输出 | resume 后重复或串号 | 状态表记录 step 状态，resume 前做完整性检查 |
| 并发调度器本身失败 | 实验状态停在 running | stale running 检查和手工 cancel / resume 流程 |

并发不等于无限并行。默认配置必须保守，实际并发数由 owner 根据 BigQuery 配额、实验重要性和排查成本调整。

## 12. 当前约束与过渡规则

在本文实现前，现有约束继续有效：

1. 不并发执行同一 `run_id` 的 BQML 训练步骤。
2. 清理模型对象前必须确认无 RUNNING job。
3. 不直接本地多进程裸跑 SQL。
4. 不碰正在运行的 A3 实验。
5. 不删除或覆盖 `reports/strategy1` 下已有产物。

本文实现后，允许并发的前提是：

1. 状态表上线。
2. 调度器上线。
3. SQL runner 满足参数化、写隔离和幂等要求。
4. 并发 QA 通过。
5. owner 配置 `max_parallel` / `max_parallel_backtest`。
