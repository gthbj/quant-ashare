# PRD：Agent 记忆体系归档压缩

> 状态：草案，待 Codex review 收敛后定稿。
> 范围声明：只动 `.agent/memory/**` 与其归档，外加 `docs/prd/`（本文）。不改任何代码 / SQL / 配置 / TODO 实质内容，不触碰 BigQuery / Cloud Run。
> 关联：2026-06-12 重构评估（owner 已采纳）；`UPDATE_PROTOCOL.md` §归档与瘦身；既有归档先例 `archive/AGENT_HANDOFF_2026-05.md` / `_2026-06.md` / `CLOSED_QUESTIONS.md`。

---

## 1. 背景与问题

CLAUDE.md / AGENTS.md 要求每会话开始必读 `MEMORY_INDEX` + `PROJECT_CONTEXT` + `IMPLEMENTATION_STATUS` + `KNOWN_CONSTRAINTS` + `OPEN_QUESTIONS`，每任务开始另读 `AGENT_HANDOFF`。实测 `main@d45560d`（2026-06-12）：

| 文件 | 行数 | 字节 | 问题 |
|---|---|---|---|
| `IMPLEMENTATION_STATUS.md` | 686 | 145,264 | 90% 是 58+ 节「### 最新补充（日期）」追加式编年史，与 `AGENT_HANDOFF.md` 职责重复（同一事件双份详记）；自身定位本应是「整体状态快照」 |
| `AGENT_HANDOFF.md` | 2,010 | 154,532 | `UPDATE_PROTOCOL` §归档与瘦身规定「只保留摘要 + 最近 2-3 条」，实际累积 20+ 条，规则存在但未被执行 |
| `DECISION_LOG.md` | 3,185 | 228,168 | 无任何归档轮转；按 `MEMORY_INDEX` 非每会话必读，但「涉及关键取舍时」读取的成本已不可控 |
| `KNOWN_CONSTRAINTS.md` | 122 | 52,448 | 单行最长 5,106 字符，多条约束把操作性红线与历史叙事 / 论证黏在同一行 |
| `PROJECT_CONTEXT.md` | 77 | ~12,000 | 「当前阶段」小节已长成第三份编年史摘要，与 `IMPLEMENTATION_STATUS` 重复 |

每会话强制必读集合计约 **340KB**（约 8–10 万 CJK token），占主流 200K 上下文窗口的 4–5 成，直接侵蚀本仓库所有 agent 会话（多 agent 高频迭代，2026-06-10 至 06-12 三天 24 条 handoff）的可用上下文。该体量两天内净增约 44KB（296KB → 340KB），只增不减的趋势已确认。

`UPDATE_PROTOCOL.md` §归档与瘦身现有三条规则未覆盖 `IMPLEMENTATION_STATUS` 与 `KNOWN_CONSTRAINTS` 这两个最重的必读文件，也没有规定任何触发节奏，导致瘦身从未发生。

## 2. 目标

1. 必读集（5 必读文件 + `AGENT_HANDOFF`）总体积压到 **≤ 120KB**（参考值，作为验收报告项而非阻断门；若 KNOWN_CONSTRAINTS 保守压缩后超预期，如实报告即可）。
2. **零信息丢失**：所有历史内容原样搬入 `archive/`，所有 `DECISION-YYYYMMDD-NN` 编号在非 archive 主文件中仍可 grep 命中（索引行），所有操作性约束语义逐条保留。
3. `UPDATE_PROTOCOL.md` 补上缺失的瘦身规则与触发节奏，防止回涨。

## 3. 方案

### 3.1 `IMPLEMENTATION_STATUS.md` → 重写为纯快照

- 新结构：`Last updated` + 「当前状态快照」（按域分小节：数仓分层与数据覆盖 / 采集与调度 / Strategy1 执行层与 baseline / QA 与契约 / 开放主线，每域若干 bullet，写**现状**而非过程）+ 「最近补充」区（只保留最近 **7 个自然日**内的小节）。
- 全部更早的「### 最新补充」小节**原样**（不改写、不删减）按月搬入 `archive/IMPLEMENTATION_STATUS_2026-05.md`、`archive/IMPLEMENTATION_STATUS_2026-06.md`（倒序保持原顺序）。
- 快照区内容从现有编年史中提炼，提炼时只做归并陈述，不引入新结论；每条快照 bullet 涉及具体数字 / id（如 baseline run id、CAGR）时必须与最新编年史小节一致。

### 3.2 `AGENT_HANDOFF.md` → 执行既有规则

- 顶部「当前交接摘要」+ 最近 **3 条**交接保留；其余条目原样追加到 `archive/AGENT_HANDOFF_2026-06.md`（已存在，append 保持时间顺序）。

### 3.3 `DECISION_LOG.md` → 索引 + 月度归档

- 新建 `archive/DECISION_LOG_2026-05.md`、`archive/DECISION_LOG_2026-06.md`，按月原样搬入完整决策条目。
- 主文件保留两部分：
  1. **决策索引**：每条一行 `DECISION-YYYYMMDD-NN | 日期 | 一句话标题 | active/superseded(被谁) | archive 文件名`，覆盖**全部**历史决策——保证任何文档 / 代码注释 / 正则引用某 DECISION id 时，主文件 grep 必命中并指向归档全文。
  2. **近期完整条目**：最近 **30 天**内的决策保留全文（当前即 2026-05-31 之后的条目；首次执行时以此切线，之后按协议滚动）。

### 3.4 `KNOWN_CONSTRAINTS.md` → 保守重构（review 重点）

- **不删除任何操作性语义**。这是本 PRD 红线：每一条现存的「必须 / 不得 / 只能 / fail-fast / 默认值」语句逐条保留。
- 允许的操作仅三类：
  1. 把千字黏行**拆分**为小节标题 + bullet（纯排版）；
  2. 把**历史叙事 / 论证 / 实跑记录**（如「2026-06-04 已用 INFORMATION_SCHEMA 复核…」「事故经过：…」）替换为一行指针：`详见 <PRD/DECISION 编号> / IMPLEMENTATION_STATUS archive`——前提是指向的文档确实包含该内容，否则原文保留；
  3. 合并字面重复的条款（PR #199 拆黏行后若仍有残留）。
- 实现 PR 必须附**语义保全清单**：逐条列出被移走的叙事段及其指针落点，供 review 核对。

### 3.5 `PROJECT_CONTEXT.md` → 「当前阶段」瘦身

- 「当前阶段」小节压缩为 ≤ 10 行：核心已完成里程碑一句话 + 当前研究 baseline 一句话 + 开放主线指针（指向 `OPEN_QUESTIONS` / `IMPLEMENTATION_STATUS` 快照区）。历史 bullet 不需归档（内容已在 IMPLEMENTATION_STATUS 编年史中），直接删除重复。

### 3.6 `UPDATE_PROTOCOL.md` → 新增瘦身规则

在 §归档与瘦身追加：

1. `IMPLEMENTATION_STATUS.md` 只保留快照区 + 最近 7 天「最新补充」；任何会话在追加新小节时，顺带把超期小节搬入对应月份 archive，并同步快照区。
2. `DECISION_LOG.md` 主文件 = 全量索引行 + 最近 30 天全文；追加新决策时同时写索引行；超期条目搬入月度 archive。
3. `KNOWN_CONSTRAINTS.md` 新增 / 修改约束时只写操作性语义 + 出处指针（PRD / DECISION 编号），不内嵌论证与实跑记录；单条约束以一个短段落为上限。
4. `AGENT_HANDOFF.md` 规则不变，但明确执行时机：追加新条目时若主文件超过摘要 + 3 条，当场归档最旧条目。

### 3.7 `MEMORY_INDEX.md`

- 登记新增 archive 文件；`IMPLEMENTATION_STATUS` / `DECISION_LOG` 行的「何时更新」补一句滚动归档义务。

## 4. 不做什么（Out of scope）

- 不动 `OPEN_QUESTIONS.md`（现仅 2 条 open）、`GLOSSARY.md`、`DOC_CONVENTIONS.md`、`templates/`。
- 不改 `CLAUDE.md` / `AGENTS.md`（必读文件清单与路径不变）。
- 不改写任何归档内容（archive 内容 = 原文搬运）。
- 不动代码 / SQL / TODO.md（TODO 由本批三个 PRD 的常规记忆同步覆盖，不属于本 PRD 改动面）。

## 5. 验收

1. **体积对比表**：PR body 给出 6 个文件前后行数 / 字节及必读集合计。
2. **DECISION id 追溯**：`grep -o 'DECISION-[0-9]\{8\}-[0-9]\{2\}' .agent/memory/DECISION_LOG.md | sort -u` 的集合 ⊇ 改动前主文件同口径集合（脚本输出贴 PR）。
3. **约束语义保全**：§3.4 语义保全清单逐条核对（Codex review + Claude 复核双签）。
4. **编年史无损**：archive 文件小节数之和 + 主文件保留小节数 = 改动前小节总数（`grep -c '^### 最新补充'` 前后对账，贴 PR）。
5. 既有 pytest 全量通过（记忆文件被 `analyze_topdown_lot_phase0.py` resolver 及其 fixture 测试消费——重排后 `tests/strategy1/test_topdown_lot_phase0.py` 必须通过；若 resolver 对主文件结构有隐含依赖，以测试为准修正搬运边界，**不得修改 resolver 代码**，必要时在主文件保留 resolver 所需的 baseline 段落并在 PR 说明）。

## 6. 风险与红线

- 红线：不丢任何 DECISION 编号追溯性；不丢任何操作性约束语义；不把归档当删除。
- 风险：`analyze_topdown_lot_phase0.py` 默认 baseline resolver 从记忆 markdown 解析 run/backtest id（DECISION 语义段优先）。§5.5 已设为硬验收；实现时先跑该测试确认边界。
- 风险：并行会话可能在实施期间追加新 handoff / 决策——实现 PR 合并前 rebase 并把新增条目按同规则处置。
