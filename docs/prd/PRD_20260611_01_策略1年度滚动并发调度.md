# PRD：策略 1 年度滚动并发调度

> 状态：草案，待 owner review。
> 范围声明：本文只定义年度滚动执行调度改造方案；不改模型、不扩候选池、不执行 BigQuery / Cloud Run、不生成或覆盖实验产物。
> 关联：`docs/prd/PRD_20260610_03_策略1年度滚动选参.md`、`docs/prd/PRD_20260610_04_策略1年度滚动执行工程化.md`、`configs/strategy1/annual_rolling_lgbm_regression_v0.yml`、`.agent/memory/OPEN_QUESTIONS.md` OQ-010。

---

## 1. 背景

当前年度滚动执行路径按年份串行推进：

```text
2021 build panel -> prepare matrix -> 11 candidate fanout -> select/register/predict -> yearly diagnostic backtest
2022 build panel -> prepare matrix -> 11 candidate fanout -> select/register/predict -> yearly diagnostic backtest
...
```

每个年度内部的候选训练不是串行的。`strategy1-train-candidate-fanout-job` 会按 `--tasks=N` 拉起多个 Cloud Run task 并行训练候选；当前年度滚动候选池为 11 个 LightGBM regression candidate，配置中的 `candidate_parallelism=11`，正式 fanout job 的 job spec 上限为 `parallelism=20`。

问题在于年份之间串行，导致每年都会被最慢候选拖尾。2026-06-10 实跑中，`unit_index=6` 对应候选：

```text
risk_lgbm_prd_attack_lr005_n600_l63_lr005_n600_leaf800_ff07_bf09_l1_1_l2_1
```

该候选 `n_estimators=600`、`num_leaves=63`、`num_threads=1`，在多个年份明显慢于其它候选。串行年度链路必须等该候选完成后才启动下一年，浪费了已经释放的 Cloud Run 并发容量。

但直接把 6 年全部 fanout 同时打开也不可行：

```text
6 年 * 11 candidate = 66 candidate task
```

当前资源口径最多支持约 20 个 candidate task 并发，且每个 LightGBM candidate task 为 `2 CPU / 8Gi`。因此需要的是跨年度流水线调度，而不是无上限并发。

## 2. 目标

P0 目标：

1. 允许不同年度的可独立阶段流水线并发，避免上一年慢候选阻塞下一年训练。
2. 保持全局 candidate task 并发不超过 owner 指定上限，默认 `20`。
3. 保持资源预算可审计：candidate task、prepare、select 和 backtest/report 都要进入调度资源模型。
4. 本年度 `select_register_predict` 必须等待本年度所有候选完成，不能因为慢候选未完成就提前选参。
5. 年度 diagnostic backtest 不阻塞后续年份训练。
6. 最终 official 结果仍必须来自单一 continuous ledger，或通过 resume-continuous QA 的 segment ledger。
7. 调度器必须可恢复、可重入、可只读 dry-run，并能输出每个年度 / 每个 candidate unit 的状态。

P1 目标：

1. 支持按历史运行时长对候选排序，把慢候选优先提交，减少尾部等待。
2. 支持基于 Cloud Run / BigQuery 最近配额状态动态调整并发。
3. 支持失败 candidate 的显式重试策略，但不得自动覆盖 succeeded artifact。

## 3. 非目标

本 PRD 不做以下事情：

- 不改变 11 个年度滚动 LightGBM regression 候选池。
- 不改变 valid hard filter、排序规则或 acceptance gate。
- 不把 `unit_index=6` 排除或降级；慢不等于无效。
- 不把本年度未完成候选排除后提前选参。
- 不把年度 fresh-run NAV 拼接为正式结果。
- 不提高 Cloud Run quota，不改 job spec，不改 IAM。
- 不做真实 experiment promotion。

## 4. 核心约束

### 4.1 依赖约束

年度滚动的依赖 DAG：

```text
build_training_panel(year)
  -> prepare_matrix(year)
  -> train_candidate_unit(year, unit=0..10)
  -> select_register_predict(year)
  -> yearly_diagnostic_backtest(year)
```

跨年度依赖：

- `build_training_panel(year+1)` 不依赖 `select_register_predict(year)`。
- `prepare_matrix(year+1)` 不依赖 `train_candidate_unit(year, *)`。
- `train_candidate_unit(year+1, *)` 不依赖 `train_candidate_unit(year, *)`。
- `select_register_predict(year)` 只依赖本年全部候选完成。
- `yearly_diagnostic_backtest(year)` 只依赖本年 prediction 完成，不阻塞其它年份训练。
- `continuous_ledger` 依赖所有年度 prediction stream 或全部 resume segment 产物完成。

因此允许的并发示例：

```text
2025 unit6 still running
2026 build_training_panel running
2026 prepare_matrix running
2026 candidate unit0..10 running, if global resource budget allows
```

不允许的并发示例：

```text
2025 unit6 still running
2025 select_register_predict starts
```

因为这会把 unit6 从 2025 年度选参中隐式排除，破坏实验口径。

### 4.2 资源约束

P0 默认资源预算：

| 资源 | 默认上限 | 说明 |
|---|---:|---|
| `candidate_task_slots` | `20` | 同时 running 的 LightGBM candidate task 数 |
| `cloudrun_cpu_tokens` | `40` | 以 vCPU 计；所有 Cloud Run stage 共享 |
| `cloudrun_memory_gib_tokens` | `160` | 以 GiB 计；所有 Cloud Run stage 共享 |
| `active_fanout_executions` | `4` | 防止碎片化 execution 过多 |
| `active_prepare_jobs` | `1` | prepare matrix 主要消耗 BigQuery / GCS，P0 保守串行 |
| `active_select_jobs` | `1` | select/register/predict 写 research 表，P0 保守串行 |
| `active_backtest_jobs` | `1` | yearly diagnostic backtest 低优先级，P0 保守串行 |

预算不是 Cloud Run job spec 的替代品。Cloud Run `parallelism=20` 只限制单个 job execution；多个 execution 同时启动时仍可能叠加超过总资源。因此 orchestrator 必须自己做全局资源调度。

P0 stage token 默认值：

| Stage | CPU token | Memory token | Candidate task slot | 说明 |
|---|---:|---:|---:|---|
| `train_candidate_unit` | `2` | `8Gi` | `1` | LightGBM candidate task |
| `prepare_matrix` | `8` | `32Gi` | `0` | 现有 prepare matrix job spec |
| `select_register_predict` | `4` | `16Gi` | `0` | 写 research registry / prediction |
| `yearly_diagnostic_backtest` | `4` | `16Gi` | `0` | 年度 fresh diagnostic backtest/report |

CPU / memory token 是跨 stage 共享池，不是 candidate 专用池。举例：20 个 candidate task running 时已经占满 `40 vCPU / 160Gi`，即使 `active_prepare_jobs=0`，也不得再启动 `prepare_matrix`，否则会形成 `48 vCPU / 192Gi` 的实际需求。调度器判断 ready task 时必须同时检查 stage 并发上限、candidate slot、CPU token 和 memory token。

### 4.3 写入隔离约束

每个年度必须使用独立：

- `run_id`
- `matrix_id`
- `prediction_run_id`
- `backtest_id`
- GCS artifact prefix

并发执行不得共享同一年度的可写 artifact 路径。`--force-replace` 只能用于同一 run 的显式重跑，不得在 scheduler 自动恢复中默认开启。

### 4.4 Scheduler 互斥约束

同一个 `scheduler_run_id` / 年度滚动实验版本在同一时间只能有一个 active scheduler 实例。

P0 必须复用现有 Cloud Run orchestrator / `pipeline_control` 已验证的 GCS generation-guarded lease lock 语义：

- acquire 使用 `if_generation_match=0`，只有第一个实例能创建 scheduler-level lock。
- lock payload 至少记录 `scheduler_run_id`、`lock_owner`、`plan_hash`、`started_at`、`lease_expires_at`、`last_heartbeat_at`。
- 主循环按固定间隔 heartbeat；heartbeat 读取和写回 lock object 必须使用当前 object generation precondition。
- heartbeat 发现 generation ownership 丢失时，实例必须停止提交新的 Cloud Run execution，只允许只读 reconciliation 后退出。
- stale lock 只能在 lease 过期且 lock 中记录的 Cloud Run executions 均已进入 terminal 状态后回收。

建议 lock key：

```text
annual_pipeline:{candidate_set_id}:{start_year}:{end_year}:{run_version}
```

Scheduler-level lock 不替代 per-stage / per-unit idempotency。即使只有一个 scheduler 实例，也必须对每个 `(year, stage, unit_index)` 做状态 skip，防止恢复过程重复提交已成功 task。

## 5. 调度设计

### 5.1 新入口

建议新增稳定入口：

```text
quant_ashare.strategy1.annual_pipeline_scheduler
```

CLI 示例：

```bash
python -m quant_ashare.strategy1.annual_pipeline_scheduler \
  --start-year 2021 \
  --end-year 2026 \
  --as-of-date 2026-06-09 \
  --run-version vYYYYMMDD_NN \
  --global-candidate-task-limit 20 \
  --global-cpu-limit 40 \
  --global-memory-gib-limit 160 \
  --dry-run
```

兼容层可临时提供：

```text
scripts/strategy1_cloudrun/orchestrate_annual_rolling_pipeline.py
```

但 active runbook 和后续新引用应优先使用 package entrypoint，避免重新引入已退役的 job wrapper 命名债。

### 5.2 状态模型

P0 scheduler 必须维护状态，不得只依赖本地 stdout。

最低状态字段：

| 字段 | 说明 |
|---|---|
| `scheduler_run_id` | 本次调度 run |
| `backtest_year` | 年份 |
| `stage` | `panel` / `matrix` / `candidate` / `select` / `diagnostic_backtest` / `continuous_ledger` |
| `unit_index` | candidate 阶段必填 |
| `candidate_id` | candidate 阶段必填 |
| `status` | `planned` / `running` / `succeeded` / `failed` / `skipped` |
| `cloudrun_execution` | Cloud Run execution name |
| `started_at` / `completed_at` | UTC timestamp |
| `resource_tokens` | 本阶段声明的 CPU / memory / task slots |
| `artifact_uri` | GCS artifact prefix 或关键输出 URI |
| `error_summary` | 失败摘要，不写原始敏感日志 |
| `state_generation` | 最近一次写入 GCS state JSON 的 object generation |
| `scheduler_lock_generation` | 当前 scheduler-level lock generation；用于判断是否仍拥有提交权 |

状态落点建议：

1. GCS generation-conditioned state JSON：用于本地恢复和人工审计。
2. `research_experiment_run_status` 或后续专用 scheduler state 表：用于跨机器恢复和查询。

P0 可先写 GCS + local mirror；如果使用 BigQuery state 表，必须遵守 `ashare_research` research-first 约束。

GCS state 写入必须使用 generation precondition：

- 首次创建 state object 使用 `if_generation_match=0`。
- 更新 state 前先读取当前 object generation。
- 写回 state 时使用 `if_generation_match=<current_generation>`。
- generation mismatch 时必须重新读取、reconcile running executions 和 GCS artifacts 后再决定下一步，不得 blind overwrite。

禁止把无条件 overwrite 的 GCS JSON 当作 authoritative scheduler state。无条件 local mirror 只能作人工审计副本，不能驱动恢复提交。

### 5.3 Stage 队列

Scheduler 把每个年度拆成可调度 task：

```text
panel:y2021
matrix:y2021
candidate:y2021:u0
...
candidate:y2021:u10
select:y2021
diagnostic_backtest:y2021
```

每轮循环：

1. 刷新所有 running Cloud Run execution 和 task 状态。
2. 标记完成 / 失败 task。
3. 释放已完成 task 的资源 token。
4. 根据依赖 DAG 找出 ready task。
5. 按优先级、scheduler-level lock ownership 和资源预算提交新 task。
6. 写状态快照。

### 5.4 Fanout 提交方式

现有 `train_candidate_task` 已支持：

```text
--task-index-offset
--tasks
```

P0 可以复用这个能力，把一个年度的 11 个候选拆成一个或多个 Cloud Run execution。

推荐策略：

1. 若空闲 slot >= 11，提交整年 11 task。
2. 若空闲 slot < 11 但 >= `min_candidate_batch_size`，提交部分 task。
3. 默认 `min_candidate_batch_size=2`，避免过多 1-task execution。
4. 若只剩慢候选或尾部填缝，允许提交 1-task execution。
5. 同一 `(year, unit_index)` 已有 `succeeded` 状态时不得重复提交。

示例：

```text
t0: submit 2021 units 0..10                    active=11
t1: 2021 有 8 个完成，unit6 等慢候选仍 running     active=3
t1: submit 2022 units 0..10                    active=14
t2: 2021 unit6 still running, 2022 释放 6 slots  active=8
t2: submit 2023 units 0..9                     active=18
```

该策略不会超过 20 个 candidate task，同时不会让某一年的慢候选阻塞后续年份候选训练。

### 5.5 优先级

P0 默认优先级：

1. 补齐已开始年份的剩余 candidate units。
2. 启动下一年度 candidate units。
3. 对已 11/11 成功的年份启动 `select_register_predict`。
4. 启动下一年度 `panel` / `matrix`，前提是不会挤占 candidate 资源。
5. 启动 yearly diagnostic backtest。
6. 所有年度 prediction 完成后启动 continuous ledger。

“不会挤占 candidate 资源”必须落实为共享 token 检查，而不是文字优先级。例如 `candidate_task_slots` 还有空位但 CPU / memory token 已满时，不得启动 `prepare_matrix`；CPU / memory token 有空位但 scheduler-level lock 已失效时，也不得提交任何新 execution。

P1 可调整为慢候选优先：

```text
unit6 -> unit1/unit4/unit10 -> other units
```

但 P0 不要求按历史慢候选重排。P0 只要求全局资源不空转。

## 6. 与现有年度 orchestrator 的关系

当前 `orchestrate_annual_rolling_selection.py` 已能生成 resolved annual plan，但它输出的是按年度顺序排列的 command plan，适合人工或串行脚本执行。

并发 scheduler 不应复制窗口生成逻辑。P0 应复用现有能力：

- 年度窗口生成
- `Experiment` payload 生成
- `matrix_id` / `matrix_uri` 规则
- training panel 参数生成
- `build_task_fanout_steps` 或等价 command builder
- continuous ledger metadata

新增部分只负责：

- 把 command plan 转成跨年度 DAG。
- 按资源预算提交 Cloud Run execution。
- 追踪每个 candidate unit 的完成状态。
- 在每年候选完整后触发 select。
- 在所有年度 prediction 完成后触发 official continuous ledger 或 resume-continuous 方案。

## 7. Continuous ledger 规则

年度 diagnostic backtest 可以在本年度 prediction 完成后运行，但它不是正式结果。

正式结果只接受两条路径：

### 路径 A：单一 continuous backtest

1. 每年独立训练 / 预测。
2. 把各年度 prediction stream 合并为一个 synthetic continuous prediction run。
3. 从 `2021-01-04` fresh-start 跑到 `2026-06-09`。
4. 只生成一个 official continuous backtest id。

### 路径 B：resume-continuous segment

1. 2021 fresh-start。
2. 2022 从 2021 年末真实 ledger state resume。
3. 逐年 resume 到 2026。
4. 每段必须通过 resume continuity QA。
5. 最终报告必须明确这是 resume-continuous，不是年度 fresh NAV 拼接。

Scheduler 必须在状态和报告中把以下三类结果分开：

- 年度选参 / 年度 prediction。
- 年度 fresh diagnostic backtest。
- official continuous ledger result。

## 8. Failure / Resume 语义

### 8.1 Candidate task

Candidate task 成功的最低证据：

- Cloud Run task `Completed=True`。
- GCS `task_status.json` 存在且 `status='succeeded'`。
- `candidate_metrics.json` 存在且包含 expected `candidate_id` / `unit_index` / `matrix_id`。

Cloud Run 控制面可能短时滞后。Scheduler 不得只靠 `gcloud --wait` 返回码判断失败；必须在异常时用 execution describe / task list / GCS artifact 做二次确认。

### 8.2 Select

`select_register_predict(year)` 启动前必须检查：

```text
count(succeeded candidate units for year) == work_unit_count
```

任何 `failed` / `running` / `missing` candidate 都必须阻止 select。

### 8.3 Retry

P0 默认：

- Cloud Run job spec 仍保持 `maxRetries=0`。
- Scheduler 不自动重试 failed candidate。
- owner 或 operator 显式传 `--retry-failed-candidates` 后，只重提 failed / missing unit。
- 不重提 succeeded unit，除非显式 `--force-replace`.

### 8.4 本地进程中断

Scheduler 进程中断后，再次启动必须：

1. 读取 state。
2. 查询仍 running 的 Cloud Run executions。
3. 查询 GCS artifact 确认 candidate 成功状态。
4. 只提交缺失或失败且被允许重试的 task。

不得因为本地 `gcloud --wait` 超时而直接重跑同一 execution。

## 10. Runbook 草案

P0 执行顺序：

1. 运行 research schema readiness QA。
2. 生成年度 resolved plan dry-run。
3. 运行 pipeline scheduler dry-run，确认 DAG 和资源峰值。
4. 获取 scheduler-level GCS lease lock，并启动 heartbeat。
5. 启动 pipeline scheduler。
6. 观察 candidate task 全局并发不超过 `20`，且 CPU / memory 共享 token 不超过 `40 / 160Gi`。
7. 每年 11/11 candidate 成功后自动启动本年 select/register/predict。
8. 年度 diagnostic backtest 可延后批量执行。
9. 全部年度 prediction 完成后，执行 official continuous ledger。
10. 跑 continuous ledger / resume continuity QA。
11. 汇总年度选参表、diagnostic 年度结果和 official continuous result。

## 11. 验收标准

| 验收项 | 要求 |
|---|---|
| dry-run DAG | 输出 2021-2026 全部 stage / unit，依赖关系正确 |
| 资源计划 | dry-run 显示每个 stage token，峰值 candidate task <= 20、CPU <= 40、memory <= 160Gi |
| 跨阶段资源池 | prepare/select/backtest 与 candidate 共用 CPU / memory token；20 个 candidate running 时不得启动 prepare |
| scheduler 互斥 | 两个实例不能同时提交同一 scheduler run；第二实例必须 fail-fast 或只读 reconciliation |
| state generation guard | state 更新必须使用 generation precondition；generation mismatch 触发重新读取和 reconciliation |
| 不提前 select | 任一年 candidate 未 11/11 succeeded 时 select 不会启动 |
| 跨年度流水线 | 上一年慢 candidate running 时，下一年 ready candidate 可在资源空位内启动 |
| 状态恢复 | 本地进程中断后可从 state + Cloud Run + GCS artifact 恢复 |
| gcloud wait 超时处理 | `gcloud --wait` 或轮询超时不会误判已成功 execution 为失败 |
| research-first | 默认写 `ashare_research`，ADS 写入仍只走 promotion |
| continuous ledger | 正式结果来自单一 continuous ledger 或 resume-continuous QA |
| 年度 diagnostic | 年度 fresh backtest 明确标记为 diagnostic，不阻塞后续年份训练 |

## 12. 风险与控制

| 风险 | 控制 |
|---|---|
| 两个 scheduler 实例并发运行导致重复提交 | Scheduler-level GCS generation-guarded lease lock、heartbeat、lost-ownership 停止提交、stale reclaim |
| GCS state 无条件覆盖导致恢复状态倒退 | GCS state JSON 使用 generation-conditioned create/update；generation mismatch 后重新读取并 reconcile |
| 多个 fanout execution 合计超过资源上限 | Scheduler 维护全局 task / CPU / memory token，不只依赖 Cloud Run `parallelism` |
| candidate 已占满 CPU / memory 时又启动 prepare | prepare/select/backtest 与 candidate 共用资源池；ready task 必须通过 stage token 检查 |
| Cloud Run task 状态滞后导致误判空闲 slot | P0 用 task list + GCS artifact 双确认；可配置 safety buffer |
| 已成功 candidate 被重复提交 | 状态和 GCS `task_status.json` 双重 skip；默认不重跑 succeeded unit |
| 单年度 select 漏掉慢候选 | select 前强制 `work_unit_count == succeeded_count` |
| BigQuery 写入限流 | P0 限制 select / backtest 串行；必要时沿用 BigQuery load retry |
| 年度 diagnostic 被误认为 official result | 报告字段和 summary artifact 显式写 `diagnostic_only=true` |
| official result 仍被年度 NAV 拼接污染 | continuous ledger / resume continuity QA 作为 hard gate |

## 13. 分阶段实施

### Phase 0：只写 PRD

- 新增本文档。
- 不改执行代码。
- 不运行 Cloud Run。

### Phase 1：Scheduler dry-run

- 新增 package entrypoint。
- 复用年度 resolved plan。
- 输出跨年度 DAG、资源计划和预计提交顺序。
- 单元测试覆盖依赖、跨阶段共享资源上限、scheduler 互斥、lock lost-ownership 停止提交，以及 GCS state generation mismatch 后重新读取。

### Phase 2：Candidate-only live smoke

- 选 2 个年度、每年 2-3 个 candidate unit 做小规模 live smoke。
- 验证部分 batch、状态恢复、GCS artifact skip 和超时二次确认。
- 不跑 select/register/predict。

### Phase 3：完整 2021-2026 pipeline

- 全年度 panel / matrix / fanout / select/register/predict。
- 年度 diagnostic backtest 低优先级执行。
- 生成 official continuous ledger 或 resume-continuous 结果。

### Phase 4：优化

- 慢候选优先。
- 动态并发调节。
- 失败 candidate 显式重试。
- 更完整的 BigQuery state table。
