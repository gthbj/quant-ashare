# PRD: 策略 1 回测复合年化收益指标口径

## 1. 背景

当前 Strategy1 回测 summary 表已经有字段：

```text
ads_backtest_performance_summary.annual_return
```

但该字段是历史 legacy 口径，不应被直接视为本项目后续默认的复合年化收益率。近期 R14 长训练 `2020-2022` 回测中，基于 NAV 交易日数补算出的复合年化收益与表内 `annual_return` 明显不同。

owner 已确认项目中讨论“年化 / 月化 / 日化”时默认使用复利口径；`v3` 验收门也要求 Sharpe、Calmar 和相对年化收益使用复合年化收益。若继续复用旧字段名 `annual_return`，后续 report、CSV 导出、acceptance gate 和人工分析容易混用两种口径。

因此需要先写清楚新的字段和迁移方案，再改代码。

## 2. 目标

### 2.1 P0 目标

1. 为策略回测新增明确的复合年化收益字段，不再依赖旧 `annual_return` 字段表达复利口径。
2. 让 report、diagnosis、v3 acceptance gate 和对比 artifact 默认读取复合年化字段。
3. 保留旧 `annual_return` 字段为 legacy 字段，避免静默改变历史字段含义。
4. 新增 QA，校验 summary 中的复合年化收益能从 NAV / total return / 有效收益期数重算出来。
5. 新 run 默认写出复合年化收益；历史 run 不在 P0 中追溯覆盖。

### 2.2 P1 目标

1. 按 owner 决策选择是否批量回填历史 run 的复合年化字段。
2. 在所有导出 CSV / Markdown / HTML report 中把复合年化标为主口径，把 legacy 年化标为可选审计字段。
3. 若后续确认没有兼容风险，再单独评估是否重命名或退役旧 `annual_return`。

## 3. 非目标

1. 不执行 BigQuery / Cloud Run，不在本文档中追溯重算历史产物。
2. 不在 legacy 字段中静默改变 total return、NAV、daily return、年化波动率、Sharpe 或 MaxDD 的既有计算定义；若 v3 gate 需要使用复合年化作为 Sharpe 分子，必须作为 contract-specific 口径单独计算 / 校验。
3. 不追溯覆盖历史 `ads_backtest_performance_summary.annual_return`。
4. 不把旧 BQML / SQL ledger fallback 恢复为执行路径。
5. 不把复合年化字段作为单独策略优化目标；它只是收益度量口径统一。

## 4. 指标定义

### 4.1 标准复合年化收益

策略全周期复合年化收益定义为：

```text
compound_annual_return = (1 + total_return) ^ (target_period_count / return_period_count) - 1
```

其中：

| 参数 | 含义 |
|---|---|
| `total_return` | 策略在同一窗口内的累计总收益，等价于 `ending_nav / starting_nav - 1` |
| `target_period_count` | 年化目标期数；交易日口径默认 `252` |
| `return_period_count` | 样本窗口内实际日收益区间数，等于 NAV 有效交易日观测数减 1；必须来自策略 NAV / daily return 的有效交易日口径，不使用自然日天数 |

实现要求：

1. `return_period_count` 固定定义为日收益区间数，不是 NAV 观测天数。例如一个窗口有 728 个有效 NAV 交易日，则 `return_period_count = 727`。
2. `return_period_count` 必须与 annual volatility、Sharpe、Calmar 和相对年化计算使用的收益序列窗口一致。
3. 不允许使用自然日跨度，例如 `DATEDIFF(end_date, start_date)`。
4. 若 `return_period_count <= 0`，复合年化收益应为 `NULL`，并由 QA 拦截不完整产物。
5. 若 `total_return = -100%`，复合年化收益为 `-100%`。
6. 若出现 `total_return < -100%`，视为异常数据，QA 应失败。

### 4.2 月化和日化扩展

当前 P0 只要求落地年化字段。若后续报告需要月化或日化，统一使用同一公式：

```text
compound_period_return = (1 + total_return) ^ (target_period_count / return_period_count) - 1
```

默认换算因子：

| 口径 | `target_period_count` |
|---|---:|
| 年化 | 252 |
| 月化，交易日口径 | 21 |
| 日化 | 1 |

说明：`252` 和 `21` 是交易日换算因子，不表示每个自然年或自然月固定有这些真实交易日。

### 4.3 相对复合年化收益

相对 benchmark 的复合年化收益定义为：

```text
excess_compound_annual_return = strategy_compound_annual_return - benchmark_compound_annual_return
```

要求：

1. 策略和 benchmark 必须使用同一 `window_start_date`、`window_end_date`、`target_period_count` 和有效交易日集合。
2. 五指数相对门中，不允许策略按一套交易日年化、指数按另一套交易日年化后直接相减。
3. benchmark 缺数据时，该 benchmark 的相对复合年化门应标记为缺证据，不能默认为通过。

## 5. 字段方案

### 5.1 ADS summary 字段

P0 推荐在 `ads_backtest_performance_summary` 新增 nullable 字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `compound_annual_return` | `FLOAT64` | 策略复合年化收益 |
| `return_period_count` | `INT64` | 用于年化的有效日收益区间数，等于 NAV 有效交易日数减 1 |
| `annualization_target_period_count` | `INT64` | 年化目标期数，默认 `252` |
| `annualization_method` | `STRING` | 固定写 `compound` |

旧字段处理：

| 字段 | P0 处理 |
|---|---|
| `annual_return` | 保留，视为 legacy annual return；不静默改义 |
| `sharpe` | 保留，视为 legacy daily-return-statistics Sharpe；v3 若使用复合分子，不得假定该字段已是复合 Sharpe |
| `excess_return` | 保留现有累计超额语义；不改名 |
| `information_ratio` | 不因本 PRD 恢复为 gate 指标 |

### 5.2 `metrics_json` 字段

`metrics_json` 应同步写入：

```json
{
  "annualization": {
    "method": "compound",
    "target_period_count": 252,
    "return_period_count": 727,
    "compound_annual_return": 0.0255,
    "legacy_annual_return": 0.0468
  }
}
```

说明：上面的数值只是结构示例，不是固定验收值。

### 5.3 report / artifact 字段

Markdown、HTML、evidence pack、comparison artifact 和 CSV 导出中：

1. 默认展示名使用 `复合年化收益` / `Compound Annual Return`。
2. 若展示 legacy 字段，必须标注 `Legacy annual_return`，不得直接称为“年化收益”。
3. v3 relative artifact 中的相对收益字段必须基于 `compound_annual_return`。

## 7. 验收标准

### 7.1 单 backtest summary QA

给定任一新 backtest：

1. `compound_annual_return IS NOT NULL`。
2. `annualization_method = 'compound'`。
3. `annualization_target_period_count = 252`。
4. `return_period_count > 0`。
5. `return_period_count = NAV 有效交易日观测数 - 1`。
6. 从 `total_return` 和 `return_period_count` 重算出的复合年化收益，与 summary 字段误差在容忍范围内。

推荐容忍范围：

```text
abs(recomputed_compound_annual_return - compound_annual_return) <= 1e-9
```

### 7.2 v3 gate QA

v3 gate 必须满足：

1. Sharpe 分子使用 `compound_annual_return`。
2. Calmar 分子使用 `compound_annual_return`。
3. 五指数 `策略超额复合年化收益` 使用策略和指数的复合年化收益差。
4. 若 v3 path 缺 `compound_annual_return`，不能 fallback 到 legacy `annual_return` 并继续 accepted。

影响分析：

1. 当前 legacy summary Sharpe 通常等价于 `AVG(daily_return) * 252 / annual_vol`，分子是算术年化日均收益。
2. v3 若改用 `compound_annual_return / annual_vol`，在存在波动时通常会低于 legacy Sharpe；接近阈值的候选可能因此从通过变为拒绝。
3. 实现 PR 不得把既有 `sharpe` 字段静默改义为 compound Sharpe；应在 v3 acceptance / artifact 中显式标注 `sharpe_annualization_method = compound`，或新增等价的 v3 gate 字段。
4. 启用 v3 compound Sharpe 硬门前，必须对近期候选或五次正式搜索 replay 输出 legacy Sharpe 与 compound Sharpe 的差异表；若继续沿用 `min_sharpe = 0.70`，必须作为 owner 明确接受的阈值决策记录。
5. v1 / v2 / legacy historical artifact 不因本 PRD 自动重算 Sharpe，也不追溯改写历史 accepted / rejected 结论。

### 7.3 报告 QA

报告产物必须满足：

1. 主展示字段为 `复合年化收益`。
2. 旧 `annual_return` 若被展示，必须带 `legacy` 标记。
3. evidence pack 中必须包含 `annualization.method` 和 `return_period_count`，方便人工复核。

## 8. 迁移策略

### 8.1 P0: 新 run 生效

1. 对 summary 表做向后兼容 schema 扩展，新增字段 nullable。
2. 新回测写出复合年化字段。
3. report / v3 gate 新路径读复合字段。
4. 历史行字段为空，不自动改变历史报告结论。

### 8.2 P1: 历史 run 回填，可选

若 owner 后续需要对历史 20 个候选、5 次正式搜索或 R14 长训练结果统一比较，可单独执行 replay / backfill：

1. 从历史 NAV 重算 `compound_annual_return`。
2. 写入 nullable 新字段或独立 replay artifact。
3. 不覆盖旧 `annual_return`。
4. 在报告中明确标注“复合年化字段为后补计算”。

## 9. 回滚方案

如果实现后发现 report 或 gate 兼容问题：

1. 保留新增 nullable 字段，不删除 schema。
2. report 可临时退回 legacy 展示，但必须在输出中标注 legacy。
3. v3 gate 不应退回 legacy `annual_return`；若复合字段不可用，应将候选标记为 gate evidence 不完整，而不是接受。
