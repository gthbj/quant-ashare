# PRD: quant-ashare 项目结构重构方案

> 状态：待 owner review，未实现。
> 范围：工程结构、模块边界、命名与迁移顺序设计；不改变数据仓库表语义、Strategy1 策略逻辑、acceptance gate 阈值或线上调度入口。
> 关联前置：`docs/prd/PRD_20260609_02_策略1旧BQMLSQLRunner退役.md` 已完成旧 BQML-only SQL、SQL ledger fallback 与旧调度器退役；本 PRD 只处理退役之后仍残留的结构混乱。

## 1. 背景

`quant-ashare` 已从早期 BigQuery SQL / BQML runner、Composer 调度、单策略探索，演进为 BigQuery ODS -> DIM/DWD -> DWS/ADS 数据仓库、Cloud Scheduler + Workflows 控制面、Cloud Run Python Strategy1 执行链路并存的项目。

这类演进留下了几个工程问题：

1. 当前 active path 与 historical / retired path 的边界需要靠记忆和上下文判断，代码目录本身表达不够清楚。
2. Strategy1 active runner 仍使用部分位于 `sql/ml/strategy1/` 的共享 SQL。旧 BQML-only SQL 已删除，但 `ml` 路径名会让后续 agent 误以为这些 SQL 仍属于旧 runner。
3. `scripts/strategy1_cloudrun/` 同时承载 CLI、编排、模型训练、预测、ledger、报告、QA 辅助和 artifact 处理，模块边界偏宽。
4. 文件名、函数名和变量名中有阶段性命名，例如 `oq010`、`p0`、`r14`、`cloudrun`、`bqml`、`v3`，其中一部分是持久业务口径，一部分只是历史阶段标签。
5. 路径依赖分散在 Python、SQL 文件名、README 和 runbook 中，缺少一个稳定的 path catalog / step catalog 来表达“哪些 SQL 是 active contract，哪些只是历史文档”。
6. 文档、记忆和 TODO 的信息密度很高，但结构上仍混有历史完成记录、当前 runbook、未来实验计划和 retired 口径，阅读成本偏高。

## 2. 目标

1. 建立清晰的 active / retired / generated / experimental 边界，使新 agent 能从目录结构直接判断可维护面。
2. 把 Strategy1 active runner 的共享 SQL 从“旧 BQML runner 残留”表达，迁移为“Strategy1 execution SQL contract”表达。
3. 将 Python runner 从脚本堆叠拆为稳定包模块，CLI 只保留薄入口。
4. 收敛阶段性命名，保留必要的持久 ID，避免把历史阶段标签继续写入新模块和新接口。
5. 引入路径目录和轻量测试，防止旧 SQL、旧 runner、旧调度入口被重新引用。
6. 采用分阶段迁移，每个 PR 都可独立验证，避免一次性大搬家破坏 Cloud Run、BigQuery 或 Dataform 链路。

## 3. 非目标

1. 不在本 PRD 阶段重写 Strategy1 策略逻辑、特征、模型、ledger 交易语义或 acceptance gate 阈值。
2. 不把 BigQuery DWS/ADS contract 改写成 Python 逻辑；候选池、组合、订单、报告输入和 QA SQL 仍可继续作为 BigQuery contract 存在。
3. 不追溯修改历史 ADS / GCS artifact / run id / search id。
4. 不手工修改 Dataform generated definitions。若后续迁移 canonical SQL 且 manifest 覆盖对应文件，必须通过现有 generator 重新生成并执行 `--check`。
5. 不把 `v3`、`ml_pv_clf_v0`、`strategy1_pv_v0_20260601`、`market_state_v0_20260606` 等已持久化或用于审计的 ID 静默改名。

## 4. 当前模块边界判断

### 4.1 应保持稳定的核心边界

| 边界 | 当前职责 | 重构原则 |
|---|---|---|
| `sql/dim`, `sql/dwd`, `sql/dws`, `sql/ads`, `sql/incremental` | BigQuery warehouse canonical SQL | 保持数据仓库 contract 稳定，只做清晰命名和必要 runbook 同步 |
| `sql/qa`, `sql/metadata`, `sql/meta` | QA、metadata、状态表 | 保持 BigQuery contract，避免被策略 runner 重构牵连 |
| `orchestration/workflows/**` | 当前生产控制面 | 保持为 active scheduling/control-plane 唯一入口 |
| `orchestration/composer/**` | retired / audit-only 历史快照 | 不再新增生产逻辑 |
| `scripts/strategy1_cloudrun/**` | 当前 Strategy1 Cloud Run Python runner | 逐步拆为包模块，脚本层最终只保留 CLI wrapper |
| `sql/ml/strategy1/**` 剩余文件 | 当前 Strategy1 active path 仍在使用的共享 SQL | 改名迁移到稳定 Strategy1 SQL 命名空间，迁移前通过 catalog 明确 active 文件 |
| `configs/strategy1/**` | acceptance contract、实验配置、运行配置 | 保留 contract 事实来源；新增路径目录或 step catalog |

### 4.2 Strategy1 剩余共享 SQL 的定位

旧 BQML-only SQL `02-04`、SQL ledger fallback `08` 和旧 `run_oq010_experiments.py` 已退役删除。当前仍保留的 Strategy1 SQL 不应被称为“过去 runner 的 SQL”，而应拆成两类：

1. Active shared SQL contract：training panel、candidate、portfolio target、order plan、metrics/report input、runner QA、diagnostics、v3 replay/acceptance QA。
2. Historical reference：只存在于文档、历史 PRD、历史 artifact 或 archived runbook 中的旧 BQML / SQL ledger 口径。

重构重点不是删除 active shared SQL，而是把路径名和引用关系改到稳定命名空间，避免 `sql/ml/strategy1` 继续制造误解。

## 5. 目标结构

目标结构应按“数据仓库、策略执行、控制面、研究配置、文档历史”拆分，而不是按旧阶段编号拆分。

```text
sql/
  dim/
  dwd/
  dws/
  ads/
  incremental/
  metadata/
  meta/
  qa/
  strategy1/
    panel/
    execution/
    reporting/
    acceptance/
    qa/

src/
  quant_ashare/
    common/
      bq/
      gcs/
      config/
      artifacts/
    ingestion/
    pipeline_control/
    strategy1/
      config/
      data_access/
      features/
      model_training/
      prediction/
      selection/
      portfolio/
      ledger/
      reporting/
      diagnostics/
      acceptance/
      orchestration/

scripts/
  strategy1/
    run_search.py
    run_backtest.py
    render_report.py
    replay_acceptance_gate.py
  pipeline/
  dataform/
  qa/

docs/
  prd/
  runbooks/
  retired/
```

说明：

- `src/quant_ashare/**` 是可测试、可导入的库代码。
- `scripts/**` 只保留命令行入口、参数解析和对包模块的调用。
- `sql/strategy1/**` 是 Strategy1 active BigQuery SQL contract，不再挂在 `ml` 路径下。
- `docs/retired/` 是否创建可后置；如果不创建，也至少要在现有历史文档标题或开头标注 retired / historical。

## 6. 分阶段方案

### Phase A: Active path catalog 与防误用护栏

目标：先不移动文件，建立事实目录和检查，降低后续迁移风险。

交付：

1. 新增 Strategy1 active step catalog，例如 `configs/strategy1/active_step_catalog.yml` 或 Python 等价结构。
2. catalog 明确每个 active step 的稳定名称、当前 SQL 路径、输入/输出表、调用方和 retired 替代关系。
3. 在 `sql/ml/strategy1/README.md` 或迁移说明中明确剩余 SQL 是 active shared SQL，不是 BQML-only legacy。
4. 增加轻量检查，禁止 active code / active runbook 重新引用已退役的 `02_train_bqml_logistic_candidates.sql`、`03_select_model_and_register.sql`、`04_predict_daily.sql`、`08_run_backtest.sql`、`--use-bq-ledger`、`run_oq010_experiments.py`。
5. 统一在文档中把旧 Composer 和旧 BQML runner 标成 retired / audit-only。

验收：

- `git grep` 不应在 active code / active runbook 中发现已退役入口。
- Cloud Run runner 的 SQL 读取路径来自 catalog 或集中常量，不再散落硬编码。
- 不改变 BigQuery SQL 内容和线上运行行为。

### Phase B: Strategy1 SQL 稳定命名空间迁移

目标：把 active shared SQL 从 `sql/ml/strategy1` 迁移到 `sql/strategy1`，先改表达，再考虑深层逻辑拆分。

建议映射：

| 当前路径 | 目标路径 |
|---|---|
| `01_build_training_panel.sql` | `sql/strategy1/panel/build_training_panel.sql` |
| `05_build_candidates.sql` | `sql/strategy1/execution/build_candidates.sql` |
| `06_build_portfolio_targets.sql` | `sql/strategy1/execution/build_portfolio_targets.sql` |
| `07_build_order_plan.sql` | `sql/strategy1/execution/build_order_plan.sql` |
| `09_build_metrics_and_report_inputs.sql` | `sql/strategy1/reporting/build_metrics_and_report_inputs.sql` |
| `10_qa_runner_outputs.sql` | `sql/strategy1/qa/qa_runner_outputs.sql` |
| `12_qa_model_diagnosis_outputs.sql` | `sql/strategy1/qa/qa_model_diagnosis_outputs.sql` |
| `16-24` replay / acceptance SQL | `sql/strategy1/acceptance/` 或 `sql/strategy1/qa/acceptance/` |

迁移要求：

1. 由 catalog 驱动路径替换，避免在多个 Python 文件中手写新路径。
2. 若 Cloud Run image 或历史 wrapper 需要兼容旧路径，可只保留一轮兼容 shim，但 shim 必须有 retired 注释和删除 TODO。
3. `sql/README.md`、Strategy1 runbook、相关 PRD 和 `.agent/memory/KNOWN_CONSTRAINTS.md` 同步更新。
4. 如迁移影响 Dataform manifest 管理的 canonical SQL，必须运行 Dataform generator 和 `--check`。

验收：

- active runner 全量读取 `sql/strategy1/**`。
- `sql/ml/strategy1/**` 不再作为 active 执行路径；若短期保留，只能是 compatibility shim 或 historical note。
- 现有 unit/import tests 通过；至少完成一次 dry-run 级 SQL render 检查。

### Phase C: Python runner 包拆分

目标：从 `scripts/strategy1_cloudrun` 脚本目录抽取可测试包模块，降低单文件复杂度。

建议拆分：

| 目标模块 | 职责 |
|---|---|
| `quant_ashare.strategy1.config` | contract/config 加载、path catalog、参数校验 |
| `quant_ashare.strategy1.data_access` | BigQuery / GCS 读写封装、SQL render、artifact URI |
| `quant_ashare.strategy1.features` | feature set、sample mask、label embargo 辅助 |
| `quant_ashare.strategy1.model_training` | sklearn / LightGBM 训练与模型 artifact |
| `quant_ashare.strategy1.prediction` | prediction 生成、score orientation、candidate materialization |
| `quant_ashare.strategy1.portfolio` | target holdings、权重、rebalance 输入输出 |
| `quant_ashare.strategy1.ledger` | `ledger_exec_v1_lot100`、resume state、lot100 / float audit |
| `quant_ashare.strategy1.reporting` | report inputs、HTML/CSV evidence、summary 展示 |
| `quant_ashare.strategy1.acceptance` | v1/v2/v3 contract parsing、live gate、replay gate |
| `quant_ashare.strategy1.orchestration` | search/backtest/report/replay 高层编排 |

脚本层保留：

```text
scripts/strategy1/run_search.py
scripts/strategy1/run_backtest.py
scripts/strategy1/render_report.py
scripts/strategy1/replay_acceptance_gate.py
```

过渡策略：

1. 第一轮只抽无副作用 helper 和 config/data_access，保持原 CLI 不变。
2. 第二轮抽 ledger/report/acceptance，并为现有 CLI 增加 import smoke tests。
3. 第三轮将 `scripts/strategy1_cloudrun/**` 标为 compatibility wrappers，新增入口改用 `scripts/strategy1/**`。
4. Cloud Run Job entrypoint 迁移必须单独 PR，并附最小 smoke 或 dry-run 证据。

验收：

- 包模块可以被 `pytest` 直接导入。
- CLI wrapper 只负责参数解析和调用包模块。
- Cloud Run entrypoint 未经验证不得删除旧 wrapper。

### Phase D: 阶段性命名收敛

目标：清理新代码中的历史阶段标签，保留持久化和审计必要的名字。

命名规则：

1. 新目录、新模块、新函数优先使用稳定业务名：`strategy1`、`acceptance`、`ledger`、`reporting`、`pipeline_control`、`warehouse`。
2. `oq010`、`p0`、`r14` 只用于文档、TODO、实验配置或历史 artifact 说明，不用于通用运行模块名。
3. `cloudrun` 只用于平台适配层或部署入口，不用于领域逻辑模块名。
4. `bqml` 只用于历史审计或确实调用 BigQuery ML 的路径。
5. `v1/v2/v3` 可用于明确 contract version，不可泛化成模块职责名。
6. 已持久化到 BigQuery / GCS / registry 的 ID 不在重构 PR 中静默修改。

示例：

| 旧表达 | 推荐表达 |
|---|---|
| `run_oq010_*` | `run_strategy1_search` / `run_strategy1_backtest` |
| `strategy1_cloudrun` 领域逻辑 | `quant_ashare.strategy1.*` |
| `sql/ml/strategy1` active shared SQL | `sql/strategy1/*` |
| `p0_*` 通用变量 | `baseline_*` / `default_*` / 具体业务名 |
| `v3_gate_helper` | `acceptance_contract_replay` 或保留 `contract_v3` 作为版本字段 |

验收：

- 新增代码不引入非必要阶段标签。
- 对必须保留的历史名字，在 README 或代码注释中说明保留原因。
- 不修改持久 ID，除非另开 migration PRD。

### Phase E: 测试与发布护栏

目标：让重构不会依赖人工记忆。

建议测试：

1. import smoke：覆盖 `quant_ashare.strategy1.*` 核心模块和 CLI wrapper。
2. path catalog test：所有 catalog SQL 路径存在，retired SQL 不在 active step 中。
3. SQL render smoke：关键 Strategy1 SQL 可在无 BigQuery 执行的情况下完成参数渲染。
4. retired reference linter：active docs/code 不引用旧 BQML-only SQL、SQL ledger fallback、旧 Composer helper。
5. ledger unit tests：lot100、cash/position、pending sell、resume state 的纯 Python 单测。
6. report snapshot smoke：HTML/CSV evidence 生成路径不依赖旧目录。

发布要求：

- 每个迁移 PR 只做一个边界：catalog、SQL 路径、Python 包、Cloud Run entrypoint、命名 cleanup 分开。
- 涉及 Cloud Run entrypoint 的 PR 必须有镜像构建和最小 smoke 证据，或明确标注未部署。
- 涉及 SQL contract 的 PR 必须说明是否需要 BigQuery dry-run / QA / Dataform generator。

## 7. 推荐实施顺序

1. **先做 Phase A**：建立 active step catalog 和 retired reference linter。这一步价值最高、风险最低。
2. **再做 Phase B**：迁移 Strategy1 active shared SQL 到 `sql/strategy1/**`，解决“runner 依赖旧 runner SQL 文件”的主要观感问题。
3. **随后做 Phase C 的前半段**：抽 config、path、data_access、SQL render，不碰 Cloud Run entrypoint。
4. **再拆 ledger/report/acceptance**：这些模块风险较高，应配套 unit tests。
5. **最后做入口与命名 cleanup**：Cloud Run Job entrypoint、`scripts/strategy1_cloudrun` 退役、文档 retired 归档。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| SQL 路径迁移导致 Cloud Run image 找不到文件 | Strategy1 runner 失败 | 先用 catalog 统一路径，再迁移；entrypoint PR 必须 smoke |
| `sql/ml/strategy1` 直接删除导致 active QA / replay 断链 | v3 replay、report、QA 失败 | 明确 active shared SQL 清单，迁移而非删除 |
| 重命名持久 ID 导致历史 artifact 无法对账 | registry / ADS / GCS 难以审计 | 持久 ID 不在结构重构中修改 |
| Python 包拆分引入循环依赖 | CLI import 失败 | 先抽无副作用 helper，增加 import smoke |
| Dataform generated SQL 与 canonical SQL 漂移 | 发布链路不一致 | 只改 canonical SQL；必要时运行 generator 与 `--check` |
| 文档历史被过度整理，丢失审计线索 | 难以追溯旧决策 | historical / retired 文档保留，只改变标签和入口索引 |

## 9. PR 验收标准

### 本 PRD PR

- 新增本 PRD。
- 同步 `.agent/memory/IMPLEMENTATION_STATUS.md`、`.agent/memory/AGENT_HANDOFF.md` 和 `TODO.md`。
- 不改代码、不改 SQL、不执行 BigQuery / Cloud Run。

### 后续 Phase A PR

- 有 active step catalog。
- 有 retired reference linter 或等价检查。
- README / runbook 能明确 active shared SQL 与 retired BQML SQL 的区别。
- 不改变 runner 行为。

### 后续 Phase B PR

- active Strategy1 SQL 迁移到 `sql/strategy1/**` 或通过明确 shim 过渡。
- runner 从 catalog 读取新路径。
- active docs 不再把 `sql/ml/strategy1` 描述为当前执行 SQL 根目录。
- 通过 import / path / SQL render 检查。

### 后续 Phase C/D/E PR

- 包模块 import smoke 通过。
- CLI wrapper 行为保持兼容。
- Cloud Run entrypoint 迁移单独验证。
- 新增命名符合本 PRD 的稳定命名规则。

## 10. 决策点

需要 owner 在 review 时确认：

1. 是否接受 `sql/strategy1/**` 作为 Strategy1 active shared SQL 的目标命名空间。
2. 是否接受 `src/quant_ashare/**` 作为 Python 包根目录。
3. 是否允许短期保留 `scripts/strategy1_cloudrun/**` wrapper，直到 Cloud Run entrypoint 单独迁移。
4. 是否需要创建 `docs/retired/`，还是只在现有历史文档顶部标注 retired / historical。

## 11. 结论

当前项目不适合一次性“大重构”。推荐把重构拆成“先 catalog 和护栏，再 SQL 命名空间，再 Python 包拆分，再入口迁移和命名 cleanup”的顺序。

对 owner 提到的具体问题，本 PRD 的判断是：

- `runner 依赖过去 runner 的 sql 文件`：实际剩余 SQL 多数是 active shared SQL，不应直接删除；应迁移到 `sql/strategy1/**` 并由 catalog 管理。
- `模块应该拆分出来`：应把 Strategy1 Python 逻辑抽到 `src/quant_ashare/strategy1/**`，脚本只保留 CLI。
- `文件名变量名函数名阶段性命名`：新增代码应避免 `oq010/p0/r14/cloudrun/bqml` 这类阶段标签进入通用模块，持久 ID 另行保留。
- `项目混乱`：先让结构表达事实，再做行为迁移；用小 PR 和检查替代一次性重排。
