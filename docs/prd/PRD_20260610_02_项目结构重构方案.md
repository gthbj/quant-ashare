# PRD: quant-ashare 项目结构重构方案

> 状态：待 owner review，未实现。
> 范围：工程结构、BigQuery 实验/正式产物生命周期边界、模块边界、命名与迁移顺序设计；不改变数据仓库表语义、Strategy1 策略逻辑、acceptance gate 阈值或线上调度入口。
> 关联前置：`docs/prd/PRD_20260609_02_策略1旧BQMLSQLRunner退役.md` 已完成旧 BQML-only SQL、SQL ledger fallback 与旧调度器退役；本 PRD 只处理退役之后仍残留的结构混乱。

## 1. 背景

`quant-ashare` 已从早期 BigQuery SQL / BQML runner、Composer 调度、单策略探索，演进为 BigQuery ODS -> DIM/DWD -> DWS/ADS 数据仓库、Cloud Scheduler + Workflows 控制面、Cloud Run Python Strategy1 执行链路并存的项目。

这类演进留下了几个工程问题：

1. 当前 active path 与 historical / retired path 的边界需要靠记忆和上下文判断，代码目录本身表达不够清楚。
2. Strategy1 active runner 仍使用部分位于 `sql/ml/strategy1/` 的共享 SQL，risk-feature search 还使用 `sql/cloudrun/strategy1/01_build_training_panel.sql`。旧 BQML-only SQL 已删除，但 `ml` / `cloudrun` 路径名会让后续 agent 误以为 Strategy1 SQL 仍按历史执行 backend 分裂。
3. `scripts/strategy1_cloudrun/` 同时承载 CLI、编排、模型训练、预测、ledger、报告、QA 辅助和 artifact 处理，模块边界偏宽。
4. 文件名、函数名和变量名中有阶段性命名，例如 `oq010`、`p0`、`r14`、`cloudrun`、`bqml`、`v3`，其中一部分是持久业务口径，一部分只是历史阶段标签。
5. 路径依赖分散在 Python、SQL 文件名、README 和 runbook 中，缺少一个稳定的 path catalog / step catalog 来表达“哪些 SQL 是 active contract，哪些只是历史文档”。
6. 文档、记忆和 TODO 的信息密度很高，但结构上仍混有历史完成记录、当前 runbook、未来实验计划和 retired 口径，阅读成本偏高。
7. `ashare_ads` 当前承载了正式契约表、历史实验、rejected candidate、diagnostic backtest 与报告元数据，后续容易让“实验结果”和“正式投产产物”混在同一物理层。

## 2. 目标

1. 建立清晰的 active / retired / generated / experimental 边界，使新 agent 能从目录结构直接判断可维护面。
2. 把 Strategy1 active runner 的共享 SQL 从“旧 BQML runner 残留”表达，迁移为“Strategy1 execution SQL contract”表达。
3. 将 Python runner 从脚本堆叠拆为稳定包模块，CLI 只保留薄入口。
4. 收敛阶段性命名，保留必要的持久 ID，避免把历史阶段标签继续写入新模块和新接口。
5. 新增 BigQuery 研究层边界：默认实验、候选模型、诊断回测和 acceptance replay 写 `ashare_research`；只有 owner 明确 promotion 的正式产物写 `ashare_ads`。
6. 引入路径目录和轻量测试，防止旧 SQL、旧 runner、旧调度入口被重新引用。
7. 采用分阶段迁移，每个 PR 都可独立验证，避免一次性大搬家破坏 Cloud Run、BigQuery 或 Dataform 链路。

## 3. 非目标

1. 不在本 PRD 阶段重写 Strategy1 策略逻辑、特征、模型、ledger 交易语义或 acceptance gate 阈值。
2. 不把 BigQuery DWS/ADS contract 改写成 Python 逻辑；候选池、组合、订单、报告输入和 QA SQL 仍可继续作为 BigQuery contract 存在。
3. 不追溯修改历史 ADS / GCS artifact / run id / search id。
4. 不手工修改 Dataform generated definitions。若后续迁移 canonical SQL 且 manifest 覆盖对应文件，必须通过现有 generator 重新生成并执行 `--check`。
5. 不把 `v3`、`ml_pv_clf_v0`、`strategy1_pv_v0_20260601`、`market_state_v0_20260606` 等已持久化或用于审计的 ID 静默改名。
6. 本 PRD PR 不创建 `ashare_research` dataset、不迁移历史 ADS 数据、不改变当前 runner 默认写入目标；这些属于后续单独实现 PR。

## 4. 当前模块边界判断

### 4.1 应保持稳定的核心边界

| 边界 | 当前职责 | 重构原则 |
|---|---|---|
| `sql/dim`, `sql/dwd`, `sql/dws`, `sql/ads`, `sql/incremental` | BigQuery warehouse canonical SQL | 保持数据仓库 contract 稳定，只做清晰命名和必要 runbook 同步 |
| `sql/qa`, `sql/metadata`, `sql/meta` | QA、metadata、状态表 | 保持 BigQuery contract，避免被策略 runner 重构牵连 |
| `orchestration/workflows/**` | 当前生产控制面 | 保持为 active scheduling/control-plane 唯一入口 |
| `orchestration/composer/**` | retired / audit-only 历史快照 | 不再新增生产逻辑 |
| `scripts/strategy1_cloudrun/**` | 当前 Strategy1 Cloud Run Python runner | 逐步拆为包模块，脚本层最终只保留 CLI wrapper |
| `sql/ml/strategy1/**` 剩余文件 | 当前 Strategy1 active path 仍在使用的一部分共享 SQL | 改名迁移到稳定 Strategy1 SQL 命名空间，迁移前通过 catalog 明确 active / historical |
| `sql/cloudrun/strategy1/**` | 当前 risk-feature Cloud Run path 使用的 training panel SQL | 纳入同一 Strategy1 SQL catalog 和迁移计划，避免形成第三个 active SQL namespace |
| `configs/strategy1/**` | acceptance contract、实验配置、运行配置 | 保留 contract 事实来源；新增路径目录或 step catalog |
| BigQuery `ashare_research` | 建议新增；当前尚不存在 | 作为实验、候选、诊断和 acceptance replay 的默认写入层 |
| BigQuery `ashare_ads` | 当前正式契约和历史实验混合 | 后续收敛为 owner promotion 后的正式模型、正式信号、正式回测和生产监控层 |

### 4.2 Strategy1 剩余共享 SQL 的定位

旧 BQML-only SQL `02-04`、SQL ledger fallback `08` 和旧 `run_oq010_experiments.py` 已退役删除。当前仍保留的 Strategy1 SQL 不应被称为“过去 runner 的 SQL”，而应按实际调用方和产物角色拆成两类：

1. Active shared SQL contract：training panel、candidate、portfolio target、order plan、metrics/report input、runner QA、diagnostics、v3 replay/acceptance QA。
2. Historical reference：只存在于文档、历史 PRD、历史 artifact 或 archived runbook 中的旧 BQML / SQL ledger 口径。

Catalog 不能手写 `sql/ml/strategy1` 清单后就结束，必须从现行 Python 调用方、config 和 runbook 反推 active SQL。至少要覆盖并分类：

- `sql/ml/strategy1/01_build_training_panel.sql`
- `sql/cloudrun/strategy1/01_build_training_panel.sql`
- `sql/ml/strategy1/05_build_candidates.sql`
- `sql/ml/strategy1/06_build_portfolio_targets.sql`
- `sql/ml/strategy1/07_build_order_plan.sql`
- `sql/ml/strategy1/09_build_metrics_and_report_inputs.sql`
- `sql/ml/strategy1/10_qa_runner_outputs.sql`
- `sql/ml/strategy1/12_qa_model_diagnosis_outputs.sql`
- `sql/ml/strategy1/16-25` 的 Cloud Run / native search / risk / acceptance / ledger resume QA SQL
- `sql/ml/strategy1/11`、`14`、`15` 等历史诊断或审计 SQL也必须显式分类，不能因不在默认路径中就遗漏。

重构重点不是删除 active shared SQL，而是把路径名和引用关系改到稳定命名空间，避免 `sql/ml/strategy1` 和 `sql/cloudrun/strategy1` 继续制造误解。

### 4.3 `ashare_research` / `ashare_ads` 生命周期边界

建议新增 BigQuery dataset：`ashare_research`。

目标口径：

```text
ashare_dws
  -> ashare_research   实验、候选模型、诊断、未投产回测、acceptance replay
  -> ashare_ads        已 promotion 的正式模型、正式信号、正式回测、生产监控
```

这里的 `ashare_research` 不是 DWS 到 ADS 之间强制串行转换的“中间层”，而是策略研究生命周期隔离层。实验 runner 默认写 `ashare_research`；只有通过 acceptance gate 且经 owner 明确 promotion 的产物，才进入 `ashare_ads`。

推荐命名为 `ashare_research`，不推荐：

| 候选名 | 不推荐原因 |
|---|---|
| `ashare_experiment` / `ashare_experiments` | 过窄，不能自然覆盖诊断、报告、replay 和模型比较 |
| `ashare_sandbox` | 语义过临时，不适合承载可复现研究结果 |
| `ashare_staging` | 容易和 ETL staging / 发布暂存混淆 |
| `ashare_lab` | 简短但正式性和 BigQuery IAM / runbook 语义弱于 research |
| `ashare_candidate` | 只表达候选模型，不覆盖回测、诊断和报告 |

表命名建议继续遵守“dataset 层级 + 表名前缀一致”的现有风格：

| 层 | 示例 |
|---|---|
| DWD | `ashare_dwd.dwd_stock_eod_price` |
| DWS | `ashare_dws.dws_stock_feature_daily_v0` |
| ADS | `ashare_ads.ads_model_registry` |
| Research | `ashare_research.research_model_registry` |

因此 research 表目标命名优先使用 `research_*`，而不是在 `ashare_research` 里继续创建 `ads_*` 表。这样 promotion SQL 能明确把 `research_*` 映射到 `ads_*`，避免研究表被误认为正式 ADS。

建议 P0 research 表族：

| Research 表族 | 职责 |
|---|---|
| `research_ml_training_panel_daily` | 每个实验 run 的冻结训练/验证/测试/预测输入面板 |
| `research_model_registry` | 候选模型、参数、artifact URI、acceptance 状态、promotion 状态 |
| `research_model_prediction_daily` | 研究 run 的预测分数、rank、score orientation |
| `research_stock_candidate_daily` | 研究 run 的候选池和过滤原因 |
| `research_portfolio_target_daily` | 研究 run 的目标组合 |
| `research_order_plan_daily` | 研究 run 的订单计划 |
| `research_backtest_*` | 研究 run 的成交、持仓、NAV、ledger state、summary |
| `research_signal_monitor_daily` | 研究 run 的信号质量、覆盖率、异常监控 |
| `research_acceptance_result` | acceptance gate / replay / comparison 的结构化结果 |

目标态写入规则：

1. 普通 Strategy1 search/backtest/report/replay 默认只写 `ashare_research` 和 GCS research artifact。
2. `ashare_ads` 只接收 owner 明确 promotion 的最小正式产物：模型 registry、正式 prediction/signals、正式 portfolio/backtest summary、生产监控和 promotion manifest。
3. `accepted` 不等于 `promoted`。一个模型可以通过 acceptance gate 但仍停留在 `ashare_research`，直到 owner 决定是否正式投产。
4. 历史已写入 `ashare_ads` 的实验产物先保留为 mixed historical state，不在结构重构中追溯迁移。
5. IAM 上，实验 runner 只需要 `ashare_research` 写权限；写 `ashare_ads` 应收敛到 promotion job / owner-approved service account。
6. Research 表必须继续遵守 BigQuery 分区过滤约束；按 `trade_date`、`predict_date`、`rebalance_date`、`run_started_date` 等实际分区列设计，并按 `run_id`、`backtest_id`、`model_id`、`strategy_id` 聚簇。

状态语义必须拆开：

| 状态 | 含义 |
|---|---|
| `rejected` | acceptance gate 不通过 |
| `accepted` | gate 通过，但仍只是 research 结果 |
| `promoted` | owner 批准进入 ADS |
| `active` | 当前生产默认使用 |
| `deprecated` | 曾 promoted，后续被替换 |

Research 表不是 ADS clone，而是 run-scoped lifecycle 表。字段可以与 ADS 相近，但 description 必须强调 candidate / unpromoted / replay / diagnostic。Promotion 应写 append-only manifest，不靠覆盖 `status` 字段表达发布历史。最小 manifest 字段建议：

- `promotion_id`
- `source_dataset`
- `source_run_id`
- `source_backtest_id`
- `source_model_id`
- `source_artifact_uri`
- `target_ads_tables`
- `acceptance_contract_version`
- `acceptance_contract_sha256`
- `approval_ref`
- `approved_by`
- `approved_at`
- `source_git_commit`
- `promotion_code_version`
- `promotion_status`
- `created_at`

### 4.4 当前实施范围收敛

本 PRD 描述长期项目结构，但 P0/P1 实施范围应收敛到：

- Strategy1 execution runner。
- Strategy1 shared SQL。
- Strategy1 research / ADS lifecycle。
- Strategy1 report、diagnostics、QA、acceptance、promotion 边界。

以下内容只作为 future scope，不应在本轮重构 PR 链路中顺手搬迁：

- `scripts/ingestion/**`。
- `orchestration/workflows/**`。
- `ashare-pipeline-control`。
- 生产 Scheduler / Workflows / Cloud Run ingestion 入口。
- 通用 `pipeline_control` 包大搬家。

原因是调度侧刚完成 `Cloud Scheduler + Cloud Workflows` production cutover，当前稳定性优先级高于目录统一；本 PRD 不应把策略 runner 重构扩大成调度控制面重构。

目标结构应按“数据仓库、策略执行、控制面、研究配置、文档历史”拆分，而不是按旧阶段编号拆分。

```text
sql/
  dim/
  dwd/
  dws/
  ads/
  research/
  incremental/
  metadata/
  meta/
  qa/
  strategy1/
    panel/
    execution/
    reporting/
    acceptance/
    qa/

src/
  quant_ashare/
    common/
      bq/
      gcs/
      config/
      artifacts/
    ingestion/          # future scope
    pipeline_control/   # future scope
    strategy1/
      config/
      data_access/
      features/
      model_training/
      prediction/
      selection/
      portfolio/
      ledger/
      reporting/
      diagnostics/
      acceptance/
      orchestration/

scripts/
  strategy1/

  pipeline/
  dataform/

  prd/
  runbooks/

```

说明：

- `src/quant_ashare/**` 是可测试、可导入的库代码。
- `scripts/**` 只保留命令行入口、参数解析和对包模块的调用。

- `sql/research/**` 存放 `ashare_research` dataset / table / QA / promotion-readiness contract。
- `sql/strategy1/**` 是 Strategy1 active BigQuery SQL contract，不再挂在 `ml` 路径下。
- `sql/ads/**` 后续应只表达正式 ADS contract 和 promotion 后产物，不再作为默认实验写入层。
- `docs/retired/` 是否创建可后置；如果不创建，也至少要在现有历史文档标题或开头标注 retired / historical。

## 6. 分阶段方案

### Phase A: Active path catalog 与防误用护栏

目标：先不移动文件、不改行为，建立事实目录和检查，降低后续迁移风险。

交付：

1. 新增 Strategy1 active step catalog，例如 `configs/strategy1/active_step_catalog.yml` 或 Python 等价结构。
2. Catalog 必须从实际调用方反推 active SQL，覆盖 `sql/ml/strategy1/**` 和 `sql/cloudrun/strategy1/**`，并为每个文件标记 `active`、`historical`、`retired` 或 `audit_only`。
3. Catalog 明确每个 active step 的稳定名称、当前 SQL 路径、目标 SQL 路径、调用方、输入/输出表角色、输出 dataset role（当前 `ads`，目标可能为 `research`）、分区列和 retired 替代关系。
4. 在 `sql/ml/strategy1/README.md`、`sql/cloudrun/strategy1/README.md` 或迁移说明中明确剩余 SQL 是 active shared SQL 或 historical/audit SQL，不是 BQML-only legacy。
5. 增加 retired reference linter，禁止 active code / active runbook 重新引用已退役的 `02_train_bqml_logistic_candidates.sql`、`03_select_model_and_register.sql`、`04_predict_daily.sql`、`08_run_backtest.sql`、`--use-bq-ledger`、`run_oq010_experiments.py`。
6. 统一在文档中把旧 Composer 和旧 BQML runner 标成 retired / audit-only。
7. 把现有 Strategy1 写入步骤标注为 current-mixed ADS，并给出后续 research 默认写入目标；Phase A 只标注，不迁移。

Retired linter 不能做全仓裸 grep，必须带作用域和 allowlist。建议配置：

```yaml
banned_active_refs:
  - sql/ml/strategy1/02_train_bqml_logistic_candidates.sql
  - sql/ml/strategy1/03_select_model_and_register.sql
  - sql/ml/strategy1/04_predict_daily.sql
  - sql/ml/strategy1/08_run_backtest.sql
  - --use-bq-ledger
  - scripts/strategy1/run_oq010_experiments.py
active_scopes:
  - scripts/**
  - sql/**
  - configs/**
  - docs/runbooks/**
  - orchestration/workflows/**
  - sql/ml/strategy1/README.md
  - sql/cloudrun/strategy1/README.md
historical_allowed_scopes:
  - docs/prd/**
  - docs/retired/**
  - .agent/memory/archive/**
  - .agent/memory/AGENT_HANDOFF.md
  - .agent/memory/IMPLEMENTATION_STATUS.md
required_marker_for_historical_refs:
  - retired
  - historical
  - audit
```

验收：

- `git grep` 不应在 active code / active runbook 中发现已退役入口。
- Cloud Run runner 的 SQL 读取路径来自 catalog 或集中常量，不再散落硬编码。
- Catalog 能区分 experiment/research output 与 promoted/ADS output。
- Catalog 覆盖 `sql/cloudrun/strategy1/01_build_training_panel.sql`，不能只覆盖 `sql/ml/strategy1/**`。
- 不改变 BigQuery SQL 内容和线上运行行为。

### Phase A2: Table role / dataset role abstraction

目标：先建立逻辑表角色和 dataset role resolver，但仍全部解析到当前 `ashare_ads`，为 research 迁移铺路。

交付：

1. 新增 table role / dataset role 概念，例如 `training_panel`、`model_registry`、`model_prediction_daily`、`stock_candidate_daily`、`portfolio_target_daily`、`backtest_nav_daily`、`backtest_summary`、`acceptance_result`。
2. 现阶段所有 role 仍解析到 `data-aquarium.ashare_ads.*`，不改变运行行为。
3. Python 和 SQL runner 不再散落硬编码表路径；通过 resolver 或 catalog 生成 SQL 参数 / table names。
4. Catalog 同时管理 SQL 参数契约，避免 `DECLARE p_* DEFAULT ...` 的历史示例值在漏传参数时静默生效。

建议 catalog schema 至少包含：

```yaml
steps:
  build_candidates:
    status: active
    current_path: sql/ml/strategy1/05_build_candidates.sql
    target_path: sql/strategy1/execution/build_candidates.sql
    caller:
      - scripts.strategy1_cloudrun.backtest_report
    required_params:
      - p_run_id
      - p_prediction_run_id
      - p_strategy_id
      - p_predict_start
      - p_predict_end
      - p_force_replace
    optional_params:
      p_tail_risk_profile_id: diagnostic_only
    inputs:
      - role: model_registry
      - role: model_prediction_daily
      - role: stock_universe_daily
    outputs:
      - role: stock_candidate_daily
        dataset_role_current: ads
        dataset_role_target: research
    partition_columns:
      - rebalance_date
```

验收：

- Catalog 中每个 SQL path 必须存在。
- Active SQL 中每个 `DECLARE p_*` 必须在 `required_params` 或 `optional_params` 中出现。
- Active runner 不能直接传裸路径，必须从 catalog resolve。
- Render 后不允许保留带示例 run id / backtest id / date range / feature version 的默认值，除非 catalog 标记 `allow_default: true` 并说明原因。
- Report、diagnosis、QA、acceptance 后续必须从同一 table-role resolver 读取。
- 行为仍等价于写 `ashare_ads`，不创建或写入 `ashare_research`。

### Phase B: Strategy1 SQL 稳定命名空间迁移

目标：把 active shared SQL 从 `sql/ml/strategy1` 和 `sql/cloudrun/strategy1` 迁移到 `sql/strategy1`，先改表达，再考虑深层逻辑拆分。

建议映射：

| 当前路径 | 目标路径 |
|---|---|
| `sql/ml/strategy1/01_build_training_panel.sql` | `sql/strategy1/panel/build_training_panel_base.sql` 或由 catalog 定名 |
| `sql/cloudrun/strategy1/01_build_training_panel.sql` | `sql/strategy1/panel/build_training_panel_risk_feature.sql` 或由 catalog 定名 |
| `sql/ml/strategy1/05_build_candidates.sql` | `sql/strategy1/execution/build_candidates.sql` |

5. 第二轮才抽 ledger/report/acceptance/orchestration。

验收：

- 包模块可以被 `pytest` 直接导入。
- CLI wrapper 只负责参数解析和调用包模块。
- Cloud Run image 的 package 安装 / import path 策略明确。
- Cloud Run entrypoint 未经验证不得删除旧 wrapper。

### Phase D: `ashare_research` / `ashare_ads` 生命周期隔离

目标：在 table-role abstraction 和 SQL namespace 收敛之后，分段实现 research 物理表、可选 research routing、默认 research-first 和 explicit promotion。

分段：

1. **Phase D0: research table contract**
   新增 `data-aquarium.ashare_research` dataset DDL 和 `sql/research/**` research 表契约；默认 runner 不切换。
2. **Phase D1: optional research routing**
   Runner 显式 config 可写 research；report、diagnosis、QA、acceptance、comparison 必须能从同一 table-role resolver 读取 research output。
3. **Phase D2: default research-first**
   只有 D1 的 QA/report/acceptance 都支持 research source 后，普通 experiment 默认写 research；ADS 写入不再是 runner 默认副作用。
4. **Phase D3: explicit promotion job**
   owner-approved promotion 把 research 产物复制/映射到 ADS，写 append-only promotion manifest。

交付：

1. Research 表命名使用 `research_*` 前缀。
2. Research 表按实际查询模式设置分区/聚簇/description，继续遵守 BigQuery partition filter 约束。
3. Promotion manifest 记录 source research run、source artifact、approval metadata、promotion code version、target ADS table 和状态。
4. IAM 梳理：实验 runner 写 `ashare_research`；promotion job / owner-approved service account 写 `ashare_ads`。
5. 历史 ADS 实验产物不追溯迁移；只从新 runner / 新实验执行 research-first。

验收：

- `ashare_research` 表契约存在，并有分区/聚簇/description 口径。
- 显式 research routing 可完成 report / diagnosis / QA / acceptance。
- 新实验默认写 research 前，所有读取方都已支持 research source。
- Promotion 路径是显式步骤，不是普通 runner 隐式副作用。
- 不搬迁历史 ADS，不破坏现有报告和 QA 的可追溯性。

### Phase E: 阶段性命名收敛与深层包拆分

目标：清理新代码中的历史阶段标签，保留持久化和审计必要的名字。

命名规则：

1. 新目录、新模块、新函数优先使用稳定业务名：`strategy1`、`acceptance`、`ledger`、`reporting`、`pipeline_control`、`warehouse`。
2. `oq010`、`p0`、`r14` 只用于文档、TODO、实验配置或历史 artifact 说明，不用于通用运行模块名。
3. `cloudrun` 只用于平台适配层或部署入口，不用于领域逻辑模块名。
4. `bqml` 只用于历史审计或确实调用 BigQuery ML 的路径。
5. `v1/v2/v3` 可用于明确 contract version，不可泛化成模块职责名。
6. 已持久化到 BigQuery / GCS / registry 的 ID 不在重构 PR 中静默修改。

命名 cleanup 必须带 exception registry，避免把有审计意义的字段过度重命名。示例：

```yaml
allowed_legacy_names:
  bqml_reference_run_id:
    reason: historical reference / parity evidence
    allowed_in:
      - configs/strategy1/**
      - scripts/strategy1_cloudrun/**
    removal_condition: parity/audit fields no longer consumed by report or QA
```

示例：

| 旧表达 | 推荐表达 |
|---|---|
| `run_oq010_*` | `run_strategy1_search` / `run_strategy1_backtest` |
| `strategy1_cloudrun` 领域逻辑 | `quant_ashare.strategy1.*` |
| `sql/ml/strategy1` active shared SQL | `sql/strategy1/*` |
| `p0_*` 通用变量 | `baseline_*` / `default_*` / 具体业务名 |
| `v3_gate_helper` | `acceptance_contract_replay` 或保留 `contract_v3` 作为版本字段 |

验收：

- 新增代码不引入非必要阶段标签。
- 对必须保留的历史名字，在 README 或代码注释中说明保留原因。
- `bqml_reference_run_id` 这类 audit 字段只能按 exception registry 保留，不能被误删或静默改义。
- 不修改持久 ID，除非另开 migration PRD。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| SQL 路径迁移导致 Cloud Run image 找不到文件 | Strategy1 runner 失败 | 先用 catalog 统一路径，再迁移；entrypoint PR 必须 smoke |
| `sql/ml/strategy1` 或 `sql/cloudrun/strategy1` 直接删除导致 active QA / replay 断链 | v3 replay、report、QA 失败 | 从调用方反推 active SQL，迁移而非删除 |
| SQL 参数漏传后保留历史 `DECLARE DEFAULT` | 静默使用旧 run id / date / feature version | Catalog 管理 required/optional params，render 后禁止未授权默认值 |
| 重命名持久 ID 导致历史 artifact 无法对账 | registry / ADS / GCS 难以审计 | 持久 ID 不在结构重构中修改 |
| Research / ADS 边界迁移过急 | 现有报告、QA、历史回测读取断链 | 新实验 research-first，历史 ADS 不追溯搬迁；promotion 单独实现 |
| Research 表命名继续使用 `ads_*` | 实验表被误认为正式产物 | research dataset 内优先使用 `research_*` 表名前缀 |
| Python 包拆分引入循环依赖或 Cloud Run image import 失败 | CLI / Cloud Run 入口失败 | 先抽无副作用 helper，明确 packaging 方案，保留 wrapper 并增加 import smoke |
| Dataform generated SQL 与 canonical SQL 漂移 | 发布链路不一致 | 只改 canonical SQL；必要时运行 generator 与 `--check` |
| 文档历史被过度整理，丢失审计线索 | 难以追溯旧决策 | historical / retired 文档保留，只改变标签和入口索引 |

## 10. 决策点

需要 owner 在 review 时确认：

1. 是否接受 `ashare_research` 作为 BigQuery 研究/实验/诊断产物 dataset 名。
2. 是否接受 research 表使用 `research_*` 前缀，而不是在 `ashare_research` 中继续使用 `ads_*` 表名。
3. 是否接受 `accepted != promoted`：通过 acceptance gate 的模型仍需 owner promotion 才写 `ashare_ads`。
4. 是否接受先做 table role / dataset role abstraction，再创建和默认启用 `ashare_research`。
5. 是否接受 `sql/strategy1/**` 作为 Strategy1 active shared SQL 的目标命名空间，并把 `sql/ml/strategy1/**` 与 `sql/cloudrun/strategy1/**` 一并迁入。
6. 是否接受 `src/quant_ashare/**` 作为 Python 包根目录，以及新增 `pyproject.toml` 或等价 package install 策略。
7. 是否允许短期保留 `scripts/strategy1_cloudrun/**` wrapper，直到 Cloud Run entrypoint 单独迁移。
8. 是否需要创建 `docs/retired/`，还是只在现有历史文档顶部标注 retired / historical。

## 11. 结论

当前项目不适合一次性“大重构”。推荐把重构拆成“先 catalog 和护栏，再 table-role 抽象，再 SQL 命名空间，再 Python package foundation，再 research/ADS 物理隔离，再深层包拆分和入口迁移”的顺序。

对 owner 提到的具体问题，本 PRD 的判断是：

- `runner 依赖过去 runner 的 sql 文件`：实际剩余 SQL 多数是 active shared SQL，不应直接删除；应把 `sql/ml/strategy1/**` 和 `sql/cloudrun/strategy1/**` 一并纳入 catalog，再迁移到 `sql/strategy1/**`。
- `实验结果是否继续写 ADS`：建议新增 `ashare_research`，但必须先做 table-role resolver；新实验、候选、诊断和 replay 默认写 research，`ashare_ads` 只保留 owner promotion 后的正式产物。
- `模块应该拆分出来`：应把 Strategy1 Python 逻辑抽到 `src/quant_ashare/strategy1/**`，但 first pass 只抽 config/data_access/sql_render/path_catalog；ledger/report/acceptance/orchestration 后置。
- `文件名变量名函数名阶段性命名`：新增代码应避免 `oq010/p0/r14/cloudrun/bqml` 这类阶段标签进入通用模块，持久 ID 另行保留。
- `项目混乱`：先让结构表达事实，再做行为迁移；用小 PR 和检查替代一次性重排。
