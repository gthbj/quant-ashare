# 分析：topdown 单票上限探针 —— 上限能否救回 topdown vs v1

> 文档维护：Claude Opus 4.8（2026-06-13）
> 范围：纯只读 paper 探针（复用已验证的 Phase 0 paper harness，仅开关 opt-in `single_weight_cap`）。不跑 live、不写 BigQuery/GCS、不改 ledger 默认、不改记忆。
> 关联：PR #217（topdown retained 持仓记账 bug 修复 + `_v02` 重跑，证伪基于干净 run 成立）；`docs/分析-topdownPhase2三方对比-20260613.md`；`scripts/strategy1/analyze_topdown_single_weight_cap_probe.py`；`docs/analysis_topdown_single_weight_cap_probe_20260613.csv`。

## 1. 背景与问题

PR #217 修掉 topdown ledger 的 retained 持仓销毁 bug 后，`_v02` 重跑给出**干净**的真实结果：topdown T0（无单票上限）全维度劣于 v1 baseline（CAGR 11.96% vs 15.35%、Calmar 0.21 vs 0.41、MaxDD -56.85% vs -37.43%）。三方对比同时显示：topdown **确实消灭了现金拖累**（现金 2.5% vs v1 的 ~30%），但 ceil-lot **单票集中**很高（最大单票 46%）。

由此提出的假设：**给 topdown 加 `max_single_weight` 上限、压低集中度，能否在保留"消灭现金拖累"红利的同时把回撤压回 v1 水平、从而反超？**

## 2. 方法

复用 Phase 0 paper harness（`analyze_topdown_lot_phase0.py`，CSI1000 域、同一 prediction 流、ceil-lot 整手、贪心 top-down），新增 opt-in 配置 `single_weight_cap`（默认 `None` = 行为不变）。上限语义两部分：

- **新买入**：若某票最小整手（100 股）市值 > `cap × NAV`，跳过（无法在上限内持有）；
- **retained 漂移**：调仓时若 retained 持仓权重 > cap，按整手削减回 cap（释放现金供贪心再分配）。

对 T0 臂（walk_depth=50、primary matched-cost）在 cap ∈ {None, 0.20, 0.15, 0.10} 各跑一遍。

**harness 可信度锚点**：`cap=None` 必须复现已提交的 paper T0 数字。实跑结果 `cap=None` = CAGR 11.81% / Calmar 0.2013 / MaxDD -58.67% / 最大单票 47.19% / 持仓 14.54，与 `docs/分析-topdownPhase2三方对比-20260613.md` 报告的 paper T0 行**逐项一致** → harness 可信，capped 行可采信。paper 无上限 11.81% 亦与 live `_v02` 11.96% 吻合到 0.15pp。

## 4. 判读

1. **上限机制有效，但救不回 topdown**：上限把最大单票从 47% 压到 16-22%、p95 从 33% 压到 11-19%，集中度确实下来了；但 risk-adjusted 收益几乎没动——最好的 cap=15% 也只有 Calmar 0.2147（v1 是 0.4101，差一半），cap=20%/10% 反而比无上限更差（非单调，0.15 是噪声级小幅领先）。
2. **回撤根本没被上限改善**：四档 cap 的 MaxDD 仍在 -55% ~ -61%，cap=15% 甚至比无上限略深。这说明 topdown 的深回撤**不是单票暴雷（idiosyncratic）**造成的，而是**满仓持有的整篮小盘股在崩盘时一起跌（systematic / beta）**——压单票上限对系统性回撤无能为力。
3. **更深的结构性结论（topdown 的核心前提是错的）**：v1 用 ~30% 现金把 MaxDD 压到 -37%，那 30% 现金不是"拖累"，是**回撤保险**。topdown 满仓（现金 2.5%）把保险丢了，于是无论加不加上限，Calmar 都只有 v1 的一半。在这个中证1000 域 + 含两轮深熊的 2021-2026 窗口，**满仓的回撤代价 > 满仓多赚的收益**。

## 5. 结论与建议

- **topdown 构造路线收口**：retained bug 修复后的干净 `_v02` 已证伪 topdown（劣于 v1）；本探针进一步证明**加单票上限也救不回**。不值得再花 ledger 改动 + Cloud Run 周期跑 live capped 重测。
- **现金拖累问题仍然真实，但解法不是 topdown（满仓）**：应转向**显式的 market-state 条件化现金/仓位管理**——在风险偏好高时降现金、深熊时留现金缓冲，而不是无脑满仓。这与 v1 的"~30% 现金恰好提供缓冲"一致，方向是把这个缓冲做成**条件化、可控**的，而非固定或归零。
- 本探针为只读分析，未跑 live、未改 ledger 默认语义；按本会话约定未改 `.agent/memory`。`single_weight_cap` 作为 opt-in 研究开关保留在 paper harness 中（默认关闭，14 个既有 paper 单测仍通过），供未来复核或其它构造实验复用。
